Prefill / Decode 性能拆解实验笔记

把一次黑盒 model.generate() 拆成可观测的 prefill 和 decode 两个阶段，分别记录 TTFT、单 token 延迟、吞吐和显存，理解长上下文推理为什么首 token 变慢，以及 KV Cache 为什么能支撑稳定 decode。

1. 一句话结论

LLM 推理可以拆成两个阶段：prefill 一次性处理完整 prompt，决定首 token 延迟和 KV Cache 大小；decode 逐 token 生成，主要看每 token 延迟和 tokens/s。

长 prompt 实测 TTFT 1.4493s
长 prompt 输入长度 4148 tokens
decode 延迟区间 18-20 ms/token
长 prompt 峰值显存 3.15 GB

2. 核心概念

Prompt 输入 用户输入会先被 tokenizer 转成 token ids。输入越长，后续 prefill 要处理的序列越长。
Prefill 模型一次 forward 完整 prompt，计算每层 attention 的 Key/Value，并得到第一个 next token。
Decode 后续每一步只输入上一个 token，同时复用 prefill 和历史 decode 产生的 KV Cache。
输出文本 重复 decode，直到达到 max_new_tokens 或遇到 EOS 结束符。
阶段 做什么 主要指标 瓶颈倾向
Prefill 完整 prompt 一次 forward，建立 KV Cache TTFT、prefill_seconds、输入 tokens、峰值显存 计算密集，对 prompt 长度非常敏感
Decode 一次生成一个新 token，并更新 KV Cache avg_decode_latency_ms、decode_tokens_per_second 访存密集，受 KV Cache 读写和 batch 调度影响
为什么 generated_tokens=128 但 decode_tokens=127？ 第一个新 token 是 prefill 后直接从最后一个位置的 logits 里选出来的；后面 127 个 token 才进入逐 token decode 循环，所以这是正常计数。

PD 分离技术原理图

PD 分离到底分离了什么？

PD 分离中的 P 是 Prefill，D 是 Decode。它不是改变 Transformer 数学结构，而是把一次请求生命周期里两种性质完全不同的工作拆开调度：prefill 处理整段输入，吞吐依赖 GPU 并行矩阵计算；decode 按 token 串行推进，性能更依赖 KV Cache 读写、显存带宽、请求调度和 batch 组织。

1. Prefill 是“大块计算” 输入 prompt 长度为 N 时，模型可以一次性对 N 个 token 做 Q/K/V 投影和 attention 计算。这个阶段 GPU 利用率通常更高，适合把多个请求的 prefill 合并成较大的 batch 来跑。它直接决定 TTFT，也就是用户等多久看到第一个 token。
2. Decode 是“细粒度循环” Decode 第 t 步只会输入上一步生成的 1 个 token，但是它要读取之前所有 token 的 KV Cache。每一步计算量不大，却要反复读写缓存，所以更容易受显存带宽、KV Cache 布局和调度开销影响。
3. KV Cache 是两阶段的交接物 Prefill 结束后会产出 past_key_values。Decode 不再重算 prompt 的 K/V，而是把这个缓存作为历史上下文继续使用。也就是说，PD 分离的关键不是文本交接，而是 KV Cache 的交接、复用和生命周期管理。
4. 分离的目标是资源匹配 长 prompt 请求会占用大量 prefill 计算；长输出请求会占用大量 decode 步数。把两者拆开后，可以让 prefill worker 专注吃满算力，让 decode worker 专注低延迟逐 token 服务，避免两类工作互相拖慢。

工程系统里为什么常做 PD 分离？

  • 降低排队干扰：长 prompt 的 prefill 可能很慢，如果和 decode 混在同一队列里，会让已经开始输出的请求卡住，用户会看到断断续续的流式输出。
  • 提升吞吐：prefill 可以聚合成大 batch，更容易吃满 GPU 算力；decode 可以按 token 小步调度，用 continuous batching 维持稳定 tokens/s。
  • 独立扩容：如果业务是长输入短输出，可以增加 prefill 资源；如果业务是短输入长输出，可以增加 decode 资源，而不是整套模型副本一起扩。
  • 缓存管理更可控：服务框架可以围绕 KV Cache 做页式分配、复用、迁移和淘汰，减少显存碎片，并在多请求并发时保持稳定。
注意： PD 分离会带来额外工程复杂度，例如 KV Cache 跨设备传输、缓存地址映射、请求状态机、失败恢复和调度策略。小模型、低并发、本地学习实验里不一定需要真正拆成两个服务；但理解这个分离，对理解 vLLM、SGLang、TensorRT-LLM 等推理系统非常关键。

核心代码思路

# 1. Prefill: prompt 一次完整 forward，拿到 past_key_values
outputs = model(
    **inputs,
    use_cache=True,
    return_dict=True,
)
past_key_values = outputs.past_key_values
next_token = torch.argmax(outputs.logits[:, -1, :], dim=-1, keepdim=True)

# 2. Decode: 每次只喂一个 token，复用 KV Cache
outputs = model(
    input_ids=next_token,
    past_key_values=past_key_values,
    use_cache=True,
    return_dict=True,
)

6. 结果怎么解释

为什么长 prompt 的 prefill 明显变慢？

Prefill 要对完整 prompt 做一次前向计算，输入 token 越多，attention 要处理的序列越长，每层都要为这些 token 计算 Key 和 Value。长上下文会让首 token 时间明显变大，这就是线上服务常说的 TTFT 变差。

为什么 decode 延迟没有随 prompt 长度暴涨？

因为 KV Cache 已经保存了历史 token 的 Key/Value。decode 时模型每步只处理最新 token，然后和缓存里的历史 K/V 做 attention。上下文变长后，decode 会有更多缓存读写，长 prompt 下从 18ms/token 涨到接近 20ms/token，但没有像 prefill 那样大幅增加。

为什么 medium 有时比 short 快？

这是实验噪声，不是理论趋势。short 是第一组真实推理，可能吃到了 CUDA kernel 初始化、GPU 升频、显存缓存预热等开销。更严谨的做法是加 warmup，并对每组多跑几次取平均。

实践注意： 如果要写进正式实验报告，建议让脚本支持 --warmup 和 --runs，然后报告平均值和标准差。单次结果适合学习观察，不适合做严格性能结论。

7. 指标口径

指标 含义 怎么看
TTFT Time To First Token，本实验里近似等于 prefill 时间。 用户体感的首字响应速度，长 prompt 会明显拉高。
avg_decode_latency_ms decode 阶段每个 token 的平均生成延迟。 越低说明逐 token 生成越快。
decode_tokens_per_second decode 阶段吞吐，不包含 prefill。 更适合观察 KV Cache 下的持续生成速度。
end_to_end_tokens_per_second 把 prefill 和 decode 都算进来的整体吞吐。 短输出或长输入场景下，会被 TTFT 明显影响。
peak_cuda_memory_gb 运行过程中的 CUDA 峰值显存。 prompt 越长，KV Cache 越大，显存越高。

8. 面试表达

我把一次 LLM 推理拆成了 prefill 和 decode 两个阶段来测。Prefill 阶段对完整 prompt 做一次 forward，并建立每层 attention 的 KV Cache，所以它主要影响 TTFT，输入越长越慢；decode 阶段每次只输入一个新 token，并复用历史 KV Cache，所以每 token 延迟相对稳定。在我的实验里，输入从 116 tokens 增加到 4148 tokens 后，TTFT 从约 0.24s 增加到 1.45s，峰值显存从 0.96GB 增加到 3.15GB；但 decode 延迟基本保持在 18-20ms/token。这说明长上下文的主要成本集中在 prefill，decode 的稳定性主要来自 KV Cache。
