RAG 是什么？

RAG（Retrieval-Augmented Generation，检索增强生成）先从外部知识库检索相关资料，再将检索结果与用户问题一起交给 LLM 生成答案。

核心流程

用户问题 -> 检索 -> 组装上下文 -> LLM 生成 -> 引用溯源

解决什么问题

  • 降低幻觉：答案基于检索资料，而非纯参数记忆。
  • 知识可更新：文档变更后重新索引即可，无需重训模型。
  • 可溯源：展示引用来源，满足企业合规审计。

企业知识库 RAG 链路是什么？

生产级 RAG 是端到端数据与推理流水线。

离线链路

  1. 文档采集：Wiki、PDF、工单、数据库导出。
  2. 解析清洗：去噪、OCR、表格/标题结构提取。
  3. 切片（Chunking）：按语义或结构切分，保留 overlap 和元数据。
  4. Embedding：文本转向量，写入向量库。

在线链路

  1. Query Rewrite：多轮指代消解、关键词补全。
  2. 检索：向量 + BM25 混合召回，权限/租户过滤。
  3. Rerank：Cross-Encoder 精排 TopN。
  4. 上下文组装 + 生成 + 引用 + 日志/评测。

生产必备

增量更新、软删除、版本管理、权限隔离、质量评估、可观测性。

RAG 搭建有哪些开源框架？

常见有两类：代码框架灵活定制，低代码平台快速上线。面试提 1–2 个即可。

LangChain

Python / JS 生态最流行的 LLM 应用框架，RAG 链路组件齐全：Document Loader → Text Splitter → Embedding → VectorStore → Retriever → Prompt → LLM，用 LCEL 管道串联。

  • 适合：需要深度定制检索策略、Agent 编排、与企业系统集成的团队。
  • 特点：生态大、组件多、灵活度高；生产需自行补评测、权限、运维。
# LangChain RAG 最小链路
loader | splitter | vectorstore
retriever | prompt | llm | parser

Dify

开源 LLM 应用平台，知识库 RAG 开箱即用：上传文档 → 自动切片 / Embedding → 对话应用，可视化配置，支持自托管。

  • 适合：PoC、企业内部知识库问答、产品/运营也能参与配置。
  • 特点：低代码、上线快；复杂逻辑和深度集成不如 LangChain 灵活。

面试回答：常见 LangChain（代码灵活，组件化搭 RAG）和 Dify（低代码，知识库开箱即用）。选型：要定制用 LangChain，要快速验证用 Dify。

MCP 与 Tool Calling 有什么区别？

MCP 全称 Model Context Protocol（模型上下文协议），是 AI Agent 连接外部工具、数据库、文件系统、业务系统的标准化协议。Tool Calling 则是 LLM API 提供的模型调用工具的能力。两者层次不同、互补而非替代。

MCP 是什么

AI 应用 / Agent
  └── MCP Client
          ↓
      MCP Server
          ├── Tool：执行动作
          ├── Resource：读取资源
          └── Prompt：提供提示词模板

Client 通过 MCP 协议连接 Server，发现可用工具并调用；Server 负责把业务能力（查库、读文件、调 API）以统一格式暴露出来。

注册一个 MCP 服务通常需提供：

类型 内容
连接信息服务地址、启动命令、传输方式、环境变量
能力清单tools、resources、prompts
参数规范每个 tool 的入参 schema、出参格式
权限信息鉴权方式、工具权限、读写范围

与 Tool Calling 的区别

对比项 Tool Calling MCP
定位LLM 选择并调用工具的能力工具/数据源的标准化暴露协议
层次模型 API 层应用基础设施层
关注点模型如何决策调用哪个工具、填什么参数工具如何注册、鉴权、治理、跨应用复用
类比「模型会调工具」「工具以统一协议对外服务」
 
典型协作链路：
用户提问 → LLM Tool Calling 决策（选 search_docs）
         → MCP Client 转发 tools/call
         → MCP Server 执行真实查询
         → 结果回传 LLM → 生成答案

面试回答：MCP 是连接 Agent 与外部系统的标准协议，定义 Client/Server 如何发现 Tool、读 Resource；Tool Calling 是模型 API 能力，决定「调哪个工具、参数是什么」。一句话：Tool Calling 管决策，MCP 管工具基础设施，两者配合使用。

MCP 如何通信与工具 Schema 定义？

MCP 落地涉及三件事：Client 如何连接 Server（注册与鉴权）、双方如何通信、工具参数如何描述（Schema）。

1. Server 注册与鉴权

MCP 没有中心注册表，由 Client 在本地配置中声明 Server，启动时拉起或连接，握手后纳入可用工具池。

部署形态 配置要点 典型场景
本地 stdiocommand + args + envCursor / Claude Desktop 拉起子进程
远程 HTTPurl + 鉴权头 / OAuth团队共享、企业级工具服务
 
// .cursor/mcp.json 示例
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/data"],
      "env": { "API_KEY": "xxx" }
    },
    "remote-api": {
      "url": "https://mcp.example.com/mcp",
      "headers": { "Authorization": "Bearer <token>" }
    }
  }
}
  • 本地 stdio：依赖 OS 进程隔离；敏感凭据放 env。
  • 远程：Bearer Token / API Key / OAuth 2.0（授权码 + PKCE）。
  • 权限治理：Client 控制启用哪些 Server；Server 按 scope 限制可调工具。

连接后 Client 发 initialize 交换 capabilities，再 tools/list 发现工具，完成「注册」。

2. Client 与 Server 通信

消息层统一 JSON-RPC 2.0（说什么），传输层选 Transport（怎么传）。

Transport 方式 场景
stdio子进程 stdin/stdout，一行一条 JSON本地 IDE 最常见
Streamable HTTPHTTP POST，响应 JSON 或 SSE 流远程共享，支持 Bearer / OAuth
 
典型流程：
initialize → initialized → tools/list → tools/call → 结果回传

// tools/call 请求示例
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "search_docs",
    "arguments": { "query": "MCP 鉴权", "top_k": 5 }
  }
}

3. 工具 Schema 定义

Server 声明 name、description、inputSchema；inputSchema 为标准 JSON Schema，经 tools/list 暴露，Client 转成模型 Tool 定义。

{
  "name": "search_docs",
  "description": "搜索企业知识库文档",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": { "type": "string", "description": "搜索关键词" },
      "top_k": { "type": "integer", "minimum": 1, "maximum": 20, "default": 5 },
      "category": { "type": "string", "enum": ["tech", "hr", "finance"] }
    },
    "required": ["query"]
  }
}
MCP 字段 模型 Tool 字段
namename
descriptiondescription
inputSchemaparameters

写好 schema：description 写清语义；required 标必填；enum / minimum 约束取值；Server 执行前再做 schema 校验，防模型幻觉参数。

面试回答：MCP Server 由 Client 本地配置注册（stdio 或远程 HTTP），鉴权靠 env / Bearer / OAuth。通信用 JSON-RPC 2.0，传输走 stdio 或 Streamable HTTP。工具用 name + description + inputSchema（JSON Schema）描述，与 Function Calling 的 parameters 一一对应；流程是 initialize → tools/list → tools/call。

如何减少大模型幻觉？

幻觉（Hallucination）指模型生成看似合理但无事实依据的内容——编造数据、虚构引用、张冠李戴。无法彻底消除，但可通过「给依据、限发挥、做校验、能拒答」系统性降低。

RAG 层面

  • 提供可靠上下文：检索质量是上限；Chunk 合理、Hybrid + Rerank、过滤低相关文档。
  • Prompt 约束：明确要求「仅基于以下资料回答，资料未提及则说不知道」。
  • 强制引用：输出必须带 doc_id / 页码 / 原文片段，便于校验与溯源。
  • 拒答机制：检索分数低于阈值、Context 为空时，走固定拒答话术，不让模型硬编。
  • 忠实度评测：用 RAGAS Faithfulness、人工抽检 bad case，持续迭代检索与 Prompt。

生成层面

  • 降低 temperature：事实类问答用 0–0.3，减少随机「发挥」。
  • Structured Output：JSON Schema / Function Calling 约束字段，减少自由文本胡编。
  • 分步生成：先抽取事实，再组织答案（Extract-then-Generate），比一步到位更稳。
  • Self-RAG / CRAG：生成后自检是否与 Context 一致，不一致则重写或拒答。
  • 避免误导性 Prompt：不要暗示「一定有答案」；少问开放臆测题（「你觉得 CEO 怎么想」）。

工具与 Agent 层面

  • 能查就不猜：数值、状态、实时信息走 API / SQL / 搜索工具，不让模型凭记忆回答。
  • 工具结果优先：Prompt 规定「工具返回为准，不得覆盖或篡改」。
  • 参数校验：Function Calling 的 schema 加 enum、required，Server 端二次校验再执行。

工程与治理层面

  • 规则后处理：金额、日期、身份证号等关键字段用正则 / 规则引擎校验。
  • HITL 人工审核：高风险场景（医疗、法务、金融）低置信度转人工。
  • LLM-as-Judge：用独立模型或 RAGAS 对 Faithfulness 打分，低分拦截。
  • 监控与回归：记录幻觉反馈、低分 case，纳入评测集持续回归。
  • 权限与数据隔离：metadata 过滤，避免模型把 A 部门文档内容答给 B 部门用户。
System: 你是企业知识库助手。严格遵守：
1. 仅根据【检索资料】回答，不得使用外部知识。
2. 每条结论必须标注来源 [doc_id]。
3. 资料不足以回答时，回复：「根据现有资料无法确认，建议联系人工客服。」
4. 禁止编造政策条款、数值、日期、人名。

【检索资料】
{context}

【用户问题】
{question}
手段 解决哪类幻觉 典型场景
RAG + 引用事实性胡编制度问答、产品手册
工具调用实时数据编造订单状态、库存查询
Structured Output格式/字段乱填工单分类、参数抽取
拒答 + 阈值无依据硬答检索 miss 时
Faithfulness 评测上线后漂移Prompt / Rerank 迭代

面试回答：幻觉无法根治，核心是四层防控——RAG 给依据并强制引用与拒答；生成层降 temperature、结构化输出、生成后自检；Agent 层能查就不猜、工具结果为准；工程层规则校验、人工审核、Faithfulness 监控。记住：检索质量决定上限，Prompt 约束减少发挥，校验与拒答是最后防线。

Dify 是什么？

Dify 是一个开源 LLM 应用开发平台，提供可视化编排、知识库 RAG、Agent、模型接入与 API 发布，让团队低代码 / 无代码快速搭建生产级 AI 应用，也支持自托管部署。

核心能力

知识库（Knowledge）上传文档，自动解析、切片、Embedding、向量检索，支撑 RAG
工作流（Workflow）拖拽式编排 LLM、检索、条件分支、代码节点、工具调用
模型供应商统一接入 OpenAI、Azure、Anthropic、本地 Ollama 等，集中管理 Key
插件 / 工具内置或自定义 API 工具，供 Agent / Workflow 调用
观测与日志对话日志、Token 消耗、Tracing，便于调试与运营

  • Agent：模型自主选工具，适合开放域任务。
  • Workflow：固定流程编排（检索 → 生成 → 后处理），适合可控业务链路。

企业使用注意点

RAG 中如何处理结构化数据查询？

结构化数据（数据库表、Excel、CSV、JSON）有明确的 schema 和精确字段，向量检索不适合做聚合/过滤/排序。核心策略是路由分流：结构化走精确查询，非结构化走向量 RAG。

用户问题 → 意图分类（Classifier / LLM Router）
  ├─ 结构化（统计/过滤/聚合/精确字段）→ Text-to-SQL / API 查询 → 格式化结果
  ├─ 半结构化（表格文档）→ 表格切片 RAG + metadata 过滤
  └─ 非结构化（文档/段落）→ 向量 RAG

三种处理方式

方式 适用数据 做法 示例问题
Text-to-SQLMySQL / PG 等关系库NL → SQL → 执行 → 结果格式化「Q3 华东区销售额 Top5 客户」
表格 RAGExcel / CSV 文档按行/列切片 + 表名 metadata「产品手册中 XX 型号参数是多少」
API / Tool 调用业务系统接口LLM 选 Tool + 填参调用「查订单 12345 的物流状态」

Text-to-SQL 关键步骤

  1. Schema 注入：将表名、字段、类型、注释、样例行注入 Prompt 或向量库。
  2. SQL 生成：LLM 输出 SQL，需约束只读（禁止 DROP/UPDATE）。
  3. 校验执行：SQL 语法检查 → 权限校验 → 沙箱执行 → 结果截断。
  4. 结果解释：将查询结果转为自然语言回答，附 SQL 溯源。

混合路由设计

  • 用轻量 Classifier 或 LLM 判断问题类型：「需要精确数字」→ SQL；「需要文档解释」→ RAG。
  • 复杂问题可组合：先 SQL 查数据，再 RAG 查文档解释背景。
  • 表格文档（PDF 中的表）：解析为 Markdown/HTML 保留行列结构，metadata 标记 table_name、sheet。

常见坑

  • 把数据库表全量 Embedding 走向量检索 → 聚合/排序类问题答不对。
  • Text-to-SQL 不做权限控制 → 用户可能查到无权限的数据。
  • Schema 过大时全量注入 Prompt → 用 Schema Linking 只选相关表/字段。

面试回答：结构化数据优先 Text-to-SQL 精确查询，非结构化走向量 RAG；用意图路由分流，表格文档按行列切片保留结构；Text-to-SQL 需 Schema 注入、SQL 校验和权限控制。

Text-to-SQL 在 AI 应用中如何落地？

落地链路

  1. Schema 注入：表结构、字段注释、样例行写入 Prompt。
  2. SQL 生成：LLM 输出 SQL，限 SELECT、禁止 DDL/DML。
  3. 校验执行：语法检查、权限过滤、超时限制、结果行数上限。
  4. 结果解释：将查询结果转为自然语言。

风险管控

  • 只读账号、行级权限、SQL 注入防护。
  • 错误 SQL 重试 + 人工兜底。

RAG 如何评估答案忠实度？

Faithfulness（忠实度）衡量生成答案是否被检索上下文支持，而非模型臆造。

评估方法

  • LLM-as-Judge：用一个大模型充当裁判，去评价另一个大模型的回答质量。
  • RAGAS faithfulness：一个专门评估 RAG / LLM 应用的开源框架。它提供多种指标和测试数据生成能力

忠实度低说明检索不准或 Prompt 约束不足；需分别排查 retrieve 和 generate 环节。

LLM-as-a-Judge 是什么？

用 LLM 作为评判者，对另一个 LLM 的输出打分或排序，替代人工标注。

典型场景

  • 答案质量评分（相关性、完整性、安全性）。
  • 忠实度验证（是否 grounded）。
  • A/B 测试自动对比两个 Prompt 版本。

你可以给 LLM-as-Judge模型这样的输入：

你可以给 Judge 模型这样的输入：

用户问题：
{question}

检索到的上下文：
{contexts}

模型回答：
{answer}

请判断模型回答中的事实是否都能被上下文支持。
如果有 unsupported claim，请指出。
最后给出 0~1 分。

注意点

Judge 模型应有足够能力；Position bias（倾向第一个答案）需交换顺序消除；关键场景仍需人工抽检。

RAGAS 可以评估哪些指标？

RAGAS（Retrieval Augmented Generation Assessment）是 RAG 系统的开源自动化评测框架，用 LLM 作为评判器，从检索质量和生成质量两个维度打分，无需大量人工标注即可做回归对比。

用户问题 → 检索 Context → 生成 Answer
              ↓                    ↓
        检索类指标            生成类指标
   Context Precision/Recall   Faithfulness / Answer Relevancy

核心指标

指标 维度 含义
Context Precision检索召回的 Chunk 中，真正与问题相关的占比（排序质量）
Context Recall检索回答问题所需的信息，是否被检索上下文覆盖（需 ground truth）
Faithfulness生成答案中的陈述是否都能被检索上下文支撑，衡量幻觉程度
Answer Relevancy生成答案与问题的相关程度，是否答非所问
Answer Correctness生成答案与标准答案的语义一致性（需 ground truth）

Faithfulness 怎么算？

  1. 从 Answer 中抽取若干 statement（陈述句）。
  2. 逐条判断该 statement 能否被 retrieved context 支撑。
  3. 可支撑的比例即为 Faithfulness 分数。

使用场景

  • 无标注：Faithfulness + Answer Relevancy + Context Precision 可直接跑，适合 Prompt / Rerank 迭代。
  • 有标注：加上 Context Recall、Answer Correctness，全面评估检索与生成。
  • CI 回归：每次改 Chunk、Embedding、Prompt 后跑评测集，对比分数变化。

面试回答：RAGAS 核心看四件事——检索准不准（Precision/Recall）、答案有没有胡编（Faithfulness）、答没答到点（Relevancy）。

RAGAS 在代码项目中如何评价召回率和回答准确率？

在工程里接入 RAGAS，本质是准备评测数据集 → 跑 RAG 链路拿到 contexts / answer → 调用 ragas.evaluate() 批量打分 → 对比版本回归。指标与业务含义的对应关系如下：

业务关注点 RAGAS 指标 所需字段 说明
检索召回率context_recallquestion、contexts、ground_truth标准答案中的信息，是否被检索到的 Context 覆盖
检索排序质量context_precisionquestion、contexts、ground_truth召回 Chunk 里无关内容占比，反映 Rerank 效果
回答准确率answer_correctnessquestion、answer、ground_truth生成答案与标准答案的语义/事实一致性
是否胡编faithfulnessquestion、answer、contexts答案是否被 Context 支撑（无标注可跑）
是否答非所问answer_relevancyquestion、answer答案与问题的相关度（无标注可跑）

注意：召回率在 RAGAS 里对应 context_recall，不是传统 IR 的 recall@K；若评测集标注了相关 doc_id，可同时算工程指标 recall@K = 命中 doc 数 / 应命中 doc 数，与 RAGAS 互补。

评测数据集格式

每条样本至少包含：

  • question：用户问题
  • contexts：RAG 实际检索到的 Chunk 列表（字符串数组）
  • answer：RAG 系统生成的最终答案
  • ground_truth：人工标注的标准答案（算召回率、准确率时必填）

代码接入（离线评测）

# pip install ragas datasets langchain-openai

from datasets import Dataset
from ragas import evaluate
from ragas.metrics import context_recall, answer_correctness, faithfulness

# 1. 准备评测集（可从 JSON / CSV 加载）
eval_rows = []
for item in eval_dataset:
    # 2. 跑你的 RAG 链路，收集实际输出
    contexts = retriever.retrieve(item["question"])   # ["chunk1", "chunk2", ...]
    answer = rag_pipeline.generate(item["question"], contexts)

    eval_rows.append({
        "question": item["question"],
        "contexts": contexts,
        "answer": answer,
        "ground_truth": item["ground_truth"],  # 人工标注
    })

dataset = Dataset.from_list(eval_rows)

# 3. 批量评测：召回率 + 回答准确率 + 忠实度
result = evaluate(
    dataset,
    metrics=[context_recall, answer_correctness, faithfulness],
)

print(result)  # 各指标均值，如 context_recall=0.82, answer_correctness=0.76
print(result.to_pandas())  # 逐条明细，便于定位 bad case

工程化实践

  1. 基线版本：当前线上 Prompt / Embedding / Rerank 跑一遍，记录分数作为 baseline。
  2. 变更对比：改 Chunk 策略、换模型、调 Rerank 后重跑，对比 context_recall 和 answer_correctness 升降。
  3. 分项排查：context_recall 低 → 查检索/切片/Embedding；answer_correctness 低但 recall 高 → 查 Prompt / 生成模型；faithfulness 低 → 幻觉，加强引用约束。
  4. CI 门禁：评测集固定 100+ 条，PR 合并前跑 RAGAS，核心指标不得低于 baseline 阈值（如 recall -2%、correctness -3%）。
  5. bad case 回流：result.to_pandas() 筛低分样本，人工复核后加入回归集。

无 ground_truth 时的替代方案

  • 无法标注标准答案时，不能算 context_recall / answer_correctness。
  • 可先用 faithfulness + answer_relevancy + context_precision 做相对对比。
  • 有 doc_id 标注时，自研 recall@K / MRR 评估检索，生成侧仍用 Faithfulness。

面试回答：代码里用 RAGAS 评召回率看 context_recall（需 ground_truth），评回答准确率看 answer_correctness；流程是构造评测集 → 跑 RAG 收集 contexts 和 answer → evaluate() 批量打分 → 对比 baseline 做回归。召回低查检索，准确率低查生成，faithfulness 低查幻觉；有 doc_id 时可额外算 recall@K 与 RAGAS 互补。

如何构造 RAG 离线评测集？

构造步骤

  1. 从真实用户日志采样高频问题。
  2. 人工标注期望答案 + 相关 doc_id（ground truth）。
  3. 覆盖边界：多轮指代、无答案、权限受限、复合问题。
  4. 定期补充 bad case 回归。

评测集 100–500 条即可启动；每次改 Chunk/Embedding/Rerank 后跑 recall@K 和 faithfulness 对比。

Few-shot Prompt 适合什么场景？

Few-shot 在 Prompt 中提供若干输入-输出示例（通常 2–5 个），让模型通过「照猫画虎」学习期望的格式、术语和决策边界，而不必 Fine-tuning。

适合的场景

场景 为什么用 Few-shot 示例
固定输出格式示例比文字描述更能约束 JSON / 标签 / 字段结构工单分类输出 {"category": "退款", "priority": "高"}
领域术语与口径通用模型不懂企业内部叫法，示例可统一翻译/归类规则「SLA 违约」→「服务质量」类
边界 case 分类规则难写清，用正反例讲清「什么算 A、什么不算」情感分析中「还行吧」算中性还是负向
信息抽取 / 结构化填槽字段含义、空值处理、多实体拆分需一致从客服对话抽 order_id、issue_type
风格 / 话术模仿语气、长度、礼貌程度用示例比形容词更好控电商客服回复模板、摘要长度 50 字内
小模型 / 弱模型补齐Zero-shot 不稳定时，少量示例显著提升准确率7B 本地模型做意图识别

不适合的场景

  • 复杂多步推理：数学、逻辑链、规划类任务优先用 CoT（或 Few-shot CoT），单纯模仿输入输出不够。
  • 示例过多：占 Token、拉高成本与延迟，还可能挤占检索上下文窗口。
  • 示例引入偏见：示例分布不均时，模型会过拟合示例模式（如示例全是「退款」就乱判退款）。
  • 强模型 + 简单通用任务：GPT-4 等做「翻译成英文」往往 Zero-shot 已足够，不必加示例。
  • 示例与指令矛盾：System Prompt 说「只输出 JSON」，示例却带解释文字，模型行为会混乱。

示例设计要点

  1. 数量：一般 2–5 个；覆盖典型 case + 1 个边界 case 即可，不必堆砌。
  2. 多样性：不同输入模式都要有（短句/长句、肯定/否定、含噪/不含噪）。
  3. 一致性：所有示例的输出格式、字段名、枚举值必须完全一致。
  4. 与指令对齐：示例是「标准答案」，System / User 指令不应与示例冲突。
你是工单分类助手。严格按示例格式输出 JSON，不要其他文字。

示例 1：
输入：我要退上周买的耳机，订单号 12345
输出：{"category": "退款", "priority": "中"}

示例 2：
输入：App 一直闪退，重装也没用
输出：{"category": "技术故障", "priority": "高"}

示例 3：
输入：请问会员有什么权益？
输出：{"category": "咨询", "priority": "低"}

现在请分类：
输入：{user_message}

面试回答：Few-shot 适合「格式固定、规则靠示例讲清、小模型需要补齐」的任务，如分类、抽取、风格模仿；不适合长推理（用 CoT）和示例过多占 Token 的场景。关键是示例少而精、格式一致、覆盖边界 case。

Skills 具体是怎么工作的？

Skills（Agent Skills）是按需加载的领域操作手册，以 Markdown 文件（如 SKILL.md）形式存放，教 Agent「遇到某类任务时该怎么想、怎么做」，而不是把所有规则塞进 System Prompt。

Agent 启动：只加载 Skills 清单（名称 + 简短描述），不占大量 Token
用户任务进来：Agent 判断任务类型 → 匹配相关 Skill → 读取完整 SKILL.md → 按 Skill 指引执行
执行完毕：Skill 内容可不常驻上下文，下次需要再加载

核心机制：渐进式披露（Progressive Disclosure）

  • 第一层：Skills 目录/元数据（name + description），Agent 知道「有哪些能力可选」。
  • 第二层：任务匹配后，Agent 主动 Read 完整 Skill 文件，获取详细 workflow。
  • 第三层：Skill 内引用的脚本、模板、示例文件，执行时再按需加载。

好处：100 个 Skill 不会一次性塞爆 Context Window，只有相关的才加载。

Skill 文件里通常有什么？

组成部分 内容
触发条件什么任务场景应该启用此 Skill（description 字段）
Workflow分步骤操作指南（先查什么、再改什么、如何验证）
规范约束命名约定、代码风格、项目特有规则
工具指引该场景下优先用哪些 MCP Tool / 脚本 / API
示例输入输出样例、常见错误处理

工作流程示例（Cursor Agent）

  1. 用户在项目里配置了 .cursor/skills/ 目录，每个子目录一个 Skill。
  2. Agent 收到任务「帮我创建 PR」，扫描 Skills 清单，匹配到 create-pull-request Skill。
  3. Agent 调用 Read 工具加载 SKILL.md，获取 PR 创建的标准流程（git status → diff → push → gh pr create）。
  4. Agent 按 Skill 步骤依次执行 Shell / gh 命令，而非凭空猜测流程。

与 System Prompt / Rules 的区别

  • System Prompt：全局、常驻、通用行为约束。
  • Rules：项目级持久规则（编码规范、提交规范），始终生效。
  • Skills：任务级、按需加载、专精某个 workflow（如「拆 PR」「写 Canvas」「调 CI」）。

设计 Skill 的最佳实践

  • description 写清楚何时触发，Agent 靠它做 Skill 路由。
  • 步骤要可执行、可验证，避免模糊描述（「适当优化」→ 改为具体检查项）。
  • 单 Skill 聚焦一个场景，不要把所有知识塞进一个文件。
  • 配合 MCP Tool 使用：Skill 定义流程，Tool 提供能力。

面试回答：Skills 是按需加载的领域操作手册，通过渐进式披露避免 Context 膨胀；Agent 先扫 Skill 清单匹配任务，再读取完整 SKILL.md 按 workflow 执行；与 System Prompt 全局常驻不同，Skills 是任务级专精指引。

Skills 和 Function Calling 有什么区别？

两者都扩展 Agent 能力，但层次完全不同：Skills 教 Agent「怎么做」，Function Calling 让 Agent「能调用」。

Skills     = 操作手册 / SOP（知识 + 流程，只读文本）
Function Calling = 工具接口 / API（可执行，JSON Schema 定义入参）

核心对比

对比项 Skills Function Calling
本质领域知识与操作流程（Markdown 文档）可执行的外部函数/API 调用
载体SKILL.md 文件，Agent Read 加载JSON Schema 描述，模型输出结构化参数
作用指导推理路径、决策顺序、规范约束触发真实动作（查库、发请求、写文件）
加载时机任务匹配时按需读取模型推理时实时选择并调用
是否执行代码否，仅提供指引是，Runtime 执行并返回结果
类比员工培训手册 / SOP电话/API 拨号能力

协作关系

Skills 和 Function Calling 互补而非替代：

  • Skill 定义「创建 PR 的流程：先 git status，再 diff，再 push，最后 gh pr create」。
  • Function Calling / Tool 提供实际能力：Shell 执行 git、gh 创建 PR。
  • 没有 Skill，Agent 可能调用 Tool 但顺序混乱、遗漏步骤。
  • 没有 Tool，Skill 只是空有流程无法执行。

与 MCP 的三角关系

Skills        → 告诉 Agent「什么时候、按什么顺序、用什么规范」
Function Calling → 模型决策层：选哪个 Tool、填什么参数
MCP           → 工具基础设施层：Tool 如何注册、鉴权、暴露

选型建议

  • 重复性 workflow、多步骤 SOP → 写 Skill（如 PR 流程、Canvas 规范、CI 排查）。
  • 需要访问外部系统、读写数据 → 定义 Function / MCP Tool。
  • 复杂 Agent：Skills 编排流程 + Function Calling 执行动作 + MCP 统一工具治理。

常见误区

  • 把 Skill 当 Tool：Skill 本身不可执行，不能替代 API 调用。
  • 把 Function Schema 当 Skill：Schema 只描述参数，不包含业务流程和决策逻辑。
  • 把所有规则写进 System Prompt：应拆分为全局 Rules + 按需 Skills，控制 Token 成本。

面试回答：Skills 是按需加载的操作手册，指导 Agent 怎么做；Function Calling 是模型输出结构化参数触发真实 API 执行；前者管流程与规范，后者管动作与数据，配合 MCP 构成完整 Agent 能力栈。

Agent 工具调用失败如何降级？

降级策略

  1. 重试：指数退避，最多 3 次。
  2. 降级：备用工具主 API 失败切缓存或降级 API。
  3. 跳过：告知模型工具不可用，请基于已有信息回答。
  4. 人工兜底：HITL 介入或返回友好错误。
try: result = call_tool(params)
except Timeout: result = {"error": "timeout", "fallback": cached_data}
except Exception: result = {"error": "unavailable"}

如何控制 LLM 应用的 Token 成本？

优化手段

  • 检索 TopK 控制：减少进 Prompt 的 Chunk 数。
  • 历史摘要压缩：长对话不塞全量 messages。
  • 小模型路由：简单任务用小模型，复杂才用大模型。
  • 缓存：相同 query 命中 Semantic Cache。
  • Prompt 精简：去除冗余 System Prompt。

按 tenant/user 设 Token 配额，超限降级或拒服；监控 input/output Token 分布。

上下文窗口是什么？

上下文窗口是模型单次推理能处理的最大 Token 数（input + output）。

模型 典型窗口
GPT-4o128K
Claude 3.5200K
开源 7B8K–32K

超出窗口需截断、摘要或 RAG 外挂知识；长窗口不等于模型等效利用全部内容（Lost in the Middle）。

ReAct 是什么？

ReAct（Reasoning + Acting）交替思考和行动：Thought -> Action -> Observation 循环。

Thought: 需要查员工手册中的年假规定
Action: search_docs("年假天数")
Observation: [1] 年假为15天...
Thought: 已有足够信息
Action: FINISH
Answer: 年假为15天

比纯 CoT 能获取外部信息；比纯 Tool Calling 有显式推理链。

Agent 如何做可观测性和调试？

  • Trace：每步 Thought/Action/Observation 记录 span。
  • LangSmith / Langfuse：Agent 专用追踪平台。
  • 日志：tool 入参、出参、Token 用量、延迟。
  • 指标：成功率、平均步数、Tool 失败率。

Debug 时回放完整 Agent 轨迹，定位是哪一步决策错误。
