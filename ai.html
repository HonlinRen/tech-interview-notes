RAG 是什么？

RAG（Retrieval-Augmented Generation，检索增强生成）先从外部知识库检索相关资料，再将检索结果与用户问题一起交给 LLM 生成答案。

核心流程

用户问题 -> 检索 -> 组装上下文 -> LLM 生成 -> 引用溯源

解决什么问题

  • 降低幻觉：答案基于检索资料，而非纯参数记忆。
  • 知识可更新：文档变更后重新索引即可，无需重训模型。
  • 可溯源：展示引用来源，满足企业合规审计。

企业知识库 RAG 链路是什么？

生产级 RAG 是端到端数据与推理流水线。

离线链路

  1. 文档采集：Wiki、PDF、工单、数据库导出。
  2. 解析清洗：去噪、OCR、表格/标题结构提取。
  3. 切片（Chunking）：按语义或结构切分，保留 overlap 和元数据。
  4. Embedding：文本转向量，写入向量库。

在线链路

  1. Query Rewrite：多轮指代消解、关键词补全。
  2. 检索：向量 + BM25 混合召回，权限/租户过滤。
  3. Rerank：Cross-Encoder 精排 TopN。
  4. 上下文组装 + 生成 + 引用 + 日志/评测。

生产必备

增量更新、软删除、版本管理、权限隔离、质量评估、可观测性。

Chunk 设计为什么重要？

Chunk 是检索的最小语义单元，直接影响召回率和生成质量。

Chunk 大小 优点 缺点
太大上下文完整噪声多、占 Token、语义不聚焦
太小检索精准语义断裂、丢失跨段关系

最佳实践

  • 按结构切：标题、段落、表格、代码块。
  • 保留标题路径（breadcrumb）作为 metadata。
  • 设置 10%–20% overlap，避免边界截断。
  • 常见大小 256–512 Token，按文档类型 A/B 调优。

Metadata 有什么作用？

Metadata 描述每个 Chunk 的来源与属性，是过滤、溯源和权限控制的基础。

类型 示例字段 用途
来源doc_id、url、page、section引用溯源
权限tenant_id、dept、acl检索前过滤
版本version、effective_date冲突消解
技术embedding_model、chunk_rule重建索引

多租户场景下，tenant_id 必须在检索层强制过滤，不能仅靠 Prompt 约束。

向量检索的基本原理是什么？

Embedding 模型将文本映射到高维向量空间，语义相近的文本向量距离更近。

文档 -> Embedding -> 向量库
查询 -> Embedding -> ANN 搜索 -> TopK 相似片段

优势与局限

  • 优势：语义相似、同义词、跨语言。
  • 局限：精确编号、SKU、专有名词弱；需 Hybrid Search 补充。

TopK 和召回率是什么？

TopK：检索返回相似度最高的 K 个 Chunk。召回率（Recall）：含正确答案的文档是否被检索到。

参数 K 太小 K 太大
召回漏掉正确答案召回率高
成本Rerank/生成 Token 少噪声多、延迟高

典型策略：向量召回 K=50–100，Rerank 后取 Top5–10 进上下文。用离线评测集调 recall@K。

Query Rewrite 解决什么问题？

用户原始问题往往不适合直接检索：口语化、省略指代、复合问题。

常见改写策略

  • 指代消解：「它多少钱」→「XX 产品多少钱」。
  • 关键词提取：补充领域术语。
  • 问题拆分：复合问题拆成多个子查询。
  • HyDE：生成假设性答案再检索。

多轮对话需结合 history 或摘要后再改写，否则检索命中率低。

Rerank 是什么？

Rerank（重排）对初步召回的 TopN 候选用更精确模型重新打分排序。

Bi-Encoder 召回 Top100 -> Cross-Encoder Rerank -> Top5 进 Prompt
阶段 模型 特点
召回Bi-Encoder快，可离线建索引
重排Cross-Encoder准，需在线逐对计算

只对 TopN（50–100）重排，平衡精度与延迟。

Tool Calling 和 MCP 区别？

对比项 Tool Calling MCP
定位LLM 选择并调用工具的能力工具/数据源的标准化暴露协议
层次模型 API 层应用基础设施层
关注点模型如何决策调用哪个工具工具如何注册、鉴权、治理、复用

Tool Calling 是「模型会调工具」；MCP 是「工具以统一协议对外服务」，两者互补。

MCP Server 怎么注册和鉴权？

MCP Server 不由中心注册表统一登记，而是由 Client 在本地配置中声明，启动时按配置拉起或连接 Server，完成握手后纳入可用工具池。

注册方式

部署形态 配置要点 典型场景
本地 stdiocommand + args + envCursor / Claude Desktop 拉起子进程
远程 HTTPurl + 鉴权头 / OAuth团队共享、SaaS 类 MCP 服务
 
// Cursor: .cursor/mcp.json 示例
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/data"],
      "env": { "API_KEY": "xxx" }
    },
    "remote-api": {
      "url": "https://mcp.example.com/mcp",
      "headers": { "Authorization": "Bearer <token>" }
    }
  }
}

鉴权机制

  • 本地 stdio：依赖 OS 进程隔离，通常不单独鉴权；敏感凭据放 env，由 Server 进程读取。
  • API Key / Bearer Token：远程 Server 在 HTTP Header 或环境变量中校验。
  • OAuth 2.0：远程 MCP 按规范走授权码 + PKCE，Client 拿 access token 后带在请求头访问。
  • 权限治理：Client 侧控制启用哪些 Server；Server 侧按 scope / 白名单限制可调工具与资源。

连接建立后 Client 发 initialize，双方交换 capabilities，再拉取 tools/list 完成「注册」。

MCP Client 和 Server 通信方式是什么？

底层统一采用 JSON-RPC 2.0 消息格式，通过不同 Transport 承载；同一连接上可发 request、response、notification 三类消息。

Transport 方式 特点
stdioClient 启动子进程，stdin/stdout 传 JSON-RPC 行本地、低延迟、最常见
Streamable HTTPHTTP POST 发请求，可选 SSE 流式收响应远程部署、可多 Client 共享
SSE（旧）独立 SSE 端点推送事件逐步被 Streamable HTTP 取代

典型交互流程

  1. initialize → Server 返回协议版本与 capabilities。
  2. Client 发 initialized 通知，会话就绪。
  3. Client 按需调用 tools/list、resources/list、prompts/list 发现能力。
  4. 模型决策后 Client 发 tools/call，Server 执行并返回结构化结果。
// tools/call 请求示例
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "search_docs",
    "arguments": { "query": "MCP 鉴权", "top_k": 5 }
  }
}

MCP 管「Client ↔ Server」通道；模型侧的 Tool Calling 管「LLM 选哪个工具、填什么参」，Client 负责把模型输出映射为 tools/call。

MCP工具 schema 怎么定义？

MCP 工具在 Server 中声明 name、description、inputSchema 三要素；inputSchema 为标准 JSON Schema，描述参数类型、必填项与约束，经 tools/list 暴露给 Client，再转成模型可用的 Tool 定义。

// Server 侧工具定义（逻辑结构）
{
  "name": "search_docs",
  "description": "搜索企业知识库文档",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜索关键词"
      },
      "top_k": {
        "type": "integer",
        "minimum": 1,
        "maximum": 20,
        "default": 5,
        "description": "返回条数"
      },
      "category": {
        "type": "string",
        "enum": ["tech", "hr", "finance"],
        "description": "文档分类"
      }
    },
    "required": ["query"]
  }
}

与 Function Calling 的对应关系

MCP 字段 模型 Tool 字段 说明
namename工具唯一标识
descriptiondescription供模型理解用途
inputSchemaparameters均为 JSON Schema，语义一致

写好 schema 的关键：description 写清业务语义；required 标必填；用 enum / minimum 约束取值；Server 执行前再做一次 schema 校验，防止模型幻觉参数。

如何减少大模型幻觉？

RAG 层面

  • 提供可靠检索上下文，要求「仅基于资料回答」。
  • 输出引用来源，无依据则拒答或澄清。
  • Rerank + 忠实度评测（Faithfulness）。

生成层面

  • 降低 temperature，约束输出格式。
  • Structured Output / JSON Schema 减少自由发挥。

工程层面

  • 关键业务答案规则校验或 HITL 审核。
  • LLM-as-Judge 或 RAGAS 持续监控。

LangChain 常见结构有哪些？

LangChain 按职责分层，核心是把 Model、Prompt、数据、编排 拆成可复用组件，再用 Chain / Agent 串联。

结构 典型类 / API 作用
ModelChatOpenAI、OpenAIEmbeddings封装 LLM 调用与向量化
PromptChatPromptTemplate、FewShotPromptTemplate模板化拼装 System / Human 消息
Output ParserStrOutputParser、JsonOutputParser把模型输出解析为字符串或结构化对象
Document LoaderPyPDFLoader、WebBaseLoader加载原始文档
Text SplitterRecursiveCharacterTextSplitter切分 Chunk，控制检索粒度
VectorStoreChroma、FAISS、PGVector存储与检索向量
Retrieveras_retriever()、MultiQueryRetriever按 query 召回相关文档
Chain（LCEL）prompt | llm | parser用管道符组合 Runnable，线性编排
Tool@tool 装饰函数把业务能力暴露为可调用工具
Agentcreate_react_agent、AgentExecutor模型自主选工具、多步推理
MemoryChatMessageHistory、RunnableWithMessageHistory维护多轮对话上下文

现代 LangChain 推荐 LCEL（LangChain Expression Language）：各组件实现 Runnable 接口，支持 invoke / stream / batch 统一调用。

LangChain 标准构建流程是什么？

从 0 到 1 搭一个 LangChain 应用，通常按「模型 → 数据 → 提示词 → 链 → 调用」五步走；RAG 和 Agent 在此基础上加 Retriever 或 Tool。

基础流程（5 步）

  1. 选 Model：配置 LLM 和（如需 RAG）Embedding 模型。
  2. 准备数据：Loader 加载 → Splitter 切分 → Embedding → 写入 VectorStore。
  3. 定义 Prompt：写清角色、上下文占位符、输出格式要求。
  4. 组装 Chain：用 LCEL 管道串联各组件。
  5. 调用与观测：invoke() / stream()，挂 Callback 记录 Token 与延迟。

简单问答 Chain

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(model="gpt-4o")
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是企业知识库助手，简洁回答。"),
    ("human", "{question}"),
])
chain = prompt | llm | StrOutputParser()

answer = chain.invoke({"question": "年假有多少天？"})

RAG Chain 扩展

from langchain_core.runnables import RunnablePassthrough

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

rag_prompt = ChatPromptTemplate.from_messages([
    ("system", "仅根据以下资料回答，无依据则说不知道。\n\n{context}"),
    ("human", "{question}"),
])
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | rag_prompt
    | llm
    | StrOutputParser()
)

Agent 扩展

  1. 用 @tool 定义搜索、查库等业务函数。
  2. create_react_agent(llm, tools, prompt) 创建 Agent。
  3. AgentExecutor(agent, tools) 执行多轮 think-act 循环。

流程固定、无分支时用 Chain；需要自主选工具、多步推理时升级为 Agent；有循环、人工审批、状态持久化时再考虑 LangGraph。

LangChain 和 LangGraph 区别是什么？

LangChain 是 LLM 应用组件库；LangGraph 是有状态图工作流编排框架。

对比项 LangChain LangGraph
定位组件与链式编排有状态图工作流与 Agent 编排
流程形态线性链路或简单 Agent分支、循环、路由、状态机
适用场景RAG 原型、工具调用复杂 Agent、审批流、可恢复任务

简单 RAG 用 LangChain 足够；需要循环、人工介入、状态持久化时选 LangGraph。

LangGraph 的 Node 怎么设计？

Node 是工作流中的可执行步骤，应职责单一、输入输出清晰

设计原则

  • 按能力拆分：rewrite_query、retrieve、rerank、generate、call_tool、human_review。
  • 避免一个 Node 做太多事，便于观测、重试和替换。
  • 只返回 State 增量更新，不覆盖整个状态。
  • 条件分支、循环通过 conditional edge 表达。
def retrieve_node(state):
    docs = retriever.search(state["rewritten_query"])
    return {"retrieved_docs": docs}

LangGraph 中 Edge 怎么设计？

Edge 决定 Node 之间的流转方向，分固定边和条件边两类。

固定边（add_edge）

无条件跳转，适合线性流水线。

builder.add_edge("retrieve", "rerank")
builder.add_edge("rerank", "generate")
builder.add_edge("generate", END)

条件边（add_conditional_edges）

路由函数读取 State，返回下一个 Node 名称，实现分支、循环、重试。

def route_after_generate(state):
    if state.get("need_tool"):
        return "call_tool"
    if state.get("retry_count", 0) > 3:
        return END
    return "human_review"

builder.add_conditional_edges(
    "generate",
    route_after_generate,
    {
        "call_tool": "call_tool",
        "human_review": "human_review",
        END: END,
    },
)

常见 Edge 模式

场景 Edge 设计
线性 RAGrewrite → retrieve → rerank → generate → END
Tool Calling 循环agent → tools → agent（条件边检测是否继续调工具）
重试 / 降级generate → 失败则 retry_node，超限则 fallback → END
人工审核interrupt_before + 条件边路由到 human_review
Agentic RAGretrieve → grade → 质量不足则 rewrite 再检索（形成环）

设计原则

  • 路由函数只读 State、无副作用，分支逻辑集中、可单测。
  • 返回值必须在映射表中存在，避免运行时找不到 Node。
  • 循环边务必设退出条件（步数上限、置信度阈值），防止死循环。
  • END 是内置终止节点，任何可结束的路径都应能到达 END。

LangGraph 中 State 存储哪些字段？

State 是跨节点共享的状态对象，存储执行过程中需读写的数据

字段类型 常见字段
用户输入question、session_id、user_id、tenant_id
对话上下文messages、history_summary、memory
检索过程rewritten_query、retrieved_docs、reranked_docs、citations
工具调用tool_calls、tool_results、tool_errors
控制字段route、step、retry_count、need_human_review
输出结果answer、confidence、error

State 不宜膨胀：大对象、原始全文、敏感数据只存引用 ID 或摘要。

LangGraph 中 Builder 怎么设计？

Builder 负责把 Node、Edge、State 组装成可执行的图，典型流程是：定义 State → 注册 Node → 连 Edge → compile()。

标准构建步骤

  1. 定义 State：规定图运行有哪些字段
  2. 创建图：绑定状态结构 schema
  3. 注册 Node：每个 Node 只做一件事，把业务步骤注册到图
  4. 连接 Edge：固定边 、条件边 ，定义节点执行顺序和分支条件
  5. 编译运行：编译成可执行Graph
from langgraph.graph import StateGraph, END
# 创建一个图构建器 builder
builder = StateGraph(AgentState)
# 注册 Node 节点
builder.add_node("rewrite", rewrite_node)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
# 设置入口节点
builder.set_entry_point("rewrite")
# 连接 Edge 边，定义节点之间的执行顺序
builder.add_edge("rewrite", "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)
# 编译图，生成真正可以执行的 graph
graph = builder.compile(checkpointer=memory)

设计要点

  • Builder 只做编排，业务逻辑放在 Node 函数里，便于单测。
  • 复杂路由用 conditional edge 抽成独立 route_fn，Builder 保持可读。
  • 子图（Subgraph）可封装 RAG、Tool 等模块，主 Builder 只负责拼装。
  • 编译后再注入运行时依赖（retriever、llm），Node 通过闭包或依赖注入获取。

Embedding 模型如何选型？

选型维度

维度 说明
语言中文场景选 bge-m3、text-embedding-v3 等
维度768/1024/1536，与向量库兼容
任务检索 vs 分类 vs 聚类
成本自托管 vs API 调用

用 MTEB/C-MTEB 榜单 + 自有评测集 recall@K 做最终决策；切换模型需全量重建索引。

向量维度对检索效果和成本有什么影响？

维度 效果 成本
低维（256–384）语义表达弱存储小、检索快
中维（768–1024）平衡较好主流选择
高维（1536+）细粒度语义好内存/延迟增加

维度由 Embedding 模型决定，不可随意更改。百万级文档下，1536 维 vs 768 维存储差约 2 倍。

余弦相似度、欧氏距离、点积有什么区别？

度量 公式特点 适用
余弦相似度只看方向，不看模长文本 Embedding 最常用
欧氏距离看绝对距离未归一化向量
点积方向 + 模长归一化后等价余弦

大多数 Embedding 模型输出已 L2 归一化，余弦与点积排序一致。选型时与向量库默认度量对齐。

Hybrid Search 是什么？为什么要结合向量检索和 BM25？

Hybrid Search 融合稠密检索（语义）和稀疏检索（关键词），互补各自弱点。

检索方式 擅长 弱项
向量（Dense）语义相似、同义词精确编号、SKU
BM25（Sparse）精确关键词、专有名词语义理解弱

融合方式

  • RRF（Reciprocal Rank Fusion）：按排名融合，无需分数归一化。
  • 加权融合：α·dense_score + (1-α)·sparse_score。

GraphRAG 是什么？和普通 RAG 有什么区别？

GraphRAG 在向量检索基础上构建知识图谱，利用实体关系做社区摘要和多跳推理。

对比 普通 RAG GraphRAG
索引Chunk 向量实体 + 关系 + 社区摘要
擅长单文档问答全局总结、跨文档关联
成本低建图与社区摘要成本高

适合「整个知识库的主题概览」类问题；简单 FAQ 仍用普通 RAG 更经济。

Multi-hop RAG 是什么？

Multi-hop RAG 处理需多步检索才能回答的问题：先检索 A，再从 A 中提取线索检索 B，最终综合生成。

Q: 发明 Transformer 的公司 CEO 是谁？
Hop1: 检索 -> Transformer 由 Google 提出
Hop2: 检索 -> Google CEO 是 Sundar Pichai
Generate: 综合回答

实现方式：迭代检索、子问题分解、Agent 驱动多轮 retrieve。每跳增加延迟，需设 max_hops 上限。

Agentic RAG 是什么？

Agentic RAG 让 Agent 自主决定何时检索、检索什么、是否重试，而非固定流水线。

与普通 RAG 区别

  • 普通 RAG：固定 query -> retrieve -> generate。
  • Agentic RAG：Agent 可多次检索、调用工具、改写策略、自我反思。

灵活但成本高、难调试；适合复杂开放域问题，简单 FAQ 用固定链路更稳。

Self-RAG 是什么？

Self-RAG 让模型在生成过程中自我评估：是否需要检索、检索结果是否相关、生成是否忠实。

反思 Token

  • [Retrieve]：判断是否需要检索。
  • [IsRel]：检索结果是否相关。
  • [IsSup]：生成是否有依据。
  • [IsUse]：回答是否有用。

减少无效检索和幻觉，但增加推理 Token 和延迟。

RAG 中如何处理结构化数据查询？

策略

  • Text-to-SQL：自然语言转 SQL 查数据库。
  • 表格 RAG：行/列切片 + metadata 标记表名。
  • 混合路由：Classifier 判断走向量检索还是 SQL。
用户问题 -> 意图分类
  ├─ 结构化 -> Text-to-SQL -> 执行 -> 格式化
  └─ 非结构化 -> 向量 RAG

结构化数据优先精确查询，向量检索仅作补充。

Text-to-SQL 在 AI 应用中如何落地？

落地链路

  1. Schema 注入：表结构、字段注释、样例行写入 Prompt。
  2. SQL 生成：LLM 输出 SQL，限 SELECT、禁止 DDL/DML。
  3. 校验执行：语法检查、权限过滤、超时限制、结果行数上限。
  4. 结果解释：将查询结果转为自然语言。

风险管控

  • 只读账号、行级权限、SQL 注入防护。
  • 错误 SQL 重试 + 人工兜底。

RAG 中如何处理表格、图片、PDF 复杂版面？

类型 处理方式
PDF 复杂版面Unstructured、MinerU、LayoutLM 做版面分析
表格转 Markdown/HTML 或按行切片，保留表头 metadata
图片OCR + 多模态 Embedding 或 VLM 描述后入库

表格切忌纯文本乱序提取；保留行列结构才能正确回答「第 3 行销售额」类问题。

RAG 如何做增量更新？

增量策略

  1. 文档变更检测：Webhook / 定时扫描 diff。
  2. 按 doc_id 删除旧 Chunk（软删除标记 is_deleted=true）。
  3. 重新解析、切片、Embedding 新内容。
  4. 写入向量库，更新 metadata 版本号。

避免全量重建；Embedding 模型变更时才需全量 re-index。用消息队列异步处理，保证最终一致。

RAG 如何处理文档版本冲突？

策略

  • metadata 存 version、effective_date、is_latest。
  • 检索时优先 is_latest=true，或按日期降序。
  • 同 doc_id 多版本共存，旧版本软删除而非物理删除。
  • 生成时 Prompt 要求「以最新版本为准」。

审计场景可保留历史版本供溯源，但默认检索只返回最新有效版。

RAG 如何设计引用和溯源？

设计要点

  • 每个 Chunk 带 doc_id、page、section、url。
  • 生成时要求标注 [1][2] 引用编号，映射到来源列表。
  • 前端展示可点击跳转原文。
上下文格式:
[1] 来源: 员工手册 v3.2 第5页
内容: 年假天数为...

回答: 根据规定，年假为15天 [1]。

日志记录每次检索命中和最终引用，支持审计回放。

RAG 如何评估答案忠实度？

Faithfulness（忠实度）衡量生成答案是否被检索上下文支持，而非模型臆造。

评估方法

  • LLM-as-Judge：让 Judge 模型逐句判断是否有依据。
  • RAGAS faithfulness：自动分解 claim 并验证。
  • NLI 模型：entailment 判断答案是否被 context 蕴含。

忠实度低说明检索不准或 Prompt 约束不足；需分别排查 retrieve 和 generate 环节。

LLM-as-a-Judge 是什么？

用 LLM 作为评判者，对另一个 LLM 的输出打分或排序，替代人工标注。

典型场景

  • 答案质量评分（相关性、完整性、安全性）。
  • 忠实度验证（是否 grounded）。
  • A/B 测试自动对比两个 Prompt 版本。

注意点

Judge 模型应有足够能力；Position bias（倾向第一个答案）需交换顺序消除；关键场景仍需人工抽检。

RAGAS 可以评估哪些指标？

指标 含义
Context Precision检索结果中相关 Chunk 占比
Context Recall ground truth 所需信息是否被检索到
Faithfulness答案是否被 context 支持
Answer Relevancy答案与问题的相关程度

RAGAS 无需人工标注 ground truth 即可部分自动化评估，适合 CI 回归和 Prompt 迭代。

如何构造 RAG 离线评测集？

构造步骤

  1. 从真实用户日志采样高频问题。
  2. 人工标注期望答案 + 相关 doc_id（ground truth）。
  3. 覆盖边界：多轮指代、无答案、权限受限、复合问题。
  4. 定期补充 bad case 回归。

评测集 100–500 条即可启动；每次改 Chunk/Embedding/Rerank 后跑 recall@K 和 faithfulness 对比。

如何做 Prompt 版本管理？

实践

  • Prompt 存配置中心，带 version、author、changelog。
  • Git 管理 Prompt 模板，PR Review 变更。
  • 请求日志记录 prompt_version，便于回溯。
  • 灰度发布：10% 流量走新版本，对比评测指标。

System Prompt、Developer Prompt、User Prompt 的职责如何划分？

角色 职责 示例
System全局行为、安全边界、输出格式「你是企业知识助手，仅基于资料回答」
Developer应用层指令、工具定义、RAG 上下文模板检索结果拼接格式、Function Schema
User用户实际问题「年假有多少天？」

System 优先级最高；Developer 放应用逻辑；User 只放用户输入，避免 Prompt 注入。

Few-shot Prompt 适合什么场景？

Few-shot 在 Prompt 中提供若干输入-输出示例，引导模型模仿格式和行为。

适合

  • 固定输出格式（JSON、分类标签）。
  • 领域术语翻译、工单分类。
  • 小模型能力补齐。

不适合

  • 示例过多占 Token、引入偏见。
  • 强推理任务（用 CoT 更好）。

Chain-of-Thought 是否应该直接暴露给用户？

CoT 让模型逐步推理，提升复杂题准确率，但是否展示给用户需权衡。

策略 做法 场景
隐藏 CoT内部推理，只输出结论面向终端用户、防泄露
展示 CoT流式输出思考过程教育、调试、可解释性要求高

o1 等推理模型已内置 CoT；生产环境通常隐藏，避免用户迷惑或泄露系统逻辑。

Function Calling 和 JSON Schema 有什么关系？

Function Calling 用 JSON Schema 描述工具的 name、description、parameters，模型据此生成结构化调用参数。

{
  "name": "search_docs",
  "description": "搜索企业知识库",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "搜索关键词"},
      "top_k": {"type": "integer", "default": 5}
    },
    "required": ["query"]
  }
}

Schema 越清晰，模型填参越准；required 字段、enum 约束可减少幻觉参数。

Agent 工具调用失败如何降级？

降级策略

  1. 重试：指数退避，最多 3 次。
  2. 备用工具：主 API 失败切缓存或降级 API。
  3. 跳过：告知模型工具不可用，请基于已有信息回答。
  4. 人工兜底：HITL 介入或返回友好错误。
try: result = call_tool(params)
except Timeout: result = {"error": "timeout", "fallback": cached_data}
except Exception: result = {"error": "unavailable"}

Agent 如何做长期记忆和短期记忆？

类型 存储 用途
短期记忆对话 messages、滑动窗口当前会话上下文
工作记忆State 字段、Scratchpad当前任务中间结果
长期记忆向量库、Profile DB跨会话用户偏好、历史摘要

短期记忆超窗口时做摘要压缩；长期记忆通过 RAG 检索注入，避免全量塞入 Prompt。

Agent 如何避免无限循环？

防护机制

  • max_iterations：最多 N 轮 think-act 循环。
  • max_tool_calls：单次请求工具调用上限。
  • 重复检测：相同 action 连续出现则终止。
  • 总 Token / 时间预算：超限强制结束。
  • 终止条件：模型输出 FINAL_ANSWER 或达到目标状态。

LangGraph 用 conditional edge 检测 step > MAX 路由到 END 节点。

如何控制 LLM 应用的 Token 成本？

优化手段

  • 检索 TopK 控制：减少进 Prompt 的 Chunk 数。
  • 历史摘要压缩：长对话不塞全量 messages。
  • 小模型路由：简单任务用小模型，复杂才用大模型。
  • 缓存：相同 query 命中 Semantic Cache。
  • Prompt 精简：去除冗余 System Prompt。

按 tenant/user 设 Token 配额，超限降级或拒服；监控 input/output Token 分布。

如何做模型降级和多模型路由？

路由策略

  • 按任务类型：分类用小模型，生成用大模型。
  • 按复杂度：规则/Classifier 判断难度路由。
  • 按 SLA：超时自动切备用模型。

降级链

GPT-4o -> GPT-4o-mini -> 本地 Qwen -> 缓存/static 回复

模型网关统一路由、熔断、A/B 测试；降级时告知用户或静默切换视场景而定。

什么是大模型？

大模型（LLM）是基于 Transformer 架构、在海量文本上预训练的大规模语言模型，通过预测下一个 Token 学习语言规律和世界知识。

关键特征

  • 参数量亿级到千亿级（7B、70B、405B 等）。
  • 涌现能力：In-context Learning、推理、工具调用。
  • 通过 API 或自托管提供文本生成、Embedding 等能力。

大模型和传统机器学习模型有什么区别？

对比 传统 ML 大模型
特征工程人工设计特征端到端，无需特征工程
数据需求任务级标注数据预训练海量无标注 + 少量微调
泛化单任务专用通用 + Prompt/Few-shot 适配多任务
部署轻量GPU 资源大、延迟高

Transformer 架构是什么？

Transformer（2017）用Self-Attention替代 RNN，并行处理序列，成为 LLM 基础架构。

核心组件

  • Multi-Head Self-Attention：捕捉 token 间依赖。
  • Feed-Forward Network：非线性变换。
  • Layer Norm + Residual：稳定训练。
  • Positional Encoding：注入位置信息。

Decoder-only（GPT）、Encoder-only（BERT）、Encoder-Decoder（T5）三种变体。

Attention 和 Self-Attention 是什么？

Attention 计算 Query 对 Key 的权重，加权求和 Value，让模型关注相关信息。

Attention(Q,K,V) = softmax(QK^T / sqrt(d_k)) · V

Self-Attention

Q、K、V 来自同一序列，每个 token 关注序列中所有 token（含自身），捕捉长距离依赖。

Multi-Head 将 Attention 分多个头并行，学习不同子空间关系。

Token 是什么？

Token 是 LLM 处理文本的最小单位，由 BPE/SentencePiece 等分词器切分。

  • 英文约 1 Token ≈ 4 字符；中文约 1 字 ≈ 1–2 Token。
  • 计费、上下文窗口、速率限制均以 Token 计。
  • 特殊 Token：<|im_start|>、<|endoftext|> 等。

上下文窗口是什么？

上下文窗口是模型单次推理能处理的最大 Token 数（input + output）。

模型 典型窗口
GPT-4o128K
Claude 3.5200K
开源 7B8K–32K

超出窗口需截断、摘要或 RAG 外挂知识；长窗口不等于模型等效利用全部内容（Lost in the Middle）。

SFT、RLHF、DPO 有什么区别？

方法 原理 用途
SFT监督微调，instruction-output 对教模型遵循指令、特定格式
RLHF人类偏好奖励模型 + PPO 强化学习对齐人类价值观、安全性
DPO直接偏好优化，无需奖励模型简化 RLHF，效果接近

典型流程：预训练 -> SFT -> RLHF/DPO。企业应用多数直接用 API 模型 + RAG/Prompt，无需自训。

LoRA 和 QLoRA 是什么？

LoRA（Low-Rank Adaptation）在冻结原模型权重基础上，插入低秩矩阵做高效微调。

方法 特点
LoRA只训练 ~1% 参数，多任务可切换 LoRA 适配器
QLoRA4-bit 量化 + LoRA，单卡可微调 70B

适合领域风格、格式遵循；不适合注入大量新事实（用 RAG 更好）。

模型量化是什么？

量化将 FP16/FP32 权重压缩为 INT8/INT4，降低显存和推理延迟。

精度 显存 精度损失
FP16基准无
INT8~50%很小
INT4 (GPTQ/AWQ)~25%轻微，可接受

生产自托管常用 vLLM + AWQ/GPTQ 量化模型平衡成本与质量。

MoE 是什么？

MoE（Mixture of Experts）将 FFN 层替换为多个 Expert，每次推理只激活部分 Expert（如 8 选 2）。

  • 总参数量大（如 Mixtral 8x7B），但激活参数少，推理成本接近小模型。
  • 不同 Expert 可专精不同任务/语言。

代表：Mixtral、DeepSeek-V2/V3。部署需注意 Expert 负载均衡。

多模态大模型是什么？

多模态模型同时处理文本、图像、音频、视频等多种输入模态。

  • 架构：Vision Encoder + LLM 融合（如 LLaVA、GPT-4V）。
  • 应用：图片理解、OCR、图表分析、视频摘要。
  • RAG 扩展：多模态 Embedding 统一检索文本和图片。

Prompt Engineering 是什么？

Prompt Engineering 是通过设计输入文本引导 LLM 输出 desired 结果的技术，无需修改模型参数。

常用技巧

  • 角色设定、任务描述、输出格式约束。
  • Zero-shot / Few-shot / CoT。
  • Structured Output、Function Calling。
  • 分隔符、XML 标签组织上下文。

Zero-shot、Few-shot、CoT 有什么区别？

方法 做法 适用
Zero-shot仅指令，无示例通用任务、大模型足够强
Few-shot提供若干示例格式固定、小模型
CoT「逐步思考」引导推理链数学、逻辑、多步推理

可组合：Few-shot CoT = 示例中包含推理过程。

如何设计结构化输出 Prompt？

方法

  • JSON Schema 约束：API 层 response_format: json_schema。
  • Prompt 明确字段、类型、示例。
  • 输出包裹在 ```json 代码块中便于解析。
请严格输出 JSON，不要其他文字:
{"answer": "...", "confidence": 0.0-1.0, "sources": ["doc_id"]}

RAG 和 Fine-tuning 有什么区别？

对比 RAG Fine-tuning
知识来源外部检索，实时更新写入模型参数
成本索引成本低训练 GPU 成本高
溯源可引用来源不可溯源

可组合：RAG 供知识，微调优化风格和任务习惯。

为什么企业知识库问答通常优先选择 RAG？

  • 知识更新快：制度、产品文档频繁变更，RAG 重新索引即可。
  • 可溯源审计：金融、医疗、法务需引用来源。
  • 权限隔离：metadata 过滤实现租户/部门级访问控制。
  • 成本低：无需 GPU 集群微调大模型。
  • 幻觉可控：基于检索资料生成，比纯参数记忆更可靠。

微调适合固化回答风格或专业术语表达，不适合注入大量易变事实。

RAG 检索召回不准如何排查？

排查顺序

  1. 确认问题：是检索没命中还是生成没用好？
  2. 查 Embedding：模型是否与索引一致？
  3. 查 Chunk：切片是否截断关键信息？
  4. 查 Query：是否需要 Rewrite？
  5. 查 TopK / 过滤：K 太小？权限误过滤？
  6. 查 Hybrid：是否缺 BM25 补关键词？
  7. 加 Rerank 对比前后 recall@K。

用离线评测集定位：retrieve 准但 answer 差 → 生成/Prompt 问题；retrieve 就不准 → 索引/切片问题。

Chunk overlap 如何设置？

Overlap 是相邻 Chunk 之间的重叠 Token 数，防止关键信息被切在边界上。

  • 常见：Chunk 512 Token，overlap 50–100（10%–20%）。
  • 技术文档、法律条文：overlap 可更大。
  • overlap 过大：冗余存储、检索重复命中。

用评测集对比 overlap=0 vs 10% vs 20% 的 recall@K 选最优值。

向量数据库选型看哪些指标？

指标 说明
规模百万 vs 亿级向量
延迟P99 查询 ms 级要求
Hybrid是否原生支持 BM25 + 向量
过滤metadata 过滤性能

常见：Milvus、Qdrant、Weaviate、pgvector、Elasticsearch kNN。小规模 pgvector 够用；大规模专用向量库。

HNSW 和 IVF 有什么区别？

百万级以下 HNSW 常用；亿级可 IVF+PQ 压缩。无绝对最优，需 benchmark recall-latency 曲线。

Lost in the Middle 是什么？

研究发现 LLM 对上下文中间位置的信息利用率最低，首尾关注度更高。

对 RAG 的影响

  • 检索到正确 Chunk 但排在中间，模型可能忽略。
  • 大量 Chunk 塞入 Prompt 反而降低准确率。

缓解

  • Rerank 后只取 Top3–5 高质量 Chunk。
  • 最相关 Chunk 放 Prompt 首尾。
  • 摘要压缩而非全文堆叠。

Agent 和普通 Workflow 有什么区别？

对比 Workflow Agent
流程预定义固定步骤LLM 动态决策下一步
灵活性低，可预测高，适应开放任务
成本/延迟低高（多轮 LLM 调用）
适用ETL、审批流、固定 RAG开放域问答、复杂工具编排

能用 Workflow 就不用 Agent；Agent 适合路径不确定的任务。

Agent 的核心组成有哪些？

  • LLM：推理与决策大脑。
  • Tools：搜索、API、数据库、代码执行。
  • Memory：短期对话 + 长期知识。
  • Planning：任务分解、ReAct、Plan-and-Execute。
  • State：执行状态、中间结果。

工程还需：超时、重试、权限、审计、HITL、可观测性。

ReAct 是什么？

ReAct（Reasoning + Acting）交替思考和行动：Thought -> Action -> Observation 循环。

Thought: 需要查员工手册中的年假规定
Action: search_docs("年假天数")
Observation: [1] 年假为15天...
Thought: 已有足够信息
Action: FINISH
Answer: 年假为15天

比纯 CoT 能获取外部信息；比纯 Tool Calling 有显式推理链。

Plan-and-Execute 是什么？

先由 Planner 生成完整计划，再由 Executor 逐步执行，必要时 Replan。

Plan: 1. 查销售数据 2. 生成图表 3. 写分析报告
Execute Step1 -> Execute Step2 -> ...
Replan (if step failed)

适合步骤清晰的长任务；比 ReAct 计划性更强，但灵活性稍低。

Agent 如何做任务拆解？

方法

  • LLM 分解：「将任务拆成可执行子任务列表」。
  • Map-Reduce：并行处理子任务后汇总。
  • 层次分解：大任务 -> 中任务 -> 原子操作。

每个子任务应有明确输入输出和完成条件；Supervisor Agent 协调子 Agent 执行。

Agent 工具调用如何做参数校验？

  1. JSON Schema 定义参数类型、required、enum。
  2. 模型输出后 Pydantic/Zod 校验。
  3. 校验失败：将错误反馈给模型重试（最多 N 次）。
  4. 业务规则：范围检查、权限检查、SQL 注入防护。
class SearchParams(BaseModel):
    query: str = Field(min_length=1, max_length=200)
    top_k: int = Field(default=5, ge=1, le=20)

Agent 如何处理工具调用超时？

  • 每个 Tool 设独立 timeout（如 API 5s、SQL 10s）。
  • 超时返回结构化错误：{"error": "timeout", "tool": "search"}。
  • 模型收到 error 后可换策略或告知用户。
  • 全局 request timeout 兜底，防止 Agent 无限等待。

异步 Tool 用 callback/webhook + 轮询，避免阻塞 Agent 循环。

Agent 如何做人工确认 HITL？

HITL（Human-in-the-Loop）在关键步骤暂停，等待人工审批后继续。

场景

  • 发送邮件/消息前确认。
  • 金融交易、权限变更。
  • 低置信度答案审核。

LangGraph 用 interrupt_before 暂停图执行，持久化 State，人工输入后 resume。

Agent 如何做可观测性和调试？

  • Trace：每步 Thought/Action/Observation 记录 span。
  • LangSmith / Langfuse：Agent 专用追踪平台。
  • 日志：tool 入参、出参、Token 用量、延迟。
  • 指标：成功率、平均步数、Tool 失败率。

Debug 时回放完整 Agent 轨迹，定位是哪一步决策错误。

模型网关是什么？

模型网关是 LLM 调用的统一入口，屏蔽多模型/多厂商差异，提供治理能力。

功能

  • 路由：按任务/租户/成本选模型。
  • 鉴权：API Key 管理、配额。
  • 限流、熔断、重试、降级。
  • 日志、计费、A/B 测试。

类似 API Gateway，代表：LiteLLM、One API、自研 Gateway。

如何设计模型调用重试、降级和熔断？

机制 策略
重试429/5xx 指数退避，最多 3 次；4xx 不重试
降级主模型失败切备用模型或缓存/static 回复
熔断错误率超阈值暂停调用，半开探测恢复

区分可重试错误（超时、限流）与不可重试（鉴权失败、内容违规）。

如何设计多租户 LLM 应用？

  • 数据隔离：向量库 metadata 过滤 tenant_id；会话数据分库分表。
  • 配额隔离：每租户 Token 限额、QPS 限流。
  • 配置隔离：租户级 Prompt、模型、知识库。
  • 计费：按 Token/请求数计量。

检索层必须强制 tenant 过滤，不能依赖 Prompt 约束防越权。

如何设计 Prompt / Model / Tool 配置中心？

设计要点

  • 统一管理 Prompt 模板、模型路由规则、Tool Schema。
  • 版本化 + 灰度发布 + 回滚。
  • 多环境：dev/staging/prod 隔离。
  • 变更审计：谁改了什么、何时生效。

运行时热加载配置，无需重启服务；A/B 测试按 user_id hash 分流。

如何设计 AI 应用灰度发布和 A/B 测试？

  1. 定义变量：Prompt 版本、模型、Rerank 策略等。
  2. 分流：user_id/session_id hash 分桶（如 90/10）。
  3. 采集指标：faithfulness、延迟、用户满意度、Token 成本。
  4. 统计显著性判断后全量或回滚。

日志记录 experiment_id + variant，支持离线分析。

如何缓存 LLM 响应？

缓存层级

  • Exact Cache：相同 Prompt hash 直接返回。
  • Semantic Cache：Embedding 相似 query 命中（如 GPTCache）。

注意

  • 带用户上下文/时间的 query 不宜缓存。
  • 设 TTL，知识更新后失效。
  • 缓存 key 含 model + prompt_version。

如何处理流式输出？

Streaming 逐 Token 返回，降低首字延迟（TTFT），提升用户体验。

  • API：stream=true，SSE 或 WebSocket 推送。
  • 前端：逐字渲染，支持中断（AbortController）。
  • 后端：流式写日志/审计需注意完整性。
  • Structured Output 流式需累积后 JSON 解析。

如何设计会话历史压缩？

策略

  • 滑动窗口：只保留最近 N 轮。
  • 摘要压缩：旧对话 LLM 摘要后替换。
  • Token 预算：超阈值触发压缩。
  • 重要信息提取：存 long-term memory，不丢关键事实。
if total_tokens > 8000:
    summary = summarize(old_messages)
    messages = [summary] + recent_messages

如何做敏感信息脱敏再送模型？

  1. 识别：正则 + NER 检测手机、身份证、银行卡、邮箱。
  2. 替换：138****1234、[NAME_1] 占位符。
  3. 映射表：占位符 -> 原值，回复时还原（如需）。
  4. 日志脱敏：审计日志也不存明文 PII。

第三方 API 调用前必须脱敏；自托管也需防 Prompt 注入泄露。

如何评估线上 AI 应用质量？

  • 用户反馈：点赞/踩、纠错。
  • LLM-as-Judge：采样请求自动评分。
  • 指标监控：faithfulness 趋势、拒答率、Tool 失败率。
  • Bad Case 队列：低分请求人工复核，加入评测集。

离线评测 + 在线监控双轨；A/B 测试验证 Prompt 变更效果。

如何做模型输出结构化校验？

  1. JSON Schema / Pydantic 定义期望结构。
  2. 解析模型输出，校验类型、required 字段。
  3. 失败：JSON repair -> 重试 -> 降级纯文本。
  4. API 层 response_format 约束（OpenAI Structured Outputs）。
try:
    data = AnswerModel.model_validate_json(llm_output)
except ValidationError:
    data = retry_with_fix_prompt(llm_output, error)

如何处理 LLM 请求超时和取消？

  • 客户端 AbortController / context.cancel 取消请求。
  • 服务端设 read timeout（如 60s），超时返回 504。
  • Agent 全局 deadline：所有步骤共享时间预算。
  • 取消后清理：停止 Token 生成、释放资源。

流式场景用户关闭页面应触发 cancel，避免浪费 Token。

如何设计 AI 应用的反馈闭环？

用户提问 -> 生成回答 -> 用户反馈(👍/👎/纠错)
    -> Bad Case 入库 -> 人工标注 -> 评测集更新
    -> Prompt/RAG 优化 -> A/B 验证 -> 全量发布

闭环关键：反馈数据结构化存储，定期分析 Top 失败模式，优先修复高频 bad case。

如何设计 AI 应用的成本预算和限额？

  • 按 tenant/user 设 Token 日/月配额。
  • 超限：降级小模型、拒服或告警。
  • 成本归因：日志记录 model、input/output tokens、tenant。
  • 预算告警：80% 阈值通知，100% 熔断。
