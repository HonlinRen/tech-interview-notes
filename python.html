Python GIL 是什么？

GIL 全称 Global Interpreter Lock（全局解释器锁）。

它是 CPython 解释器中的一把全局互斥锁，保证同一时刻只有一个线程执行 Python 字节码。

为什么有 GIL？

主要是为了简化 CPython 内存管理。引用计数需要线程安全，GIL 让解释器内部不必为每个对象加细粒度锁。

a = []
b = a

如果多线程同时修改引用计数，可能出现并发竞争。GIL 用粗粒度锁换实现简单和整体安全。

GIL 的影响

  • 多线程可以并发，但在 CPython 中同一时刻通常只有一个线程执行 Python 字节码。
  • 对 CPU 密集型任务，多线程难以利用多核。
  • 对 IO 密集型任务影响较小，线程等待 IO 时会释放 GIL。

Python 多线程、多进程、协程区别？

三种并发模型解决不同问题：多线程适合 IO 密集，多进程适合 CPU 密集，协程适合高并发 IO 且成本更低。

模型 实现方式 是否绕过 GIL 适用场景 成本
多线程多个线程共享进程内存否IO 密集型较低
多进程多个独立进程是CPU 密集型较高
协程单线程内协作式切换不涉及 GIL 竞争高并发 IO很低

多线程

使用 threading，同一进程内多线程共享内存。适合网络请求、文件读写等 IO 场景，线程等待 IO 时会释放 GIL。

import threading

def task():
    print("hello")

t = threading.Thread(target=task)
t.start()
t.join()

注意线程安全问题，共享可变状态需要锁或原子操作。

多进程

使用 multiprocessing，每个进程有独立内存和 GIL，可真正并行利用多核 CPU。

from multiprocessing import Process

def task():
    print("hello")

p = Process(target=task)
p.start()
p.join()

进程间通信成本更高，常用 Queue、Pipe 等方式。

协程

使用 asyncio 和 async / await，在单线程事件循环中切换，遇到 await 时让出执行权。

import asyncio

async def task():
    await asyncio.sleep(1)
    print("done")

asyncio.run(task())

协程切换成本低，但不适合长时间占用 CPU 的同步代码。

三者对比

对比项 多进程 多线程 协程
调度操作系统操作系统事件循环 / 应用层
内存独立地址空间共享进程内存共享单线程资源
切换成本高中低
并行度支持多核受 CPython GIL 限制单线程并发
典型场景CPU 计算、科学计算阻塞 IO、简单并发高并发网络 IO、WebSocket

口诀：CPU 密集优先多进程，阻塞 IO 可用多线程，高并发异步 IO 优先协程。

multiprocessing 怎么绕过 GIL？

multiprocessing 通过 多进程 绕过 GIL。

GIL 是 CPython 解释器进程级别的锁。每个子进程有自己的解释器和 GIL，因此可以并行执行。

线程 1、线程 2、线程 3
共享同一个进程的 GIL

进程 1 有自己的 GIL
进程 2 有自己的 GIL
进程 3 有自己的 GIL

因此多进程可以真正利用多核 CPU 并行执行。

适用场景

  • 数值计算、图像处理、机器学习预处理
  • CPU 密集型批处理任务
  • 希望利用多核 CPU 加速

使用成本

  • 进程之间内存隔离，不能像线程一样直接共享对象
  • 进程间通信需要序列化，有额外开销
  • Windows 默认 spawn，注意 if __name__ == "__main__": 保护

多进程适合“算得重、通信少”的任务，不适合频繁共享大状态。

async / await 原理与适用场景

async / await 是 Python 协程语法糖，基于单线程事件循环，在 IO 等待时切换任务。

事件循环 + 协程 + 非阻塞 IO + 主动让出执行权

执行到 await some_io() 时，若 IO 未完成，当前协程挂起，事件循环去执行其他任务。

import asyncio

async def fetch():
    await asyncio.sleep(1)
    return "ok"

async def main():
    result = await fetch()
    print(result)

asyncio.run(main())

执行流程

async def 定义协程函数
-> 调用后得到协程对象
-> create_task / await 交给事件循环
-> await 遇到未完成 IO 时挂起当前协程
-> 事件循环调度其他协程
-> IO 完成后恢复原协程继续执行

适用场景

  • 高并发网络服务
  • 异步 Web 框架（FastAPI、Starlette）
  • WebSocket、长连接
  • 爬虫、消息队列消费
  • 数据库异步访问

注意点

  • 协程链路中的库也要支持异步
  • 不要在协程中直接执行长时间阻塞 CPU 代码
  • CPU 密集任务应使用多进程或线程池

CPython GC 机制

CPython 垃圾回收机制主要包括：

引用计数 为主， 分代回收 处理循环引用。

引用计数

每个对象维护引用计数，计数为 0 时立即释放。无法单独处理循环引用。

a = []
b = a
del a

循环引用问题

引用计数无法回收互相引用的对象。

a = []
b = []
a.append(b)
b.append(a)

分代回收

第 0 代 Young Generation：新创建对象
第 1 代 Middle Generation：经历过一次 GC 的对象
第 2 代 Old Generation：长期存活的对象

GC 通过扫描容器对象引用关系，找出无法到达的对象并回收循环引用。

Python 深拷贝和浅拷贝区别？

浅拷贝只复制最外层容器，嵌套对象仍共享；深拷贝递归复制所有层级，完全独立。

对比项 浅拷贝 深拷贝
复制范围只复制外层容器递归复制内部对象
嵌套可变对象仍然共享通常完全独立
性能成本较低较高，可能复制大量对象
适用场景只需要复制外层结构需要完全隔离修改影响
 
import copy

a = [[1], [2]]
b = copy.copy(a)
c = copy.deepcopy(a)

a[0].append(99)
print(b)  # [[1, 99], [2]]
print(c)  # [[1], [2]]

不可变对象通常不需要深拷贝；修改嵌套结构且要隔离影响时使用 deepcopy。

Python 装饰器是什么？

装饰器是在不修改原函数代码的前提下，给函数增加日志、鉴权、缓存、计时等横切能力的高阶写法。

本质是函数接收函数、返回函数。@decorator 等价于 func = decorator(func)。

from functools import wraps

def log(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        print("before")
        return func(*args, **kwargs)
    return wrapper

@log
def hello():
    print("hello")

建议使用 functools.wraps 保留原函数元信息。

Python 装饰器与 Java 动态代理 / Spring AOP 的区别？

三者都用于在不改业务代码的前提下增强行为，但实现机制不同。

对比项 Python 装饰器 Java 动态代理 / Spring AOP
实现方式函数/类包装运行时生成代理对象
作用对象函数 / 类类 / 方法
底层机制闭包 + 高阶函数代理 + 拦截器链
常见用途日志、鉴权、缓存、重试日志、事务、权限、监控

Python 装饰器更轻量；Spring AOP 基于容器和代理，适合企业级横切治理。

迭代器和生成器区别？

迭代器实现 __iter__() 和 __next__()；生成器是用 yield 简化创建的迭代器。

def nums():
    yield 1
    yield 2

for x in nums():
    print(x)

生成器适合大文件读取、流式处理、分页拉取等按需产生数据的场景，节省内存。

类型 特点 查找复杂度

Python 上下文管理器是什么？

上下文管理器配合 with 使用，进入时调用 __enter__，退出时调用 __exit__，保证资源释放。

with open("demo.txt", "w", encoding="utf-8") as f:
    f.write("hello")

执行流程

进入 with
-> 调用 __enter__
-> 执行业务代码
-> 正常或异常退出
-> 调用 __exit__ 做清理

with 能保证即使发生异常也会执行清理逻辑。

Python 内存泄漏常见原因

Python 有自动垃圾回收机制，但如果对象仍然被引用，就不会被释放。 内存泄漏本质上通常是：对象已经不再需要，但仍被全局变量、缓存、闭包、任务或第三方库持有。

常见原因

  • 全局变量、单例对象、模块级 list / dict 持续增长
  • 缓存没有设置容量上限或过期时间，例如无上限的本地缓存
  • 闭包、装饰器、回调函数意外持有大对象
  • 循环引用中包含复杂析构逻辑，例如不当使用 __del__
  • 线程、协程、定时任务长期不结束，持续持有上下文对象
  • 异常对象和 traceback 被长期保存，间接持有局部变量
  • 文件、数据库连接、HTTP Client、Session 等资源没有及时关闭
  • C 扩展或第三方库内存未释放，例如 numpy、pandas、torch、opencv
  • Java 静态集合长期持有对象，例如 static Map 只放不删。
  • 缓存没有设置过期时间、最大容量或淘汰策略。
  • Java中ThreadLocal 使用后没有 remove，在线程池场景下尤其容易泄漏。
  • 监听器、回调、事件订阅注册后没有取消。
  • Java中 对象仍然存在于 List、Map、Queue 等集合中，没有被及时移除。

排查思路

观察进程 RSS / Python heap / 对象数量是否持续增长
      -> 用 tracemalloc 对比内存快照
      -> 用 gc 查看不可达对象和引用关系
      -> 检查全局变量、缓存、闭包、回调、后台任务
      -> 检查异常日志、traceback 是否长期保存大对象
      -> 如果 Python 对象不多但 RSS 很高，重点关注 C 扩展和 native 内存

常用工具

  • tracemalloc：定位 Python 内存分配来源
  • gc：查看对象回收和循环引用情况

什么是 Python 的 __del__ 析构方法？

__del__ 是 Python 中的析构方法，当对象即将被销毁时， Python 会尝试自动调用该方法。它通常用于对象销毁前的收尾逻辑。

基本示例

class User:
          def __init__(self, name):
              self.name = name
              print(f"{self.name} 创建了")
      
          def __del__(self):
              print(f"{self.name} 被销毁了")
      
      
      u = User("Tom")
      del u

需要注意的是，del u 删除的是变量 u 对对象的引用， 并不一定代表对象会立即销毁。只有当对象没有其他引用时，才可能触发 __del__。

为什么不建议依赖 __del__ 释放资源？

  • __del__ 的执行时机不完全可控
  • 循环引用场景下，对象回收会变得更复杂
  • 解释器退出时，部分全局变量可能已经被清理，导致析构逻辑异常
  • 如果析构方法中出现异常，通常不会像普通异常一样正常抛出

推荐做法

实际开发中，不建议依赖 __del__ 释放文件、数据库连接、 HTTP 连接等资源，更推荐使用 with 上下文管理器或显式 close() 方法。 

# 推荐：使用 with 自动释放资源
      with open("data.txt", "r") as f:
          data = f.read()
      
      
      # 推荐：使用 try...finally 显式关闭连接
      client = HttpClient()
      try:
          client.request()
      finally:
          client.close()

面试中可以总结为：__del__ 是对象销毁前自动调用的析构方法， 但由于执行时机不稳定，生产环境中不应依赖它管理关键资源。 文件、连接、锁等资源更适合通过 with 或 close() 主动释放。

Python 闭包是什么？

闭包是指：内部函数引用了外部函数中的变量，并且这个内部函数被返回到外部继续使用。 即使外部函数已经执行结束，被引用的外部变量仍然会被保留下来。

闭包的本质

闭包 = 函数 + 被函数捕获的外部变量

保存状态示例

def counter():
          count = 0
      
          def add():
              nonlocal count
              count += 1
              return count
      
          return add
      
      
      c = counter()
      
      print(c())  # 1
      print(c())  # 2
      print(c())  # 3

这里的 count 没有定义成全局变量， 但每次调用 c() 时都能记住上一次的值。 这说明闭包可以让函数保存自己的状态。

常见使用场景

  • 装饰器：在不修改原函数代码的情况下增强功能
  • 回调函数：延迟执行时仍然保留上下文变量
  • 封装状态：避免使用全局变量
  • 函数工厂：根据不同参数生成不同行为的函数

和 Go 闭包的关系

Python 闭包和 Go 闭包本质上是同一个概念， 都是函数捕获外部作用域中的变量，并在外部函数结束后继续使用这些变量。 区别主要在语法上：Python 修改外层变量需要使用 nonlocal， Go 的匿名函数可以直接修改外层变量。

面试总结

Python 闭包是指内部函数引用外部函数的变量，并且内部函数被返回后仍然可以访问这些变量。 它的核心是函数能够记住创建它时的上下文，常用于装饰器、回调函数、状态封装和延迟执行等场景。

nonlocal 关键字是什么？

nonlocal 用于在内层函数中修改外层非全局变量。没有它时，赋值会被视为创建局部变量，可能触发 UnboundLocalError。

def make_counter():
    count = 0

    def add():
        nonlocal count
        count += 1
        return count

    return add
关键字 修改范围 典型场景
nonlocal外层函数作用域闭包中的计数器、状态累加
global模块级全局变量修改模块级共享状态

Pydantic 适合解决什么问题？

Pydantic 是 Python 中常用的数据校验和数据解析工具， 主要用于解决“外部数据结构不可靠”的问题。 它可以基于类型注解，对接口参数、配置项、JSON 数据和大模型输出进行类型校验、类型转换、必填校验和默认值处理。

常见使用场景

  • FastAPI 接口入参和响应体校验
  • 配置文件或环境变量校验
  • 嵌套 JSON 数据校验
  • AI Agent / RAG 中校验大模型输出结构

常用功能

1. BaseModel：定义数据模型

BaseModel 是 Pydantic 的核心，用来定义数据结构，并自动完成字段校验、类型转换和默认值处理。 

from pydantic import BaseModel
      
      class User(BaseModel):
          name: str
          age: int
          is_active: bool = True
      
      user = User(name="Tom", age="18")
      print(user.age)  # 18

2. Field：字段约束

Field 用于设置字段限制，例如长度、范围、默认值和描述信息。 

from pydantic import BaseModel, Field
      
      class Product(BaseModel):
          name: str = Field(min_length=2, max_length=50)
          price: float = Field(gt=0)
          stock: int = Field(default=0, ge=0)

3. Validator：自定义校验

Pydantic v2 常用 field_validator 和 model_validator， 分别用于单字段校验和多字段联合校验。 

from pydantic import BaseModel, field_validator
      
      class UserRegister(BaseModel):
          username: str
      
          @field_validator("username")
          @classmethod
          def check_username(cls, value):
              if " " in value:
                  raise ValueError("用户名不能包含空格")
              return value

AI 场景示例

在 AI Agent / RAG 项目中，Pydantic 常用于校验大模型返回的 JSON 结构， 避免字段缺失、类型错误或格式不稳定导致后续流程异常。 

from pydantic import BaseModel
      from typing import List
      
      class LLMAnswer(BaseModel):
          answer: str
          confidence: float
          sources: List[str]
      
      result = LLMAnswer.model_validate_json(llm_output)

Python 协程是什么？

Python 协程是一种可暂停和恢复的函数，通过 async def 定义，用 await 等待 IO 并让出执行权。

协程不是操作系统线程，适合高并发 IO 场景；不适合直接承担 CPU 密集计算

线程是操作系统管理的 切换时需要保存和恢复寄存器、程序计数器、栈指针等上下文，成本较高; 协程主要是程序自己管理的。

但协程底层的网络 IO、文件 IO 仍然需要操作系统支持，只是协程之间的调度切换不像线程那样依赖内核线程调度。 

import asyncio

async def fetch():
    await asyncio.sleep(1)
    return "ok"

asyncio.run(fetch())

Python 字节码是什么？

Python 字节码是 CPython 将 .py 源码编译后得到的一组指令，运行时由解释器执行。通常缓存在 __pycache__ 的 .pyc 文件中。

  • 编译链路：源码 → AST → code object → 字节码指令
  • .pyc 是缓存，不是加密，也不是跨 Python 版本通用
  • 可用 dis 模块查看字节码，理解装饰器、闭包等执行行为
import dis

def add(a, b):
    return a + b

dis.dis(add)

Python 中 is 和 == 有什么区别？

is 比较对象身份（是否同一对象），== 比较值是否相等（通常调用 __eq__）。

  • a is b 等价于 id(a) == id(b)
  • 两个内容相同的 list 通常 == 为 True，但 is 为 False
  • 判断是否为 None 时推荐 is None
a = [1, 2]
b = [1, 2]

print(a == b)  # True
print(a is b)  # False

业务比较值相等用 ==，判断对象身份用 is。

Python 中可变类型和不可变类型有什么区别？

可变对象创建后内容可原地修改；不可变对象一旦创建，任何“修改”都会绑定到新对象。

  • 常见可变类型：list、dict、set
  • 常见不可变类型：int、float、str、tuple、frozenset
  • 不可变类型通常可作为 dict 的 key
nums = [1, 2]
nums.append(3)      # 原地修改

s = "ab"
s = s + "c"         # 绑定到新字符串对象

函数默认参数不要直接使用可变对象，避免共享同一默认实例。

Python 参数传递是值传递还是引用传递？

Python 更准确说是“传对象引用”：传递的是对象引用的副本，既不是纯值传递，也不是 C++ 那种引用别名。

  • 对可变对象，函数内原地修改会影响外部
  • 对参数重新赋值，只是局部变量指向新对象，不影响外部绑定
  • 不可变对象看起来像“值传递”，因为无法原地修改
def add_item(items):
    items.append(3)

def rebind(items):
    items = [9, 9]

a = [1, 2]
add_item(a)
print(a)  # [1, 2, 3]
rebind(a)
print(a)  # [1, 2, 3]

Python 中 MRO 是什么？

MRO（Method Resolution Order，方法解析顺序）规定多继承时，Python 查找属性/方法的先后顺序。

为什么需要 MRO？

多继承常出现「菱形继承」：两个父类继承同一祖先。没有统一顺序，同一方法可能被重复调用或漏调。

      A
     /   \
    B     C
     \   /
      D

Python 3 用 C3 线性化 把继承树排成一条线，规则可记三句：

  • 子类优先于父类
  • 多个父类时，保持声明顺序（如 class D(B, C) 则 B 在 C 前）
  • 全局唯一、可预测，不会因调用路径不同而变

查看 MRO

class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print(D.mro())
# [<class 'D'>, <class 'B'>, <class 'C'>, <class 'A'>, <class 'object'>]

调用 d.method() 时，按 D → B → C → A → object 依次查找，找到即停。

与 super() 的关系

super() 不是固定找「直接父类」，而是沿 MRO 找下一个类，适合多父类协作调用。

class A:
    def hello(self):
        print("A")

class B(A):
    def hello(self):
        print("B")
        super().hello()

class C(A):
    def hello(self):
        print("C")
        super().hello()

class D(B, C): pass

D().hello()
# 输出：B -> C -> A（各一次，A 不会被调两次）

口诀：多继承看 MRO，查方法按顺序；super() 跟着 MRO 走，菱形继承不重复。

Python __new__ 和 __init__ 有什么区别？

__new__ 负责创建实例并返回对象；

__init__ 负责初始化已创建实例。

创建时先 __new__ 再 __init__。

  • __new__ 是静态方法，必须返回实例；__init__ 不能返回值
  • __new__ 可用于不可变类型子类化、单例、元类控制
  • 普通业务类通常只写 __init__
class User:
    def __new__(cls, *args, **kwargs):
        print("create")
        return super().__new__(cls)

    def __init__(self, name):
        print("init")
        self.name = name

Python 元类是什么？

元类是“类的类”。普通类默认由 type 创建，元类可以控制类的创建过程。

  • 元类可在类创建时修改属性、注入校验、自动注册
  • Django ORM、Pydantic 等框架大量使用元类思想
  • 元类能力强但复杂，普通业务优先装饰器、继承等更简单方式
class User:
    pass

print(type(User))      # <class 'type'>
print(type(User()))    # <class '__main__.User'>

Python 中 property 装饰器是什么？

property 把方法包装成属性访问形式，可配合 getter、setter、deleter 做校验和封装。

  • 对外像属性，对内可控制读写逻辑
  • 读取触发 __get__，赋值触发 __set__
  • 适合延迟计算、字段校验、兼容旧接口
class User:
    def __init__(self, age):
        self._age = age

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if value < 0:
            raise ValueError("age must be positive")
        self._age = value

Python dataclass 适合什么场景？

dataclass 适合定义以数据为主的类，自动生成 __init__、__repr__、__eq__ 等样板代码。

  • 适合 DTO、配置对象、简单值对象
  • frozen=True 可模拟不可变对象
  • 可变默认值应使用 default_factory
from dataclasses import dataclass, field

@dataclass
class User:
    name: str
    tags: list[str] = field(default_factory=list)

Python 异常处理最佳实践？

Python 使用异常传递错误，从调用栈向上传播，直到被匹配的 except 捕获或终止程序。

  • try/except 捕获异常；else 在无异常时执行；finally 无论是否异常都执行
  • 不要裸写 except:，应捕获具体异常类型
  • 异常应作为错误信号，配合日志和可观测性
try:
    result = 10 / x
except ZeroDivisionError:
    result = 0
else:
    print("success")
finally:
    print("cleanup")

asyncio.gather 和 asyncio.create_task 有什么区别？

asyncio.create_task 把协程包装成 Task 并调度到事件循环；asyncio.gather 并发等待多个 awaitable 并收集结果。

  • create_task(coro) 创建单个后台任务，可单独 await、cancel
  • gather(a, b, c) 并发执行并等待全部完成，返回结果列表
  • 默认任一异常会导致 gather 失败，可用 return_exceptions=True 收集异常
task = asyncio.create_task(fetch_user())
result1, result2 = await asyncio.gather(fetch_a(), fetch_b())

Python 协程如何做超时、取消和异常？

协程超时和取消需要显式处理，避免任务泄漏和异常被吞掉。常用 asyncio.wait_for、asyncio.timeout、Task.cancel()。

  • 对外部 HTTP、数据库、LLM 调用设置超时，避免无限等待
  • 取消时要捕获 asyncio.CancelledError，在 finally 中释放连接
  • 区分全部失败和部分失败，必要时做降级或补偿
try:
    result = await asyncio.wait_for(fetch(), timeout=3)
except asyncio.TimeoutError:
    result = None
finally:
    await cleanup()

FastAPI 依赖注入机制是什么？

FastAPI 通过 Depends 声明依赖，在请求进入路由前解析并注入，用于鉴权、数据库会话、配置等横切逻辑。

  • 适合获取当前用户、校验权限、创建 DB Session
  • 依赖可以嵌套，FastAPI 会自动解析依赖图
  • 使用 yield 可实现资源获取与释放（如关闭 DB 连接）
from fastapi import Depends

def get_current_user():
    return {"name": "ren"}

@app.get("/profile")
def profile(user = Depends(get_current_user)):
    return user

FastAPI 与 Flask / Django 的差异是什么？

三者都是 Python Web 框架，但定位不同：FastAPI 偏现代 API 和异步，Flask 偏轻量灵活，Django 偏全栈后台。

  • FastAPI：ASGI、类型注解、Pydantic 校验、OpenAPI 文档，适合 API 和 AI/RAG 服务
  • Flask：轻量、生态丰富，适合中小型服务和自定义架构
  • Django：内置 ORM、Admin、认证等，适合传统后台和业务系统

FastAPI 的异步优势在高并发 IO 场景更明显；CPU 密集仍要配合进程/线程池。

Python 项目依赖管理最佳实践？

项目应通过虚拟环境和锁文件隔离依赖，保证开发、测试、生产环境一致。

  • 推荐 venv + requirements.txt，或 Poetry、PDM、uv 等现代工具
  • 锁文件应提交到版本库，CI/CD 使用相同版本安装
  • 固定 Python 版本，避免“本地能跑、线上不行”
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt

生产环境建议使用可复现安装方式，避免临时升级导致版本漂移。

Python 3 自 2008 年发布以来持续演进。面试常考 3.5 ~ 3.12 的代表性特性；3.0~3.4 已 EOL，了解「彻底告别 Python 2」即可。

async/await 成为原生语法，奠定 asyncio 生态基础（FastAPI、aiohttp 等）。

f-string 简洁高效，是日常字符串格式化的首选；3.7 起字典插入顺序正式保证。

减少样板代码，类似 Pydantic 出现前的轻量数据类方案。

在表达式中赋值并复用，减少重复计算。

不再必须从 typing 导入 List、Dict。

类似 switch，但支持解构、守卫条件，适合解析命令/AST/JSON 结构。

  • CPython 解释器专项优化，整体提速约 10%~60%（视场景而定）。
  • 异常栈精确指向出错表达式，排障体验明显提升。
