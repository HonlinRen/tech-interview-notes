1. Python GIL 是什么？

GIL 全称 Global Interpreter Lock（全局解释器锁）。

它是 CPython 解释器中的一把全局互斥锁，保证同一时刻只有一个线程执行 Python 字节码。

为什么有 GIL？

主要是为了简化 CPython 内存管理。引用计数需要线程安全，GIL 让解释器内部不必为每个对象加细粒度锁。

a = []
b = a

如果多线程同时修改引用计数，可能出现竞态。GIL 用粗粒度锁换实现简单和整体安全。

GIL 的影响

  • 多线程可以并发，但在 CPython 中同一时刻通常只有一个线程执行 Python 字节码。
  • 对 CPU 密集型任务，多线程难以利用多核。
  • 对 IO 密集型任务影响较小，线程等待 IO 时会释放 GIL。

2. Python 多线程、多进程、协程区别？

三种并发模型解决不同问题：多线程适合 IO 密集，多进程适合 CPU 密集，协程适合高并发 IO 且成本更低。

模型 实现方式 是否绕过 GIL 适用场景 成本
多线程 多个线程共享进程内存 否 IO 密集型 较低
多进程 多个独立进程 是 CPU 密集型 较高
协程 单线程内协作式切换 不涉及 GIL 竞争 高并发 IO 很低

多线程

使用 threading，同一进程内多线程共享内存。适合网络请求、文件读写等 IO 场景，线程等待 IO 时会释放 GIL。

import threading

def task():
    print("hello")

t = threading.Thread(target=task)
t.start()
t.join()

注意线程安全问题，共享可变状态需要锁或原子操作。

多进程

使用 multiprocessing，每个进程有独立内存和 GIL，可真正并行利用多核 CPU。

from multiprocessing import Process

def task():
    print("hello")

p = Process(target=task)
p.start()
p.join()

进程间通信成本更高，常用 Queue、Pipe 等方式。

协程

使用 asyncio 和 async / await，在单线程事件循环中切换，遇到 await 时让出执行权。

import asyncio

async def task():
    await asyncio.sleep(1)
    print("done")

asyncio.run(task())

协程切换成本低，但不适合长时间占用 CPU 的同步代码。

三者对比

对比项 多进程 多线程 协程
调度操作系统操作系统事件循环 / 应用层
内存独立地址空间共享进程内存共享单线程资源
切换成本高中低
并行度支持多核受 CPython GIL 限制单线程并发
典型场景CPU 计算、科学计算阻塞 IO、简单并发高并发网络 IO、WebSocket

口诀：CPU 密集优先多进程，阻塞 IO 可用多线程，高并发异步 IO 优先协程。

3. multiprocessing 怎么绕过 GIL？

multiprocessing 通过 多进程 绕过 GIL。

GIL 是 CPython 解释器进程级别的锁。每个子进程有自己的解释器和 GIL，因此可以并行执行。

线程 1、线程 2、线程 3
共享同一个进程的 GIL

进程 1 有自己的 GIL
进程 2 有自己的 GIL
进程 3 有自己的 GIL

因此多进程可以真正利用多核 CPU 并行执行。

适用场景

  • 数值计算、图像处理、机器学习预处理
  • CPU 密集型批处理任务
  • 希望利用多核 CPU 加速

使用成本

  • 进程之间内存隔离，不能像线程一样直接共享对象
  • 进程间通信需要序列化，有额外开销
  • Windows 默认 spawn，注意 if __name__ == "__main__": 保护

多进程适合“算得重、通信少”的任务，不适合频繁共享大状态。

4. async / await 原理与适用场景

async / await 是 Python 协程语法糖，基于单线程事件循环，在 IO 等待时切换任务。

事件循环 + 协程 + 非阻塞 IO + 主动让出执行权

执行到 await some_io() 时，若 IO 未完成，当前协程挂起，事件循环去执行其他任务。

import asyncio

async def fetch():
    await asyncio.sleep(1)
    return "ok"

async def main():
    result = await fetch()
    print(result)

asyncio.run(main())

执行流程

async def 定义协程函数
-> 调用后得到协程对象
-> create_task / await 交给事件循环
-> await 遇到未完成 IO 时挂起当前协程
-> 事件循环调度其他协程
-> IO 完成后恢复原协程继续执行

适用场景

  • 高并发网络服务
  • 异步 Web 框架（FastAPI、Starlette）
  • WebSocket、长连接
  • 爬虫、消息队列消费
  • 数据库异步访问

注意点

  • 协程链路中的库也要支持异步
  • 不要在协程中直接执行长时间阻塞 CPU 代码
  • CPU 密集任务应使用多进程或线程池

5. CPython GC 机制

CPython 垃圾回收机制主要包括：

引用计数 为主， 分代回收 处理循环引用。

引用计数

每个对象维护引用计数，计数为 0 时立即释放。无法单独处理循环引用。

a = []
b = a
del a

循环引用问题

引用计数无法回收互相引用的对象。

a = []
b = []
a.append(b)
b.append(a)

分代回收

第 0 代 Young Generation：新创建对象
第 1 代 Middle Generation：经历过一次 GC 的对象
第 2 代 Old Generation：长期存活的对象

GC 通过扫描容器对象引用关系，找出无法到达的对象并回收循环引用。

16. Python 深拷贝和浅拷贝区别？

浅拷贝只复制最外层容器，嵌套对象仍共享；深拷贝递归复制所有层级，完全独立。

对比项 浅拷贝 深拷贝
复制范围只复制外层容器递归复制内部对象
嵌套可变对象仍然共享通常完全独立
性能成本较低较高，可能复制大量对象
适用场景只需要复制外层结构需要完全隔离修改影响
 
import copy

a = [[1], [2]]
b = copy.copy(a)
c = copy.deepcopy(a)

a[0].append(99)
print(b)  # [[1, 99], [2]]
print(c)  # [[1], [2]]

不可变对象通常不需要深拷贝；修改嵌套结构且要隔离影响时使用 deepcopy。

17. Python 装饰器是什么？

装饰器是在不修改原函数代码的前提下，给函数增加日志、鉴权、缓存、计时等横切能力的高阶写法。

本质是函数接收函数、返回函数。@decorator 等价于 func = decorator(func)。

from functools import wraps

def log(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        print("before")
        return func(*args, **kwargs)
    return wrapper

@log
def hello():
    print("hello")

建议使用 functools.wraps 保留原函数元信息。

494. Python 装饰器与 Java 动态代理 / Spring AOP 的区别？

三者都用于在不改业务代码的前提下增强行为，但实现机制不同。

对比项 Python 装饰器 Java 动态代理 / Spring AOP
实现方式函数/类包装运行时生成代理对象
作用对象函数 / 类类 / 方法
底层机制闭包 + 高阶函数代理 + 拦截器链
常见用途日志、鉴权、缓存、重试日志、事务、权限、监控

Python 装饰器更轻量；Spring AOP 基于容器和代理，适合企业级横切治理。

18. 迭代器和生成器区别？

迭代器实现 __iter__() 和 __next__()；生成器是用 yield 简化创建的迭代器。

def nums():
    yield 1
    yield 2

for x in nums():
    print(x)

生成器适合大文件读取、流式处理、分页拉取等按需产生数据的场景，节省内存。

20. Python 上下文管理器是什么？

上下文管理器配合 with 使用，进入时调用 __enter__，退出时调用 __exit__，保证资源释放。

with open("demo.txt", "w", encoding="utf-8") as f:
    f.write("hello")

执行流程

进入 with
-> 调用 __enter__
-> 执行业务代码
-> 正常或异常退出
-> 调用 __exit__ 做清理

with 能保证即使发生异常也会执行清理逻辑。

21. Python 内存泄漏常见原因

Python 有 GC，但仍可能出现对象长期无法回收的情况，排查时要区分 Python 堆增长和 native 内存增长。

  • 全局缓存、单例、模块级容器长期持有引用
  • 闭包捕获大对象
  • 循环引用中绑定 __del__ 等复杂析构
  • C 扩展未正确释放内存
  • 长生命周期协程、线程、回调未清理

排查思路

观察 RSS / heap / 对象数量是否持续增长
-> 用 tracemalloc 对比快照
-> 用 gc 查看不可达对象和引用关系
-> 检查缓存、全局变量、后台任务
-> 若 Python 对象不多但 RSS 高，关注 C 扩展和 native 内存

常用工具：tracemalloc、gc、objgraph、heap dump、RSS 监控。

51. Python 闭包是什么？

闭包是内部函数引用了外部函数作用域中的变量，即使外部函数已经返回，内部函数仍能访问这些变量。

def make_counter():
    count = 0

    def add():
        nonlocal count
        count += 1
        return count

    return add

形成条件

  • 函数嵌套
  • 内部函数引用外部局部变量
  • 外部函数返回内部函数或将其传递到外部使用

闭包常用于工厂函数、装饰器、延迟绑定状态。

52. nonlocal 关键字是什么？

nonlocal 用于在内层函数中修改外层非全局变量。没有它时，赋值会被视为创建局部变量，可能触发 UnboundLocalError。

def make_counter():
    count = 0

    def add():
        nonlocal count
        count += 1
        return count

    return add
关键字 修改范围 典型场景
nonlocal外层函数作用域闭包中的计数器、状态累加
global模块级全局变量修改模块级共享状态

53. Pydantic 适合解决什么问题？

Pydantic 用于数据建模、类型转换和校验，适合接口入参、配置、消息体等结构化数据在进入业务逻辑前被“洗干净”。

from pydantic import BaseModel, Field

class UserIn(BaseModel):

常见用途

  • Web 接口请求体、响应模型
  • 配置文件和环境变量校验
  • 消息队列、外部 API、LLM 输出结构校验

在 FastAPI 中，Pydantic 自动校验请求并生成 OpenAPI 文档。

54. field_validator 和 model_validator 区别

field_validator 校验单个字段；model_validator 校验整个模型或字段间关系。

对比项 field_validator model_validator
粒度单字段整个模型
适合格式、范围、单字段转换字段间约束
时机字段赋值时模型初始化或校验时
 
from pydantic import BaseModel, field_validator, model_validator

class Period(BaseModel):
    start: int
    end: int

    @field_validator("start", "end")
    @classmethod
    def positive(cls, value):
        if value < 0:
            raise ValueError("must be positive")
        return value

    @model_validator(mode="after")
    def check_order(self):
        if self.start >= self.end:
            raise ValueError("start must be less than end")
        return self

114. Python 协程是什么？

Python 协程是一种可暂停和恢复的函数，通过 async def 定义，用 await 等待 IO 并让出执行权。

协程不是操作系统线程，也不能直接利用多核 CPU，但适合高并发 IO 场景。

import asyncio

async def fetch():
    await asyncio.sleep(1)
    return "ok"

asyncio.run(fetch())

116. Python 协程有什么优点？

协程切换成本极低，适合大量 IO 等待场景，比线程更省资源。

  • 适合高并发 IO：网络请求、WebSocket、爬虫、消息消费
  • 资源占用低，单线程可调度大量协程
  • async / await 写法接近同步代码，可读性较好
  • 便于超时、取消、并发编排（asyncio.gather 等）

注意：协程不适合直接承担 CPU 密集计算，应交给进程或线程池。

324. Python 字节码是什么？

Python 字节码是 CPython 将 .py 源码编译后得到的一组指令，运行时由解释器执行。通常缓存在 __pycache__ 的 .pyc 文件中。

  • 编译链路：源码 → AST → code object → 字节码指令
  • .pyc 是缓存，不是加密，也不是跨 Python 版本通用
  • 可用 dis 模块查看字节码，理解装饰器、闭包等执行行为
import dis

def add(a, b):
    return a + b

dis.dis(add)

325. Python 中 is 和 == 有什么区别？

is 比较对象身份（是否同一对象），== 比较值是否相等（通常调用 __eq__）。

  • a is b 等价于 id(a) == id(b)
  • 两个内容相同的 list 通常 == 为 True，但 is 为 False
  • 判断是否为 None 时推荐 is None
a = [1, 2]
b = [1, 2]

print(a == b)  # True
print(a is b)  # False

业务比较值相等用 ==，判断对象身份用 is。

326. Python 中可变类型和不可变类型有什么区别？

可变对象创建后内容可原地修改；不可变对象一旦创建，任何“修改”都会绑定到新对象。

  • 常见可变类型：list、dict、set
  • 常见不可变类型：int、float、str、tuple、frozenset
  • 不可变类型通常可作为 dict 的 key
nums = [1, 2]
nums.append(3)      # 原地修改

s = "ab"
s = s + "c"         # 绑定到新字符串对象

函数默认参数不要直接使用可变对象，避免共享同一默认实例。

327. Python 参数传递是值传递还是引用传递？

Python 更准确说是“传对象引用”：传递的是对象引用的副本，既不是纯值传递，也不是 C++ 那种引用别名。

  • 对可变对象，函数内原地修改会影响外部
  • 对参数重新赋值，只是局部变量指向新对象，不影响外部绑定
  • 不可变对象看起来像“值传递”，因为无法原地修改
def add_item(items):
    items.append(3)

def rebind(items):
    items = [9, 9]

a = [1, 2]
add_item(a)
print(a)  # [1, 2, 3]
rebind(a)
print(a)  # [1, 2, 3]

328. Python 中 MRO 是什么？

MRO（Method Resolution Order）是方法解析顺序，决定多继承时 Python 按什么顺序查找父类方法。

  • Python 3 使用 C3 线性化算法，保证顺序一致且符合局部优先原则
  • 可通过 Class.__mro__ 或 Class.mro() 查看
  • super() 按 MRO 查找下一个父类方法
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print(D.mro())  # D -> B -> C -> A -> object

329. Python __new__ 和 __init__ 有什么区别？

__new__ 负责创建实例并返回对象；__init__ 负责初始化已创建实例。创建时先 __new__ 再 __init__。

  • __new__ 是静态方法，必须返回实例；__init__ 不能返回值
  • __new__ 可用于不可变类型子类化、单例、元类控制
  • 普通业务类通常只写 __init__
class User:
    def __new__(cls, *args, **kwargs):
        print("create")
        return super().__new__(cls)

    def __init__(self, name):
        print("init")
        self.name = name

330. Python 元类是什么？

元类是“类的类”。普通类默认由 type 创建，元类可以控制类的创建过程。

  • 元类可在类创建时修改属性、注入校验、自动注册
  • Django ORM、Pydantic 等框架大量使用元类思想
  • 元类能力强但复杂，普通业务优先装饰器、继承等更简单方式
class User:
    pass

print(type(User))      # <class 'type'>
print(type(User()))    # <class '__main__.User'>

331. Python 中 property 装饰器是什么？

property 把方法包装成属性访问形式，可配合 getter、setter、deleter 做校验和封装。

  • 对外像属性，对内可控制读写逻辑
  • 读取触发 __get__，赋值触发 __set__
  • 适合延迟计算、字段校验、兼容旧接口
class User:
    def __init__(self, age):
        self._age = age

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if value < 0:
            raise ValueError("age must be positive")
        self._age = value

332. Python dataclass 适合什么场景？

dataclass 适合定义以数据为主的类，自动生成 __init__、__repr__、__eq__ 等样板代码。

  • 适合 DTO、配置对象、简单值对象
  • frozen=True 可模拟不可变对象
  • 可变默认值应使用 default_factory
from dataclasses import dataclass, field

@dataclass
class User:
    name: str
    tags: list[str] = field(default_factory=list)

333. Python 异常处理最佳实践？

Python 使用异常传递错误，从调用栈向上传播，直到被匹配的 except 捕获或终止程序。

  • try/except 捕获异常；else 在无异常时执行；finally 无论是否异常都执行
  • 不要裸写 except:，应捕获具体异常类型
  • 异常应作为错误信号，配合日志和可观测性
try:
    result = 10 / x
except ZeroDivisionError:
    result = 0
else:
    print("success")
finally:
    print("cleanup")

334. asyncio.gather 和 asyncio.create_task 有什么区别？

asyncio.create_task 把协程包装成 Task 并调度到事件循环；asyncio.gather 并发等待多个 awaitable 并收集结果。

  • create_task(coro) 创建单个后台任务，可单独 await、cancel
  • gather(a, b, c) 并发执行并等待全部完成，返回结果列表
  • 默认任一异常会导致 gather 失败，可用 return_exceptions=True 收集异常
task = asyncio.create_task(fetch_user())
result1, result2 = await asyncio.gather(fetch_a(), fetch_b())

335. Python 协程如何做超时、取消和异常？

协程超时和取消需要显式处理，避免任务泄漏和异常被吞掉。常用 asyncio.wait_for、asyncio.timeout、Task.cancel()。

  • 对外部 HTTP、数据库、LLM 调用设置超时，避免无限等待
  • 取消时要捕获 asyncio.CancelledError，在 finally 中释放连接
  • 区分全部失败和部分失败，必要时做降级或补偿
try:
    result = await asyncio.wait_for(fetch(), timeout=3)
except asyncio.TimeoutError:
    result = None
finally:
    await cleanup()

336. FastAPI 依赖注入机制是什么？

FastAPI 通过 Depends 声明依赖，在请求进入路由前解析并注入，用于鉴权、数据库会话、配置等横切逻辑。

  • 适合获取当前用户、校验权限、创建 DB Session
  • 依赖可以嵌套，FastAPI 会自动解析依赖图
  • 使用 yield 可实现资源获取与释放（如关闭 DB 连接）
from fastapi import Depends

def get_current_user():
    return {"name": "ren"}

@app.get("/profile")
def profile(user = Depends(get_current_user)):
    return user

337. FastAPI 与 Flask / Django 的差异是什么？

三者都是 Python Web 框架，但定位不同：FastAPI 偏现代 API 和异步，Flask 偏轻量灵活，Django 偏全栈后台。

  • FastAPI：ASGI、类型注解、Pydantic 校验、OpenAPI 文档，适合 API 和 AI/RAG 服务
  • Flask：轻量、生态丰富，适合中小型服务和自定义架构
  • Django：内置 ORM、Admin、认证等，适合传统后台和业务系统

FastAPI 的异步优势在高并发 IO 场景更明显；CPU 密集仍要配合进程/线程池。

338. Python 项目依赖管理最佳实践？

项目应通过虚拟环境和锁文件隔离依赖，保证开发、测试、生产环境一致。

  • 推荐 venv + requirements.txt，或 Poetry、PDM、uv 等现代工具
  • 锁文件应提交到版本库，CI/CD 使用相同版本安装
  • 固定 Python 版本，避免“本地能跑、线上不行”
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt

生产环境建议使用可复现安装方式，避免临时升级导致版本漂移。
