JVM 堆里面年轻代、老年代是什么？

JVM 堆通常分为年轻代和老年代。

年轻代

新对象大多数先进入年轻代。年轻代又分为：

Eden 区
Survivor From 区
Survivor To 区

对象一般先分配到 Eden 区。当年轻代空间不足时，会触发 Minor GC。存活下来的对象会进入 Survivor 区，多次存活后晋升到老年代。

老年代

老年代存放生命周期较长的对象，例如长期存在的缓存对象、Spring 单例 Bean、大集合、多次 GC 后仍存活的对象。

新对象 -> Eden
Minor GC 后存活 -> Survivor
多次存活 -> 老年代
老年代不足 -> Major GC / Full GC

volatile 作用

volatile 是 Java 中的轻量级同步机制，主要解决变量在多线程之间的可见性和指令重排序问题。它适合状态标记、配置开关、单例双重检查等场景，但不能替代锁。

保证可见性
禁止指令重排序
不保证复合操作的原子性

保证可见性

普通变量可能被线程缓存到工作内存中，一个线程修改后，另一个线程不一定立刻看到。volatile 会要求写入及时刷新到主内存，读取时从主内存获取最新值。

private volatile boolean running = true;

这类写法常用于线程停止标记：一个线程把 running 改成 false，另一个循环线程能尽快感知。

禁止指令重排序

volatile 会在读写前后加入内存屏障，限制编译器和 CPU 对相关指令进行危险重排。经典例子是双重检查单例，避免对象引用先暴露、构造尚未完成的问题。

public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

不保证原子性

volatile int count = 0;
count++;

count++ 包含读取、加一、写回三个步骤，volatile 不能保证整体原子性。

需求 推荐方式
状态开关、配置刷新volatile
计数自增、复合更新AtomicInteger、LongAdder 或锁
多个变量保持一致synchronized、ReentrantLock 或事务性设计

synchronized 和 ReentrantLock 区别

synchronized 和 ReentrantLock 都能实现互斥访问，并且都是可重入锁。区别在于：synchronized 是 JVM 关键字，写法简单、自动释放；ReentrantLock 是 JUC 工具类，功能更灵活，但必须手动释放。

    // 临界区
}

    lock.lock();

        // 临界区

一般优先使用 synchronized。需要尝试加锁、可中断锁、公平锁、多条件队列或复杂同步控制时，再使用 ReentrantLock。

选择建议

  • 普通临界区保护、代码简单优先用 synchronized。
  • 需要 tryLock() 避免长时间等待，或需要等待可中断时，用 ReentrantLock。
  • 需要多个条件队列，例如生产者消费者中“队列非空”和“队列未满”分别唤醒时，用 ReentrantLock 更合适。
  • 使用 ReentrantLock 必须在 finally 中释放锁，否则异常路径容易死锁。

Java 动态代理是什么？

代理模式是在不直接访问目标对象的情况下，通过代理对象间接访问目标对象，并在调用前后增强额外逻辑。Java 动态代理是在运行时生成代理对象，常见用途包括 AOP、日志、权限校验、事务管理、RPC、MyBatis Mapper 代理。

代理类型 生成时机 特点

动态代理运行阶段运行时生成代理类或子类，适合框架统一增强。

JDK 动态代理要求目标类实现接口，底层通过 Proxy 和 InvocationHandler 拦截接口方法调用。

HashMap 原理是什么？

HashMap 是基于数组、链表和红黑树实现的 key-value 容器。核心思路是先通过 key 的 hashCode() 计算哈希值，再定位到数组桶位；如果多个 key 落到同一个桶，就通过链表或红黑树解决哈希冲突。

put 流程：
计算 key 的 hash
-> 定位数组下标
-> 桶为空：直接放入
-> 桶不为空：比较 hash 和 equals
-> key 已存在：覆盖 value
-> key 不存在：追加到链表或红黑树
-> 元素数量超过阈值：触发扩容

JDK 8 后的结构变化

JDK 8 之后，桶内元素较少时使用链表；当链表长度达到阈值且数组容量足够大时，会转成红黑树，把极端冲突下的查询复杂度从 O(n) 降到 O(log n)。

点 说明
默认容量常见默认初始容量为 16，容量通常保持 2 的幂，方便用位运算定位桶位。
负载因子默认 0.75，在空间占用和冲突概率之间做平衡。
扩容元素数量超过容量乘负载因子后扩容，扩容会重新分布节点。
null允许一个 null key，也允许 null value。

面试里要强调：自定义对象作为 key 时必须正确重写 equals() 和 hashCode()；HashMap 本身不是线程安全的，并发写入要使用外部锁或并发容器。

HashMap 和 ConcurrentHashMap 区别？

二者都用于保存 key-value 映射，但核心差异是并发语义。HashMap 适合单线程或外部同步场景，ConcurrentHashMap 面向多线程并发读写。

对比项 HashMap ConcurrentHashMap
线程安全否是
null允许 null key / value不允许 null key / value
并发控制没有内置同步通过 CAS、synchronized、volatile 等机制控制并发
读操作并发修改时结果不可靠读操作通常不加全局锁，弱一致但可用
适用场景局部变量、单线程缓存、请求内临时 Map共享缓存、在线状态表、计数映射、高并发读写表

JDK 8 的 ConcurrentHashMap 不再使用分段锁 Segment，而是以数组桶为粒度进行并发控制。桶为空时常用 CAS 放入节点，桶不为空时对桶头节点加锁，降低全表锁竞争。

它不允许 null 的原因也常被问到：并发场景下 map.get(key) == null 无法区分“key 不存在”和“value 本身就是 null”，会造成语义歧义。

Java 线程池核心参数有哪些？

ThreadPoolExecutor 通过线程数、队列和拒绝策略共同控制任务执行能力。核心参数不是孤立配置，而是决定“任务来了以后先创建线程、先排队，还是直接拒绝”。

线程池优先用核心线程处理任务；核心线程满了才放入队列；队列也满了才创建非核心线程；线程数达到最大线程数后再来任务，就触发拒绝策略。

参数 作用 面试关注点
corePoolSize核心线程数长期保留的基础处理能力。
maximumPoolSize最大线程数队列满后还能扩展到的上限。
workQueue任务队列决定削峰能力，也决定任务堆积风险。
keepAliveTime空闲线程存活时间通常作用于非核心线程，也可配置核心线程超时。
threadFactory线程工厂用于命名线程、设置优先级、异常处理等。
handler拒绝策略线程数和队列都满时如何处理新任务。

常见队列和拒绝策略

  • ArrayBlockingQueue：有界队列，容量明确，生产环境更容易控制风险。
  • LinkedBlockingQueue：可指定容量，不指定时容量很大，容易造成任务堆积。
  • SynchronousQueue：不存储任务，提交任务必须直接交给工作线程。
  • AbortPolicy：直接抛异常；CallerRunsPolicy：由提交任务的线程执行，能形成一定反压。

生产环境不要只背参数名，还要能说出配置原则：CPU 密集型线程数接近 CPU 核心数；IO 密集型可以适当增大；队列要有界；线程名要可观测；拒绝策略要结合业务可降级、可重试或可失败的语义设计。

Java 线程池执行流程是什么？

线程池提交任务后，不是直接无限创建线程，而是按照核心线程、任务队列、最大线程、拒绝策略的顺序控制资源。

提交任务
-> 核心线程未满：创建核心线程执行
-> 核心线程已满：进入任务队列
-> 队列已满：创建非核心线程
-> 线程达到最大值且队列仍满：执行拒绝策略

配置思路

  • CPU 密集型任务：线程数通常接近 CPU 核心数，避免过多上下文切换。
  • IO 密集型任务：线程数可以适当增大，但要结合下游连接池、限流和超时。
  • 队列建议有界，避免请求堆积导致 OOM。
  • 拒绝策略要和业务语义匹配：可失败、可降级、可重试或调用方反压。

排查线程池问题时，重点看活跃线程数、队列长度、拒绝次数、任务耗时和下游阻塞情况。

ThreadLocal 原理和风险？

ThreadLocal 的作用是让每个线程拥有一份独立变量副本，避免多个线程共享同一个变量。

常见场景包括用户上下文、链路追踪traceId、数据库连接等。

Thread
-> ThreadLocalMap
-> Entry(ThreadLocal 弱引用, value 强引用)

它并不是把数据存在 ThreadLocal 对象里，而是存在当前线程的 ThreadLocalMap 中，ThreadLocal 只是访问这个 Map 的 key。

主要风险

  • 污染 线程池线程会被复用，如果请求结束不清理，旧上下文可能污染下一个请求。
  • 泄漏 ThreadLocalMap 的 key 是弱引用，但 value 是强引用；key 被回收后 value 仍可能留在线程里，造成内存泄漏。
  • 异步无法使用 异步执行或线程切换后，原线程的 ThreadLocal 值不会自动传递到新线程。
try {
    context.set(userId);
    // business
} finally {
    context.remove();
}

所以使用原则很简单：谁设置，谁清理；线程池、Web 请求拦截器、异步任务尤其要放在 finally 中调用 remove()。

Java 类加载过程是什么？

类加载是 JVM 把 .class 字节码变成运行时 Class 对象的过程。完整过程通常分为加载、验证、准备、解析、初始化，其中验证、准备、解析合称连接。

加载：读取 class 字节码
验证：校验字节码安全性
准备：为静态变量分配内存并设置默认值
解析：符号引用转直接引用
初始化：执行静态变量赋值和 static 代码块
阶段 重点
加载通过类加载器读取字节码，生成 java.lang.Class 对象。
验证检查文件格式、元数据、字节码和符号引用是否安全合法。
准备为类变量分配内存并赋默认值，例如 static int a = 1 先得到默认值 0。
解析把常量池中的符号引用转换成直接引用。
初始化执行静态变量显式赋值和 static 代码块。

类加载就是：把 class 搬进 JVM，检查一下，准备静态变量，执行 static 初始化，然后就能用了

类初始化通常发生在首次主动使用时，例如 new 对象、访问静态字段、调用静态方法、反射调用等。

Java 虚拟线程是什么？

虚拟线程是 JDK 21 正式引入的轻量级线程，属于 Project Loom 的核心能力。它仍然是 java.lang.Thread，但不再一一绑定操作系统线程，而是由 JDK 调度到少量平台线程上执行。

传统平台线程：
Java Thread -> OS Thread

虚拟线程：
大量 Virtual Thread -> 少量 Platform Thread -> OS Thread

适合场景

  • 大量阻塞式 IO：HTTP 调用、数据库访问、RPC 调用、文件 IO 等。
  • 一请求一线程模型：保留同步代码的可读性，同时降低线程创建和切换成本。
  • 高并发但单个请求大部分时间在等待外部资源的服务。

不适合误解

虚拟线程不是让 CPU 密集型计算变快。CPU 计算仍然受核心数限制，如果任务一直占用 CPU，虚拟线程再多也只是增加调度压力。它真正改善的是阻塞 IO 场景下平台线程被长时间占住的问题。

使用时还要注意 synchronized 临界区、阻塞 native 调用、连接池容量等限制。虚拟线程能让线程更便宜，但数据库连接、下游服务吞吐和限流策略仍然要单独设计。

Pattern Matching 解决什么问题？

Pattern Matching 解决的是“先判断类型，再强制转换，再定义变量”这类重复代码。它把类型判断、类型转换和局部变量绑定合在一起，让分支逻辑更清晰。

if (obj instanceof String s) {
    System.out.println(s.length());
}

传统写法需要先 instanceof 判断，再手动强转：

if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

常见收益

  • 减少重复强转，降低类型转换错误。
  • 让 if、switch 中的类型分支更紧凑。
  • 配合 sealed class / sealed interface 表达有限类型集合时，分支覆盖关系更清楚。

面试可以这样说：Pattern Matching 不只是语法糖，它让 Java 在处理多态数据、协议对象、表达式树、领域事件时更容易写出类型安全且可读的分支代码。

Minor GC 和 Full GC 区别？

Minor GC 和 Full GC 的区别主要在回收范围、触发原因和停顿成本。

Minor GC 通常回收年轻代，Full GC 往往会回收整个堆以及相关元数据区域，停顿更重。

对比项 Minor GC Full GC
回收范围主要是年轻代通常涉及年轻代、老年代、元空间等更多区域
触发原因Eden 空间不足、对象分配速度快老年代不足、元空间不足、晋升失败、显式 System.gc() 等

排查重点对象分配速率、年轻代大小、短命对象内存泄漏、大对象、缓存无界、老年代增长、类加载异常
 
新对象大量分配
-> Eden 不足
-> Minor GC
-> 存活对象进入 Survivor 或晋升老年代
-> 老年代压力过大

频繁 Minor GC 不一定是严重问题，但会增加 CPU 消耗；

频繁 Full GC 通常是明显风险信号，需要结合 GC 日志、jstat、heap dump 和对象分配链路排查。

GC日志：GC频率、 GC停顿时间、 GC前后堆大小变化

jstat：YongGC FullGC 次数 、老年代 元空间使用率

heap dump：具体对象内存占用和引用 、异常大的Map List

Java 语言有哪些特点？

Java 是一门面向对象、静态类型、跨平台的通用编程语言。它通过 JVM 屏蔽底层操作系统差异，通过 GC 管理内存，通过丰富的标准库和生态支持 Web、后端、中间件、大数据等场景。

  • 面向对象：封装、继承、多态是核心抽象方式，接口和抽象类常用于解耦。
  • 跨平台：源码编译成字节码，字节码运行在不同平台的 JVM 上。
  • 工程生态成熟：Spring、Netty、Dubbo、MyBatis、JVM 诊断工具链都非常完善。
  • 高并发 多线程 事务支持

JVM、JDK、JRE 有什么区别？

JVM、JRE、JDK 是 Java 运行体系的三个层次。JVM 负责执行字节码，JRE 提供运行环境，JDK 提供开发、编译和诊断工具。

JDK包含 JRE 以及 javac、jar、javadoc、jstack、jmap、jstat 等工具开发、编译、诊断 Java 程序

简单说：运行程序最低需要 JRE，开发程序需要 JDK；JVM 是真正执行字节码的核心。

Java 字节码是什么？为什么 Java 可以跨平台？

Java 字节码是 .java 文件经过 javac 编译后生成的 .class 指令集。

JVM 负责加载、校验、解释执行或 JIT 编译这些字节码。

  • 跨平台的关键是字节码格式统一，不同平台提供对应 JVM 实现。
  • 字节码在执行前会经过类加载和字节码校验，防止非法指令破坏运行时安全。
  • 热点代码会被 JIT 编译成本地机器码，提高长期运行服务的执行效率。

为什么说 Java 是编译与解释并存的语言？

Java 先通过 javac 编译成字节码，运行时 JVM 可以解释执行字节码，也可以把热点代码 JIT 编译成本地机器码，因此是编译与解释并存。

  • 编译阶段把源码转换成平台无关字节码，便于分发和校验。
  • 解释执行让程序可以快速启动并保持动态能力。
  • JIT 根据运行时 profiling 信息优化热点路径，例如方法内联、逃逸分析、锁消除。

GraalVM Native Image ：构建时提前把 Java 应用编译成本地可执行文件。

JIT 和 AOT 有什么区别？

JIT 是运行时编译，AOT 是运行前编译。JIT 更依赖运行时信息，峰值性能和动态优化能力更强；AOT 启动更快、运行时依赖更少。

对比项 JIT AOT
编译时机程序运行过程中程序运行前
优化依据真实运行时 profiling 信息静态分析和构建期信息
启动速度需要解释执行和预热启动更快
峰值性能长期运行后动态优化能力强运行时优化空间较少
典型场景长期运行的后端服务Serverless、CLI、短生命周期任务

HotSpot 的常见执行路径可以理解为：先解释执行，发现热点代码后 JIT 编译成本地机器码，再根据运行时信息继续优化。

AOT 有什么优点？为什么 Java 不全部使用 AOT？

AOT 的主要优点是启动快、内存占用低、运行时依赖少，适合冷启动敏感或短生命周期场景。但 Java 不全部使用 AOT，是因为 Java 生态高度依赖运行时动态能力。

  • 优点：减少启动预热时间，降低部分运行时开销，部署产物更接近原生可执行程序。
  • 限制：反射、动态代理、运行期类加载、SPI、序列化等能力需要额外配置或受限。
  • 取舍：长期运行服务通常更看重 JIT 的动态优化和峰值性能，短生命周期服务更看重 AOT 的启动速度。

所以 AOT 是重要补充，不是对 JIT 的简单替代。实际选型要看启动时间、内存预算、框架兼容性和服务生命周期。

Java 和 C++ 有什么区别？

Java 和 C++ 都支持面向对象，但运行模型和内存管理差异很大。Java 运行在 JVM 上，自动 GC；C++ 编译成本地代码，开发者直接管理对象生命周期。

  • Java 没有指针算术和多继承实现类，内存安全边界更强。
  • C++ 更接近系统底层，适合高性能系统、游戏引擎、数据库内核、基础设施。
  • Java 更强调跨平台、工程生态、运行时诊断和服务端开发效率。

Java 标识符和关键字有什么区别？

标识符是开发者给类、方法、变量、包起的名字；关键字是 Java 语言预留的语法单词，不能作为普通标识符使用。

  • 标识符可以包含字母、数字、下划线、美元符号，但不能以数字开头。
  • class、public、static、void、if、return 等都是关键字。
  • 命名规范通常要求类名大驼峰，方法和变量小驼峰，常量全大写加下划线。

自动装箱和拆箱是什么？

自动装箱是基本类型自动转换为包装类型，拆箱是包装类型自动转换为基本类型。编译器会在字节码层面插入 valueOf() 和 xxxValue() 调用。

  • Integer i = 1 会调用 Integer.valueOf(1)。
  • int x = i 会调用 i.intValue()。
  • 包装类型为 null 时自动拆箱会抛出 NullPointerException。
Integer count = null;
int total = count; // NullPointerException

工程中尤其要小心接口返回的包装类型、数据库字段映射和三元表达式中的隐式拆箱。

包装类型的缓存机制是什么？

包装类型缓存是 valueOf() 对常用值复用对象，减少频繁创建包装对象。最常见的是 Integer 默认缓存 -128 到 127。

  • Integer.valueOf(127) 多次调用可能返回同一个对象。
  • Integer.valueOf(128) 通常返回不同对象，不能依赖 == 比较数值。
  • 包装类型比较数值应使用 equals() 或显式拆箱。
Integer a = 127;
Integer b = 127;
Integer c = 128;
Integer d = 128;

System.out.println(a == b); // true，命中缓存
System.out.println(c == d); // false，通常未命中缓存

面试里要强调：缓存是实现优化，不是业务语义。包装类型数值比较优先使用 equals()，但调用前要处理 null。

Java 泛型是什么？类型擦除是什么？

Java 泛型用于在编译期提供类型约束，减少强制类型转换和运行时类型错误。

类型擦除指大部分泛型信息在编译后被擦除为原始类型或边界类型。

  • List<String> 和 List<Integer> 运行时通常都是 List。
  • 类型擦除导致不能直接 new T()，不能创建泛型数组，运行时也难以直接判断完整泛型实参。
  • 框架需要泛型信息时常用 ParameterizedType、TypeReference 等方式保留类型描述。
List<String> names = new ArrayList<>();
names.add("ren");

// 编译后主要保留为 List，泛型约束由编译器检查

泛型的价值主要在编译期类型安全。运行期如果需要完整泛型信息，通常要借助匿名子类、反射 Type 或框架提供的类型引用对象。

Java 反射是什么？有哪些使用场景？

反射（Reflection）允许程序在运行时动态获取类的元信息（字段、方法、构造器、注解），并创建对象、调用方法、读写属性，无需编译期知道具体类型。

// 获取 Class 对象的三种方式
Class<?> clazz1 = Class.forName("com.example.User");
Class<?> clazz2 = User.class;
Class<?> clazz3 = user.getClass();

// 动态创建对象 + 调用方法
Object obj = clazz1.getDeclaredConstructor().newInstance();
Method method = clazz1.getDeclaredMethod("setName", String.class);
method.setAccessible(true);   // 突破 private 限制
method.invoke(obj, "Tom");

核心 API

API 作用
Class.forName() / .class获取 Class 对象
getDeclaredFields/Methods/Constructors获取字段、方法、构造器（含 private）
newInstance() / Constructor.newInstance()动态创建对象
Method.invoke()动态调用方法
Field.get/set()动态读写属性
getAnnotation()读取运行时注解

典型使用场景

  • 框架底层：Spring IoC 根据注解/配置创建 Bean、注入依赖；AOP 代理拦截方法。
  • ORM / 序列化：MyBatis 映射结果集到对象；Jackson 通过反射读写 JSON 字段。
  • 注解处理：JUnit 扫描 @Test；JPA 解析 @Entity、@Column。
  • 插件/扩展：SPI 加载实现类；动态加载外部 JAR 中的类。
  • 通用工具：BeanUtils 属性拷贝、对象比较、调试打印。

缺点与优化

  • 性能：反射调用比直接调用慢，JVM 可通过 setAccessible(true) + 缓存 Method/Field 对象优化。
  • 封装破坏：可访问 private 成员，破坏 OOP 封装边界。
  • 类型安全：编译期无法检查，错误推迟到运行期。
  • 框架通常用字节码生成（CGLIB、ASM）替代高频反射路径。

面试回答：反射是运行时动态获取类信息并操作对象的机制，是 Spring/MyBatis/Jackson 等框架的基础；业务代码不应滥用，高频路径应缓存反射对象或用字节码优化。

Java SPI 机制是什么？

SPI 是 Service Provider Interface，用于把接口定义和具体实现解耦。调用方依赖接口，运行时通过配置发现实现类。

  • Java 原生 SPI 通过 META-INF/services/接口全限定名 文件声明实现类。
  • ServiceLoader 会扫描并加载这些实现，典型例子是 JDBC Driver。
  • SPI 适合插件化扩展，但原生 SPI 缺少按条件加载、依赖注入和优先级控制，复杂框架通常会增强这套机制。

Java 序列化和反序列化是什么？

序列化是把对象转换成可存储或可传输的数据格式，反序列化是把数据还原成对象。Java 原生序列化、JSON、Protobuf、Kryo 都属于不同序列化方案。

  • Java 原生序列化要求实现 Serializable，但性能、可读性和安全性都不适合很多现代服务场景。
  • 服务接口常用 JSON，内部高性能通信可用 Protobuf、Thrift 等二进制协议。
  • 反序列化不可信数据可能触发安全漏洞，应限制类型、升级依赖并避免反序列化任意对象。

Java 集合框架整体结构是什么？

Java 集合框架（JCF）位于 java.util，分为 Collection（单列元素）和 Map（键值对）两大体系，提供统一的数据结构抽象和算法工具。

Collection 三大分支

  • List：有序、可重复、有下标。ArrayList（数组，查快增删慢）、LinkedList（链表，增删快查慢）。
  • Set：不允许重复。HashSet（Hash 表 O(1)）、LinkedHashSet（保插入序）、TreeSet（红黑树，有序 O(logN)）。
  • Queue/Deque：FIFO 队列 / 双端队列。PriorityQueue（堆排序）、BlockingQueue（阻塞队列，生产者-消费者）。

Map 体系

  • HashMap：数组 + 链表/红黑树，O(1) 均摊，无序，线程不安全。
  • LinkedHashMap：HashMap + 双向链表，保插入序或访问序。
  • TreeMap：红黑树，key 有序 O(logN)。
  • ConcurrentHashMap：分段锁/CAS，线程安全，高并发首选。

工具类

Collections 提供排序、查找、同步包装（synchronizedList）、不可变包装等静态方法；Arrays 提供数组转 List、排序等。

面试回答：JCF 分 Collection（List/Set/Queue）和 Map 两大体系；List 有序可重复，Set 去重，Queue 排队，Map 键值映射；根据是否需要有序、去重、并发选具体实现。

Collection 和 Collections 有什么区别？

Collection 是集合接口，表示一组元素；Collections 是工具类，提供排序、查找、同步包装、不可变包装等静态方法。

  • List、Set、Queue 都继承自 Collection。
  • Collections.sort(list)、Collections.unmodifiableList(list) 是工具方法。
  • 二者名字相近但职责完全不同：一个是数据结构抽象，一个是工具方法集合。

List、Set、Queue、Map 分别适合什么场景？

四者的核心区别来自数据语义：是否有序、是否允许重复、是否按 key 访问、是否表达排队关系。选型时先明确数据特征，再选实现类。

对比总览

接口 有序 重复 访问方式 典型实现
List是（插入序）允许下标 indexArrayList, LinkedList
Set视实现不允许迭代/containsHashSet, TreeSet
Queue是（FIFO）允许队头 poll/peekPriorityQueue, LinkedBlockingQueue
Map视实现key 唯一key → valueHashMap, ConcurrentHashMap

各接口适用场景

  • List：需要按顺序存取、允许重复、通过下标访问。如订单列表、分页结果、需要保序的业务数据。频繁查用 ArrayList，频繁头尾增删用 LinkedList。
  • Set：需要去重或做集合运算（交集/并集）。如用户标签去重、已访问 URL 集合、权限 ID 集合。要 O(1) 查找用 HashSet，要排序用 TreeSet。
  • Queue：任务排队、生产者-消费者、BFS 广度优先。如线程池任务队列（BlockingQueue）、延迟任务（DelayQueue）、TopK（PriorityQueue）。
  • Map：按 key 快速定位 value。如缓存（userId → User）、配置项、索引表、计数器（word → count）。高并发用 ConcurrentHashMap。

要去重                 → Set（HashSet）
要排序去重             → TreeSet / TreeMap

要延迟执行             → DelayQueue

要 key-value 映射      → HashMap
要高并发 key-value     → ConcurrentHashMap
要保插入/访问顺序      → LinkedHashMap

  • Map<K, List<V>>：一对多分组，如部门 → 员工列表。
  • Map<K, Set<V>>：一对多去重分组，如用户 → 标签集合。
  • Queue + Set：BFS 去重遍历，Queue 存待访问，Set 存已访问。

面试回答：List 有序可重复适合列表；Set 去重；Queue 排队和生产消费；Map 按 key 查 value；先定数据语义再选实现，并发场景优先 ConcurrentHashMap / CopyOnWriteArrayList。

ArrayList 扩容机制是什么？

ArrayList 底层是数组。容量不足时会创建更大的新数组，并把旧数组元素复制过去，因此扩容会带来 O(n) 的复制成本。

  • JDK 8 中常见扩容约为原容量的 1.5 倍。
  • 尾部追加均摊复杂度是 O(1)，中间插入或删除需要移动元素，复杂度是 O(n)。
  • 已知数据量时设置初始容量可以减少扩容次数。

ArrayList 为什么线程不安全？

ArrayList 没有同步控制，多线程同时修改时，size 更新、数组写入、扩容复制都可能交叉执行，导致数据覆盖、丢失或越界。

  • 两个线程同时 add 可能读到相同 size，写入同一个位置。
  • 扩容期间另一个线程读写旧数组或新数组，可能出现不一致状态。
  • 并发场景可使用外部加锁、Collections.synchronizedList、CopyOnWriteArrayList 或并发队列。

LinkedList 为什么不一定比 ArrayList 插入删除快？

LinkedList 插入删除只是在已定位节点后修改指针，但定位节点本身通常需要 O(n) 遍历；ArrayList 虽然移动元素，但数组内存连续，CPU 缓存友好。

  • 随机访问场景 ArrayList 明显更快。
  • LinkedList 每个节点都要额外保存前后指针，内存开销更大。
  • 只有在已经持有节点位置、频繁头尾操作等场景，链表优势才更明显。

HashSet 底层为什么基于 HashMap？

HashSet 底层基于 HashMap 实现，Set 中的元素作为 HashMap 的 key，value 使用一个固定的占位对象。

  • Set 需要去重，而 HashMap 的 key 天然唯一。
  • 判断重复依赖元素的 hashCode() 和 equals()。
  • 自定义对象放入 HashSet 时，如果 equals/hashCode 实现不一致，会导致去重失败或查找异常。

LinkedHashMap 如何保证插入顺序？

LinkedHashMap 在 HashMap 的哈希表结构基础上，额外维护一条双向链表，用来记录插入顺序或访问顺序。

  • 默认按插入顺序遍历。
  • 构造时开启 accessOrder 后，可以按访问顺序维护，适合实现 LRU 缓存。
  • 它比 HashMap 多维护链表指针，内存开销略高。

TreeMap 和 HashMap 有什么区别？

HashMap 基于哈希表，关注快速查找；TreeMap 基于红黑树，关注 key 的有序性。

  • HashMap 平均 put/get 是 O(1)，不保证遍历顺序。
  • TreeMap put/get/remove 是 O(log n)，会按 key 的自然顺序或 Comparator 排序。
  • 需要范围查询、排序遍历、最近 key 查询时 TreeMap 更合适。

PriorityQueue 底层原理是什么？

PriorityQueue 是优先队列，底层通常是二叉堆。出队时返回优先级最高或最低的元素，具体顺序由自然排序或 Comparator 决定。

  • 插入和删除堆顶通常是 O(log n)。
  • 获取堆顶是 O(1)，但遍历结果不保证整体有序。
  • 适合 TopK、任务优先级调度、合并有序流等场景。

ArrayBlockingQueue 和 LinkedBlockingQueue 有什么区别？

ArrayBlockingQueue 基于数组，是有界阻塞队列；LinkedBlockingQueue 基于链表，可以指定容量，不指定时容量非常大。

  • ArrayBlockingQueue 内存连续、容量固定，生产消费共用一把锁。
  • LinkedBlockingQueue 链表节点动态创建，put/take 通常使用不同锁，吞吐可能更好。
  • 线程池中使用无界 LinkedBlockingQueue 容易造成任务堆积和内存风险。

DelayQueue 适合什么场景？

DelayQueue 是延迟队列，元素必须实现 Delayed 接口，只有到期元素才能被取出。

  • 底层通常用优先队列按过期时间排序。
  • 适合订单超时关闭、缓存过期、延迟重试、定时任务触发等场景。
  • 它是内存队列，进程重启会丢失任务；可靠延迟任务通常要结合数据库或 MQ。

fail-fast 和 fail-safe 是什么？

fail-fast 是快速失败机制，集合在迭代过程中检测到结构性修改会抛出 ConcurrentModificationException；fail-safe 通常基于副本或弱一致迭代，不直接抛该异常。

  • ArrayList、HashMap 的普通迭代器通常是 fail-fast。
  • CopyOnWriteArrayList 迭代的是快照，属于 fail-safe 风格。
  • fail-fast 不能作为并发安全保证，它只是尽早暴露错误修改。

Iterator 删除元素为什么比 for 循环更安全？

Iterator 的 remove 方法会同步更新迭代器内部状态，避免集合结构修改计数和迭代器期望计数不一致。

  • 增强 for 本质也是迭代器，循环中直接调用集合 remove 容易触发 ConcurrentModificationException。
  • 使用 iterator.remove() 删除的是当前迭代到的元素，状态可控。
  • 并发集合有自己的弱一致迭代语义，不能和普通集合混用理解。

集合判空为什么推荐 isEmpty()？

集合判空推荐使用 isEmpty()，语义明确，并避免通过 size() == 0 暴露不必要的计数细节。

  • isEmpty() 可读性更强，直接表达是否为空。
  • 某些集合或视图计算 size 可能不是常数成本。
  • 判空前仍要处理 null；常见写法是 collection != null && !collection.isEmpty()。

线程的生命周期有哪些状态？

Java 线程状态定义在 Thread.State 枚举中，共 6 种。它们描述的是 JVM 视角下的线程调度状态，与操作系统原生线程状态并非一一对应（例如 RUNNABLE 同时包含「正在 CPU 上执行」和「就绪等待调度」）。

NEW → start() → RUNNABLE ⇄ BLOCKED / WAITING / TIMED_WAITING → run 结束 → TERMINATED

六种状态详解

状态 含义 典型触发
NEWThread 对象已创建，尚未调用 start()new Thread(runnable)
RUNNABLE可运行：正在执行或等待 CPU 时间片start() 之后、未被阻塞/等待时
BLOCKED等待获取 synchronized 监视器锁竞争已进入的 synchronized 块/方法
WAITING无限期等待，不占用 CPUObject.wait()、Thread.join()、LockSupport.park()
TIMED_WAITING带超时的等待Thread.sleep()、wait(timeout)、join(timeout)
TERMINATEDrun() 执行完毕或异常退出正常结束、未捕获异常、被 interrupt 后退出

常见状态转换

  • start() 只能调用一次，重复调用会抛 IllegalThreadStateException。
  • 进入 BLOCKED：线程 A 持有锁，线程 B 在 synchronized 入口等待 → B 为 BLOCKED。
  • 进入 WAITING：线程在 synchronized 内调用 wait() 会释放锁并进入 WAITING，需 notify 唤醒。
  • sleep() 不释放锁，线程进入 TIMED_WAITING；wait() 释放锁，进入 WAITING/TIMED_WAITING。
  • 线程池中的 Worker 线程在取不到任务时会 park，表现为 WAITING/TIMED_WAITING。

排查工具

// 查看当前线程状态
Thread.currentThread().getState();

// 线上排查：jstack <pid> 输出每个线程 STATE
// 大量 BLOCKED → 锁竞争；大量 WAITING → 线程池/条件等待；大量 RUNNABLE → CPU 密集或忙等

面试回答：Java 线程 6 态：NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED；BLOCKED 等 synchronized 锁，WAITING/TIMED_WAITING 等通知或超时，RUNNABLE 含就绪和运行；排查用 getState() 或 jstack。

sleep() 和 wait() 有什么区别？

sleep() 和 wait() 都会让当前线程暂停，但所属类、锁语义、唤醒方式、使用场景完全不同。混用是并发面试的高频考点。

核心对比

对比项 Thread.sleep() Object.wait()
定义位置Thread 静态方法Object 实例方法
是否释放锁不释放释放当前对象的 monitor 锁
调用前提任意上下文必须在 synchronized 块/方法内
设计目的单纯延迟 / 限流线程间条件协作（生产者-消费者）

代码示例

// sleep：持有锁时睡眠，其他线程仍无法进入 synchronized
synchronized (lock) {
    Thread.sleep(1000);  // 不释放 lock
}

// wait：释放 lock，进入等待队列，被 notify 后需重新竞争锁
synchronized (lock) {
    while (!condition) {   // 必须用 while，防虚假唤醒
        lock.wait();
    }
    // 执行业务
}
lock.notifyAll();

常见误区

  • 在 synchronized 外调用 wait() 抛 IllegalMonitorStateException。
  • 用 sleep 模拟线程协作无法释放锁，容易造成死锁或长时间阻塞。
  • wait() 被唤醒后不会立刻继续执行，需重新获取 monitor，因此条件判断要用 while 而非 if。
  • 两者都会响应 interrupt：sleep 抛 InterruptedException；wait 抛异常并退出等待。

面试回答：sleep 是 Thread 方法、不释放锁、超时自动醒，用于延迟；wait 是 Object 方法、释放锁、需 synchronized 内调用、靠 notify 唤醒，用于线程协作；条件等待用 wait + while，单纯休眠用 sleep。

notify() 和 notifyAll() 有什么区别？

notify() 随机唤醒一个等待在同一对象 monitor 上的线程，notifyAll() 唤醒所有等待线程，让它们重新竞争锁。

  • notify 可能唤醒的不是期望条件对应的线程，复杂条件下容易造成线程继续等待。
  • notifyAll 更安全但会带来更多无效唤醒和锁竞争。
  • 等待条件应使用 while 循环检查，避免虚假唤醒或条件不满足。

Java 多线程有哪些实现方式？

Java 中「创建线程」本质是定义任务 + 启动执行。常见有四种写法，生产环境应优先线程池，避免无节制 new Thread()。

四种方式

// 1. 继承 Thread（不推荐：Java 单继承，任务与线程耦合）
class MyThread extends Thread {
    public void run() { /* ... */ }
}
new MyThread().start();

// 2. 实现 Runnable（推荐：任务与线程解耦，可复用）
Runnable task = () -> { /* ... */ };
new Thread(task).start();

// 3. 实现 Callable + FutureTask（有返回值、可抛异常）
Callable<String> callable = () -> "result";
FutureTask<String> ft = new FutureTask<>(callable);
new Thread(ft).start();
String result = ft.get();

// 4. 线程池提交（生产首选）
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(callable);
pool.execute(runnable);

方式对比

方式 优点 缺点
继承 Thread写法直观占用继承位、任务难复用、不利于线程池管理
实现 Runnable解耦、可 Lambda、可提交线程池无返回值
Callable + Future有返回值、可声明受检异常需 Future 阻塞取结果或回调
线程池复用线程、控并发、统一队列与拒绝策略需正确配置参数与优雅关闭

生产实践

  • 不要显式 new Thread() 处理业务任务，用 ThreadPoolExecutor 自定义核心/最大线程数、队列、拒绝策略。
  • 避免 Executors.newFixedThreadPool 等工厂方法的隐藏风险（如无界队列导致 OOM）。
  • 虚拟线程（Java 21+）适合 IO 密集：用 Executors.newVirtualThreadPerTaskExecutor() 轻量创建海量任务。
  • 线程必须调用 start() 才会启动；直接调用 run() 只是普通方法调用，不会新起线程。

面试回答：四种方式：继承 Thread、实现 Runnable、Callable+FutureTask、线程池；推荐 Runnable/Callable 解耦任务，生产用 ThreadPoolExecutor 统一管理；Callable 有返回值，Runnable 无；虚拟线程适合高并发 IO。

Runnable 和 Callable 有什么区别？

二者都是「任务」的抽象，但返回值、异常声明、与 Future 的配合不同。

线程池的 execute() 接收 Runnable，submit() 接收 Runnable 或 Callable 并返回 Future。

接口对比

对比项 Runnable Callable<V>
方法void run()V call() throws Exception
返回值无泛型 V
受检异常不能在 run 签名中抛出，需 try-catch 包装可直接 throws Exception

代码示例

// Runnable：fire-and-forget
executor.execute(() -> {
    sendEmail();  // 异常需在内部处理，否则线程池吞掉
});

// Callable：需要结果
Future<User> future = executor.submit(() -> userDao.findById(1L));
User user = future.get(3, TimeUnit.SECONDS);  // 超时等待

// Runnable 也可 submit，返回 Future<?>，get() 为 null
Future<?> f = executor.submit(() -> log.info("done"));

使用建议

  • 只需异步执行、不关心结果 → Runnable + execute。
  • 需要返回值、异常传播、超时控制 → Callable + submit + Future。

面试回答：Runnable 无返回值、run 不能抛受检异常，用 execute 提交；

Callable 有泛型返回值、call 可抛异常，用 submit 得 Future；

需要结果或异常传播选 Callable，纯异步副作用选 Runnable。

Future 和 CompletableFuture 有什么区别？

Future 表示异步计算结果，但组合能力弱；CompletableFuture 在 Future 基础上提供回调、编排、组合和异常处理能力。

  • Future 主要通过 get 阻塞等待结果。
  • CompletableFuture 支持 thenApply、thenCompose、thenCombine、exceptionally 等链式操作。
  • 复杂异步流程、并行聚合、回调编排更适合 CompletableFuture。

乐观锁和悲观锁有什么区别？

悲观锁和乐观锁是并发冲突处理策略，不是具体锁实现。

悲观锁「先占坑再操作」，乐观锁「先操作再校验是否冲突」。

核心对比

对比项 悲观锁 乐观锁
冲突假设冲突频繁，先加锁冲突少，更新时再检测
Java 实现synchronized、ReentrantLock、数据库 SELECT FOR UPDATECAS（Compare And Swap）版本号 UPDATE … WHERE version=?
读是否阻塞写是（互斥）读通常不加锁
适用场景写多、强一致、库存扣减读多写少、配置更新、计数器
典型代价锁等待、死锁风险自旋CPU、ABA问题

Java 层示例

// 悲观锁
synchronized (this) {
    balance -= amount;
}

// 乐观锁：CAS
AtomicInteger counter = new AtomicInteger(0);
counter.compareAndSet(old, old + 1);  // 失败则重试或放弃

// 数据库乐观锁
// UPDATE account SET balance = ?, version = version + 1
// WHERE id = ? AND version = ?   -- 影响行数 0 表示冲突，业务重试

选型与注意点

  • 写冲突高（秒杀库存）：悲观锁或 Redis 分布式锁更稳妥；纯乐观锁大量重试反而更慢。
  • 读多写少（文章点赞数）：AtomicLong / LongAdder 或 version 字段乐观更新。
  • CAS 的 ABA 问题：值从 A→B→A，CAS 误判未变；可用 AtomicStampedReference 带版本戳解决。
  • 热点 key 下 CAS 自旋激烈 很多线程同时抢着修改同一个变量，大家都用 CAS 乐观更新，但因为竞争太激烈，大量线程不断失败、不断重试，导致 CPU 空转升高
  • 乐观锁更新失败要有重试上限或降级，避免无限循环。

面试回答：悲观锁先加锁再操作（synchronized/Lock/FOR UPDATE），适合写多；乐观锁更新时校验版本或 CAS，适合读多写少；CAS 注意 ABA 和热点自旋，DB 乐观锁用 version 字段，冲突时重试或返回失败。

Java 中 Atomic 原子类的原理是什么？

Atomic 原子类通过 volatile 保证可见性，通过 CAS 保证单变量更新的原子性。

  • CAS 比较内存旧值是否等于期望值，匹配则写入新值。
  • AtomicInteger、AtomicLong、AtomicReference 是常见原子类。
  • 高并发计数热点下 AtomicLong 竞争严重时，可以考虑 LongAdder 分散竞争。

AQS 的核心思想是什么？

AQS（AbstractQueuedSynchronizer）解决的是：多个线程竞争资源时，等待、唤醒和排队机制。它把「资源能否被获取」抽象为 state，把「获取失败怎么办」抽象为CLH 双向队列 + park/unpark，子类只需实现少量模板方法即可构建锁或同步器。

// AQS 核心字段（简化理解）
volatile int state;              // 同步状态：0=未占用，>0=已占用/计数
Node head, tail;                 // CLH 等待队列
// 获取：tryAcquire → 成功则返回；失败则入队 + park
// 释放：tryRelease → 唤醒后继节点

三大核心机制

机制 作用
state + CAS用 volatile state 表示锁/许可/计数，CAS 原子修改，避免重量级 synchronized
CLH 队列获取资源失败的线程封装为 Node 入队，FIFO 排队，避免惊群
park / unpark队列中线程挂起让出 CPU，释放资源时按序 unpark 唤醒后继

两种资源共享模式

  • 独占模式（Exclusive）：同一时刻只有一个线程成功，如 ReentrantLock、ReentrantReadWriteLock 写锁。
  • 共享模式（Shared）：多个线程可同时获取，如 Semaphore 许可、CountDownLatch 倒计时、ReadWriteLock 读锁。

基于 AQS 的实现

类 state 含义 模式
ReentrantLock0=未锁，1=已锁，>1=重入次数独占
Semaphore剩余许可数共享
CountDownLatch剩余计数共享
ReentrantReadWriteLock高 16 位读计数，低 16 位写计数读共享 / 写独占

模板方法（子类实现）

  • 独占：tryAcquire / tryRelease
  • 共享：tryAcquireShared / tryReleaseShared
  • 是否可被当前线程独占：isHeldExclusively
  • AQS 负责队列管理、阻塞、唤醒；子类只定义「什么条件下算获取/释放成功」。

面试回答：AQS 用 volatile state + CAS 表示同步状态，失败线程入 CLH 队列并 park；支持独占/共享两种模式；ReentrantLock、Semaphore、CountDownLatch、读写锁都基于它，子类实现 tryAcquire/tryRelease 等模板方法即可。

ReentrantReadWriteLock 适合什么场景？

ReentrantReadWriteLock 是读写锁，允许多个读线程并发访问，但写线程需要独占访问。

  • 适合读多写少、读操作耗时明显、共享数据结构需要一致保护的场景。
  • 读写锁不是所有场景都更快，写多或锁竞争复杂时开销可能更高。
  • 写锁可以降级为读锁，但读锁不能直接升级为写锁。

StampedLock 是什么？

StampedLock 是 Java 8 引入的锁，支持写锁、悲观读锁和乐观读。它通过 stamp 标识锁状态。

  • 乐观读不阻塞写，但读取后需要 validate 检查期间是否发生写入。
  • 适合读多写少且读操作可以重试的场景。
  • StampedLock 不可重入，使用不当容易死锁或忘记释放 stamp。

并发容器有哪些？

并发容器是为多线程访问设计的数据结构，常见有 ConcurrentHashMap、CopyOnWriteArrayList、BlockingQueue、ConcurrentLinkedQueue 等。

  • ConcurrentHashMap 适合高并发 key-value 读写。
  • CopyOnWriteArrayList 适合读多写少的监听器列表、配置快照。
  • BlockingQueue 适合生产者消费者和线程池任务队列。

ForkJoinPool 适合什么场景？

ForkJoinPool 适合把大任务拆成多个小任务并行执行，再合并结果。它使用工作窃取算法提高线程利用率。

  • 典型场景包括递归计算、分治任务、并行流。
  • 任务应尽量 CPU 密集且可拆分，避免在 ForkJoin 任务中长时间阻塞 IO。
  • Java parallelStream 默认使用公共 ForkJoinPool，阻塞任务可能影响其它并行流。

线程池为什么不建议使用 Executors 快捷方法？

Executors 提供的工厂方法看起来方便，但隐藏了队列类型、容量和线程上限，不符合《阿里巴巴 Java 开发手册》「线程池不允许使用 Executors 创建，应通过 ThreadPoolExecutor 方式」的规范，生产环境容易 OOM 或线程耗尽。

各快捷方法的风险

方法 实际构造 风险
newFixedThreadPool(n)固定 n 线程 + 无界 LinkedBlockingQueue任务提交速度 > 处理速度时队列无限堆积 → OOM
newCachedThreadPool()SynchronousQueue + 最大线程 Integer.MAX_VALUE突发流量创建海量线程 → CPU 上下文切换、OOM
newSingleThreadExecutor()1 线程 + 无界 LinkedBlockingQueue同 Fixed，且无法并行
newScheduledThreadPool(n)DelayedWorkQueue（无界）定时任务堆积撑爆内存

推荐写法

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    8,                              // corePoolSize：常驻线程
    16,                             // maximumPoolSize：峰值线程
    60, TimeUnit.SECONDS,           // keepAliveTime：非核心线程空闲回收
    new ArrayBlockingQueue<>(500),   // 有界队列，背压可控
    new ThreadFactoryBuilder().setNameFormat("biz-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);

参数选型要点

  • CPU 密集：核心线程 ≈ CPU 核数 + 1。
  • IO 密集：核心线程可更大，或结合队列长度与超时监控。
  • 队列选型：ArrayBlockingQueue（有界、数组）、LinkedBlockingQueue（需设容量）、SynchronousQueue（直接交接，适合 Cached 场景但需限 max）。
  • 拒绝策略：AbortPolicy（抛异常）、CallerRunsPolicy（调用方执行，天然限流）、DiscardOldestPolicy 等按业务选。
  • 必须自定义 ThreadFactory，便于 jstack 识别线程来源（如 order-pool-3）。

面试回答：Executors 工厂方法用无界队列或无界线程数，任务堆积或突发流量会导致 OOM/线程耗尽；生产应显式 new ThreadPoolExecutor，设核心/最大线程、有界队列、拒绝策略和命名 ThreadFactory。

线程池如何优雅关闭？

应用停机或容器销毁时，若粗暴 kill -9 或只调 shutdownNow()，可能导致任务丢失、事务半提交、连接未释放。

停止接收新任务 → 等待已在执行和队列中的任务完成 → 超时再强制中断 → 最后处理异常和恢复中断标记。

关闭 API 对比

方法 行为
shutdown()不再接受新任务，已提交任务（含队列中）继续执行
shutdownNow()尝试中断所有线程，返回未开始执行的任务列表
awaitTermination(timeout, unit)阻塞等待线程池完全终止，返回是否超时
isTerminated()线程池是否已完全停止

标准关闭流程

ExecutorService pool = ...;
pool.shutdown();                          // 1. 停止接收新任务
try {
    if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {  // 2. 等待 60s
        List<Runnable> dropped = pool.shutdownNow();   // 3. 超时则中断
        if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
            log.error("线程池未能终止");
        }
    }
} catch (InterruptedException e) {
    pool.shutdownNow();                   // 4. 当前线程被中断，也触发强制关闭
    Thread.currentThread().interrupt();
}

Spring / 生产注意点

  • Spring Boot 中可在 @PreDestroy 或 DisposableBean 中执行上述逻辑。
  • Tomcat / 网关的 graceful shutdown：先摘流量，再关线程池，最后停 JVM。
  • 任务代码必须响应中断：阻塞 IO、Thread.sleep、Future.get 会抛 InterruptedException，不能吞掉。
  • 长循环任务应定期检查 Thread.currentThread().isInterrupted() 或 InterruptedException。
  • 配合 shutdownNow() 返回的未执行任务，可持久化到 MQ/DB 以便重启后补偿。

面试回答：先 shutdown 拒新任务，awaitTermination 等待完成，超时再 shutdownNow 中断；捕获 InterruptedException 时也要 shutdownNow 并恢复中断标志；任务内要正确处理中断，Spring 停机钩子里做同样流程。

类文件结构包含哪些内容？

Class 文件是 JVM 识别的二进制格式，包含魔数、版本号、常量池、访问标志、类索引、字段表、方法表和属性表。

  • 魔数 0xCAFEBABE 用于识别 class 文件。
  • 常量池保存字面量、类名、方法名、字段名、符号引用等。
  • 方法表中的 Code 属性保存字节码指令、操作数栈深度、局部变量表等执行信息。

方法区和永久代、元空间是什么关系？

方法区是 JVM 规范中的运行时内存区域，用于存放类元数据、常量、静态变量等；永久代和元空间是 HotSpot 对方法区的具体实现。

  • JDK 8 之前 HotSpot 使用永久代，受 JVM 堆相关参数影响。
  • JDK 8 起使用元空间，元数据主要分配在本地内存。
  • 元空间仍可能 OOM，常见原因是动态生成大量类、类加载器泄漏。

堆外内存是什么？

堆外内存是不在 Java 堆中的内存，通常通过 DirectByteBuffer、Unsafe、JNI 或本地库分配。

  • Netty、NIO、数据库驱动、压缩加密库都可能使用堆外内存。
  • 堆外内存减少某些 IO 拷贝，但不受普通堆 dump 直接展示。
  • 需要关注 -XX:MaxDirectMemorySize、进程 RSS、Native Memory Tracking。

直接内存有什么风险？

直接内存主要通过 DirectByteBuffer 使用，适合减少 Java 堆和 native IO 之间的数据复制。

  • 它的释放依赖 Cleaner 或显式管理，释放不及时可能导致进程内存持续上涨。
  • 直接内存 OOM 不一定体现在 Java heap 使用率高。
  • 大量 direct buffer 分配释放会带来 native 内存压力和排查复杂度。

什么是 Stop The World？

Stop The World 指 JVM 暂停所有用户线程，让 GC 或某些 JVM 内部操作在安全点执行。

  • STW 期间业务线程无法继续处理请求，会体现为延迟尖刺。
  • Minor GC、Full GC、某些阶段的 G1/CMS/ZGC 操作都可能产生 STW，只是时长不同。
  • 降低 STW 要从堆大小、对象分配速率、GC 选择、晋升失败、大对象和引用处理等方向优化。

如何选择垃圾收集器？

垃圾收集器选择取决于堆大小、延迟目标、吞吐目标、JDK 版本和服务类型。

  • 吞吐优先的批处理任务可考虑 Parallel GC。
  • 通用服务端应用在 JDK 11/17 常用 G1，兼顾吞吐和可控停顿。
  • 超低延迟、大堆场景可评估 ZGC 或 Shenandoah。

G1 的 Region 是什么？

G1 把 Java 堆划分为多个大小相等的 Region，不再固定划分连续的新生代和老年代区域。

  • Region 可以扮演 Eden、Survivor、Old、Humongous 等角色。
  • G1 根据每个 Region 的垃圾比例和回收收益，优先回收价值最高的 Region。
  • 这种结构让 G1 能在目标停顿时间内做增量式回收。

Minor GC、Major GC、Mixed GC、Full GC 分别是什么？

Minor GC 通常回收年轻代；Major GC 常被用来表示老年代回收；Mixed GC 是 G1 中同时回收年轻代和部分老年代 Region；Full GC 会更彻底地回收整个堆，停顿通常更长。

  • Minor GC 频繁通常说明对象分配速率高或年轻代过小。
  • Mixed GC 是 G1 控制老年代垃圾的主要方式。
  • Full GC 常与晋升失败、元空间不足、显式 System.gc 或内存压力有关。

这四个是 JDK 自带的命令行诊断工具，位于 $JAVA_HOME/bin，线上排查 CPU 高、死锁、OOM、频繁 GC 时组合使用，通常流程：jps 找 PID → jstat 看 GC → jstack 看线程 → jmap 导堆。

如何分析线程死锁？

死锁（Deadlock）是多个线程循环等待彼此持有的锁，导致全部永久阻塞。典型四条件：互斥、占有且等待、不可抢占、循环等待。线上表现为接口超时、线程池耗尽、CPU 不高但吞吐为 0。

排查步骤

  1. jps -l 找到 Java 进程 PID。
  2. jstack -l <pid> 导出线程栈（-l 显示锁持有者信息）。
  3. 搜索输出末尾的 Found one Java-level deadlock，JVM 会自动列出参与死锁的线程和锁依赖链。
  4. 若无自动提示，手动找大量 BLOCKED 线程，对比 waiting to lock 与 locked 是否形成环。

jstack 关键字段

"pool-1-thread-2" #15 prio=5 os_prio=0 tid=0x... nid=0x1234 waiting for monitor entry
   java.lang.Thread.State: BLOCKED (on object monitor)
   at com.example.ServiceB.methodB(ServiceB.java:20)
   - waiting to lock <0x000000076ab62208> (a java.lang.Object)   // 等这把锁
   - locked <0x000000076ab62218> (a java.lang.Object)             // 已持有这把锁

常见死锁场景

场景 示例
锁顺序不一致线程 A 先锁 account1 再锁 account2，线程 B 相反
嵌套锁synchronized 内再调外部 synchronized 方法，交叉调用
数据库 + 本地锁持 DB 行锁等本地锁，另一线程持本地锁等 DB
线程池 + 同步等待主线程 submit 后 get() 阻塞，Worker 又等主线程释放锁

修复与预防

  • 统一加锁顺序：多资源加锁按固定 ID 排序，破坏循环等待。
  • 缩小临界区：锁内不做 IO、RPC，减少持锁时间。
  • tryLock 超时：ReentrantLock.tryLock(timeout) 失败则回滚/重试，避免无限等。
  • 减少锁嵌套；能用并发容器（ConcurrentHashMap）就不用 synchronized 包大段逻辑。
  • 开发期用 jconsole / IDEA 线程分析 / 单元测试压测提前发现。

面试回答：jstack 导出线程栈，看 BLOCKED 和 deadlock 提示，分析 waiting to lock 与 locked 是否成环；修复靠统一加锁顺序、缩小临界区、tryLock 超时；预防减少嵌套锁和锁内 IO。

如何分析频繁 GC？

频繁 GC 表现为 RT 抖动、吞吐下降、GC 日志密集甚至 Full GC 连环触发。先要区分 GC 类型（Minor / Mixed / Full），再定位是分配过快、晋升失败、内存泄漏还是参数不合理。

排查流程

1. 开 GC 日志（-Xlog:gc* 或 -XX:+PrintGCDetails）
2. jstat -gcutil <pid> 1000     → 看 Eden/S0/S1/Old/Metaspace 变化
3. 判断类型：
   - Young GC 频繁、Old 低    → 对象分配过快 / 年轻代过小
   - Old 持续上升不回落       → 内存泄漏或大对象晋升
   - Full GC 后 Old 仍高       → 泄漏或 MaxMetaspaceSize 不足
4. jmap -dump → MAT 分析 Dominator Tree / GC Roots

常见原因

现象 可能原因 代码/配置层面
Minor GC 极频繁Eden 过小、短生命周期对象暴增循环内 new 大集合、日志拼接、Stream 中间对象
Old 持续涨对象无法回收（泄漏）静态 Map 无界缓存、ThreadLocal 未 remove、连接未关
Full GC 频繁老年代满、Metaspace OOM、System.gc()大对象直接进老年代、动态代理类过多
GC 后内存不降泄漏或堆设过大MAT 找 Retained Heap 最大的对象链

GC 日志关注点

  • 频率：Young GC 间隔是否 < 1s；Full GC 是否每分钟多次。
  • 耗时：STW 停顿是否超过 SLA（如 200ms）。
  • 回收效果：GC 后 Old 使用率是否明显下降；不降则怀疑泄漏。
  • 晋升：对象过早进老年代 → 调 Survivor 或排查大对象（-XX:PretenureSizeThreshold）。

优化方向

  • 代码：缓存加 LRU/过期、批量处理改流式、避免 ThreadLocal 泄漏、及时 close 流/连接。
  • 参数：-Xms=-Xmx 固定堆；G1 设 MaxGCPauseMillis；容器用 MaxRAMPercentage。
  • 监控：Prometheus + Grafana 看 heap_used、gc_pause_seconds、allocation rate。

面试回答：先开 GC 日志和 jstat 区分 Minor/Full；Young GC 频繁查分配速度和 Eden 大小，Old 涨查泄漏用 jmap+MAT；看 GC 后内存是否回落、STW 耗时；代码层查无界缓存和 ThreadLocal，参数层调堆和 G1。

BIO、NIO、AIO 有什么区别？

BIO 是同步阻塞 IO，NIO 是同步非阻塞 IO，AIO 是异步 IO。三者的差异在于线程是否阻塞等待、是否由操作系统完成后通知。

模型 特点 适用场景
BIO同步阻塞，一个连接通常占用一个线程连接数少、逻辑简单的服务
NIO同步非阻塞，通过 Selector 管理多个连接高并发网络连接、网关、RPC 框架
AIO异步 IO，提交后由系统完成并通知理论上适合异步文件/网络 IO，但 Java 服务端使用较少

面试里重点说 NIO：它不是让单个请求更快，而是让少量线程能管理大量连接，降低线程资源成本。

同步、异步、阻塞、非阻塞有什么区别？

这是 IO 和网络面试的两个正交维度，不要混为一谈：

  • 同步 / 异步：关注结果由谁通知——调用方是否必须主动等待或轮询结果。
  • 阻塞 / 非阻塞：关注等待时线程状态——线程是否被挂起、能否去做别的事。

四概念对比

概念 含义 典型例子
同步发起调用后，自己等待或主动取结果普通方法调用、Future.get()、BIO read 直到有数据
异步发起调用后立即返回，系统/回调完成后通知CompletableFuture 回调、AIO completionHandler、MQ 消费
阻塞等结果期间线程挂起，不占 CPU 但也不能干别的Socket.read() 无数据时线程 BLOCKED
非阻塞调用立即返回（可能无数据），线程可继续执行channel.read(buffer) 返回 0，Selector 轮询

四种组合（IO 模型）

组合 说明 代表
同步 + 阻塞线程挂起等到数据，自己读结果传统 BIO：InputStream.read()
同步 + 非阻塞立即返回，线程自己反复 poll/read 直到有数据NIO：Selector + 非阻塞 Channel
异步 + 阻塞提交异步任务后，调用方仍阻塞等 Futureexecutor.submit().get()
异步 + 非阻塞提交后立即返回，完成时回调通知AIO、Netty 异步写 + Listener

易混点

  • 同步 ≠ 阻塞：NIO 是同步非阻塞——线程没挂起，但还是要自己轮询/read 取结果。
  • 异步 ≠ 一定更快：异步减少的是等待浪费，回调编排复杂度和上下文切换可能增加开销。
  • Future.get() 是异步提交 + 同步等待，并非纯异步。
  • Netty 事件循环：IO 多路复用（同步非阻塞）+ 业务线程池，高并发靠少量 IO 线程管大量连接。
// 同步阻塞：read 直到有字节
inputStream.read(buf);

// 同步非阻塞：没数据立刻返回 0，需循环或 Selector
int n = channel.read(buf);  // n==0 表示暂无数据

// 异步：提交后通过回调处理
CompletableFuture.supplyAsync(() -> fetch())
    .thenAccept(result -> handle(result));  // 不阻塞当前线程

面试回答：同步/异步看结果谁通知——同步自己等/取，异步回调通知；阻塞/非阻塞看线程是否挂起；BIO=同步阻塞，NIO=同步非阻塞，AIO=异步；四者可组合，Future.get 是异步提交同步等待。

Java NIO 的 Channel、Buffer、Selector 分别是什么？

Channel、Buffer、Selector 是 Java NIO 的核心组件。Channel 表示数据通道，Buffer 表示读写缓冲区，Selector 负责监听多个 Channel 的就绪事件。

  • 数据从 Channel 读入 Buffer，再从 Buffer 写入 Channel。
  • Selector 可以监听 accept、connect、read、write 等事件。
  • NIO 适合大量连接、每个连接活跃度不高的网络服务。

IO 多路复用是什么？

IO 多路复用是用一个或少量线程同时监听多个连接的 IO 就绪事件，避免每个连接都占用一个阻塞线程。

  • Linux 下常见实现有 select、poll、epoll。
  • Java NIO Selector 是多路复用思想在 Java 层的封装。
  • 它提升的是连接管理能力，业务处理慢仍然需要线程池或异步化配合。

Reactor 模型是什么？

Reactor 模型是一种事件驱动网络模型，由事件多路复用器监听 IO 事件，再分发给对应 Handler 处理。

Selector 监听 IO 事件
-> 事件就绪
-> 分发给 Handler
-> Handler 处理读写、编解码或业务任务
  • 单 Reactor 单线程模型简单，但 IO 和业务都在一个线程中，容易被慢任务阻塞。
  • 单 Reactor 多线程会把业务处理交给工作线程池。
  • 主从 Reactor 模型通常由 boss 线程处理连接，worker 线程处理读写事件，Netty 就是典型实现。

零拷贝是什么？

零拷贝是减少数据在内核态和用户态之间复制次数的技术总称，常用于文件传输和网络发送。

  • 传统 IO 往往需要磁盘到内核缓冲区、内核到用户缓冲区、用户到 socket 缓冲区等多次复制。
  • sendfile、mmap、DirectBuffer 都能在不同层面减少复制。
  • Kafka、Netty、文件服务器等高吞吐场景经常利用零拷贝降低 CPU 消耗。

Java 中有哪些常见 IO 设计模式？

Java IO 中常见设计模式包括装饰器模式、适配器模式和组合式封装。

  • InputStream/OutputStream 的各种包装类体现装饰器模式，例如 BufferedInputStream 增强缓冲能力。
  • Reader/Writer 处理字符流，InputStream/OutputStream 处理字节流，转换流体现字节到字符的适配。
  • 这种设计让功能可以按需叠加，但类层级较多，使用时要区分字节、字符、缓冲、对象序列化等职责。

Netty 为什么性能高？

Netty 性能高来自 NIO 多路复用、事件循环线程模型、内存池、零拷贝思想和成熟的编解码 pipeline。

能力 为什么有帮助
NIO 多路复用少量线程管理大量连接，降低线程成本。
EventLoopChannel 绑定固定线程，减少锁竞争和线程切换。
ByteBuf读写指针分离，支持池化、直接内存、切片和组合。
Pipeline编解码、业务处理、异常处理可以拆成可组合 Handler。
零拷贝思想减少不必要的数据复制，降低 CPU 和内存带宽消耗。

但 Netty 性能高不代表业务一定快。耗时业务、阻塞数据库调用、慢下游服务仍然要放到业务线程池或异步链路中处理，不能阻塞 EventLoop。

Netty 的 EventLoop 是什么？

EventLoop 是 Netty 的事件循环，一个 EventLoop 通常绑定一个线程，负责处理注册在其上的多个 Channel 的 IO 事件和任务队列。

  • 同一个 Channel 的 IO 事件通常在同一个 EventLoop 线程中执行，减少并发同步成本。
  • 耗时业务不应直接阻塞 EventLoop，否则会影响该线程负责的所有连接。
  • 耗时任务应提交到业务线程池，再把结果写回 Channel。

Netty 的 ByteBuf 相比 ByteBuffer 有什么优势？

ByteBuf 是 Netty 提供的字节缓冲区，相比 JDK ByteBuffer 更适合网络编程。

优势 说明
读写指针分离有 readerIndex 和 writerIndex，读写不需要频繁 flip()。
内存形态丰富支持堆内存、直接内存、池化、切片、组合缓冲区。
池化能力减少频繁分配和回收，降低 GC 压力。
引用计数可以减少复制，但必须正确 release()。

ByteBuf 的风险也很明确：池化和引用计数提升性能的同时，也要求开发者严格管理生命周期，否则容易出现直接内存泄漏。
