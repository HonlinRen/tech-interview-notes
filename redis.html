Redis 为什么快？

Redis 是基于内存的 KV 存储，读写不走磁盘随机 IO，延迟通常在亚毫秒到毫秒级。快的原因是多方面叠加，不只是「数据在内存里」这么简单。

核心原因

因素 说明
纯内存读写数据常驻内存，避免磁盘寻道；持久化异步进行，不阻塞主路径
单线程命令执行命令串行执行，无锁竞争、无上下文切换开销（Redis 6+ 网络 IO 可多线程辅助）
IO 多路复用基于 epoll/kqueue 的事件驱动模型，单线程处理大量并发连接
高效数据结构SDS、跳表、压缩列表等针对场景优化，编码会按数据量自动切换（如 ziplist → hashtable）
 
# 典型 QPS：单机可达 10 万+（简单 GET/SET，视硬件与命令复杂度而定）
GET key
SET key value

注意：「单线程」指命令执行主路径，持久化（AOF rewrite）、网络 IO（可选）、Lazy Free 等可有后台线程参与。复杂命令（KEYS、大 key 操作）仍会阻塞，生产应避免。

面试回答：内存 + 单线程无锁 + IO 多路复用 + 紧凑数据结构，使 Redis 读写极快；但要区分命令执行单线程与 IO/持久化多线程辅助，并注意大 key、慢命令对单线程模型的影响。

Redis 持久化 RDB 和 AOF 区别？

Redis 是内存数据库，重启后数据会丢失，需通过 RDB 快照或 AOF 日志持久化到磁盘。两者机制不同，生产常混合使用。

对比

对比项 RDB AOF
原理fork 子进程，将内存数据序列化写入 .rdb 文件追加写命令到 .aof 文件，重启时重放
触发save 900 1 等规则，或 BGSAVE 手动触发每条写命令追加；appendfsync 控制刷盘频率
恢复速度快，直接加载二进制快照慢，需重放全部写命令（可 AOF rewrite 压缩）
数据丢失可能丢失上次快照后的写入always 最多丢 1 条；everysec 最多约 1 秒
文件体积紧凑，适合备份归档较大，rewrite 后缩小

混合持久化（Redis 4.0+）

开启 aof-use-rdb-preamble yes 后，AOF 重写时先写 RDB 格式快照，再追加增量 AOF，兼顾恢复速度与数据安全。

# 推荐生产配置思路
save 900 1
save 300 10
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes

面试回答：RDB 是定时快照，恢复快但可能丢数据；AOF 是命令日志，更安全但文件大、恢复慢。生产建议混合持久化，并注意 fork 时内存翻倍（写时复制）对大数据集的影响。

缓存穿透、击穿、雪崩是什么？

三者都是缓存与数据库之间的典型故障模式，表现相似（DB 压力骤增），但成因和应对不同，面试需能区分。

对比与方案

问题 含义 典型场景 常见方案
穿透查询不存在的数据，缓存和 DB 都没有恶意刷不存在的 id缓存空值（短 TTL）、布隆过滤器、参数校验
击穿热点 key 过期，大量并发同时查 DB秒杀商品详情缓存失效互斥锁重建、逻辑过期（不设 TTL，异步刷新）
雪崩大量 key 同时失效 Redis 整体不可用批量缓存同一时刻过期、Redis 宕机过期时间加随机值、多级缓存、限流降级、集群高可用

击穿 — 互斥锁示例

# 伪代码思路
val = GET key
if val == null:
  if SET lock:key NX EX 10:   # 只有一个线程去查 DB
    val = queryDB()
    SET key val EX 3600
    DEL lock:key
  else:
    wait and retry GET

面试回答：穿透是查不存在的数据，用布隆过滤器或空值缓存；击穿是热点 key 过期，用互斥锁或逻辑过期；雪崩是批量失效或 Redis 故障，用过期随机化、多级缓存和限流。三者都要避免 DB 被瞬时打垮。

Redis 分布式锁怎么实现？

分布式锁用于多实例环境下对共享资源的互斥访问。Redis 实现的核心是原子加锁 + 唯一 value + 安全释放。

基本实现

# 加锁：NX 不存在才设，PX 毫秒过期防死锁
SET lock:order:1001 uuid-xxx NX PX 30000

# 释放：Lua 保证「判断 value + 删除」原子性，避免误删他人锁
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
else
  return 0
end

关键问题

问题 说明 应对
锁超时业务未完成锁已过期，其他线程获得锁看门狗续期（Redisson）、合理设置 TTL
误删锁线程 A 超时后删了线程 B 的锁value 存唯一标识，Lua 脚本校验后删除
主从切换锁写在 Master，宕机前未同步到 Slave，可能双写RedLock 或改用 etcd/ZooKeeper 等 CP 组件
不可重入同一线程无法再次加锁Hash 结构记录重入次数（Redisson 已实现）

面试回答：用 SET NX PX 原子加锁，value 存 UUID，释放时用 Lua 校验再删。注意超时续期、主从一致性；强一致场景 Redis 锁有局限，可结合 Redisson 或专用协调服务。

Redis 主从、哨兵、集群区别？

Redis 高可用与扩展有三种常见架构，解决不同层次的问题：复制、自动故障转移、数据分片。

三种模式

模式 作用 适用
主从复制Master 写，Slave 异步复制；读写分离提升读能力读多写少、数据量适中
哨兵 Sentinel监控主从健康，Master 宕机自动选举新主、通知客户端需要自动故障转移，数据单机可存下
Cluster 集群16384 个 hash slot 分片到多节点，节点间 Gossip 通信数据量大、需水平扩展

Cluster 分片要点

# slot = CRC16(key) % 16384
# 多 key 操作需在同一 slot，可用 hash tag：{user}:1001 与 {user}:1002

CLUSTER KEYSLOT user:1001
MGET {user}:1001 {user}:1002   # 同一 hash tag，可同节点执行
  • 至少 3 主 3 从，主节点负责 slot，从节点做副本与故障转移。
  • 客户端需支持 Cluster 协议，或使用代理（Codis、Twemproxy）。
  • 主从 + 哨兵不能分片；Cluster 能分片但跨 slot 事务/multi 受限。

面试回答：主从做读写分离和副本；哨兵在主从之上加自动选主；Cluster 通过 hash slot 做数据分片和水平扩展。选型看数据量是否需要分片、以及能否接受 Cluster 的跨 key 限制。

Redis maxmemory 有什么作用？

maxmemory 限制 Redis 进程可使用的最大内存上限。Redis 是内存数据库，不设上限可能导致 OOM 被系统 Kill，生产必须配置。

达到上限后的行为

由 maxmemory-policy 决定：按策略淘汰 key，或 noeviction 下拒绝新写入（读/删仍可用），写命令返回 OOM 错误。

指标 含义
used_memoryRedis 分配器实际占用的内存
used_memory_rss操作系统视角的物理内存（含碎片）
mem_fragmentation_ratiorss / used_memory，>1.5 需关注碎片

配置建议

# redis.conf
maxmemory 4gb
maxmemory-policy allkeys-lru

# 容器部署：maxmemory 应小于容器 limit，预留 20%~30% 给：
# 复制 backlog、AOF rewrite buffer、客户端输出缓冲、系统开销

面试回答：maxmemory 防止 Redis 无限占内存；触顶后按淘汰策略删 key 或拒绝写入。容器环境要留余量，并监控 used_memory 与碎片率。

Redis 内存淘汰策略有哪些？

当内存达到 maxmemory 时，maxmemory-policy 决定淘汰哪些 key。缓存场景必须选对策略，否则可能误删重要数据或频繁 OOM。

八种策略

策略 淘汰范围 算法
noeviction不淘汰写命令报错，默认策略
allkeys-lru所有 key最近最少使用（推荐缓存）
allkeys-lfu所有 key最不经常使用（Redis 4.0+，热点更准）
allkeys-random所有 key随机淘汰
volatile-lru设了 TTL 的 keyLRU
volatile-lfu设了 TTL 的 keyLFU
volatile-random设了 TTL 的 key随机
volatile-ttl设了 TTL 的 key优先淘汰 TTL 更短的

LRU vs LFU

  • LRU：淘汰最久未访问的 key，适合访问模式较均匀的场景。
  • LFU：淘汰访问频率最低的 key，适合有明显热点、长尾 key 多的场景。
  • Redis 的 LRU/LFU 是近似算法（采样而非精确），节省内存与 CPU。

面试回答：allkeys 面向全部 key，volatile 只淘汰带 TTL 的。纯缓存用 allkeys-lru 或 allkeys-lfu；不能丢数据的 key 要么不设 maxmemory 淘汰、要么用 noeviction 并保证内存充足。

Redis 内存异常增长如何排查？

内存持续增长可能导致 OOM、淘汰加剧、延迟升高。排查需从监控指标 → 大 key → 业务写入 → 碎片逐层定位。

排查步骤

  1. INFO memory：看 used_memory 趋势、mem_fragmentation_ratio、maxmemory 使用率。
  2. INFO clients：客户端输出缓冲区是否过大（慢消费者堆积）。
  3. 找大 key：redis-cli --bigkeys，或 RDB 分析工具（rdb-tools、redis-memory-analyzer）。
  4. SCAN + MEMORY USAGE key 抽样检查异常 key 类型与大小。
  5. 检查 List/Set/ZSet/Stream 是否只增不删；缓存 key 是否未设 TTL。
  6. 核对 AOF rewrite、主从复制 backlog 是否占用过多缓冲。
INFO memory
MEMORY USAGE mykey
redis-cli --bigkeys -i 0.1

# 碎片率过高时可尝试（Redis 4.0+，会短暂阻塞）
MEMORY PURGE
# 或重启 / 开启 activedefrag

常见原因

原因 处理
大 key / 热 key 膨胀拆分 key、压缩 value、设置合理 TTL
无过期缓存堆积统一 key 规范，强制 TTL
内存碎片active defrag、低峰重启、控制大 key 频繁更新
持久化/复制缓冲调 repl-backlog-size、避免过大 AOF

面试回答：先看 INFO memory 和碎片率，再用 bigkeys/SCAN 找大 key 和无 TTL key，结合业务看是否有结构只写不删；碎片高可 active defrag 或重启。

Redis 如何实现滑动窗口限流？

滑动窗口限流比固定窗口更平滑：在任意连续时间窗口内限制请求数，避免窗口边界处的流量突刺。

基于 ZSet 的实现

以时间戳为 score，member 为请求唯一 ID；每次请求前删除窗口外记录，统计剩余数量，未超限则写入并放行。

# key = rate:{userId}:{api}
# 窗口 60 秒，上限 100 次

now = 当前毫秒时间戳
windowStart = now - 60000

ZREMRANGEBYSCORE key 0 windowStart      # 清理过期
count = ZCARD key
if count < 100:
  ZADD key now requestId
  EXPIRE key 61
  放行
else:
  拒绝

生产注意点

要点 说明
原子性清理 + 计数 + 写入 + EXPIRE 用 Lua 脚本一次完成
member 唯一用 UUID 或 now:seq，避免同毫秒冲突
内存窗口内每条请求占一条 ZSet 记录，高 QPS 时注意 key 大小
替代方案令牌桶/漏桶（Redisson RRateLimiter）；固定窗口用 INCR + EXPIRE 更简单

面试回答：ZSet 以时间戳为 score，请求前删窗口外数据再 ZCARD 计数，Lua 保证原子性。比固定窗口更准，但内存随 QPS 增长；超高并发可用 Redisson 或近似算法（如 Redis Cell 模块）。

为什么用 Redis 而不是本地缓存？

本地缓存（Caffeine/Guava）访问快，但多实例间不共享，更新不一致，重启丢失。

Redis 提供跨进程、跨节点的共享缓存，适合分布式系统；常配合本地缓存做多级缓存。

Redis 除了缓存还能做什么？

  • 分布式锁、限流计数
  • 排行榜（ZSet）、延时任务（ZSet 或 Stream）
  • 轻量消息队列（List/Stream）
  • 会话存储、Bitmap 签到、HyperLogLog UV 统计

Redis String 适合哪些场景？

  • 缓存对象 JSON、计数器（INCR）
  • 分布式锁、限流令牌
  • Session、验证码、短链映射

Redis 事务为什么不支持回滚？

Redis 的 MULTI / EXEC 事务不是关系型数据库意义上的 ACID 事务，它只保证「一组命令打包、顺序执行、不被插队」，不支持运行时错误回滚。这是 Redis 为高性能和简单语义做出的设计取舍。

Redis 事务实际提供了什么

MULTI
SET key1 val1
INCR counter
EXEC
  • 批量执行：EXEC 前命令进入队列，EXEC 时一次性顺序执行。
  • 隔离性（有限）：EXEC 执行期间不会被其他客户端命令插入。
  • 不保证原子性回滚：某条命令执行失败，前面已执行的命令不会撤销。

为什么不支持回滚

原因 说明
设计定位Redis 是内存 KV，追求极致性能，不是通用 OLTP 数据库
实现成本回滚需要 undo log、版本链，与 Redis 单线程 + 简单数据结构理念冲突
使用场景多数场景是缓存/计数/锁，不需要复杂事务；开发者可用 Lua 保证逻辑原子性
错误类型简单Redis 命令失败多为类型错误（如对 string 做 LPUSH），属于编程错误，应在开发期发现

两类错误的不同行为

错误类型 示例 行为
入队错误语法错、命令不存在、参数个数不对整个事务不执行，EXEC 返回错误
运行时错误对 string 执行 LPUSH、对不存在 key 做特定操作出错命令失败，其余命令照常执行，不回滚
 
MULTI
SET key "hello"
LPUSH key "world"    # 运行时错误：key 是 string 不是 list
INCR counter
EXEC
# 结果：SET 和 INCR 已生效，LPUSH 报错，不会整体回滚

与 MySQL 事务对比

对比项 MySQL（InnoDB） Redis MULTI/EXEC
回滚支持 ROLLBACK不支持
隔离级别RC / RR 等仅 EXEC 期间不被插队
持久性redo log 保证取决于 AOF/RDB 配置
典型用途业务强一致事务批量命令、简单原子操作

需要原子性时怎么办

  • Lua 脚本：多条逻辑在一个脚本中原子执行，Redis 保证脚本执行期间不被打断（推荐）。
  • WATCH + MULTI + EXEC：乐观锁，被 watch 的 key 在 EXEC 前被改动则事务失败，需客户端重试。
  • 单条命令：SET NX、INCR、GETSET 等本身原子。
-- Lua 示例：扣减库存（原子）
EVAL "
  local stock = tonumber(redis.call('GET', KEYS[1]))
  if stock >= tonumber(ARGV[1]) then
    return redis.call('DECRBY', KEYS[1], ARGV[1])
  else
    return -1
  end
" 1 product:1001 1

面试回答：Redis 事务只保证 MULTI 到 EXEC 之间命令顺序执行、不被插队，不支持运行时回滚。原因是 Redis 定位高性能 KV，回滚需要 undo 机制与设计理念冲突，且多数场景用 Lua 脚本就能保证原子性。入队错误整个事务不执行；运行时错误只影响该条命令，前面已执行的不会撤销。复杂原子逻辑用 Lua，乐观并发用 WATCH。

Redis ZSet 底层结构是什么？

元素少时用 ziplist；元素多时使用跳表（按 score 排序）+ 哈希表（member → score）双结构，兼顾范围查询与 O(1) 单点查找。

Redis Big Key 和 Hot Key 分别是什么？

Big Key：value 过大或集合元素过多，导致阻塞、迁移慢。Hot Key：访问频率极高，造成单分片/单节点压力过大。

如何发现和治理 Big Key？

  • 发现：redis-cli --bigkeys、RDB 分析、内存采样
  • 治理：拆分 key、压缩、异步删除（UNLINK）、限制集合大小

如何发现和治理 Hot Key？

  • 发现：monitor 采样、hotkeys 命令、客户端上报
  • 治理：本地缓存、key 副本扩散、读写分离、限流

Redis Pipeline 是什么？

客户端批量发送多条命令再批量读取结果，减少 RTT，但不保证事务原子性。适合大量独立读写。

Redis Lua 脚本为什么能保证原子性？

Redis 单线程执行模型下，Lua 脚本执行期间不会插入其他命令，因此脚本内多条命令作为一个整体连续执行。

常用于分布式锁释放、限流、库存扣减。脚本应短小，避免阻塞主线程。

Redis 主从复制全量同步和增量同步是什么？

全量同步：从库首次连接或 offset 差距过大时，主库 RDB 快照 + 缓冲期间命令。增量同步：基于 replication backlog 传播 offset 之后的写命令。

Redis Cluster hash slot 是什么？

Cluster 将 key 空间分为 16384 个槽位，每个 master 负责一部分 slot。key 通过 CRC16 对 slot 取模路由，支持 slot 迁移实现扩缩容。

Redis 脑裂问题是什么？如何降低风险？

网络分区导致出现多个 master 同时写入，恢复后可能丢数据。治理：min-replicas-to-write、合理超时、Sentinel/Cluster 多数派选主。

Redis 连接池如何管理？

Redis 连接池本质上是：

提前创建一批 Redis TCP 连接，业务线程需要访问 Redis 时，从池里借一个连接，用完再归还，避免每次都新建/释放连接 

Redis 连接池主要是为了复用 TCP 连接，避免频繁创建和销毁连接。

应用启动后会维护一批 Redis 连接，请求访问 Redis 时从连接池借连接，执行完命令后归还连接。

连接池会通过参数控制最大连接数、空闲连接数和获取连接等待时间

如果连接池满了，新的请求会等待，超过 max-wait 就会报超时。

连接池打满需要注意：

连接泄漏、连接池配置过小、Redis 命令耗时过长，连接池大小后是否超过 Redis 服务端承载能力。

Redis 连接如何管理，避免多协程连接冲突？

Jedis 对象本身不能被多个线程同时共享使用。

所以在多线程环境下，不能全局只创建一个 Jedis 实例给所有线程用，而应该让每个线程从连接池里拿一个独立连接，用完归还。

看起来很省事：全局一个 Jedis 对象，大家都用它。

但是问题在于：多个线程可能同时操作同一个 Jedis 连接，线程之间互相读取对方Redis返回数据，导致混乱
