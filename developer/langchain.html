LangChain 常见结构有哪些？

LangChain 按职责分层，核心是把 Model、Prompt、数据、编排 拆成可复用组件，再用 Chain / Agent 串联。

结构 典型类 / API 作用
ModelChatOpenAI、OpenAIEmbeddings封装 LLM 调用与向量化
PromptChatPromptTemplate、FewShotPromptTemplate模板化拼装 System / Human 消息
Output ParserStrOutputParser、JsonOutputParser把模型输出解析为字符串或结构化对象
Document LoaderPyPDFLoader、WebBaseLoader加载原始文档
Text SplitterRecursiveCharacterTextSplitter切分 Chunk，控制检索粒度
VectorStoreChroma、FAISS、PGVector存储与检索向量
Retrieveras_retriever()、MultiQueryRetriever按 query 召回相关文档
Chain（LCEL）prompt | llm | parser用管道符组合 Runnable，线性编排
Tool@tool 装饰函数把业务能力暴露为可调用工具
Agentcreate_react_agent、AgentExecutor模型自主选工具、多步推理
MemoryChatMessageHistory、RunnableWithMessageHistory维护多轮对话上下文

现代 LangChain 推荐 LCEL（LangChain Expression Language）：各组件实现 Runnable 接口，支持 invoke / stream / batch 统一调用。

LangChain 标准构建流程是什么？

从 0 到 1 搭一个 LangChain 应用，通常按「模型 → 数据 → 提示词 → 链 → 调用」五步走；RAG 和 Agent 在此基础上加 Retriever 或 Tool。

基础流程（5 步）

  1. 选 Model：配置 LLM 和（如需 RAG）Embedding 模型。
  2. 准备数据：Loader 加载 → Splitter 切分 → Embedding → 写入 VectorStore。
  3. 定义 Prompt：写清角色、上下文占位符、输出格式要求。
  4. 组装 Chain：用 LCEL 管道串联各组件。
  5. 调用与观测：invoke() / stream()，挂 Callback 记录 Token 与延迟。

简单问答 Chain

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(model="gpt-4o")
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是企业知识库助手，简洁回答。"),
    ("human", "{question}"),
])
chain = prompt | llm | StrOutputParser()

answer = chain.invoke({"question": "年假有多少天？"})

RAG Chain 扩展

from langchain_core.runnables import RunnablePassthrough

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

rag_prompt = ChatPromptTemplate.from_messages([
    ("system", "仅根据以下资料回答，无依据则说不知道。\n\n{context}"),
    ("human", "{question}"),
])
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | rag_prompt
    | llm
    | StrOutputParser()
)

Agent 扩展

  1. 用 @tool 定义搜索、查库等业务函数。
  2. create_react_agent(llm, tools, prompt) 创建 Agent。
  3. AgentExecutor(agent, tools) 执行多轮 think-act 循环。

流程固定、无分支时用 Chain；需要自主选工具、多步推理时升级为 Agent；有循环、人工审批、状态持久化时再考虑 LangGraph。

LangChain 和 LangGraph 区别是什么？

LangChain 是 LLM 应用组件库；LangGraph 是有状态图工作流编排框架。

对比项 LangChain LangGraph
定位组件与链式编排有状态图工作流与 Agent 编排
流程形态线性链路或简单 Agent分支、循环、路由、状态机
适用场景RAG 原型、工具调用复杂 Agent、审批流、可恢复任务

简单 RAG 用 LangChain 足够；需要循环、人工介入、状态持久化时选 LangGraph。

LangGraph 的 Node 怎么设计？

Node 是工作流中的可执行步骤，应职责单一、输入输出清晰

设计原则

  • 按能力拆分：rewrite_query、retrieve、rerank、generate、call_tool、human_review。
  • 避免一个 Node 做太多事，便于观测、重试和替换。
  • 只返回 State 增量更新，不覆盖整个状态。
  • 条件分支、循环通过 conditional edge 表达。
def retrieve_node(state):
    docs = retriever.search(state["rewritten_query"])
    return {"retrieved_docs": docs}

LangGraph 中 Edge 怎么设计？

Edge 决定 Node 之间的流转方向，分固定边和条件边两类。

固定边（add_edge）

无条件跳转，适合线性流水线。

builder.add_edge("retrieve", "rerank")
builder.add_edge("rerank", "generate")
builder.add_edge("generate", END)

条件边（add_conditional_edges）

路由函数读取 State，返回下一个 Node 名称，实现分支、循环、重试。

def route_after_generate(state):
    if state.get("need_tool"):
        return "call_tool"
    if state.get("retry_count", 0) > 3:
        return END
    return "human_review"

builder.add_conditional_edges(
    "generate",
    route_after_generate,
    {
        "call_tool": "call_tool",
        "human_review": "human_review",
        END: END,
    },
)

常见 Edge 模式

场景 Edge 设计
线性 RAGrewrite → retrieve → rerank → generate → END
Tool Calling 循环agent → tools → agent（条件边检测是否继续调工具）
重试 / 降级generate → 失败则 retry_node，超限则 fallback → END
人工审核interrupt_before + 条件边路由到 human_review
Agentic RAGretrieve → grade → 质量不足则 rewrite 再检索（形成环）

设计原则

  • 路由函数只读 State、无副作用，分支逻辑集中、可单测。
  • 返回值必须在映射表中存在，避免运行时找不到 Node。
  • 循环边务必设退出条件（步数上限、置信度阈值），防止死循环。
  • END 是内置终止节点，任何可结束的路径都应能到达 END。

LangGraph 中 State 存储哪些字段？

State 是跨节点共享的状态对象，存储执行过程中需读写的数据

字段类型 常见字段
用户输入question、session_id、user_id、tenant_id
对话上下文messages、history_summary、memory
检索过程rewritten_query、retrieved_docs、reranked_docs、citations
工具调用tool_calls、tool_results、tool_errors
控制字段route、step、retry_count、need_human_review
输出结果answer、confidence、error

State 不宜膨胀：大对象、原始全文、敏感数据只存引用 ID 或摘要。

LangGraph 中 Builder 怎么设计？

Builder 负责把 Node、Edge、State 组装成可执行的图，典型流程是：定义 State → 注册 Node → 连 Edge → compile()。

标准构建步骤

  1. 定义 State：规定图运行有哪些字段
  2. 创建图：绑定状态结构 schema
  3. 注册 Node：每个 Node 只做一件事，把业务步骤注册到图
  4. 连接 Edge：固定边 、条件边 ，定义节点执行顺序和分支条件
  5. 编译运行：编译成可执行Graph
from langgraph.graph import StateGraph, END
# 创建一个图构建器 builder
builder = StateGraph(AgentState)
# 注册 Node 节点
builder.add_node("rewrite", rewrite_node)
builder.add_node("retrieve", retrieve_node)
builder.add_node("generate", generate_node)
# 设置入口节点
builder.set_entry_point("rewrite")
# 连接 Edge 边，定义节点之间的执行顺序
builder.add_edge("rewrite", "retrieve")
builder.add_edge("retrieve", "generate")
builder.add_edge("generate", END)
# 编译图，生成真正可以执行的 graph
graph = builder.compile(checkpointer=memory)

设计要点

  • Builder 只做编排，业务逻辑放在 Node 函数里，便于单测。
  • 复杂路由用 conditional edge 抽成独立 route_fn，Builder 保持可读。
  • 子图（Subgraph）可封装 RAG、Tool 等模块，主 Builder 只负责拼装。
  • 编译后再注入运行时依赖（retriever、llm），Node 通过闭包或依赖注入获取。

Agent 如何避免无限循环？

防护机制

  • max_iterations：最多 N 轮 think-act 循环。
  • max_tool_calls：单次请求工具调用上限。
  • 重复检测：相同 action 连续出现则终止。
  • 总 Token / 时间预算：超限强制结束。
  • 终止条件：模型输出 FINAL_ANSWER 或达到目标状态。

LangGraph 用 conditional edge 检测 step > MAX 路由到 END 节点。

Agent 状态如何持久化和恢复？

Agent / LangGraph 执行过程中，State（状态）会在多个 Node 之间传递。持久化就是把每一步的 State 快照写入存储，以便断点续跑、多轮对话、HITL 暂停后恢复、进程崩溃不丢进度。

为什么需要持久化

场景 说明
多轮对话用户下次进来，Agent 能接着上一轮上下文继续
HITL 暂停图执行到审批节点暂停，等人工操作后再 resume
长任务检索 → 工具调用 → 生成多步链路，中途失败可从 checkpoint 恢复
可观测 / 审计回放每一步 State，排查 Agent 为何走错分支

LangGraph Checkpointer 机制

LangGraph 在 graph.compile(checkpointer=...) 时注入 Checkpointer，每执行完一个 Node 自动写入 checkpoint。通过 thread_id 标识一次会话线程，同一 thread 的多次 invoke 共享状态历史。

Checkpointer 存储 适用
MemorySaver内存本地开发、单进程测试
SqliteSaverSQLite 文件轻量单机部署
PostgresSaverPostgreSQL生产多实例、高可靠
RedisSaverRedis低延迟、带 TTL 的会话

持久化与恢复流程

  1. 编译时绑定 Checkpointer：graph = builder.compile(checkpointer=checkpointer)
  2. 调用时传 thread_id：config = {"configurable": {"thread_id": "session-123"}}
  3. 首次 invoke：从入口 Node 开始，每步 State 自动 checkpoint
  4. 再次 invoke：同一 thread_id 下，LangGraph 加载最新 checkpoint，从断点继续（或合并新输入）
  5. 指定历史版本：可通过 checkpoint_id 回滚到某个快照（调试 / 重试）

Python 示例

from langgraph.checkpoint.postgres import PostgresSaver
from langgraph.graph import StateGraph, END

# 1. 创建 Checkpointer（生产用 Postgres / Redis）
checkpointer = PostgresSaver.from_conn_string(DB_URL)
checkpointer.setup()

# 2. 编译图
graph = builder.compile(checkpointer=checkpointer)

# 3. 首次执行：创建 thread 并持久化
config = {"configurable": {"thread_id": "user-42-session-001"}}
result = graph.invoke({"question": "查一下订单 12345 状态"}, config)

# 4. 同 thread 续跑（多轮对话 / 恢复）
result = graph.invoke(
    {"question": "那退款进度呢？"},
    config,  # 同一 thread_id，自动加载历史 messages / tool_results
)

# 5. 查看历史 checkpoint（调试）
states = list(graph.get_state_history(config))

State 里通常持久化什么

  • messages：对话历史（多轮 Agent 必备）
  • tool_calls / tool_results：已执行的工具及返回
  • retrieved_docs、answer：RAG 中间结果与输出
  • step、retry_count：控制字段，防无限循环
  • need_human_review、human_decision：HITL 相关

生产注意点

  • thread_id 设计：建议 {tenant_id}:{user_id}:{session_id}，避免跨租户串数据。
  • TTL 清理：Redis / 定时任务清理过期 checkpoint，控制存储成本。
  • 敏感数据：State 中避免存明文密码、完整 PII；必要时加密或只存 ID。
  • 多实例部署：Checkpointer 必须走共享存储（Postgres/Redis），不能只用 MemorySaver。

面试回答：LangGraph 用 Checkpointer 在每个 Node 执行后自动持久化 State，通过 thread_id 标识会话。首次 invoke 创建 checkpoint，同 thread 再次 invoke 从断点恢复。生产用 Postgres/Redis，开发用 MemorySaver；注意 thread_id 租户隔离、TTL 和敏感数据脱敏。

Agent 如何做人工确认 HITL？

HITL（Human-in-the-Loop，人在回路）指 Agent 在关键动作执行前暂停，把待确认内容展示给人工，人工批准 / 拒绝 / 修改后，Agent 再从断点继续执行。LangGraph 通过 interrupt_before / interrupt_after + Checkpointer 原生支持。

典型场景

场景 暂停时机 人工操作
发邮件 / 发消息调用 send_email 工具前确认收件人、正文无误后批准
金融交易 / 退款执行支付 API 前核对金额、账户，批准或拒绝
权限变更修改角色 / 授权前安全员审批
低置信度 RAG 答案生成最终回复前客服审核后放行或改写
危险工具调用删数据、写库、部署前必须人工确认

LangGraph HITL 实现原理

  1. 标记中断点：编译时指定 interrupt_before=["human_review"]，图执行到该 Node 前自动暂停。
  2. 持久化 State：暂停时 Checkpointer 已保存当前 State（含待审批的 tool_call、draft_answer 等）。
  3. 通知人工：后端把待审内容推送到审批 UI / 企微 / 邮件，状态设为 pending_review。
  4. 人工决策：批准 → 写入 human_decision=approved；拒绝 → rejected；修改 → 更新 State 中 draft 字段。
  5. resume 恢复：用同一 thread_id 调用 graph.invoke(None, config) 或更新 State 后 resume，图从中断点继续。
from langgraph.checkpoint.memory import MemorySaver

# 1. 编译时声明中断点 + Checkpointer
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["human_review"],  # 进入 human_review 节点前暂停
)

config = {"configurable": {"thread_id": "order-123"}}

# 2. 首次执行：Agent 规划到发邮件，图在 human_review 前暂停
result = graph.invoke({"question": "给客户发退款通知"}, config)
# 此时 result 含待审 draft_email，图状态为 interrupted

# 3. 读取当前 State，展示给人工
snapshot = graph.get_state(config)
draft = snapshot.values.get("draft_email")

# 4. 人工批准后，更新 State 并 resume
graph.update_state(config, {"human_decision": "approved"})
result = graph.invoke(None, config)  # 从断点继续，执行 send_email

条件路由配合 HITL

human_review Node 后的 conditional edge 根据 human_decision 分支：

  • approved → 执行工具 Node（send_email / refund）
  • rejected → 返回拒答或重新生成
  • timeout → 超时未审，自动取消或转人工队列

工程落地要点

  • 审批 UI：展示 Agent 意图、工具参数、草稿答案，支持一键批准 / 编辑后批准。
  • 超时策略：pending 超过 N 分钟自动 reject 或 escalate，避免任务永久挂起。
  • 审计日志：记录谁、何时、批准/拒绝了什么，满足合规要求。
  • 权限控制：只有特定角色能 approve 高危操作（财务、安全员）。
  • 与 Checkpointer 绑定：HITL 必须配合持久化，否则服务重启后中断状态丢失。

面试回答：HITL 是在危险或低置信步骤前暂停 Agent，等人工确认再继续。LangGraph 用 interrupt_before 标记中断点，Checkpointer 保存 State；人工 approve/reject 后更新 State 并 invoke(None, config) resume。生产要配审批 UI、超时、审计和权限，且必须走共享 Checkpointer 防丢状态。
