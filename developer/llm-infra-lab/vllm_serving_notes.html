vLLM Serving 实验笔记

用 Qwen2.5-1.5B-Instruct 在 WSL2 + RTX 4060 8GB 环境下完成 OpenAI-compatible API 服务、并发 benchmark、Continuous Batching 和 KV Cache 观察。

实验目标

本实验不是为了证明单请求一定比 Transformers 快，而是为了验证 vLLM 在服务化推理里的核心价值：

  • OpenAI-compatible API：让本地模型以标准 HTTP 接口对外提供服务。
  • Continuous Batching：多个请求同时到达时动态合批，提高 GPU 利用率。
  • PagedAttention / KV Cache 管理：把多请求的 KV cache 组织成更容易复用和调度的块。
  • Throughput Benchmark：观察并发上升时 req/s 和 tokens/s 如何变化。
这类实验的重点是理解“模型推理如何变成服务”，也就是从单次 generate() 走向多用户并发请求调度。

vLLM 要解决什么问题

从单机脚本到在线服务，问题不再只是“模型能不能生成”，而是“多个用户同时请求时，GPU 能不能一直高效工作”。vLLM 的核心优化可以用三个问题串起来。

问题 1：普通 generate 为什么并发吞吐低？

每个请求长度不同，prefill 和 decode 交替发生。如果一个请求一个请求处理，GPU 会频繁等待，batch 很小，吞吐上不去。

问题 2：静态 batch 为什么不够？

静态 batch 要等一组请求一起开始、一起结束。短请求完成后仍被长请求拖住，无法及时插入新请求。

问题 3：KV Cache 为什么难管理？

每个请求生成长度不同，KV cache 会不断增长。如果按连续大块分配，显存容易碎片化，多个请求之间也不好复用空闲空间。

问题 4：vLLM 的核心回答是什么？

用 continuous batching 动态调度请求，用 PagedAttention 把 KV cache 拆成固定大小 block，再通过 block table 管理每个请求的上下文。

图解：普通推理 vs Continuous Batching

普通推理更像串行处理：请求 A 做完才轮到 B，GPU 中间会出现空洞。vLLM 则在 decode 阶段不断把新请求插进当前 batch，让 GPU 更接近持续工作状态。

图解：PagedAttention 与 KV Cache

Transformer decode 时，每生成一个 token 都要保存该 token 的 Key/Value。请求越多、上下文越长，KV cache 占用越大。PagedAttention 借鉴虚拟内存分页思想，把 KV cache 拆成固定大小 block，用 block table 记录每个请求的逻辑 token 对应哪些物理 block。

Request A Request B Request C 空闲或碎片

图解：一次请求在 vLLM 中怎么走

当客户端调用 /v1/chat/completions 时，请求会经过 API Server、Scheduler、Engine、GPU Worker。Prefill 负责处理 prompt，Decode 逐 token 生成，KV cache 在两阶段之间保存上下文状态。

启动参数说明

参数或环境变量 作用
--gpu-memory-utilization 0.8 让 vLLM 以约 80% GPU 显存为目标规划权重、KV cache 和运行时内存。
--max-model-len 4096 限制最大上下文长度，降低 KV cache 预留压力，更适合 8GB 显存。
--hf-endpoint https://hf-mirror.com 在 Hugging Face 主站不可达时使用镜像下载模型配置和权重。
HF_HUB_DISABLE_XET=1 禁用 Xet/CAS 下载路径，避免 401 Unauthorized，改走普通权重下载。
--enforce-eager 关闭 Torch compile 和 CUDA graph，牺牲部分性能换取 WSL 环境稳定性。
VLLM_USE_FLASHINFER_SAMPLER=0 禁用 FlashInfer sampler，避免 WSL 下 JIT 编译 CUDA kernel 时的 toolkit header 版本冲突。

排错复盘

现象 原因 解决
vllm._C_stable_libtorch 找不到 在原生 Windows 环境运行 vLLM。 改用 WSL2 Ubuntu / Linux 环境。
Segmentation fault Python 3.14 太新，PyTorch/vLLM/CUDA 扩展不稳定。 使用 uv venv --python 3.12 创建 Python 3.12 环境。
GPU access blocked 或 cuda_available False WSL 没拿到 GPU，或驱动/WSL 状态不一致。 升级 NVIDIA 驱动到 CUDA 13.x 支持版本，执行 wsl --shutdown 并重启。
Network is unreachable WSL 无法访问 Hugging Face 主站。 使用 HF_ENDPOINT=https://hf-mirror.com。
Xet/CAS 401 Unauthorized 权重下载走了 Hugging Face Xet/CAS 路径。 设置 HF_HUB_DISABLE_XET=1。
Permission denied: nvcc PyTorch Inductor 查找 CUDA 编译器路径不稳定。 把虚拟环境里的 CUDA bin 加入 PATH，并使用 --enforce-eager。
FlashInfer JIT 编译失败 FlashInfer sampler 编译时 CUDA compiler 与 headers 版本不匹配。 设置 VLLM_USE_FLASHINFER_SAMPLER=0，走 PyTorch/native sampler。

Benchmark 结果

0.43 → 5.18 req/s，从并发 1 到并发 8
42.7 → 518.2 output tokens/s，从并发 1 到并发 8

结果分析

核心结论：并发从 1 提升到 8 时，请求吞吐从 0.427 req/s 提升到 5.182 req/s，输出 token 吞吐从 42.712 tokens/s 提升到 518.228 tokens/s。

这说明 vLLM 在多请求场景下能通过 continuous batching 更充分地利用 GPU。单请求时 batch 太小，GPU 利用率不高；多个请求同时进入后，vLLM 可以动态合批，让 GPU 持续工作。

并发 4 和并发 8 的平均延迟分别是 1.460s 和 1.495s，单请求等待时间没有明显变差，但系统总吞吐继续提升。这是服务化推理最重要的观察点。

注意：本实验不能简单表述为“vLLM 单请求更快”。它更适合说明“vLLM 在并发请求下吞吐更好”。

环境说明：为了在 WSL 学习环境中稳定跑通，使用了 --enforce-eager 并禁用了 FlashInfer sampler。生产 Linux 环境中如果启用 CUDA graph、Torch compile 和更优 kernel，性能可能更好。

服务化推理链路

Client
  |
  | OpenAI-compatible HTTP request
  v
vLLM API Server
  |
  | request scheduling + continuous batching
  v
vLLM Engine
  |
  | PagedAttention / KV cache blocks
  v
GPU
  |
  | decode tokens
  v
Streaming response

理解这条链路，比记住单个 benchmark 数字更重要。AI Infra 的核心不是只会调用模型，而是知道请求如何被调度、显存如何被规划、吞吐如何被测量。

面试表达

可以这样总结：

我在 WSL2 + RTX 4060 8GB 环境下使用 Qwen2.5-1.5B-Instruct 完成了 vLLM serving 实验。通过 OpenAI-compatible API 部署模型，并用并发请求 benchmark 对比不同 concurrency 下的 latency、TTFT、req/s 和 output tokens/s。实验中并发从 1 提升到 8 时，output token throughput 从 42.7 tokens/s 提升到 518.2 tokens/s，说明 vLLM 的 continuous batching 能在多用户场景下显著提高 GPU 利用率。同时，我也观察了 KV cache 显存规划和 PagedAttention 在服务化推理中的作用。
