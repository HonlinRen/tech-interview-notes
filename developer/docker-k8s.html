镜像 Image 和容器 Container 有什么区别？

镜像：静态模板，类似安装包。
容器：镜像运行后的实例，类似正在运行的进程。

镜像是静态的，容器是运行中的实例。一个镜像可以启动多个容器。

Dockerfile 是什么？常见指令有哪些？

FROM openjdk:17
WORKDIR /app
COPY target/demo.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
  • FROM：指定基础镜像
  • WORKDIR：工作目录
  • COPY：复制文件
  • EXPOSE：声明端口
  • CMD：启动命令

Docker 容器之间如何通信？

容器默认有独立网络命名空间。要让容器互相访问，需要理解网络模式、自定义网络和端口映射三种常见方式。

外部客户端
    │  localhost:8080（端口映射）
    ▼
宿主机 ── docker0 网桥 ──┬── 容器 A（172.18.0.2）
                        └── 容器 B（172.18.0.3）
                             同一自定义网络内可用容器名 / 服务名互访

四种网络模式

模式 特点 典型场景
bridge（默认）容器接入虚拟网桥，彼此隔离；需自定义网络或端口映射才能互访单机多容器，最常用
host容器直接使用宿主机网络栈，无独立 IP追求极致网络性能、端口不冲突时
none无网络，完全隔离安全沙箱、离线计算
container共享另一个容器的网络命名空间Sidecar、网络代理伴生容器

容器间互访的三种做法

  1. 自定义 bridge 网络（推荐）：同一网络内容器可通过容器名 DNS 互访，无需记 IP。 
    docker network create app-net
docker run -d --name redis --network app-net redis:7
docker run -d --name api  --network app-net my-api
# api 容器内直接访问 redis:6379
  2. 端口映射（-p）：把容器端口暴露到宿主机，适合容器 ↔ 宿主机 / 外部访问，不是容器间首选。 
    docker run -d -p 8080:8080 --name api my-api
curl http://localhost:8080/health
  3. Compose 服务名：同一 docker-compose.yml 下的服务自动加入同一网络，用服务名当主机名。 
    # compose 中 api 访问 mysql:3306、redis:6379
DATABASE_URL=mysql://root:pwd@mysql:3306/demo

面试要点

  • 默认 bridge 下，不同 docker run 启动的容器不能直接用容器名解析，需加入同一 user-defined network。
  • 容器访问宿主机服务：Linux 可用 host.docker.internal（Docker Desktop 内置）或宿主机网关 IP。
  • 生产多机通信靠 K8s Service / Overlay 网络，单机 Compose 靠内置 DNS 即可。

Docker 数据持久化怎么做？

容器文件系统是可写层 + 联合挂载，删除容器后默认可写层数据会丢失。有状态服务（MySQL、Redis、ES）必须把数据存到容器外部。

容器（临时可写层）  ← 删容器即丢
    │
    ├── Volume（Docker 管理，推荐）
    ├── Bind Mount（挂载宿主机目录）
    └── tmpfs（内存，重启即丢）

三种挂载方式对比

方式 存储位置 优点 注意
Volume/var/lib/docker/volumes/Docker 统一管理、易备份迁移、跨平台生产数据库首选
Bind Mount宿主机任意路径开发时直接改本地文件、所见即所得路径依赖宿主机、权限问题多
tmpfs内存极快、不落盘仅适合临时缓存、密钥

常用命令与示例

# 命名 Volume（推荐）
docker volume create mysql-data
docker run -d --name mysql \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  mysql:8

# Bind Mount（开发常用）
docker run -d -v /data/mysql:/var/lib/mysql mysql:8

# 查看 Volume
docker volume ls
docker volume inspect mysql-data

Compose 中声明持久化

services:
  mysql:
    image: mysql:8
    volumes:
      - mysql-data:/var/lib/mysql   # 命名卷
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro  # 只读绑定
volumes:
  mysql-data:   # 顶层声明，Compose 自动创建

最佳实践

  • 生产用命名 Volume，不要把数据库文件直接 bind 到随机目录。
  • 配置文件可用 bind mount 或 ConfigMap（K8s），数据与配置分离。
  • 定期备份：docker run --rm -v mysql-data:/data -v $(pwd):/backup alpine tar czf /backup/mysql.tgz /data
  • 注意文件权限：MySQL 容器内通常以 mysql 用户写数据，宿主机目录权限不对会导致启动失败。

一句话：无状态应用容器可随时重建；有状态服务的数据必须挂在 Volume / 外部存储，与容器生命周期解耦。

Docker Compose 是什么？

Docker Compose 是 Docker 官方的多容器编排工具，用一份 docker-compose.yml 描述整套应用（Web + DB + 缓存 + 网络 + 卷），一条命令拉起/销毁整个环境。

docker-compose.yml（声明式配置）
    │
    ├── services：各容器服务定义
    ├── networks：自定义网络
    ├── volumes：持久化卷
    └── 一条命令：docker compose up -d

解决什么问题？

对比 docker run Docker Compose
容器数量逐个手动启动，命令冗长多服务一键编排
网络需手动创建并 --network 关联自动建网，服务名即 DNS
依赖顺序自行控制启动顺序depends_on 声明依赖
配置管理散落在 shell 历史里YAML 版本化、团队共享
适用场景临时单容器测试本地开发、集成测试、小规模部署

典型 compose 文件结构

services:
  api:
    build: ./api
    ports:
      - "8080:8080"
    environment:
      - REDIS_URL=redis://redis:6379
    depends_on:
      - redis
      - mysql
  redis:
    image: redis:7
  mysql:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: secret
    volumes:
      - mysql-data:/var/lib/mysql
volumes:
  mysql-data:

常用命令

docker compose up -d          # 后台启动全部服务
docker compose ps             # 查看状态
docker compose logs -f api      # 跟踪某服务日志
docker compose exec api sh    # 进入容器
docker compose down           # 停止并删除容器（-v 同时删卷）
docker compose up -d --build    # 重新构建镜像后启动

与 K8s 的关系

  • Compose 适合单机、开发测试；生产多节点、弹性扩缩、滚动发布用 K8s。
  • Compose 的 service 类似 K8s 的 Deployment + Service 简化版。
  • 可用 kompose convert 将 compose 文件转为 K8s YAML（仅作起点，生产需再调优）。

面试回答：Compose 用 YAML 声明多容器应用拓扑，自动处理网络互联和服务发现，让本地环境与团队环境一键复现。

Docker 容器如何查看日志和排查问题？

Docker 镜像怎么优化？

  • 使用更小基础镜像（alpine、distroless）
  • 多阶段构建，只保留运行产物
  • .dockerignore 排除无关文件
  • 合并 RUN 层、清理缓存

Kubernetes 是什么？解决了什么问题？

K8s 是容器编排平台，管理大规模容器的部署、扩缩容、服务发现、故障自愈、滚动发布和配置管理。

Docker 解决单机容器运行，K8s 解决多机器、多服务的编排治理。

Pod 是什么？

Pod 是 K8s 最小调度单位，通常一个 Pod 运行一个业务容器。Pod 内容器共享网络和存储卷。

Pod 和 Container 有什么区别？

Container 是运行应用的进程隔离环境；Pod 是 K8s 对容器的封装和调度单位，K8s 调度 Pod 而非直接调度 Container。

Pod 是 K8S 最小调度单位，一个 Pod 里可以有一个或多个容器，共享网络和存储

Deployment 是什么？

Deployment 管理无状态应用的 Pod 副本数、滚动更新和回滚，是微服务最常用的部署对象。

spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: demo-app
          image: demo-app:1.0

Service 是什么？为什么需要 Service？

Pod IP 不稳定，Service 为一组 Pod 提供稳定访问入口和负载均衡，通过 selector 关联 Pod，集群内 DNS 访问。

ConfigMap 和 Secret 有什么区别？

ConfigMap 存放普通配置；Secret 存放密码、Token、证书等敏感信息。两者都支持环境变量或 Volume 挂载注入容器。

K8s 如何做滚动发布和回滚？

kubectl rollout status deployment/demo-app
kubectl rollout undo deployment/demo-app

Deployment 逐步创建新 Pod、下线旧 Pod，异常时可快速回滚到上一版本。

Pod 一直启动失败，你会怎么排查？

kubectl get pods
kubectl describe pod pod-name
kubectl logs pod-name
kubectl logs pod-name --previous

常见原因：ImagePullBackOff、CrashLoopBackOff、配置缺失、健康检查失败、资源不足、应用自身异常。

Slurm和K8S区别是什么？

Slurm 是 HPC （HPC = High Performance Computing，高性能计算）和 AI 训练场景常用的资源调度系统，主要负责把批处理任务调度到 CPU/GPU 节点上运行。

它和 K8S 都有资源调度能力，但 Slurm 更偏离线计算和训练任务，K8S 更偏容器编排、微服务和在线服务。

比如大模型训练、科研计算更常用 Slurm；RAG API、Agent 服务、推理服务更常部署在 K8S 上。

Slurm：调度计算任务，跑完结束K8S：编排在线服务，长期运行
