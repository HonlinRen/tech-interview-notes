14. MySQL 四种隔离级别

事务隔离级别决定并发事务之间能「看到」彼此多少数据。级别越高，一致性越强，但并发性能通常越低。

隔离级别 脏读 不可重复读 幻读
READ UNCOMMITTED可能可能可能
READ COMMITTED不会可能可能
REPEATABLE READ不会不会InnoDB 下大多可避免
SERIALIZABLE不会不会不会

READ UNCOMMITTED

读未提交。一个事务可以读到另一个事务还没提交的数据，可能产生脏读。生产环境极少使用。

READ COMMITTED

读已提交。只能读到其他事务已经提交的数据，可能产生不可重复读。Oracle 默认级别。

REPEATABLE READ

可重复读。同一个事务中多次读取同一条数据，结果一致。这是 MySQL InnoDB 默认隔离级别，配合 MVCC 和 Next-Key Lock 在多数场景下能避免幻读。

SERIALIZABLE

串行化。隔离级别最高，读操作也会加锁，相当于事务串行执行，安全性最高，但并发性能最低。

三种读异常

异常 含义
脏读读到其他事务未提交的数据，对方回滚后当前读到的数据无效
不可重复读同一事务内多次读同一行，结果不一致，通常因中间有其他事务修改并提交
幻读同一事务内多次按相同条件查询，结果集行数不一致，通常因中间有插入或删除

面试补充：InnoDB 是 MySQL 默认存储引擎，支持事务、行级锁、外键和崩溃恢复；隔离级别讨论必须结合 MVCC 与锁机制一起回答。

15. select for update 使用场景

SELECT ... FOR UPDATE 用于在事务中查询数据并对查询到的行加排他锁（X 锁），阻止其他事务同时修改这些行。

BEGIN;

SELECT * FROM account WHERE id = 1 FOR UPDATE;

UPDATE account SET balance = balance - 100 WHERE id = 1;

COMMIT;

适用场景

  • 扣库存、扣余额、订单状态流转。
  • 抢券、分布式任务领取、防止并发重复处理。
BEGIN;

SELECT stock FROM product WHERE id = 100 FOR UPDATE;

UPDATE product
SET stock = stock - 1
WHERE id = 100 AND stock > 0;

COMMIT;

注意点

  1. 必须在事务中使用，否则锁会立即释放。
  2. 查询条件最好命中索引，否则可能锁表或锁范围扩大。
  3. 只对 InnoDB 等支持事务和行锁的引擎有效。
  4. 避免长事务，锁持有时间越长，越容易造成阻塞甚至死锁。

FOR UPDATE 属于当前读，会读取最新版本并加锁，与普通快照读行为不同。

36. MySQL 索引为什么能加快查询？

InnoDB 常用 B+Tree 索引。索引按 key 有序组织，查询时通过树形结构快速定位，减少全表扫描，把查找范围缩小到少量数据页。

核心原理

机制 说明
有序结构B+Tree 叶子节点按索引列有序，支持等值和范围查找
减少 IO树高通常 3～4 层，百万级数据只需几次磁盘页读取
缩小扫描范围从 O(n) 全表扫描降为 O(log n) 索引定位
 
无索引：逐行扫描整张表
有索引：根节点 -> 中间节点 -> 叶子节点 -> 命中目标行

索引不是越多越好。索引会占用磁盘空间，并增加插入、更新、删除时维护 B+Tree 的成本；写多读少的表应谨慎建索引。

37. 聚簇索引和非聚簇索引区别？

InnoDB 中，聚簇索引的叶子节点保存整行数据；非聚簇索引（二级索引）的叶子节点保存索引列值和对应的主键值。

对比

类型 叶子节点内容 数量
聚簇索引整行数据每张表只有一个（通常是主键）
二级索引索引列 + 主键值可建多个

回表

通过二级索引查到主键后，再回到聚簇索引查整行数据，这个过程叫回表。回表次数越多，查询越慢。

-- 二级索引 idx_name 查到 id=100
-- 再回聚簇索引查 id=100 的整行
SELECT * FROM user WHERE name = 'Tom';

覆盖索引可以避免回表；没有主键时 InnoDB 会选唯一非空索引，否则自动生成 hidden row_id 作为聚簇索引。

38. 覆盖索引是什么？

覆盖索引指查询需要的列都能从索引中直接拿到，不需要回表查聚簇索引。Extra 中通常出现 Using index。

CREATE INDEX idx_user_name_age ON user(name, age);

SELECT name, age
FROM user
WHERE name = 'Tom';

这个查询只需要 name 和 age，联合索引已包含全部列，可直接从索引返回结果。

收益与注意

收益 注意
减少回表 IO索引列不宜过多，否则写入成本上升
减少随机 IOSELECT * 很难形成覆盖索引
优化 COUNT、分页等需结合最左前缀设计联合索引

39. 最左前缀原则是什么？

联合索引按照从左到右的列顺序组织。查询条件要从索引最左列开始连续命中，才能充分利用索引。

CREATE INDEX idx_a_b_c ON t(a, b, c);

-- 能用到索引
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

-- 通常无法充分使用该联合索引
WHERE b = 2 AND c = 3
WHERE c = 3

范围查询的影响

遇到范围查询（>、<、BETWEEN、LIKE 'xx%'）后，范围列右侧的列通常难以继续用于精确定位，但仍可能用于索引下推过滤。

条件 索引使用情况
WHERE a = 1 AND b > 2 AND c = 3a、b 用于定位，c 可能仅做 ICP 过滤
WHERE a = 1 ORDER BY b可能利用索引有序性避免 filesort

40. MVCC 是什么？

MVCC（Multi-Version Concurrency Control，多版本并发控制）通过保留数据的多个历史版本，让读操作读取一致性快照，写操作修改最新版本，实现读写并发。

InnoDB 实现要点

组件 作用
隐藏列 DB_TRX_ID记录最后修改该行的事务 ID
隐藏列 DB_ROLL_PTR指向 undo log 中的旧版本
undo log 版本链串联同一行的多个历史版本
Read View决定当前事务能看到哪些版本
 
快照读：普通 SELECT，读 Read View 可见的历史版本，不加锁
当前读：SELECT FOR UPDATE / UPDATE / DELETE，读最新版本并可能加锁

RC 级别每次 SELECT 生成新 Read View；RR 级别事务首次 SELECT 生成 Read View 并复用，因此 RR 下同一事务多次读结果一致。

41. MySQL 慢查询怎么排查？

慢查询排查的核心是：先定位慢 SQL，再看执行计划，最后针对性优化。

排查步骤

  1. 开启慢查询日志，找出耗时 SQL。
  2. 使用 EXPLAIN 查看访问类型、索引、扫描行数。
  3. 检查是否全表扫描、索引失效、回表过多、排序/临时表过大。
  4. 优化索引、改写 SQL、减少返回列、优化分页。
  5. 结合业务判断是否需要缓存、读写分离或归档历史数据。
-- 开启慢查询（示例）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;

EXPLAIN SELECT id, name FROM user WHERE status = 1;

重点观察字段

字段 关注点
type访问方式，尽量避免 ALL 全表扫描
key实际使用的索引
rows优化器估算扫描行数
Extra是否出现 filesort、temporary、Using index

63. 当前读和快照读有什么区别？

这是理解 InnoDB 并发控制的关键概念：普通读走 MVCC 快照，写相关读走最新数据并加锁。

类型 读取内容 是否加锁 典型语句
快照读Read View 可见的历史版本不加锁普通 SELECT
当前读最新已提交/当前事务修改的版本可能加锁SELECT ... FOR UPDATE、UPDATE、DELETE

为什么需要两种读？

  • 快照读强调读写并发，读不阻塞写、写不阻塞普通读。
  • 当前读强调读取最新数据并参与并发控制，更新前必须看到最新状态。
事务 A：SELECT age FROM user WHERE id=1;        -- 快照读，可能读到旧值
事务 B：UPDATE user SET age=30 WHERE id=1;       -- 当前读 + 加锁
事务 A：SELECT age FROM user WHERE id=1 FOR UPDATE; -- 当前读，读到 30 并加锁

64. Next-Key Lock 是什么？

Next-Key Lock 是 InnoDB 在 RR 隔离级别下使用的记录锁 + 间隙锁（Gap Lock）组合，锁住索引记录及其前面的间隙，用于避免并发插入导致的幻读。

三种锁

锁类型 锁住范围
Record Lock单条索引记录
Gap Lock两条索引记录之间的间隙（不含记录本身）
Next-Key Lock记录 + 该记录前的间隙
 
-- 假设 id 索引有 10、20、30
SELECT * FROM t WHERE id = 20 FOR UPDATE;
-- 可能锁住 (10, 20] 区间，阻止 id=15 的插入

Next-Key Lock 依赖索引。若查询未命中索引，可能退化为锁大量记录甚至锁表，严重影响并发。RC 级别通常不使用 Gap Lock。

65. InnoDB 为什么常作为默认存储引擎？

InnoDB 是 MySQL 5.5 起默认存储引擎，面向在线事务处理场景，兼顾一致性、并发与可靠性。

能力 InnoDB MyISAM（对比）
事务支持 ACID不支持
锁粒度行级锁表级锁
崩溃恢复redo log 保证易损坏，恢复能力弱
外键支持不支持
MVCC支持不支持

InnoDB 适合绝大多数需要数据一致性和高并发读写的业务系统；MyISAM 仅适合只读或低并发、对事务无要求的少数场景。

219. SQL 语句在 MySQL 中的执行流程是什么？

一条 SQL 在 MySQL 中会先经过 Server 层处理，再调用存储引擎层读写数据。整体链路可以概括为：连接 → 解析 → 优化 → 执行 → 存储引擎。

客户端
-> 连接器（权限、连接管理）
-> 解析器（词法、语法分析）
-> 优化器（选索引、生成执行计划）
-> 执行器（调用存储引擎接口）
-> 存储引擎（InnoDB 读写数据页）

各阶段职责

阶段 职责
连接器身份验证、权限校验、连接管理、超时断开
解析器词法分析、语法分析，生成语法树
优化器选择索引、确定 JOIN 顺序、生成执行计划
执行器检查权限后调用存储引擎 API 执行 SQL
存储引擎负责数据的物理读写，如 InnoDB 的 B+Tree、Buffer Pool

注意：MySQL 8.0 已移除查询缓存；优化器只负责选路，真正的数据存取发生在存储引擎层。

220. MySQL Server 层和存储引擎层如何分工？

MySQL 采用插件式存储引擎架构：Server 层负责 SQL 处理与通用能力，存储引擎层负责数据的物理组织与读写。

Server 层

  • 连接器、解析器、优化器、执行器。
  • 跨引擎能力：存储过程、视图、触发器、权限、binlog（Server 层日志）。
  • 与客户端通信、字符集转换、SQL 模式校验。

存储引擎层

  • InnoDB：B+Tree 索引、Buffer Pool、redo/undo log、行锁、MVCC。
  • MyISAM：堆表 + 索引文件，无事务。
  • Memory：数据存内存，重启丢失。
能力 所在层
SQL 解析与优化Server 层
binlogServer 层
redo log / undo logInnoDB 引擎层
索引结构与数据页存储引擎层

执行器通过统一接口调用引擎，因此同一套 Server 可以对接不同存储引擎，但实际生产几乎都用 InnoDB。

221. MySQL 查询缓存为什么被移除？

查询缓存（Query Cache）在 MySQL 8.0 中被彻底移除。它曾尝试缓存 SELECT 结果，但实践中弊大于利。

失效原因

问题 说明
命中率低OLTP 场景 SQL 模板多、参数变化大，缓存很难命中
失效成本高表有任何写操作，该表相关缓存全部失效
锁竞争高并发下访问 Query Cache 需要全局锁，反而拖慢性能
维护复杂内存管理、碎片、一致性边界难以做好

现代替代方案：应用层缓存（Redis）、InnoDB Buffer Pool 缓存数据页、合理的索引与 SQL 优化，而不是缓存 SQL 结果本身。

222. MySQL 自增主键一定连续吗？

不一定。 AUTO_INCREMENT 保证递增趋势，但不保证严格连续、无空洞。

出现间隙的常见原因

  • 事务回滚：已分配的自增值不会回收。
  • 批量插入：InnoDB 可能一次预分配多个 id 以提升性能。
  • 手动指定 id 或 INSERT ... ON DUPLICATE KEY UPDATE。
  • 主从切换、重启后 auto_increment_offset 等配置影响。
  • 删除数据后 id 不会复用。
BEGIN;
INSERT INTO t VALUES (NULL, 'a');  -- 假设分配到 id=10
ROLLBACK;
INSERT INTO t VALUES (NULL, 'b');  -- 下一条可能是 id=11，10 不会被复用

自增主键的价值在于单调递增、插入性能好、B+Tree 页分裂少，而不是业务编号连续性。需要连续业务号应单独设计发号器或序列表。

223. MySQL 隐式类型转换为什么可能导致索引失效？

当 WHERE 条件两侧类型不一致时，MySQL 可能对列做隐式类型转换，导致对索引列使用函数，从而无法走索引。

-- phone 是 VARCHAR，建了索引
SELECT * FROM user WHERE phone = 13800138000;
-- 数字常量会触发 phone 列转换为数字，索引失效

-- 正确写法
SELECT * FROM user WHERE phone = '13800138000';

常见场景

场景 结果
字符串列与数字比较对列做 CAST，索引失效
对索引列使用函数WHERE DATE(create_time) = '2024-01-01' 失效
字符集/排序规则不一致JOIN 或比较时可能隐式转换

排查方法：看 EXPLAIN 的 type=ALL，或 SHOW WARNINGS 中的 convert 提示。原则是让常量适配列类型，而不是让列适配常量。

224. MySQL 字符集和排序规则是什么？

字符集（Character Set）决定字符如何编码存储；排序规则（Collation）决定字符比较、排序和索引顺序。

层级

server 默认字符集
  -> database 字符集
    -> table 字符集
      -> column 字符集
概念 示例 说明
字符集utf8mb4支持完整 Unicode，包括 emoji
排序规则utf8mb4_general_ci_ci 表示不区分大小写
排序规则utf8mb4_bin按二进制比较，区分大小写
 
SHOW VARIABLES LIKE 'character%';
SHOW FULL COLUMNS FROM user;

不同字符集或排序规则的列 JOIN/比较时会隐式转换，可能导致索引失效或结果异常。建表时应统一使用 utf8mb4。

225. utf8 和 utf8mb4 有什么区别？

MySQL 中的 utf8 实际是阉割版 UTF-8，最多 3 字节，不支持 emoji 和部分生僻字；utf8mb4 是完整 4 字节 UTF-8。

对比项 utf8 utf8mb4
最大字节3 字节4 字节
emoji不支持支持
存储空间略小略大（按最大字节计）
推荐程度已过时生产环境首选
 
CREATE TABLE user (
  id BIGINT PRIMARY KEY,
  nickname VARCHAR(64) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci
) DEFAULT CHARSET=utf8mb4;

新项目应直接使用 utf8mb4。从 utf8 迁移时注意索引长度限制：VARCHAR(255) 在 utf8mb4 下可能超过 767 字节索引上限，需缩短字段或启用前缀索引/大索引页。

226. MySQL 时间类型 DATETIME 和 TIMESTAMP 有什么区别？

对比项 DATETIME TIMESTAMP

时区存什么读什么，不受时区影响存储 UTC，显示时按 session 时区转换
存储空间8 字节4 字节

CREATE TABLE log (
  id BIGINT PRIMARY KEY,

业务时间戳（如订单创建时间）推荐 DATETIME 或 BIGINT 毫秒时间戳，避免 2038 年问题。跨时区场景若用 TIMESTAMP，需确保连接 session 时区配置一致。

227. SQL 优化有哪些常见手段？

SQL 优化应基于执行计划和数据分布，而非盲目加索引。

常见手段

方向 做法
索引优化合理建联合索引、覆盖索引，避免冗余索引
SQL 改写避免 SELECT *、子查询改 JOIN、拆分大 SQL
分页优化延迟关联、游标分页代替 OFFSET 深分页
减少回表覆盖索引、只查必要列
批量操作批量 INSERT/UPDATE 减少 round-trip
架构层读写分离、缓存、归档冷数据
 
-- 避免
SELECT * FROM orders WHERE status = 1 ORDER BY create_time LIMIT 100000, 20;

-- 延迟关联
SELECT o.* FROM orders o
INNER JOIN (
  SELECT id FROM orders WHERE status = 1 ORDER BY create_time LIMIT 100000, 20
) t ON o.id = t.id;

228. 深分页为什么慢？如何优化？

LIMIT offset, size 中 offset 很大时，MySQL 仍需扫描并丢弃前 offset 行，成本随页码线性增长。

慢的原因

  • LIMIT 1000000, 20 需先读 1000020 行再丢弃前 1000000 行。
  • 若有 ORDER BY 且无法走索引，还会触发 filesort。

优化方案

方案 适用场景
游标/Seek 分页WHERE id > last_id LIMIT 20，无限滚动
延迟关联子查询先查 id，再 JOIN 取全列
覆盖索引排序列在索引中，减少回表
搜索引擎复杂筛选排序交给 ES 等
 
-- 游标分页（推荐）
SELECT * FROM orders
WHERE id > 1000000
ORDER BY id
LIMIT 20;

229. 数据冷热分离是什么？

冷热分离指按数据访问频率将热数据（近期、高频访问）与冷数据（历史、低频访问）分开存储，降低成本并提升热数据查询性能。

常见做法

方式 说明
分表归档按时间将历史订单迁到 archive 表或历史库
不同存储介质热数据 SSD，冷数据 HDD 或对象存储
只读从库历史查询走从库或 OLAP 引擎
TTL + 清理日志类数据定期删除或压缩
 
orders_2024  -- 热表，SSD，主库
orders_2023  -- 温表，从库可查
orders_hist  -- 冷数据，归档到对象存储 / ClickHouse

设计时需考虑：跨冷热数据查询如何路由、归档任务一致性、回查冷数据的延迟 acceptable 范围。

230. 读写分离会带来什么问题？

读写分离通过主库写、从库读来扩展读能力，但引入了主从延迟导致的一致性问题。

典型问题

问题 场景
主从延迟刚注册完读从库，查不到刚写入的数据
读己之写用户改密码后立即登录，从库还是旧密码
路由复杂度需中间件或框架决定走主还是从
从库故障读流量需 failover，可能全部压到主库

常见应对

  • 写后读主：关键操作后强制走主库。
  • 延迟检测：从库延迟超阈值时读主。
  • 会话粘滞：同一用户短时间走同一从库（不能根本解决延迟）。
  • 缓存：写时更新缓存，读优先走缓存。

231. 主从延迟如何解决？

主从延迟 = 从库回放 binlog 的速度跟不上主库写入速度。需要从架构、配置、业务三个层面治理。

排查

SHOW SLAVE STATUS\G
-- 关注 Seconds_Behind_Master、Relay_Log_Pos、SQL 线程状态

优化手段

层面 手段
硬件从库磁盘 IO、CPU 不低于主库
参数从库 log_slave_updates、并行复制 slave_parallel_workers
架构多从库分担读、大事务拆分、避免主库突发写入
业务写后读主、延迟告警、非核心读走从库

MySQL 5.7+ 支持基于 GTID 的并行复制；MySQL 8.0 的 replica_parallel_type=LOGICAL_CLOCK 可提升回放并行度。

232. 数据库如何做容量规划？

容量规划目标是：在业务增长下，数据库在性能、存储、高可用和成本之间取得平衡，并提前触发扩容或分库分表。

评估维度

维度 指标
存储单表行数、数据增长率、索引占比、磁盘使用率
性能QPS/TPS、慢 SQL 比例、P99 延迟
连接活跃连接数、连接池使用率、Threads_running
复制主从延迟、binlog 增长速率
 
规划步骤：
1. 基准测试（sysbench）得单机上限
2. 按业务峰值 QPS × 安全系数（1.5~3）
3. 估算 6~12 个月数据量
4. 设定告警阈值：磁盘 70%、CPU 60%、延迟 P99
5. 提前规划读写分离 / 分库分表 / 归档

339. B+Tree 和 B Tree 有什么区别？

B Tree 和 B+Tree 都是多路平衡查找树，但数据组织方式不同，B+Tree 更适合磁盘索引。

对比项 B Tree B+Tree
数据存放所有节点都可能存数据仅叶子节点存数据（或主键+指针）
非叶子节点存 key + 数据只存 key + 指针，扇出更高
叶子链表无叶子节点双向链表连接
范围查询需中序遍历沿叶子链表扫描，高效
单次查找可能在非叶子命中必到叶子，路径稳定

B+Tree 非叶子节点不存数据，单个磁盘页能容纳更多 key，树更矮，磁盘 IO 次数更少。

340. 为什么 MySQL 索引通常使用 B+Tree？

InnoDB 选择 B+Tree 是因为数据库索引的核心矛盾是减少磁盘 IO，而非内存中的比较次数。

原因

  • 高扇出：非叶子节点只存 key，单页可指向更多子节点，树高通常 3～4 层即可支撑千万级数据。
  • 顺序访问：叶子链表让 WHERE id BETWEEN 1 AND 1000、ORDER BY id 更高效。
  • 稳定 IO：任何查找都到叶子， IO 次数可预测。
  • 页对齐：与 InnoDB 16KB 数据页天然契合。
为什么不用 Hash？
- Hash 只支持等值，不支持范围
- Hash 无序，无法 ORDER BY
- 冲突时需要链表扫描

为什么不用红黑树？
- 树高大，磁盘 IO 多
- 不适合块设备读取

341. 索引下推是什么？

索引下推（Index Condition Pushdown，ICP）是 MySQL 5.6+ 的优化：将部分 WHERE 条件下推到存储引擎层，在索引遍历过程中直接过滤，减少回表次数。

CREATE INDEX idx_name_age ON user(name, age);

SELECT * FROM user WHERE name LIKE '张%' AND age = 25;

无 ICP

Server 层用 name LIKE '张%' 在索引定位，回表取整行后再在 Server 层过滤 age = 25。

有 ICP

引擎层在索引遍历时同时判断 age = 25（若 age 在索引中），不满足则不回表。Extra 显示 Using index condition。

ICP 适用于：联合索引中，前导列范围匹配，后续列仍可在引擎层过滤的场景。

342. 索引失效的常见场景有哪些？

场景 示例
违反最左前缀联合索引 (a,b,c)，条件只有 b、c
对列做函数/运算WHERE YEAR(create_time) = 2024
隐式类型转换字符串列与数字比较
前导模糊匹配LIKE '%abc'
OR 连接不同列WHERE a=1 OR b=2 可能全表扫
不等于 / NOT IN优化器可能认为全表扫更优
区分度低性别字段建索引，优化器可能放弃
字符集不一致JOIN 时隐式转换
 
-- 失效
WHERE DATE(create_time) = '2024-06-01'

-- 改写为范围，可走索引
WHERE create_time >= '2024-06-01' AND create_time < '2024-06-02'

是否走索引以 EXPLAIN 为准，规则是经验法则，优化器还会结合统计信息做 cost 估算。

343. EXPLAIN 中 type、key、rows、Extra 如何解读？

EXPLAIN 是查看 MySQL 执行计划最常用的工具，重点关注访问类型、索引使用和额外操作。

字段 含义 关注点
type访问类型优劣：system > const > eq_ref > ref > range > index > ALL
key实际使用的索引NULL 表示未用索引
rows估算扫描行数越小通常越好，但是估算值
Extra额外信息Using index 好；Using filesort、Using temporary 需关注

type 常见值

  • const：主键或唯一索引等值，最多一行。
  • ref：非唯一索引等值匹配。
  • range：索引范围扫描。
  • ALL：全表扫描，大数据量时需优化。

执行计划只是估算，必要时结合慢日志、实际耗时、EXPLAIN ANALYZE（8.0.18+）和数据分布判断。

344. MySQL 主从复制原理是什么？

主从复制基于主库的 binlog，从库拉取并重放 binlog 事件，使从库数据与主库保持一致（最终一致）。

主库：事务提交 -> 写 binlog
从库 IO 线程：连接主库 -> 拉取 binlog -> 写入 relay log
从库 SQL 线程：读取 relay log -> 重放 SQL/行事件 -> 更新本地数据

复制格式

格式 说明
STATEMENT记录 SQL 语句，日志小但可能不一致（如 UUID()）
ROW记录行变更，可靠，日志较大，生产推荐
MIXED混合模式，由 Server 自动选择

现代架构常用 GTID 复制，便于 failover 和主从切换；读写分离、备份、高可用都依赖复制链路稳定。

345. binlog、redo log、undo log 分别有什么作用？

日志 层级 作用
binlogServer 层归档日志，主从复制、基于时间点的恢复
redo logInnoDB重做日志，保证事务持久性，崩溃恢复已提交事务
undo logInnoDB回滚日志，事务回滚 + MVCC 多版本

协作关系

更新一行：
1. 写 undo log（保存旧值，供回滚和 MVCC）
2. 更新 Buffer Pool 中的数据页
3. 写 redo log（prepare 或 commit 阶段）
4. 写 binlog
5. redo log commit（两阶段提交）

redo 是物理/物理逻辑日志，循环写；binlog 是逻辑日志，追加写。undo 在 MVCC 可见性判断后由 purge 线程清理。

346. redo log 和 binlog 的两阶段提交是什么？

InnoDB 引擎日志（redo）与 Server 层日志（binlog）分属两层，两阶段提交（2PC）用于保证二者逻辑一致，避免崩溃后主从数据不一致。

1. Prepare：InnoDB 写 redo log，状态为 prepare
2. 写 binlog
3. Commit：InnoDB 将 redo log 标记为 commit

崩溃恢复

崩溃时机 恢复行为
redo prepare 后、binlog 写完前无 binlog，事务回滚
binlog 写完、redo commit 前有 binlog，事务提交（通过 XID 关联）

这样无论 crash 在哪个阶段，恢复后主库与 binlog 都能保持一致，从库回放 binlog 也不会丢事务。

347. MySQL 如何保证崩溃恢复？

崩溃恢复依赖 redo log + undo log + 两阶段提交，启动时 InnoDB 自动执行 crash recovery。

恢复流程

  1. 扫描 redo log，找出已 prepare 或 commit 的事务。
  2. 对比 binlog 的 XID，判断事务是否应提交。
  3. 应提交的：用 redo 重做已提交修改。
  4. 应回滚的：用 undo log 回滚未提交事务。
WAL 原则（Write-Ahead Logging）：
先写日志，再写数据页
崩溃后通过 redo 重做，通过 undo 撤销

innodb_flush_log_at_trx_commit=1 与 sync_binlog=1 配合可保证每次事务提交都刷盘，可靠性最高；设为 2 或 0 可换性能但 crash 可能丢最近事务。

348. 分库分表适合什么场景？

分库分表是解决单库单表容量或性能瓶颈的架构手段，不应作为项目初期默认方案。

适合时机

信号 说明
单表过大行数亿级、索引 B+Tree 深度增加、DDL 耗时小时级
写入瓶颈单主库 TPS 到顶，垂直扩容无效
存储瓶颈磁盘/备份/恢复窗口不可接受
热点集中单行/单分片过热，需打散

不适合

  • 数据量小、QPS 低，索引优化和读写分离即可。
  • 复杂跨分片 JOIN/事务多，改造成本极高。
  • 团队缺乏分片治理经验，运维风险大。

优先顺序：SQL/索引优化 → 缓存 → 读写分离 → 垂直拆分 → 水平分库分表。

349. 水平分表和垂直分表有什么区别？

类型 拆分方式 典型场景
垂直分表按列拆，大字段/冷字段拆到扩展表主表行宽过大，TEXT 字段访问少
垂直分库按业务模块拆库用户库、订单库、商品库
水平分表同结构按行拆到多表单表数据量过大
水平分库多库 + 每库多表写入和存储双重瓶颈
 
垂直分表：
user(id, name, phone) + user_ext(id, avatar_blob, bio)

水平分表：
orders_0, orders_1, ... orders_15
路由：shard = user_id % 16

垂直拆分改动相对小；水平拆分需解决路由、全局 ID、跨分片查询和扩容迁移。

350. 分库分表后的全局 ID 如何设计？

分库分表后不能使用单库自增 ID，需要全局唯一、趋势递增的 ID 方案。

方案 优点 缺点
雪花算法（Snowflake）本地生成、高性能、趋势递增依赖时钟、机器 ID 分配
号段模式（Leaf）批量取号、DB 压力小需号段服务、扩容需规划
Redis INCR实现简单依赖 Redis 高可用
UUID绝对唯一无序、字符串占空间、索引性能差
 
雪花 ID 结构（64 bit）：
0 | 41 bit 时间戳 | 10 bit 机器 ID | 12 bit 序列号

生产推荐雪花或号段；避免 UUID v4 做主键。若用雪花，需处理时钟回拨和多机房 workerId 分配。

351. 分库分表后的分页查询如何处理？

分片后 LIMIT offset, size 需在每个分片执行，再在中间件层归并排序，成本极高。

方案

方案 说明
游标分页WHERE id > last_id LIMIT n，避免大 offset
禁止全局深分页产品上限「只能翻到第 N 页」
异构索引表ES/ClickHouse 存排序字段，查 id 再回表
分片内分页按 user_id 分片时，用户只看自己的数据，无需跨片
 
跨分片 ORDER BY create_time LIMIT 10000, 20 的代价：
每个分片取 10020 条 -> 中间件归并 16×10020 条 -> 排序 -> 取 20 条

设计分片键时尽量让高频查询落在单分片内，避免全局排序分页。

352. 数据库连接池的核心参数有哪些？

连接池复用 TCP 连接，避免频繁建连/断连。HikariCP、Druid 是 Java 生态常见实现。

参数 含义
maximumPoolSize最大连接数，过大占满 DB max_connections
minimumIdle最小空闲连接
connectionTimeout获取连接超时时间
idleTimeout空闲连接回收时间

经验公式（仅供参考）：

353. MySQL 死锁如何排查？

InnoDB 会自动检测死锁并回滚其中一个事务（代价较小者），但频繁死锁说明并发控制设计有问题。

排查步骤

  1. 查看最近一次死锁信息：SHOW ENGINE INNODB STATUS\G 中 LATEST DETECTED DEADLOCK。
  2. 分析相互等待的锁：哪两条 SQL、哪两个索引、加锁顺序是否相反。
  3. 结合应用日志、TraceId 定位业务代码路径。
  4. 检查长事务、未命中索引导致锁范围扩大。
-- 查看当前锁等待
SELECT * FROM information_schema.innodb_locks;
SELECT * FROM performance_schema.data_locks;  -- 8.0+

常见原因与解决

原因 解决
加锁顺序不一致统一按主键升序访问多行
大事务拆小事务，尽快提交
Gap Lock 冲突调整隔离级别为 RC 或优化 SQL
重复锁竞争应用层分布式锁、重试机制

死锁不一定是错误，少量可接受；高频死锁需从索引、事务边界和加锁顺序上根治。
