大模型推理加速是什么？

大模型推理加速指在不改变或轻微影响模型效果的前提下，降低单次生成延迟、提高吞吐（QPS/TPS），并减少显存与算力成本。

推理是自回归逐 Token 生成，瓶颈通常在 Attention 计算、KV 缓存读写、显存带宽和 batch 调度，而非单纯 FLOPs。

核心优化方向

方向 原理 典型手段
Attention 加速减少 QK^T 计算与内存访问FlashAttention、FlashAttention-2
KV Cache缓存历史 K/V，避免重复计算PagedAttention、KV 量化
批处理调度提高 GPU 利用率Continuous Batching、Dynamic Batching
模型压缩降低权重与激活精度INT8/INT4 量化（GPTQ、AWQ）
解码策略用小模型辅助大模型生成Speculative Decoding
并行部署多卡分担计算Tensor Parallel、Pipeline Parallel

KV Cache

生成每个新 Token 时，历史 Token 的 Key/Value 可复用。KV Cache 将这部分存下来，避免 Prefill 之后每步重复计算 Attention。

Prefill：处理完整 prompt，建立 KV Cache
Decode：每步只算新 token，读取已有 KV Cache

长上下文下 KV Cache 显存占用大，PagedAttention（vLLM）将其分页管理，减少碎片并提高并发。

Continuous Batching

传统静态 batch 需等最长序列结束才释放；Continuous Batching 在推理过程中动态插入/移除请求，显著提升吞吐。

静态 batch：A、B、C 同时开始，等最慢的结束
Continuous Batching：A 先结束立刻释放，新请求 D 补位

量化推理

将 FP16/FP32 权重压缩为 INT8/INT4，降低显存占用与内存带宽压力，常见方案 GPTQ、AWQ、GGUF。

FP16 7B 模型 ≈ 14GB 显存
INT4 量化后 ≈ 4GB，延迟通常可接受

Speculative Decoding

用小模型（Draft Model）快速生成多个候选 Token，大模型并行验证，接受则一次前进多步，减少大模型调用次数。

常见推理框架

  • vLLM：PagedAttention + Continuous Batching，高吞吐自托管首选
  • TGI / TensorRT-LLM：NVIDIA 生态，适合生产部署
  • llama.cpp / Ollama：CPU/边缘侧轻量推理

面试要点：推理加速 ≠ 换更大模型；从 Attention、KV Cache、Batch 调度、量化、并行 五个维度回答即可。

vLLM是什么？

vLLM 是一个大模型推理服务框架，主要目标是提高推理吞吐和显存利用率。它比较核心的点是 PagedAttention，可以更高效地管理 KV Cache，提升高并发请求下的显存利用率。

模型微调 SFT 是什么？

SFT（Supervised Fine-Tuning，监督微调）是在预训练大模型基础上，用标注好的输入-输出样本继续训练，让模型学会遵循指令、输出特定格式或领域风格。

在大模型训练链路中的位置

预训练（Pre-training）
  ↓ 海量无标注文本，学语言与知识
SFT（Supervised Fine-Tuning）
  ↓ 指令-回答对，学对话与任务格式
RLHF / DPO（可选）
  ↓ 人类偏好对齐，提升安全性与有用性

SFT 解决的是「模型能说话、能按指令完成任务」，而非注入大量易变事实（后者更适合 RAG）。

训练数据格式

典型为 instruction-output 对，也可含 system prompt 和多轮对话。

{
  "instruction": "将以下句子翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today."
}

多轮对话则按 ChatML / ShareGPT 等模板组织，训练时只对 assistant 回复部分计算 loss。

SFT 能做什么

  • 教模型遵循指令（问答、摘要、分类、代码生成）
  • 固化输出格式（JSON、Markdown、企业话术）
  • 适配领域术语与表达风格（医疗、法律、客服）
  • 激发预训练阶段已学能力（In-context 能力对齐）

SFT 不适合什么

  • 注入大量实时或易变知识 → 用 RAG
  • 复杂价值观与安全对齐 → 需 RLHF/DPO
  • 数据量极少且要求强推理 → 效果有限，需更多数据或更强基座

Full Fine-tuning vs LoRA

方式 特点 适用
Full Fine-tuning更新全部参数，效果上限高充足算力、大量高质量数据
LoRA / QLoRA只训练低秩适配器，~1% 参数单卡微调 7B~70B，多任务切换

与预训练、RLHF 的区别

阶段 数据 目标
预训练海量无标注文本学语言规律与世界知识
SFT标注 instruction-output学任务执行与对话格式
RLHF/DPO人类偏好排序对齐价值观、减少有害输出

企业落地多数直接用 API 基座模型 + Prompt/RAG；仅在需要固化风格或专有任务时才做 SFT。

大模型微调是什么？

大模型微调（Fine-tuning）是在预训练基座模型上，用特定任务或领域的数据继续训练，使模型适应下游场景——而非从零训练。

预训练让模型「懂语言、有知识」；微调让模型「会做事、符合业务要求」。

在大模型训练链路中的位置

预训练（Pre-training）     海量无标注文本，学语言与通用知识
    ↓
微调（Fine-tuning）        用标注数据适配任务 / 领域 / 风格
    ├── 全参微调（Full Fine-tuning）
    ├── 参数高效微调（PEFT）：LoRA、QLoRA、Adapter
    └── 监督微调 SFT 是最常见的微调形式
    ↓
对齐（可选）               RLHF / DPO，对齐人类偏好与安全性

常见微调方式

方式 原理 适用
Full Fine-tuning更新全部参数数据多、算力足，追求效果上限
LoRA冻结原权重，只训低秩适配器（约 0.1%~1% 参数）单卡微调 7B~70B，多任务切换
QLoRA4-bit 量化 + LoRA显存受限场景
SFTinstruction-output 监督学习教指令遵循、输出格式、领域话术

微调 vs 其他方案

方案 解决什么 成本
Prompt Engineering通过提示词引导行为最低，即时生效
RAG注入实时 / 私有知识中，无需改模型权重
微调固化风格、格式、任务习惯高，需 GPU 与标注数据

什么时候该微调？

  • 需要稳定输出格式（JSON Schema、企业话术、代码规范）。
  • 领域术语密集，Prompt 难以稳定覆盖（医疗、法律、内部系统）。
  • 同一任务高频调用，微调后推理成本可能低于超长 Prompt。

什么时候不该微调？

  • 知识频繁更新 → 用 RAG，而非反复重训。
  • 数据量极少（几十条）→ 效果有限，优先 Prompt / Few-shot。
  • 安全与价值观对齐 → 需 RLHF/DPO，单靠 SFT 不够。

面试回答：微调是在预训练模型上用标注数据做二次训练；企业多数场景 API + RAG/Prompt 即可，仅在需固化风格或专有任务时才微调。

大模型 RLHF 是什么？

RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）是让大模型输出更符合人类偏好（有用、诚实、无害）的对齐技术，ChatGPT 等对话模型都经过此阶段。

SFT 教会模型「怎么回答」；RLHF 教会模型「哪种回答更好」。

为什么需要 RLHF？

SFT 只用单一标准答案训练，难以覆盖「多种回答都合理但优劣不同」的场景，也无法有效抑制有害、幻觉、啰嗦等输出。RLHF 引入人类偏好排序，让模型学习「更好 vs 更差」。

经典三阶段流程

阶段 1：SFT（监督微调）
  用 instruction-output 数据，得到会对话的基础模型

阶段 2：训练奖励模型（Reward Model, RM）
  人工对同一问题的多个回答排序（A > B > C）
  RM 学习预测人类偏好分数

阶段 3：强化学习优化（通常 PPO）
  用 RM 给模型输出打分作为奖励信号
  通过 PPO 更新策略模型，使高分回答概率增大
  同时用 KL 散度约束，防止偏离 SFT 模型太远（模型崩溃）

各阶段职责

阶段 输入 输出
SFT标注问答对能遵循指令的基础模型
奖励模型人类偏好排序数据给任意回答打偏好分的 RM
PPO 强化学习RM 奖励 + KL 惩罚对齐人类价值观的最终模型

RLHF 的难点

  • 标注成本高：偏好排序需大量人工参与。
  • 训练不稳定：PPO 超参敏感，易出现奖励黑客（reward hacking）。
  • 工程复杂：需同时维护 SFT 模型、RM、策略模型，显存与调度成本高。

RLHF 的替代：DPO

DPO（Direct Preference Optimization）跳过显式奖励模型和 PPO，直接用偏好对（chosen vs rejected）优化策略，更简单、更稳定，效果接近 RLHF，近年被广泛采用。

RLHF：SFT → 训 RM → PPO 强化学习（三模型、复杂）
DPO：  SFT → 直接偏好优化（两阶段、轻量化）

面试回答：RLHF 用人类偏好训练奖励模型，再以强化学习（PPO）让模型输出更符合人类期望；目的是安全对齐，与 SFT 的任务适配互补。企业应用通常直接用已对齐的 API 模型，无需自训 RLHF。
