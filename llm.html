大模型推理和训练有什么区别？

训练是学习参数，推理是使用已学参数生成结果。两者计算图、资源需求和优化目标完全不同。

核心对比

对比项 训练 推理
目标最小化 loss，更新权重给定输入，生成输出 Token
计算前向 + 反向传播 + 梯度更新仅前向传播
显存权重 + 梯度 + 优化器状态（约为推理 3~4 倍）权重 + KV Cache + 激活
精度通常 FP32/BF16 + 混合精度FP16/BF16/INT8/INT4 量化常见
并行Data / Tensor / Pipeline / ZeROTensor / Pipeline Parallel，侧重 batch 调度
优化指标收敛速度、loss、泛化延迟（TTFT/TPOT）、吞吐（TPS）、成本

面试回答：训练要反向传播和优化器，显存开销大，追求收敛；

推理只做前向、逐 Token 自回归生成，瓶颈在 Attention 和 KV Cache，追求低延迟和高吞吐。

大模型推理流程是什么？

大模型推理（以 Decoder-only 为例）是将用户输入编码后，自回归逐 Token 生成完整回复，直到遇到 EOS 或达到最大长度。

完整流程

1. 接收请求：prompt + 采样参数（temperature、top_p、max_tokens）
2. Tokenize：文本 → token id 序列
3. Prefill 阶段：并行处理整个 prompt，计算 Attention，建立 KV Cache
4. Decode 阶段：循环生成
   - 取上一步 logits → 采样/贪心选下一个 token
   - 将新 token 送入模型，更新 KV Cache
   - 重复直到 EOS 或 max_tokens
5. Detokenize：token 序列 → 文本
6. 流式返回（SSE/WebSocket）或一次性返回

关键指标

指标 含义
TTFTTime To First Token，首 Token 延迟（含 Prefill）
TPOTTime Per Output Token，解码阶段每 Token 耗时
TPSTokens Per Second，吞吐

面试回答：Tokenize → Prefill 处理 prompt 建 KV Cache → Decode 循环采样生成 → Detokenize 输出；关注 TTFT 和 TPOT 两个延迟指标。

Prefill 和 Decode 是什么？

大模型自回归推理分为两阶段：Prefill（预填充）和 Decode（解码），计算特征和瓶颈不同。

对比

阶段 输入 计算特点 瓶颈
Prefill完整 prompt（可能很长）所有 token 并行计算，计算密集（compute-bound）矩阵乘法 FLOPs
Decode每次仅 1 个新 token读取全部历史 KV Cache，访存密集（memory-bound）显存带宽、KV Cache 大小
 
Prefill:  prompt=[t1, t2, ..., tn]  → 一次前向，输出 logits + KV Cache
Decode:   每步输入=[t_new]           → 读历史 KV，算新 token logits，追加 KV

长 prompt 场景 TTFT 高（Prefill 慢）；长输出场景 TPOT 累积明显（Decode 步数多）。优化需分别针对两阶段：FlashAttention 加速 Prefill，PagedAttention / KV 量化降低 Decode 显存压力。

面试回答：Prefill 并行处理 prompt，计算密集；Decode 每次生成一个 token，依赖 KV Cache，访存密集。两者瓶颈不同，优化手段也不同。

为什么大模型逐 token 生成？

主流 Causal LM（GPT、LLaMA 等）采用自回归（Autoregressive）建模：用已生成序列预测下一个 Token 的概率分布，无法一次性并行输出整句。

原因

  • 训练目标：语言模型学的是 P(token_t | token_1, ..., token_{t-1})，天然是逐位预测。
  • Causal Mask：当前 token 只能 attend 到左侧历史，不能看未来，保证生成时因果性。
  • 输出空间巨大：词表通常 3 万~10 万，每步从整表采样，无法一步输出整段文本。
Step 1: "今天" → P(下一token) → 采样得 "天气"
Step 2: "今天天气" → P(下一token) → 采样得 "很"
Step 3: ... 直到 EOS

能否并行生成？

标准自回归不行，但有加速思路：Speculative Decoding（小模型草稿 + 大模型验证）、Medusa（多头并行预测）等，在不改变训练范式前提下减少大模型调用次数。

面试回答：因为 Causal LM 训练目标是逐 token 条件概率，推理必须自回归；每步依赖上一步结果，所以是串行生成，这也是 Decode 阶段延迟的主要来源。

Attention 机制是什么？

Attention（注意力）让模型在处理当前 token 时，动态加权聚合其他位置的信息，解决长距离依赖问题，是 Transformer 的核心。

计算流程

输入 X → 线性变换得 Q、K、V
Attention(Q,K,V) = softmax(QK^T / √d_k) · V

QK^T 得到每个 query 对所有 key 的相似度（注意力分数），softmax 归一化后与 V 加权求和，得到融合了上下文的新表示。

多头注意力（Multi-Head）

将 Q/K/V 拆成 h 个头并行计算，各头关注不同模式（语法、语义、指代等），最后拼接再线性变换。

概念 作用
Self-Attention序列内部互相关注，Encoder/Decoder 均用
Causal / Masked AttentionDecoder 中只能看左侧，保证自回归
Cross-AttentionDecoder 关注 Encoder 输出（Seq2Seq 场景）

面试回答：Attention 通过 QK 相似度对 V 加权求和，让模型动态关注相关位置；多头并行捕捉多种依赖关系，是 Transformer 性能的关键。

为什么 Attention 是 O(n²)？

n 为序列长度。标准 Self-Attention 需计算每个 token 与所有 token 的相似度，时间和空间均随序列长度平方增长。

复杂度分析

Q: (n, d)    K^T: (d, n)    →  QK^T: (n, n)  共 n² 次相似度计算
softmax + 乘 V: 也是 (n, n) 矩阵运算
空间：需存储 (n, n) 注意力矩阵（训练时）
长度 n 注意力矩阵大小
2K4M 元素
8K64M 元素
32K1B 元素

优化方向

  • FlashAttention：分块计算，不物化完整 n×n 矩阵，降低显存 IO。
  • 稀疏 Attention：Local、Longformer 等只算部分位置。
  • KV Cache：推理时 Decode 阶段 Q 只有 1 行，但 K/V 随 n 增长，瓶颈转为访存。

面试回答：因为要算 n×n 的注意力矩阵，计算和显存都是 O(n²)；长上下文靠 FlashAttention、稀疏注意力等技术缓解。

什么是 KV Cache？

KV Cache 是推理时对历史 token 的 Key 和 Value 向量的缓存，避免 Decode 阶段每步重复计算已处理 token 的 K/V。

为什么需要

无 KV Cache：每生成 1 token，重算全部历史 token 的 Attention → O(n²) 每步
有 KV Cache：Prefill 算一次存下来，Decode 每步只算新 token 的 Q，与缓存 K/V 做 Attention → O(n) 每步

工作流程

  1. Prefill：处理 prompt，计算每层 K/V，写入 Cache。
  2. Decode：新 token 只算自己的 Q 和新 K/V，追加到 Cache；Attention 用 Q 与全部缓存 K 算分数，加权缓存 V。

KV Cache 是推理加速的必要手段，但长上下文 + 高并发下会成为显存主要占用来源（见下一题）。

面试回答：KV Cache 缓存历史 K/V，Decode 时避免重复计算，把每步从 O(n²) 降到 O(n)；是推理必备优化，但占显存大。

KV Cache 为什么占显存？

KV Cache 为每一层、每个 token、每个请求各存一份 K 和 V 向量，随层数、序列长度、并发请求数线性增长，往往是推理显存的大头。

显存估算

单层单 token KV 大小 ≈ 2 × num_heads × head_dim × dtype_bytes
总 KV Cache ≈ 2 × num_layers × seq_len × hidden_size × dtype_bytes × batch_size

例：LLaMA-7B, FP16, seq_len=4096, batch=1
≈ 2 × 32 × 4096 × 4096 × 2 bytes ≈ 2 GB

增长因素

因素 影响
序列长度线性增长，128K 上下文 KV 可达数十 GB
模型层数/宽度更大模型 KV 更大
并发请求每个请求独立 KV Cache
精度FP16 → INT8/INT4 量化可减半或更多

缓解手段

  • PagedAttention（vLLM）：分页管理，减少碎片，提高并发。
  • KV 量化（INT8/INT4）、Multi-Query Attention（MQA）、Grouped-Query Attention（GQA）。
  • 限制 max_seq_len、Prompt 压缩、上下文裁剪。

面试回答：每层每 token 存 K 和 V，长度 × 层数 × 并发 × 精度决定显存；长上下文和高并发下 KV Cache 常超过权重本身，需 PagedAttention、GQA、量化等手段。

vLLM 是什么？

vLLM 是 UC Berkeley 开源的大模型推理服务框架，主打高吞吐、低延迟的自托管 LLM 部署，已成为工业界主流推理引擎之一。

核心特性

特性 说明
PagedAttention分页管理 KV Cache，减少显存碎片，提高并发
Continuous Batching动态插入/移除请求，GPU 利用率高
OpenAI 兼容 API/v1/chat/completions，易接入现有应用
多模型支持LLaMA、Qwen、Mistral 等 HuggingFace 模型
量化推理AWQ、GPTQ、FP8 等
 
# 启动服务
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-2-7b-chat-hf \
  --tensor-parallel-size 2

同类方案：TGI（HuggingFace）、TensorRT-LLM（NVIDIA）、SGLang。选型看生态、硬件、是否需极致延迟。

面试回答：vLLM 是高性能 LLM 推理框架，核心是 PagedAttention + Continuous Batching，提供 OpenAI 兼容 API，适合自托管高并发推理场景。

PP 分离是什么？

「PP 分离」通常指Prefill 和 Decode 分离部署（也叫 Disaggregated Prefill），或 Pipeline Parallel 中阶段分离。面试中以前者更为热点。

Prefill / Decode 分离的原因

阶段 计算特征 资源偏好
Prefill计算密集（大矩阵乘）高算力 GPU（如 A100/H100）
Decode访存密集（读 KV Cache）大显存、高带宽 GPU

混在一起时，长 Prefill 会阻塞 Decode 请求（拖高 TPOT），短 Decode 又浪费 Prefill 算力。分离后可独立扩缩容、专用调度。

请求 → Prefill 节点（处理 prompt，生成 KV Cache）
     → 传输 KV Cache → Decode 节点（逐 token 生成）
     → 返回结果

挑战

  • KV Cache 跨节点传输有网络开销。
  • 调度复杂度增加，需全局请求路由。
  • 代表方案：Mooncake、DistServe、NVIDIA NIM 等。

面试回答：Prefill 计算密集、Decode 访存密集，混在一起互相干扰；分离部署可针对性扩缩容，提升整体吞吐，但需解决 KV Cache 跨节点传输问题。

大模型推理加速是什么？

大模型推理加速指在不改变或轻微影响模型效果的前提下，降低单次生成延迟、提高吞吐（QPS/TPS），并减少显存与算力成本。

推理是自回归逐 Token 生成，瓶颈通常在 Attention 计算、KV 缓存读写、显存带宽和 batch 调度，而非单纯 FLOPs。

核心优化方向

方向 原理 典型手段
Attention 加速减少 QK^T 计算与内存访问FlashAttention、FlashAttention-2
KV Cache缓存历史 K/V，避免重复计算PagedAttention、KV 量化
批处理调度提高 GPU 利用率Continuous Batching、Dynamic Batching
模型压缩降低权重与激活精度INT8/INT4 量化（GPTQ、AWQ）
解码策略用小模型辅助大模型生成Speculative Decoding
并行部署多卡分担计算Tensor Parallel、Pipeline Parallel

KV Cache

生成每个新 Token 时，历史 Token 的 Key/Value 可复用。KV Cache 将这部分存下来，避免 Prefill 之后每步重复计算 Attention。

Prefill：处理完整 prompt，建立 KV Cache
Decode：每步只算新 token，读取已有 KV Cache

长上下文下 KV Cache 显存占用大，PagedAttention（vLLM）将其分页管理，减少碎片并提高并发。

Continuous Batching

传统静态 batch 需等最长序列结束才释放；Continuous Batching 在推理过程中动态插入/移除请求，显著提升吞吐。

静态 batch：A、B、C 同时开始，等最慢的结束
Continuous Batching：A 先结束立刻释放，新请求 D 补位

量化推理

将 FP16/FP32 权重压缩为 INT8/INT4，降低显存占用与内存带宽压力，常见方案 GPTQ、AWQ、GGUF。

FP16 7B 模型 ≈ 14GB 显存
INT4 量化后 ≈ 4GB，延迟通常可接受

Speculative Decoding

用小模型（Draft Model）快速生成多个候选 Token，大模型并行验证，接受则一次前进多步，减少大模型调用次数。

常见推理框架

  • vLLM：PagedAttention + Continuous Batching，高吞吐自托管首选
  • TGI / TensorRT-LLM：NVIDIA 生态，适合生产部署
  • llama.cpp / Ollama：CPU/边缘侧轻量推理

面试要点：推理加速 ≠ 换更大模型；从 Attention、KV Cache、Batch 调度、量化、并行 五个维度回答即可。

模型微调 SFT 是什么？

SFT（Supervised Fine-Tuning，监督微调）是在预训练大模型基础上，用标注好的输入-输出样本继续训练，让模型学会遵循指令、输出特定格式或领域风格。

在大模型训练链路中的位置

预训练（Pre-training）
  ↓ 海量无标注文本，学语言与知识
SFT（Supervised Fine-Tuning）
  ↓ 指令-回答对，学对话与任务格式
RLHF / DPO（可选）
  ↓ 人类偏好对齐，提升安全性与有用性

SFT 解决的是「模型能说话、能按指令完成任务」，而非注入大量易变事实（后者更适合 RAG）。

训练数据格式

典型为 instruction-output 对，也可含 system prompt 和多轮对话。

{
  "instruction": "将以下句子翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today."
}

多轮对话则按 ChatML / ShareGPT 等模板组织，训练时只对 assistant 回复部分计算 loss。

SFT 能做什么

  • 教模型遵循指令（问答、摘要、分类、代码生成）
  • 固化输出格式（JSON、Markdown、企业话术）
  • 适配领域术语与表达风格（医疗、法律、客服）
  • 激发预训练阶段已学能力（In-context 能力对齐）

SFT 不适合什么

  • 注入大量实时或易变知识 → 用 RAG
  • 复杂价值观与安全对齐 → 需 RLHF/DPO
  • 数据量极少且要求强推理 → 效果有限，需更多数据或更强基座

Full Fine-tuning vs LoRA

方式 特点 适用
Full Fine-tuning更新全部参数，效果上限高充足算力、大量高质量数据
LoRA / QLoRA只训练低秩适配器，~1% 参数单卡微调 7B~70B，多任务切换

与预训练、RLHF 的区别

阶段 数据 目标
预训练海量无标注文本学语言规律与世界知识
SFT标注 instruction-output学任务执行与对话格式
RLHF/DPO人类偏好排序对齐价值观、减少有害输出

企业落地多数直接用 API 基座模型 + Prompt/RAG；仅在需要固化风格或专有任务时才做 SFT。

大模型微调是什么？

大模型微调（Fine-tuning）是在预训练基座模型上，用特定任务或领域的数据继续训练，使模型适应下游场景——而非从零训练。

预训练让模型「懂语言、有知识」；微调让模型「会做事、符合业务要求」。

在大模型训练链路中的位置

预训练（Pre-training）     海量无标注文本，学语言与通用知识
    ↓
微调（Fine-tuning）        用标注数据适配任务 / 领域 / 风格
    ├── 全参微调（Full Fine-tuning）
    ├── 参数高效微调（PEFT）：LoRA、QLoRA、Adapter
    └── 监督微调 SFT 是最常见的微调形式
    ↓
对齐（可选）               RLHF / DPO，对齐人类偏好与安全性

常见微调方式

方式 原理 适用
Full Fine-tuning更新全部参数数据多、算力足，追求效果上限
LoRA冻结原权重，只训低秩适配器（约 0.1%~1% 参数）单卡微调 7B~70B，多任务切换
QLoRA4-bit 量化 + LoRA显存受限场景
SFTinstruction-output 监督学习教指令遵循、输出格式、领域话术

微调 vs 其他方案

方案 解决什么 成本
Prompt Engineering通过提示词引导行为最低，即时生效
RAG注入实时 / 私有知识中，无需改模型权重
微调固化风格、格式、任务习惯高，需 GPU 与标注数据

什么时候该微调？

  • 需要稳定输出格式（JSON Schema、企业话术、代码规范）。
  • 领域术语密集，Prompt 难以稳定覆盖（医疗、法律、内部系统）。
  • 同一任务高频调用，微调后推理成本可能低于超长 Prompt。

什么时候不该微调？

  • 知识频繁更新 → 用 RAG，而非反复重训。
  • 数据量极少（几十条）→ 效果有限，优先 Prompt / Few-shot。
  • 安全与价值观对齐 → 需 RLHF/DPO，单靠 SFT 不够。

面试回答：微调是在预训练模型上用标注数据做二次训练；企业多数场景 API + RAG/Prompt 即可，仅在需固化风格或专有任务时才微调。

大模型 RLHF 是什么？

RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）是让大模型输出更符合人类偏好（有用、诚实、无害）的对齐技术，ChatGPT 等对话模型都经过此阶段。

SFT 教会模型「怎么回答」；RLHF 教会模型「哪种回答更好」。

为什么需要 RLHF？

SFT 只用单一标准答案训练，难以覆盖「多种回答都合理但优劣不同」的场景，也无法有效抑制有害、幻觉、啰嗦等输出。RLHF 引入人类偏好排序，让模型学习「更好 vs 更差」。

经典三阶段流程

阶段 1：SFT（监督微调）
  用 instruction-output 数据，得到会对话的基础模型

阶段 2：训练奖励模型（Reward Model, RM）
  人工对同一问题的多个回答排序（A > B > C）
  RM 学习预测人类偏好分数

阶段 3：强化学习优化（通常 PPO）
  用 RM 给模型输出打分作为奖励信号
  通过 PPO 更新策略模型，使高分回答概率增大
  同时用 KL 散度约束，防止偏离 SFT 模型太远（模型崩溃）

各阶段职责

阶段 输入 输出
SFT标注问答对能遵循指令的基础模型
奖励模型人类偏好排序数据给任意回答打偏好分的 RM
PPO 强化学习RM 奖励 + KL 惩罚对齐人类价值观的最终模型

RLHF 的难点

  • 标注成本高：偏好排序需大量人工参与。
  • 训练不稳定：PPO 超参敏感，易出现奖励黑客（reward hacking）。
  • 工程复杂：需同时维护 SFT 模型、RM、策略模型，显存与调度成本高。

RLHF 的替代：DPO

DPO（Direct Preference Optimization）跳过显式奖励模型和 PPO，直接用偏好对（chosen vs rejected）优化策略，更简单、更稳定，效果接近 RLHF，近年被广泛采用。

RLHF：SFT → 训 RM → PPO 强化学习（三模型、复杂）
DPO：  SFT → 直接偏好优化（两阶段、轻量化）

面试回答：RLHF 用人类偏好训练奖励模型，再以强化学习（PPO）让模型输出更符合人类期望；目的是安全对齐，与 SFT 的任务适配互补。企业应用通常直接用已对齐的 API 模型，无需自训 RLHF。
