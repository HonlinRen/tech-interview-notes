RAG 向量检索
没有找到匹配的问题，换个关键词试试。

Chunk 设计为什么重要？

Chunk 是检索的最小语义单元，直接影响召回率和生成质量。

Chunk 大小 优点 缺点
太大上下文完整噪声多、占 Token、语义不聚焦
太小检索精准语义断裂、丢失跨段关系

最佳实践

  • 按结构切：标题、段落、表格、代码块。
  • 保留标题路径（breadcrumb）作为 metadata。
  • 设置 10%–20% overlap，避免边界截断。
  • 常见大小 256–512 Token，按文档类型 A/B 调优。

Metadata 有什么作用？

Metadata 描述每个 Chunk 的来源与属性，是过滤、溯源和权限控制的基础。

类型 示例字段 用途
来源doc_id、url、page、section引用溯源
权限tenant_id、dept、acl检索前过滤
版本version、effective_date冲突消解
技术embedding_model、chunk_rule重建索引

多租户场景下，tenant_id 必须在检索层强制过滤，不能仅靠 Prompt 约束。

向量检索的基本原理是什么？

Embedding 模型将文本映射到高维向量空间，语义相近的文本向量距离更近。

文档 -> Embedding -> 向量库
查询 -> Embedding -> ANN 搜索 -> TopK 相似片段

优势与局限

  • 优势：语义相似、同义词、跨语言。
  • 局限：精确编号、SKU、专有名词弱；需 Hybrid Search 补充。

TopK 和召回率是什么？

TopK：检索返回相似度最高的 K 个 Chunk。召回率（Recall）：含正确答案的文档是否被检索到。

参数 K 太小 K 太大
召回漏掉正确答案召回率高
成本Rerank/生成 Token 少噪声多、延迟高

典型策略：向量召回 K=50–100，Rerank 后取 Top5–10 进上下文。用离线评测集调 recall@K。

Query Rewrite 解决什么问题？

用户原始问题往往不适合直接检索：口语化、省略指代、复合问题。

常见改写策略

  • 指代消解：「它多少钱」→「XX 产品多少钱」。
  • 关键词提取：补充领域术语。
  • 问题拆分：复合问题拆成多个子查询。
  • HyDE：生成假设性答案再检索。

多轮对话需结合 history 或摘要后再改写，否则检索命中率低。

RAG HyDE 是什么？

HyDE（Hypothetical Document Embeddings，假设性文档嵌入）是一种 RAG 检索增强策略：不直接用用户问题去向量检索，而是先让 LLM 根据问题生成一篇「假想的答案文档」，再对这篇假设文档做 Embedding 去检索真实文档。

普通 RAG：  用户问题 → Embedding → 向量库 TopK
HyDE RAG：  用户问题 → LLM 生成假设文档 → Embedding → 向量库 TopK → 生成最终答案

为什么需要 HyDE？

向量检索的核心是「Query 与 Document 在 Embedding 空间足够接近」。但用户问题往往是短句、口语化，而知识库 Chunk 是长段落、陈述式，两者在向量空间中分布不对称，导致直接检索命中率低。

HyDE 的思路：让 LLM 先把问题「翻译」成一段看起来像知识库文档的文字，再用这段文字去检索，缩小 Query-Doc 的语义鸿沟。

流程示例

用户问题：「Redis 持久化 RDB 和 AOF 怎么选？」

LLM 生成假设文档（不要求事实正确，只求风格接近知识库）：
  「Redis 提供 RDB 和 AOF 两种持久化机制。RDB 通过快照方式...
   AOF 记录每条写命令... 选择时需考虑数据安全与恢复速度...」

对假设文档 Embedding → 检索真实 Chunk → 将真实资料交给 LLM 生成最终答案

与普通检索对比

对比项 直接 Query 检索 HyDE
Embedding 对象用户原始问题LLM 生成的假设文档
语义匹配短问句 vs 长文档，易失配文档 vs 文档，更接近
额外成本无多一次 LLM 调用 + 延迟
风险检索词不精准假设文档可能幻觉，带偏检索方向

适用场景

  • 问题抽象、复杂，与文档表述风格差异大（如「怎么选型」「优缺点对比」）。
  • 零样本 / 少样本场景，没有足够 Query 改写规则或训练数据。
  • 知识库以长段落技术文档为主，Chunk 风格统一。

不适用场景

  • 精确匹配类查询（订单号、SKU、错误码）→ 直接用 BM25 或原 Query 更准。
  • 对延迟敏感的高 QPS 接口 → 多一次 LLM 调用成本不可忽视。
  • 领域知识 LLM 不熟悉 → 假设文档质量差，反而降低召回。

落地建议

  • 假设文档不需要事实正确，Prompt 中强调「风格像知识库段落即可」。
  • 可与 Hybrid Search 组合：HyDE 向量召回 + BM25 关键词召回，再 Rerank 融合。
  • 对假设文档做缓存（相同/相似问题复用），降低重复 LLM 开销。

面试回答：HyDE 用 LLM 生成假想答案文档再做向量检索，解决 Query 与 Document 语义不对称问题；适合抽象问答，不适合精确查找，需权衡延迟与幻觉风险。

Rerank 是什么？

Rerank（重排）对初步召回的 TopN 候选用更精确模型重新打分排序。

Bi-Encoder 召回 Top100 -> Cross-Encoder Rerank -> Top5 进 Prompt
阶段 模型 特点
召回Bi-Encoder快，可离线建索引
重排Cross-Encoder准，需在线逐对计算

只对 TopN（50–100）重排，平衡精度与延迟。

Embedding 模型如何选型？

选型维度

维度 说明
语言中文场景选 bge-m3、text-embedding-v3 等
维度向量维度越高，通常表达能力更强，但成本也更高。
最大长度不同模型支持的文本长度不同
检索效果召回率 正确答案排序靠前等

用 MTEB/C-MTEB 榜单 + 自有评测集 recall@K 做最终决策；切换模型需全量重建索引。

text-embedding-v3 参数量和最大上下文是多少？

text-embedding-v3 是阿里云百炼（DashScope）提供的通用文本向量模型，面向中文及多语言 RAG 场景，通过 API 调用，参数量官方未公开披露（闭源 API 模型，面试关注可用规格即可）。

规格项 数值 说明
参数量未公开闭源 API，无开源权重；选型看效果与 token 上限
单条最大上下文8,192 Token超长文本需先 Chunk 再 Embedding
批量上限10 条/次字符串列表或文件最多 10 行
输出维度1024（默认）、768、512、256、128、64可通过 dimensions 参数指定
支持语种50+中英日韩德法西葡等主流语言
 
from dashscope import TextEmbedding

resp = TextEmbedding.call(
    model="text-embedding-v3",
    input="企业知识库检索示例",
    dimension=1024,  # 可选 768/512/256/128/64
)

还需要关注的点

关注点 说明
与 OpenAI 同名不同物百炼 text-embedding-v3 与 OpenAI text-embedding-3-small/large 无关，API、维度、定价均不同，勿混用
Chunk 长度单条超 8192 Token 会报错或截断，RAG 入库前必须切分；长文档别整篇 Embed
批量限制每次 API 最多 10 条，建索引需自行分批 + 重试；百万文档要算清调用次数与耗时
维度不可随意改dimensions 一旦选定写入向量库，改维度 = 全量重算 Embedding + 重建索引
Query / Doc 同模型查询与文档必须用同一模型、同一维度，混用 v2/v3 或不同 dimension 会导致召回失真

闭源 API 约束数据经 DashScope 出网，敏感知识库需评估合规；无法本地私有化部署权重

效果验证不能只看官方指标，要用自有业务 Query 集测 recall@K / MRR，尤其专有名词、中英混合场景

技术选型优点

  • 中文检索效果好：C-MTEB 等中文榜单表现靠前，国内企业知识库、客服 FAQ 场景验证充分。
  • 与 Qwen 生态一体：LLM、Embedding、Rerank 均在百炼，同一 AK/SK、同一账单，Agent / RAG 链路集成快。
  • 免运维：无需自购 GPU、部署 bge 等开源模型，中小规模索引构建成本可控，开发效率高。
  • 可调维度降本：1024 默认，可降到 512/256 减少向量库存储与检索延迟，适合百万级文档。
  • 8192 上下文：单 Chunk 上限较高，配合合理 Chunk 策略可覆盖绝大多数段落级检索。
  • 多语言：50+ 语种，适合中英混合、东南亚多语言知识库。
  • 国内访问稳定：相比 OpenAI Embedding，网络延迟低、合规路径清晰，生产环境更易落地。

适用 / 不适用

更适合 不太适合
中文 RAG、DashScope 技术栈数据严禁出域、必须离线私有化
快速 MVP、中小规模知识库超大规模索引且 API 成本敏感（考虑自托管 bge / Qwen3-Embedding）
与 qwen-plus 等同平台组合已有 OpenAI 全栈且中文要求不高

注意：与 OpenAI 的 text-embedding-3-small/large 不同，百炼的 text-embedding-v3 是独立产品。RAG 中若 Chunk 超过 8192 Token 必须切分；切换维度或模型需全量重建索引。

面试回答：text-embedding-v3 是百炼 API 向量模型，参数量未公开；单条最长 8192 Token，默认 1024 维，可降到 768/512 等，批量每次最多 10 条。关注：与 OpenAI 同名不同物、Query/Doc 同模型同维度、改维度要重建索引、闭源数据出网。选型优点：中文效果好、DashScope 生态一体、免 GPU 运维、可调维降本、国内延迟低。用自有 recall@K 验证，大规模或强合规场景考虑自托管 Embedding。

向量维度对检索效果和成本有什么影响？

维度 效果 成本
低维（256–384）语义表达弱存储小、检索快
中维（768–1024）平衡较好主流选择
高维（1536+）细粒度语义好内存/延迟增加

维度由 Embedding 模型决定，不可随意更改。百万级文档下，1536 维 vs 768 维存储差约 2 倍。

余弦相似度、欧氏距离、点积有什么区别？

度量 公式特点 适用
余弦相似度只看方向，不看模长文本 Embedding 最常用
欧氏距离看绝对距离未归一化向量
点积方向 + 模长归一化后等价余弦

大多数 Embedding 模型输出已 L2 归一化，余弦与点积排序一致。选型时与向量库默认度量对齐。

向量检索和 BM25 区别是什么？

向量检索（Dense Retrieval）和 BM25（Sparse Retrieval）是 RAG 中最常见的两种召回方式，匹配逻辑不同，生产环境通常组合使用（Hybrid Search）。

核心对比

对比项 BM25 向量召回
本质关键词 / 词项匹配语义相似度匹配
擅长精确词、型号、错误码、字段名同义表达、自然语言语义
不擅长同义词、改写表达数字、编号、精确匹配
例子E01、A100、v1.2.3「无法启动」≈「开机失败」

工作原理简述

BM25：Query 分词 → 倒排索引查词频/逆文档频率 → 算 BM25 分数 → TopK
向量：Query Embedding → ANN 近邻搜索（余弦/点积）→ TopK

面试回答：BM25 做词项精确匹配，适合错误码、型号；向量检索做语义相似，适合同义改写。两者互补，生产 RAG 常用 Hybrid Search 融合后再 Rerank。

BM25 和普通关键词匹配有什么区别？

普通关键词匹配（如 SQL LIKE、简单 contains）往往是二元命中：文档包含关键词就返回，不考虑相关程度排序。

BM25（Best Matching 25）是经典的概率检索模型，为每个文档计算相关性分数并排序，是 Elasticsearch、Lucene 等搜索引擎的默认排序算法之一。

BM25 考虑的三个因素

  • 词频（TF）：词在文档中出现越多，一般越相关（但有饱和上限，不会无限加分）。
  • 逆文档频率（IDF）：词越稀有（全库越少文档含该词），权重越高；「的、是」等常见词权重低。
  • 文档长度归一化：文档太长会被适当惩罚，避免长文档仅仅因为词多而排名靠前。
普通关键词匹配：
  Query = "Redis 持久化"
  → 文档含 "Redis" AND "持久化" → 命中（无分数或固定分）

BM25：
  Query = "Redis 持久化"
  → 分别算 "Redis"、"持久化" 的 TF-IDF 贡献
  → 短且两词都高频出现的文档排更前
  → 返回按 BM25 分数排序的 TopK

对比

对比项 普通关键词匹配 BM25
排序通常无相关性排序，或按时间/ID按相关性分数排序
词频不考虑出现次数考虑 TF，但有饱和
稀有词与普通词同等对待IDF 加权，稀有词更重要
文档长度长文档易误命中长度惩罚，更公平

面试回答：普通匹配是「包含即命中」；BM25 进一步算相关性分数，综合词频、词稀有度和文档长度，返回排序后的 TopK，检索质量明显更好。

Hybrid Search 混合检索是什么？为什么要结合向量检索和 BM25？

Hybrid Search 融合稠密检索（语义）和稀疏检索（关键词），互补各自弱点。

检索方式 擅长 弱项
向量（Dense）语义相似、同义词精确编号、SKU
BM25（Sparse）精确关键词、专有名词语义理解弱

融合方式

  • RRF（Reciprocal Rank Fusion）：按排名倒数加权融合，不依赖分数尺度，最常用。
  • 加权分数：向量分 × α + BM25 分 × (1-α)，需归一化。
  • 融合后通常再经 Cross-Encoder Rerank 精排 TopN 进 LLM 上下文。

面试回答：向量补语义、BM25 补精确词，Hybrid Search 用 RRF 或加权融合双路召回，是生产 RAG 的标配方案。

RAG 中如何处理表格、图片、PDF 复杂版面？

类型 处理方式
PDF 复杂版面Unstructured、MinerU、LayoutLM 做版面分析
表格转 Markdown/HTML 或按行切片，保留表头 metadata
图片OCR + 多模态 Embedding 或 VLM 描述后入库

表格切忌纯文本乱序提取；保留行列结构才能正确回答「第 3 行销售额」类问题。

RAG 方案中图片是怎么处理的？

图片无法直接做文本 Embedding，RAG 处理图片的核心是把视觉信息转成可检索、可进 Prompt 的表示。生产环境通常组合多种路径，而非单一方案。

PDF/Word 中的图片
  ├── 路径 A：OCR 提取图中文字 → 文本 Chunk → 文本 Embedding
  ├── 路径 B：VLM 生成图片描述/Caption → 文本 Chunk → 文本 Embedding
  └── 路径 C：多模态 Embedding（CLIP 等）→ 图文统一向量空间检索

三条主流路径

路径 做法 优点 局限
OCRPaddleOCR / Tesseract 提取图中文字成本低，适合截图、扫描件、含大量文字的图片纯视觉图表（流程图、架构图）效果差
VLM 描述GPT-4V / Qwen-VL 生成结构化 Caption能理解图表语义、流程关系多一次模型调用，描述可能遗漏细节
多模态 EmbeddingCLIP / SigLIP / BGE-VL 直接编码图片支持「以图搜图」「图文混合检索」需专用向量库，文本问答召回未必最优

离线入库流程

  1. 版面分析：MinerU / Unstructured 识别 PDF 中图片区域，记录 page、bbox、figure_id。
  2. 图片存储：原图存对象存储（S3/OSS），Chunk metadata 存 image_url、figure_caption、page_num。
  3. 内容提取：文字型图片走 OCR；图表/架构图走 VLM 生成描述（Prompt 要求输出：标题、关键元素、数据趋势）。
  4. 索引：描述文本 + OCR 文本合并为 Chunk 做文本 Embedding；高价值图片额外存多模态向量。
  5. 关联：图片 Chunk 与相邻段落 Chunk 共享 doc_id、section_path，检索时可一并召回。

在线问答怎么处理？

  • 纯文本 Query：走文本向量检索，命中图片描述 Chunk 后进 Prompt；必要时把原图 URL 交给多模态 LLM 二次理解。
  • 用户上传图片 Query：多模态 LLM 直接理解 + 多模态向量检索知识库相似图/描述。
  • 混合 Prompt：检索到的文本 Chunk + 图片 URL 一起送入 GPT-4V / Qwen-VL，让模型「看图说话」。

生产落地建议

  • 架构图、流程图、数据图表：优先 VLM 描述 + OCR 补充，描述 Prompt 要求结构化（节点、箭头、数值）。
  • 截图类文档（PPT、工单）：OCR 为主，成本低。
  • 回答需展示原图时，metadata 保留 image_url，前端渲染引用卡片。
  • 评测：准备含图表的 QA 对，单独测 image recall@K，避免被纯文本指标掩盖。

面试回答：图片 RAG 本质是视觉→可检索表示：OCR 提文字、VLM 生成描述、或多模态 Embedding 统一检索；离线存原图+描述 Chunk，在线文本 Query 检描述、复杂场景用多模态 LLM 看图；按图片类型选路径，图表类优先 VLM。

图片向量和文字向量适合放到同一个向量数据库吗？

关键不在「能不能用同一个向量库产品」，而在向量是否来自同一语义空间、同一维度、同一距离度量。混放不可比的向量会导致检索结果无意义。

三种常见情况

情况 能否同一 Collection 说明
文本 Embedding + 图片描述/OCR 文本 Embedding✅ 可以同一文本模型（如 bge-m3）编码，维度一致，Query 也是文本
多模态统一模型（CLIP / BGE-VL）✅ 可以图文映射到同一向量空间，支持以文搜图、以图搜文
文本模型向量 + 独立图片模型向量❌ 不应混搜维度、分布、语义空间不同，余弦相似度没有可比性

生产推荐架构

# 方案 A（最常见）：主索引全是「可检索文本」
图片 → OCR / VLM Caption → 文本 Chunk → 文本 Embedding → text_collection

# 方案 B：多模态统一索引（以图搜图、图文互检）
图片 → CLIP Image Encoder  ─┐
文本 → CLIP Text Encoder   ─┴→ 同一 multimodal_collection（同维度、同模型）

# 方案 C：同一向量库实例，多 Collection 隔离
Milvus / Qdrant:
  ├── collection_text      # bge-large, dim=1024
  └── collection_image     # clip-vit, dim=512
在线按 Query 类型路由，或双路召回后 RRF 融合

同一 Collection 的前提条件

  • 相同 Embedding 模型版本：换模型必须全量 re-index，不可与旧向量混用。
  • 相同维度：如都是 768 或 1024，索引结构才能共用。
  • 相同归一化与距离度量：L2 归一化后一般用余弦/内积，全库一致。
  • metadata 标 modality：type=text|image|caption，检索时可过滤或加权。

为什么不建议硬混？

  • 文本 BGE 与图片 CLIP 向量数值相近不代表语义相关，TopK 会被某一模态「霸榜」。
  • 图片向量维度和文本不同，多数向量库一个 Collection 只支持一种 schema。
  • 纯文本问答场景，图片描述已进文本索引即可；单独图片向量索引 ROI 低。

选型建议

  • 文档问答为主：图片转 Caption/OCR → 文本向量，单文本索引最简单。
  • 以图搜图 / 电商识图：多模态统一模型 + 独立 multimodal 索引。
  • 两者都要：双 Collection 或 Hybrid 双路召回，metadata 关联同一 doc_id。

面试回答：同一向量库实例可以，同一 Collection 要看向量是否同空间同维度；文本+Caption 用同一文本模型可混放；CLIP 等多模态模型图文可统一索引；不同模型（BGE+CLIP）不能混搜，应分 Collection 或把图片先转文本再入文本索引。

RAG 如何做增量更新？

增量策略

  1. 文档变更检测：Webhook / 定时扫描 diff。
  2. 按 doc_id 删除旧 Chunk（软删除标记 is_deleted=true）。
  3. 重新解析、切片、Embedding 新内容。
  4. 写入向量库，更新 metadata 版本号。

避免全量重建；Embedding 模型变更时才需全量 re-index。用消息队列异步处理，保证最终一致。

RAG 如何处理文档版本冲突？

策略

  • metadata 存 version、effective_date、is_latest。
  • 同 doc_id 多版本共存，旧版本软删除而非物理删除。
  • 生成时 Prompt 要求「以最新版本为准」。

审计场景可保留历史版本供溯源，但默认检索只返回最新有效版。

RAG 如何设计引用和溯源？

设计要点

  • 每个 Chunk 带 doc_id、page、section、url。
  • 生成时要求标注 [1][2] 引用编号，映射到来源列表。
  • 前端展示可点击跳转原文。
上下文格式:
[1] 来源: 员工手册 v3.2 第5页
内容: 年假天数为...

回答: 根据规定，年假为15天 [1]。

日志记录每次检索命中和最终引用，支持审计回放。

RAG 检索召回不准如何排查？

排查顺序

  1. 确认问题：是检索没命中还是生成没用好？
  2. 查 Embedding：模型是否与索引一致？
  3. 查 Chunk：切片是否截断关键信息？
  4. 查 Query：是否需要 Rewrite？
  5. 查 TopK / 过滤：K 太小？权限误过滤？
  6. 查 Hybrid：是否缺 BM25 补关键词？
  7. 加 Rerank 对比前后 recall@K。

用离线评测集定位：retrieve 准但 answer 差 → 生成/Prompt 问题；retrieve 就不准 → 索引/切片问题。

Chunk overlap 如何设置？

Overlap 是相邻 Chunk 之间的重叠 Token 数，防止关键信息被切在边界上。

  • 常见：Chunk 512 Token，overlap 50–100（10%–20%）。
  • 技术文档、法律条文：overlap 可更大。
  • overlap 过大：冗余存储、检索重复命中。

用评测集对比 overlap=0 vs 10% vs 20% 的 recall@K 选最优值。

RAG 中 Chunk 父子块关联方案怎么设计？

父子块（Parent-Child / Small-to-Big）解决 RAG 经典矛盾：小块检索准、大块上下文全。思路是用小子块做向量索引提高召回精度，命中后回溯父块交给 LLM 生成，避免答案缺背景。

离线：文档 → 切父块（800–1500 Token）→ 再切子块（128–256 Token）
      子块 Embedding 入向量库，metadata 存 parent_id / child_index / doc_id
在线：Query → 检索 TopK 子块 → 按 parent_id 去重 → 取对应父块进 Prompt

为什么要父子分离？

  • 单一大块：Embedding 语义被稀释，TopK 命中率低。
  • 单一小块：召回精准但缺前后文，LLM 容易断章取义。
  • 父子关联：检索用「针」，生成用「线」，兼顾精度与完整性。

数据模型设计

层级 典型大小 是否 Embedding 关键 metadata
父块 Parent800–1500 Token可选（二次召回）parent_id, doc_id, section_path, version
子块 Child128–256 Token是（主检索单元）parent_id, child_index, doc_id, char_offset

在线检索流程

  1. 向量检索 TopK 子块（如 K=20）。
  2. 按 parent_id 去重，保留每个父块最高分子块。
  3. 从文档存储 / KV / 关系库加载父块原文（向量库只存子块向量 + metadata）。
  4. 可选 Rerank：对「Query + 父块摘要」精排，取 Top3–5 父块进 Prompt。
  5. 生成时引用子块命中位置，溯源到父块 doc_id + section。

常见实现模式

  • 双索引：子块向量库 + 父块对象存储（S3/MinIO/ES），通过 parent_id 关联。
  • 单索引 + 冗余：子块 metadata 内嵌父块全文（适合父块不大、QPS 低的场景）。
  • 多级块：Document → Section → Paragraph 三层，复杂手册/法规常用。
  • LangChain ParentDocumentRetriever：框架内置 child_splitter + parent_splitter + docstore 模式。

落地注意点

  • 父块更新时必须级联重建其下所有子块向量，按 doc_id 批量软删再写入。
  • 同一父块多个子块命中时，合并为一条上下文，避免 Token 浪费。
  • 表格/代码块尽量不切进子块中间，按结构边界切，父块保留完整结构。
  • 评测时分别看 child recall@K 和 parent 级 answer 质量，定位瓶颈在检索还是上下文组装。

面试回答：父子块用小子块做精准向量检索、父块提供完整上下文；metadata 用 parent_id 关联，在线检索后去重回溯父块再生成；适合长文档和技术手册，注意级联更新与结构边界切分。

Milvus 和 Chroma 技术选型比较？

两者都是 RAG 常用向量库，但定位不同：Chroma 偏轻量开发体验，Milvus 偏大规模生产部署。选型核心看数据规模、运维能力、延迟与可用性要求。

总览对比

对比项 Chroma Milvus
定位轻量 Embedding 数据库，开发友好云原生分布式向量数据库，面向生产
典型规模万～百万级向量百万～亿级向量
部署形态嵌入式（进程内）/ 单机 ServerMilvus Lite / Standalone / K8s 集群
运维复杂度低，pip install 即用集群模式需 etcd、对象存储等组件，中高
语言生态Python 优先，与 LangChain 深度集成Python / Java / Go SDK，多语言生产接入
索引算法HNSW 等（相对精简）HNSW、IVF、IVF_PQ、SCANN、GPU 索引等，可调参空间大
Hybrid 检索需应用层自行组合 BM252.x 支持 Sparse + Dense，Hybrid 能力更强
Metadata 过滤支持，小规模够用支持，大规模过滤性能更好
高可用单机为主，生产 HA 需自建原生分布式、副本、扩缩容，适合 SLA 场景

Chroma 特点

  • 上手极快：几行代码完成 Collection 创建、写入、检索，原型验证和 Agent 项目首选。
  • 嵌入式模式：与应用同进程运行，本地开发无需额外服务（如 agent-rag 本地 Chroma）。
  • LangChain 默认搭档：教程、示例多，团队学习成本低。
  • 局限：超大规模数据、高 QPS、多副本 HA 不是强项；索引与查询调优选项相对少。
import chromadb
client = chromadb.PersistentClient(path="./chroma_db")
col = client.get_or_create_collection("docs")
col.add(ids=["1"], documents=["示例"], metadatas=[{"source": "a.pdf"}])
hits = col.query(query_texts=["示例问题"], n_results=5)

Milvus 特点

  • 为规模而生：分布式架构，支持水平扩展，亿级向量检索有成熟实践。
  • 索引丰富：HNSW 低延迟、IVF_PQ 省内存，可按 recall-latency-cost 曲线调优。
  • 企业级能力：多租户、RBAC、监控、云托管（Zilliz Cloud），适合正式生产环境。
  • 局限：Standalone / Cluster 部署组件多，小团队维护成本高于 Chroma。
from pymilvus import connections, Collection
connections.connect(host="127.0.0.1", port="19530")

col.load()
results = col.search(
    data=[query_vector],
    anns_field="embedding",
    param={"metric_type": "COSINE", "params": {"nprobe": 10}},
    limit=5,
    expr='source == "a.pdf"',  # metadata 过滤
)

选型建议

场景 推荐 原因
本地开发 / Demo / 小型 AgentChroma零运维、集成快、数据量小
企业内部知识库 < 100 万向量Chroma Server 或 pgvector 也可够用且简单；要 Hybrid 再评估 Milvus
百万～亿级、高 QPS 在线检索Milvus分布式、索引调优、过滤性能
需要 K8s 部署、多副本 HAMilvus生产级扩缩容与容灾
已有 PostgreSQL 栈、向量量不大pgvector运维统一，不一定上专用向量库
从 Chroma 迁 Milvus评估数据量与 QPS 阈值通常因规模、延迟 SLA 或 HA 触发迁移，需全量 re-index

共同注意事项

  • 距离度量一致：余弦 / 内积 / L2 与 Embedding 是否归一化对齐。
  • 换模型 = 重建索引：维度或模型版本变化需全量重算向量。
  • Collection 规划：按租户 / 业务线 / 模态分 Collection，便于隔离与扩容。
  • 评测驱动：用 recall@K + P99 延迟 benchmark，而非只看功能清单。

面试回答：Chroma 轻量、Python 友好，适合原型和小规模 RAG（万～百万级），嵌入式部署几乎零运维；Milvus 是分布式向量库，适合百万～亿级、高 QPS、K8s 生产环境，索引和 Hybrid 能力更强但运维更重。选型看规模、SLA 和团队运维能力：Agent 本地开发选 Chroma，企业级大规模检索选 Milvus；两者都要注意 metric 一致和换模型全量重建。

Chroma vs Milvus：是否支持主从/副本/HA？

生产 RAG 选型时，除了检索性能，还要看故障切换、数据副本、跨机房容灾。Chroma 与 Milvus 在 HA 能力上差距明显：Milvus 原生分层 HA；Chroma 默认单机，HA 需 K8s 或读写分离自建。

架构示意

【Chroma 典型单机 — 无 HA】
  App ──→ Chroma Server（单实例）──→ SQLite / DuckDB + Parquet
          进程挂掉 → 检索中断，需人工重启

【Chroma 自建 HA — 应用层读写分离】
  App ──write──→ Chroma Primary
     └──read───→ Replica-1 / Replica-2（共享只读卷或定时同步）
  ⚠️ 副本一致性、写入单点需自行保障

【Milvus Cluster — 原生分层 HA】
  App ──→ Proxy ──→ QueryNode × N（查询副本，节点故障自动切换）
                 └→ DataNode（Segment 多副本 replication_factor≥2）
                 └→ etcd（元数据 HA）+ Object Storage（持久化）
  可选：Primary Cluster ──CDC(WAL)──→ Standby Cluster（跨集群容灾）

分项说明

HA 层级 Chroma Milvus
节点级（单节点宕机）嵌入式 / 单机 Server 无自动切换；K8s 可重启 Pod，仍有秒级中断Segment 多副本分布在不同 Query/Data Node，副本自动接管，RTO 接近实时
服务级（读写分离）社区常见方案：Primary 写 + Read Replica 读（共享 PV 或 rsync），需应用路由Proxy 负载均衡多个 Query Node，读写路径原生支持
集群级（整集群故障）无内置跨集群复制，需备份恢复或业务双活自建CDC 备集群：WAL 流式同步到 Standby，分钟级 RTO、秒级 RPO
数据安全网定期快照 / 备份 Parquet 目录对象存储持久化 + 周期 Backup，与 CDC 互补

Chroma Distributed 模式（补充）

Chroma 较新的 Distributed 模式（K8s 微服务：Frontend / Query / Compaction / Log / SysDB）面向大规模与 HA 设计，但部署和运维复杂度显著高于默认 Client-Server，生态成熟度仍不及 Milvus Cluster。

选型建议

SLA 要求 推荐
本地开发 / Demo，可接受停机Chroma 嵌入式 / 单机
小规模生产，99% 可用性，团队有 K8sChroma Server + K8s 重启策略，或读写分离自建
在线业务 99.9%+，节点故障零感知Milvus Cluster（replication_factor ≥ 2）
跨 AZ / 跨 Region 容灾Milvus CDC 备集群 或 Zilliz Cloud 托管

面试回答：Chroma 默认单机/嵌入式，无原生 segment 副本和主从，HA 要靠 K8s 重启或自建读写分离；Distributed 模式有 HA 设计但运维较重。Milvus 原生分层 HA：节点级 segment 多副本自动 failover，集群级 CDC 备集群跨机房容灾，生产 SLA 场景更合适。要 99.9%+ 在线检索选 Milvus；可停机的小 Agent 用 Chroma 即可。

HNSW 和 IVF 有什么区别？

索引 原理 特点
HNSW分层导航小世界图召回高、查询快；内存占用大
IVF聚类分桶，搜候选桶适合大规模；依赖 nprobe 调参

百万级以下 HNSW 常用；亿级可 IVF+PQ 压缩。无绝对最优，需 benchmark recall-latency 曲线。

RAGFlow 开源框架是什么？

RAGFlow 是 InfiniFlow 开源的 RAG 引擎（Retrieval-Augmented Generation Engine），把深度文档解析、知识库管理、混合检索、Agent 编排做成一体化产品，定位为 LLM 的「上下文层」，支持 Docker 自托管，也提供 API 供业务集成。

与 LangChain（代码组件库）不同，RAGFlow 更偏开箱即用的 RAG 平台；与 Dify 类似，但核心差异化在复杂文档理解（DeepDoc）——PDF/Word/表格/版式布局的解析与可视化切片是强项。

核心能力

模块 作用
DeepDoc 文档理解OCR、版式识别、表格结构识别；支持 PDF/Word/Excel/PPT 等，MinerU、Docling 等解析器可选
知识库（Dataset）上传文档 → 解析 → 切片 → Embedding → 索引；支持可视化 Chunk，人工核对切片质量
混合检索 + Rerank关键词 + 向量多路召回，融合重排（Rerank），提升召回精度
Agent / Workflow可视化编排 Chatbot、Agent、Workflow；支持 Agentic Workflow、MCP、代码执行节点
Ingestion Pipeline可编排的数据接入流水线，从 Notion / S3 / Confluence 等同步到索引
模型接入可配置 LLM、Embedding、Rerank 模型（OpenAI、DeepSeek、本地 Ollama 等）

  • 对象存储：MinIO（存原始文件）

  • 适合 RAGFlow：大量 PDF/扫描件/表格文档；需要看切片质量、调解析与召回；希望 RAG 链路开箱即用。
  • 不太适合：纯代码深度定制、与现有微服务强耦合、只需简单 Chatbot 且文档结构简单（Dify 可能更快）。
  • 生产注意：自托管需规划 ES/MinIO/MySQL 资源；敏感数据控制出网；复杂 Agent 仍要评测与权限隔离；GraphRAG / RAPTOR 等高级能力算力开销大。

面试回答：RAGFlow 是 InfiniFlow 开源的 RAG 引擎，核心亮点是 DeepDoc 深度文档解析和可视化知识库管理，内置 Hybrid 检索 + Rerank + Agent/Workflow，Docker 自托管。和 Dify 比，RAGFlow 更专注 RAG 与复杂文档；和 LangChain 比，它是平台化开箱即用，LangChain 是代码灵活。选型：复杂 PDF/企业知识库、重视切片与召回质量用 RAGFlow；通用低代码 AI 应用用 Dify；深度定制用 LangChain/LangGraph 自研。
