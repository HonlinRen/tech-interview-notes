11. Spring AOP 原理

Spring AOP 的核心原理是 动态代理。当一个 Bean 需要被切面增强时，Spring 容器里实际暴露的通常不是原始对象，而是代理对象。调用方调用的是代理方法，执行时会先经过切面逻辑，再进入目标方法。

调用方
-> 调用代理对象方法
-> 执行切面前置逻辑
-> 调用目标方法
-> 执行切面后置 / 异常 / 返回逻辑
-> 返回结果

核心概念

概念 含义
Join Point 连接点方法执行中可以插入切面的点，在 Spring AOP 里通常指方法执行
Pointcut 切点匹配哪些连接点需要增强，例如 execution(...)、@annotation(...)
Advice 通知在连接点上执行的具体逻辑，例如前置、后置、环绕
Aspect 切面切点 + 通知的组合
Target 目标对象被增强的原始 Bean
Proxy 代理对象容器最终暴露给调用方的对象

常见通知类型

  • @Before：前置通知，在目标方法执行之前执行，常用于参数校验、日志记录、权限校验。
  • @After：后置通知，在目标方法执行完成后执行，无论是否抛出异常都会执行，类似 finally。
  • @AfterReturning：返回通知，在目标方法正常返回后执行，可获取返回值。
  • @AfterThrowing：异常通知，在目标方法抛出异常时执行，常用于异常日志和监控。
  • @Around：环绕通知，可控制目标方法是否执行，是功能最强的一种通知。
@Aspect
@Component
public class LogAspect {

    @Around("execution(* com.example.service.*.*(..))")
    public Object log(ProceedingJoinPoint joinPoint) throws Throwable {
        System.out.println("before");
        Object result = joinPoint.proceed();
        System.out.println("after");
        return result;
    }
}

JDK 动态代理 vs CGLIB

对比项 JDK 动态代理 CGLIB
前提目标类实现了接口通过继承生成子类代理
实现方式基于接口方法拦截基于字节码生成子类
限制只能代理接口方法final 类或方法无法代理
默认策略有接口时优先使用无接口时使用

可通过 spring.aop.proxy-target-class=true 强制使用 CGLIB。Spring Boot 2.x 以后默认更倾向于类代理。

面试可补充：事务、缓存、异步、@Transactional 本质上都依赖代理 AOP 完成。自调用不经过代理，会影响这些增强生效。

12. Spring 事务管理方式

Spring 事务管理建立在 PlatformTransactionManager 抽象之上，常见实现包括 DataSourceTransactionManager、JpaTransactionManager、JtaTransactionManager。业务开发里主要区分两种使用方式：编程式事务和声明式事务。

方式 特点 适用场景
编程式事务代码显式控制提交和回滚局部事务、复杂分支控制
声明式事务通过注解或 AOP 自动织入常规业务方法，最常用

编程式事务

通过 TransactionTemplate 或 PlatformTransactionManager 手动控制提交和回滚。优点是控制精细，缺点是代码侵入性强、可读性较差。

@Autowired
private TransactionTemplate transactionTemplate;

public void createOrder() {
    transactionTemplate.execute(status -> {
        orderMapper.insert(order);
        inventoryMapper.deduct(skuId, count);
        return null;
    });
}

声明式事务

使用 @Transactional 标注类或方法，底层依赖 Spring AOP + 动态代理 + TransactionInterceptor。方法执行前开启事务，正常结束提交，异常时根据配置回滚。

@Transactional(rollbackFor = Exception.class)
public void createOrder() {
    orderMapper.insert(order);
    inventoryMapper.deduct(skuId, count);
}

声明式事务是项目中最常见的方式，因为它把事务边界和具体业务逻辑解耦。需要注意：方法必须是 public、避免自调用、异常被吞掉等情况可能导致事务不按预期生效。

13. @Transactional 失效场景

@Transactional 并不是给方法加一个数据库开关，而是依赖 Spring AOP 在代理层组织事务。因此失效场景通常和代理、异常、数据库事务支持有关。

常见失效原因

  1. 方法内部自调用：同类中 this.method() 调用不会经过代理对象。
  2. 方法不是 public：默认基于 Spring AOP，非 public 方法通常不会被事务增强。
  3. 异常被捕获未抛出：事务管理器感知不到异常，就不会回滚。
  4. 受检异常默认不回滚：默认只对 RuntimeException 和 Error 回滚。
  5. 数据库引擎不支持事务：例如 MyISAM。
  6. 对象不是 Spring Bean：手动 new 出来的对象没有代理。
  7. final 类或 final 方法：CGLIB 无法代理 final 方法。
  8. 多线程调用：子线程不会自动继承当前线程的事务上下文。

自调用示例

@Service
public class OrderService {

    public void create() {
        this.save(); // 直接调用，不经过代理，事务失效
    }

    @Transactional
    public void save() {
        // ...
    }
}

解决方式：注入自身代理、拆到另一个 Bean、使用 AopContext.currentProxy()，或改用编程式事务。

异常处理示例

@Transactional
public void pay() throws Exception {
    try {
        accountMapper.debit(userId, amount);
    } catch (Exception e) {
        log.error("pay failed", e); // 异常被吞掉，不会回滚
    }
}

如果业务异常需要回滚，应显式配置：

@Transactional(rollbackFor = Exception.class)

30. Spring Boot 自动配置原理？

Spring Boot 自动配置的目标是：根据 classpath、已有 Bean、配置属性等条件，自动装配常用组件，让开发者少写大量样板配置。

入口注解

@SpringBootApplication 是一个组合注解，核心包括：

  • @SpringBootConfiguration：标识配置类。
  • @EnableAutoConfiguration：开启自动配置。
  • @ComponentScan：扫描当前包及子包组件。
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

加载流程

Spring Boot 启动
-> 读取自动配置候选类
-> 根据条件注解判断是否生效
-> 注册满足条件的 Bean
-> 应用用户自定义配置覆盖默认值

常见条件注解

注解 作用
@ConditionalOnClassclasspath 中存在指定类时生效
@ConditionalOnMissingBean容器中不存在指定 Bean 时生效
@ConditionalOnProperty配置属性满足条件时生效
@ConditionalOnWebApplication当前是 Web 应用时生效

例如引入 spring-boot-starter-data-redis 后，classpath 存在 Redis 相关类，就会自动装配 RedisTemplate 等常用 Bean。如果用户自己定义了同名 Bean，默认配置会让位。

31. Spring Boot Starter 是什么？

Starter 是 Spring Boot 的依赖聚合 + 自动配置组合包。引入一个 starter，通常就能获得某类能力所需的依赖、默认配置和 Bean，而不必手动拼一堆 jar 和 XML。

常见 Starter

Starter 常见能力
spring-boot-starter-webSpring MVC、嵌入式 Tomcat、Jackson
spring-boot-starter-data-jpaJPA、Hibernate、数据源自动配置
spring-boot-starter-data-redisRedis 连接与模板类
spring-boot-starter-actuator健康检查、指标、监控端点

命名约定

  • 官方 starter 通常以 spring-boot-starter-* 命名。
  • 第三方 starter 通常以 *-spring-boot-starter 命名，避免和官方冲突。

Starter 的本质不是魔法，而是把“依赖集合”和“自动配置类”打包好，让应用按需引入即可使用。

32. Spring Bean 生命周期是什么？

Spring Bean 生命周期描述的是：对象从创建、初始化、可用，到容器关闭时销毁的完整过程。理解生命周期有助于理解 AOP 代理、初始化回调和循环依赖等问题。

核心流程

实例化 Bean
-> 属性填充（依赖注入）
-> Aware 回调（BeanNameAware、ApplicationContextAware 等）
-> BeanPostProcessor 前置处理
-> 初始化方法（@PostConstruct / InitializingBean / init-method）
-> BeanPostProcessor 后置处理（AOP 代理常在此阶段完成）
-> Bean 可使用
-> 容器关闭时执行销毁（@PreDestroy / DisposableBean / destroy-method）
@Component
public class DemoBean implements InitializingBean, DisposableBean {

    @PostConstruct
    public void init() {
        System.out.println("post construct");
    }

    @Override
    public void afterPropertiesSet() {
        System.out.println("after properties set");
    }

    @PreDestroy
    public void destroy() {
        System.out.println("pre destroy");
    }

    @Override
    public void destroy() {
        System.out.println("disposable bean destroy");
    }
}

面试重点：AOP 代理通常在 BeanPostProcessor 后置阶段生成，因此容器里最终拿到的常常是代理对象，而不是原始目标对象。

33. Spring Bean 作用域有哪些？

Bean 作用域决定 Spring 容器如何创建和管理 Bean 实例。默认是单例，在 Web 场景下还有请求级、会话级等作用域。

作用域 含义 典型场景
singleton容器内单例，默认作用域Service、Repository、配置类
prototype每次获取都创建新实例有状态对象、短生命周期对象
request每个 HTTP 请求一个实例请求上下文对象
session每个 HTTP Session 一个实例用户会话状态
applicationServletContext 级别共享全局 Web 共享对象
 
@Scope("prototype")
@Component
public class PrototypeBean {
}

注意：singleton Bean 注入 prototype Bean 时，prototype 不会每次都新建，需通过 @Lookup、ObjectProvider 等方式解决。

34. Spring MVC 请求处理流程？

Spring MVC 以前端控制器 DispatcherServlet 为核心，把选择处理器、调用方法、参数绑定、结果渲染或写出响应串成一条协作链路。

请求进入 DispatcherServlet
-> HandlerMapping 找到处理器
-> HandlerAdapter 适配并调用目标方法
-> 参数解析、类型转换、参数校验
-> 执行业务逻辑
-> 返回 ModelAndView 或响应体
-> ViewResolver 渲染视图，或 HttpMessageConverter 写出 JSON

核心组件

组件 职责
DispatcherServlet前端控制器，统一调度请求
HandlerMapping根据 URL 和注解映射到处理器
HandlerAdapter调用具体 Controller 方法
HandlerMethodArgumentResolver解析方法参数
HttpMessageConverter读写 JSON、XML 等消息体
ViewResolver解析视图并渲染页面

REST API 场景下，常见写法是 @RestController，返回值由 HttpMessageConverter 直接序列化为 JSON，不再走传统视图解析。

35. Spring Boot 配置文件优先级怎么理解？

Spring Boot 支持多来源配置，优先级高的会覆盖优先级低的。面试时要能解释“为什么某个 yml 不生效”“为什么线上环境变量能覆盖本地配置”。

常见配置来源

  • 命令行参数
  • Java 系统属性 System.getProperties()
  • 操作系统环境变量
  • application-{profile}.yml
  • application.yml
  • 默认属性

工程实践

  • 默认配置放在 application.yml。
  • 环境差异放在 application-dev.yml、application-prod.yml。
  • 敏感信息通过环境变量、密钥系统或配置中心注入。
  • 部署环境可通过环境变量覆盖数据库地址、Redis 地址等。
spring:
  profiles:
    active: dev

---
spring:
  config:
    activate:
      on-profile: prod
server:
  port: 8080

面试可补充：Spring Boot 2.4 之后推荐用 spring.config.import 引入外部配置，而不是过度依赖复杂的加载顺序记忆。

59. Spring Boot 2 升级到 3 有哪些变化？

Spring Boot 3 不是简单版本号升级，而是建立在 JDK 17+、Spring Framework 6、Jakarta EE 10 之上的一次大版本迁移。

核心变化

变化点 说明
最低 JDK要求 JDK 17 及以上
基础框架依赖 Spring Framework 6
命名空间javax.* 迁移到 jakarta.*
自动配置注册推荐 AutoConfiguration.imports 替代 spring.factories
可观测性增强 Micrometer、Tracing 支持
Native 支持增强 GraalVM Native Image 能力

升级检查清单

  • 确认 JDK 版本满足要求。
  • 检查第三方依赖是否支持 Spring Boot 3 / Jakarta EE。
  • 替换代码中的 javax.servlet、javax.persistence 等包引用。
  • 检查自定义 starter 和自动配置注册方式。
  • 回归 Web、参数校验、安全框架等关键链路。

60. javax 到 jakarta 是什么变化？

Java EE 移交 Eclipse 基金会后更名为 Jakarta EE，很多规范包名从 javax.* 改为 jakarta.*。Spring Boot 3 全面基于 Jakarta 命名空间。

常见迁移示例

旧包名 新包名
javax.servlet.*jakarta.servlet.*
javax.persistence.*jakarta.persistence.*
javax.validation.*jakarta.validation.*
javax.annotation.*jakarta.annotation.*

这不只是文本替换，还涉及 Servlet、Validation、Persistence 等依赖版本兼容。如果中间件库仍依赖旧 javax API，升级时可能出现编译失败或运行时冲突。

61. AutoConfiguration.imports 是什么？

Spring Boot 2.7 引入、Spring Boot 3 推荐使用 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 来声明自动配置类，替代旧的 spring.factories 方式。

文件示例

com.example.autoconfigure.DemoAutoConfiguration
com.example.autoconfigure.RedisExtraAutoConfiguration

每一行写一个自动配置类的全限定名。相比 spring.factories，这种方式更聚焦、可读性更好，也更符合新版本自动配置加载机制。

旧方式对比

# META-INF/spring.factories（旧）
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.DemoAutoConfiguration

自定义 starter 在 Spring Boot 3 及以上应优先使用 AutoConfiguration.imports。

62. Spring 事务传播机制是什么？

事务传播机制定义：当一个事务方法调用另一个事务方法时，Spring 如何处理现有事务上下文。通过 @Transactional(propagation = ...) 配置。

传播行为 含义 典型场景
REQUIRED有事务就加入，没有就新建默认，最常用
REQUIRES_NEW挂起当前事务并新建独立事务审计日志、独立保存
NESTED嵌套事务，基于保存点部分回滚
SUPPORTS有事务就加入，没有就非事务执行查询类方法
NOT_SUPPORTED挂起当前事务，非事务执行不需要事务的操作
MANDATORY必须在事务中运行，否则异常强制依赖外层事务
NEVER不能在事务中运行，否则异常明确禁止事务
 
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void writeAuditLog() {
    auditMapper.insert(log);
}

常见场景：REQUIRED 和 REQUIRES_NEW 的区别是前者共享同一事务，后者子方法失败不会导致外层整体回滚，反之亦然。

194. Spring IoC 是什么？

IoC（Inversion of Control，控制反转） 是一种设计思想：对象的创建和装配不再由业务代码自己控制，而是交给 Spring 容器统一管理。

传统写法里，对象之间通过 new 耦合；Spring IoC 通过容器管理 Bean，在运行时完成依赖注入，从而降低耦合、提升可测试性和可扩展性。

IoC 容器做什么

  • 扫描并注册 Bean 定义。
  • 创建对象实例。
  • 完成依赖注入。
  • 管理 Bean 生命周期。
  • 提供扩展点，例如 BeanPostProcessor、FactoryBean、事件机制。
@Service
public class OrderService {
    private final OrderMapper orderMapper;

    public OrderService(OrderMapper orderMapper) {
        this.orderMapper = orderMapper;
    }
}

上面这种构造器注入就是 IoC 的典型体现：OrderService 不自己创建 Mapper，而是容器注入。

面试可补充：IoC 是思想，DI 是实现方式。Spring 的核心容器接口是 BeanFactory，实际开发常用 ApplicationContext。

195. DI 依赖注入有哪些方式？

DI（Dependency Injection，依赖注入） 是 IoC 的具体实现方式，本质是把对象所依赖的其他对象，在对象创建时注入进来。

三种常见注入方式

方式 写法 优点 缺点
构造器注入构造函数参数依赖明确、便于测试、推荐依赖多时构造函数较长
Setter 注入setXxx()可选依赖灵活对象可能存在不完整状态
字段注入@Autowired 直接注字段写法简单不利于测试，隐藏依赖
 
@Service
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

Spring 官方更推荐构造器注入，因为依赖在对象创建时就确定，且方便写单元测试。Setter 注入适合可选依赖，字段注入虽然常见，但工程上更偏向构造器注入。

196. Spring 中有哪些常用注解？

Spring 注解可以按职责分成组件注册、依赖注入、Web 开发、配置和条件装配等几类。面试时既要会分类，也要知道各自解决什么问题。

分类 常见注解 作用

197. @Component、@Service、@Repository、@Controller 有什么区别？

这四个注解本质上都是组件扫描注册，底层都是把类注册成 Spring Bean，区别主要在于语义分层和额外处理能力。

注解 语义 说明
@Component通用组件最基础的组件注解
@Service业务服务语义上表示业务逻辑层
@Repository数据访问层会转换持久化异常为 Spring 的 DataAccessException
@Controller控制层标识 Spring MVC 控制器

从功能上看，它们都会让类被扫描成 Bean；从工程规范看，用不同注解是为了保持分层职责清晰。REST 项目里常见的是 @RestController，它等于 @Controller + @ResponseBody。

198. @Autowired 和 @Resource 有什么区别？

两者都用于依赖注入，但来源、默认匹配规则和习惯用法不同。

对比项 @Autowired @Resource
来源Spring 提供JDK javax.annotation / Jakarta 注解
默认装配方式按类型（byType）按名称（byName）
required 属性支持 required = false无该属性
多 Bean 冲突配合 @Qualifier可通过 name 指定
 
@Autowired
@Qualifier("redisTemplate")
private RedisTemplate<String, Object> redisTemplate;

@Resource(name = "redisTemplate")
private RedisTemplate<String, Object> redisTemplate2;

Spring 项目里更常见 @Autowired，构造器注入场景更推荐它；如果需要明确按 Bean 名称注入，@Resource 也很直观。

200. Spring 事件机制是什么？

Spring 事件机制是基于观察者模式的应用，用于解耦主业务流程和附属处理，例如发送通知、写日志、刷新缓存，避免主业务代码硬编码耦合。

核心组件

  • ApplicationEvent：事件对象。
  • ApplicationListener 或 @EventListener：事件监听器。
  • ApplicationEventPublisher：事件发布器。
public class OrderCreatedEvent extends ApplicationEvent {
    public OrderCreatedEvent(Object source) {
        super(source);
    }
}

@Component
public class SmsListener {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        // 发送通知
    }
}

默认事件是同步执行的，如果希望解耦主流程，可以配合 @Async 异步处理。Spring Boot 也支持更轻量的业务事件模型，但核心思想都是“发布事件，让监听者自行处理”。

201. Spring 中 FactoryBean 和普通 Bean 有什么区别？

普通 Bean 是容器直接管理的对象；FactoryBean 是一个用来生产其他对象的特殊 Bean。

区别

对比项 普通 Bean FactoryBean
容器中的对象就是业务对象本身是工厂对象
获取结果直接注入使用通过 getObject() 得到目标对象
典型场景Service、Repository复杂对象创建，如 MyBatis 的 SqlSessionFactoryBean
 
public class DemoFactoryBean implements FactoryBean<Connection> {
    @Override
    public Connection getObject() {
        return createConnection();
    }

    @Override
    public Class<?> getObjectType() {
        return Connection.class;
    }
}

注意：如果想从容器拿到 FactoryBean 工厂对象本身，而不是它生产的对象，需要在 Bean 名前加 &，例如 &demoFactoryBean。

202. Spring MVC 的 HandlerMapping 和 HandlerAdapter 分别做什么？

两者都是 DispatcherServlet 的协作组件，但分工不同：HandlerMapping 负责找谁处理，HandlerAdapter 负责怎么调用。

HandlerMapping

根据请求 URL、HTTP 方法、注解等信息，找到对应的处理器对象。例如把 @RequestMapping("/users") 映射到某个 Controller 方法。

HandlerAdapter

由于处理器实现方式可能不同，适配器提供统一调用入口。它负责调用 Controller 方法，并完成校验、参数解析和返回值处理。

请求 /users/1
-> HandlerMapping 找到 UserController#getUser
-> HandlerAdapter 适配并调用方法
-> 参数解析器把路径变量绑定到方法参数
-> 返回值经 HttpMessageConverter 转成 JSON

这种“映射”和“调用”分离设计，是 Spring MVC 可扩展、可支持多种处理器类型的关键。

203. Spring Boot 为什么可以内嵌 Tomcat？

传统 Java Web 需要把应用打成 WAR 部署到外部 Tomcat。Spring Boot 把 Servlet 容器作为依赖嵌入应用进程，实现jar 包直接运行。

实现思路

  • 引入 spring-boot-starter-web，其中包含嵌入式 Tomcat 依赖。
  • 自动配置类创建嵌入式 Web 服务器工厂，例如 TomcatServletWebServerFactory。
  • 应用启动时，Spring Boot 启动并配置嵌入式 Tomcat。
  • Spring MVC 的 DispatcherServlet 注册到容器中。
SpringApplication.run(Application.class, args);
// 内部会创建 WebServer 并启动嵌入式 Tomcat

这样做的好处是部署简单、环境一致，更适合微服务和容器化。但代价是单个应用进程同时承担业务和 Web 容器职责，需要关注线程池、连接数、优雅停机等运维细节。

204. Spring Boot Actuator 常用于哪些监控场景？

Actuator 提供生产可用的监控和管理端点，用于观察应用健康状态、运行指标和关键运行时信息。

常见端点

端点 作用
/actuator/health健康检查，K8S 探针常用
/actuator/metrics查看 JVM、HTTP、线程池等指标
/actuator/info应用版本、构建信息
/actuator/env查看环境变量和配置
/actuator/loggers动态调整日志级别

生产环境通常只暴露必要端点，并配合 Spring Security、鉴权、Prometheus、Grafana 使用。不要把敏感端点直接无保护暴露到公网。

205. Spring Boot 如何加载外部配置？

Spring Boot 通过 Environment 抽象统一管理配置，来源可以是本地文件、环境变量、命令行参数、配置中心等。

常见方式

  • application.yml / application.properties
  • 操作系统环境变量，如 SPRING_DATASOURCE_URL
  • 命令行参数 --server.port=9090
  • spring.config.import 引入外部配置
  • 配置中心，例如 Nacos、Apollo
spring:
  config:
    import: optional:nacos:application-prod.yaml

工程上建议：默认配置放本地文件，环境差异用 profile，敏感信息走环境变量或配置中心，启动日志里确认最终生效配置。

206. Spring Boot 如何实现条件装配？

条件装配是自动配置的核心机制，本质是：只有满足条件时，才把某个组件或 Bean 注册到容器。

常见条件注解

  • @ConditionalOnClass：classpath 有某类时生效。
  • @ConditionalOnMissingBean：容器里没有某 Bean 时生效。
  • @ConditionalOnProperty：配置属性满足条件时生效。
  • @ConditionalOnWebApplication：Web 应用环境下生效。
@Configuration
@ConditionalOnClass(RedisTemplate.class)
@ConditionalOnProperty(prefix = "demo.redis", name = "enabled", havingValue = "true")
public class RedisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public RedisTemplate<String, Object> redisTemplate() {
        return new RedisTemplate<>();
    }
}

正是 starter 能“按需加载配置”、自定义 Bean 能“让位”的原因，也是 Spring Boot 约定优于配置的关键实现方式。

207. Spring Boot 自定义 Starter 如何设计？

自定义 Starter 的目标，是把某块通用能力封装成“引入依赖即可使用”的模块，通常拆分 autoconfigure 模块 + starter 模块。

设计步骤

  1. 定义配置属性类，例如 @ConfigurationProperties(prefix = "demo.xxx")。
  2. 编写自动配置类，通过条件注解控制是否装配。
  3. 在 AutoConfiguration.imports 中注册自动配置类。
  4. starter 模块聚合依赖，对使用者暴露一个坐标。
  5. 补充文档、默认值，保证开箱可用。
demo-spring-boot-autoconfigure
demo-spring-boot-starter

好的 Starter 应该提供合理默认配置、允许覆盖、边界清晰，不要把业务工程细节硬编码进 Starter。

208. Spring Boot 如何做优雅停机？

优雅停机的目标是：应用收到停止信号后，不再接受新请求，处理完已有请求，再释放线程池、连接池等资源，避免请求中断和数据不一致。

关键配置

server:
  shutdown: graceful

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s

停机流程

收到 SIGTERM / 停止命令
-> 停止接收新请求
-> 等待已有请求处理完成
-> 关闭线程池、消息消费者、数据库连接
-> 进程退出

在 K8S 中还要配合 preStop 钩子、合理设置 terminationGracePeriodSeconds，并注意消费者、定时任务、长连接资源是否在关闭前释放。

209. MyBatis 是什么？

MyBatis 是一个半自动 ORM 持久层框架，它把 JDBC 的重复代码（连接获取、参数设置、结果映射、资源关闭）封装起来，同时保留开发者对 SQL 的完全控制权。

与 Hibernate 不同，MyBatis 不会自动跟踪对象状态生成复杂 SQL，而是强调 SQL 与 Java 对象解耦、映射关系清晰、复杂查询可控。

核心组件

  • SqlSessionFactory：创建 SqlSession 的工厂。
  • SqlSession：执行 SQL 的会话对象。
  • Mapper 接口：声明数据访问方法。
  • Mapper XML：编写 SQL 和结果映射。
public interface UserMapper {
    User selectById(@Param("id") Long id);
}

在国内业务系统中，MyBatis 很常见，因为报表、联表、复杂查询多，手写 SQL 更便于审查和优化。

210. MyBatis 和 Hibernate 有什么区别？

MyBatis 是半自动 ORM，更强调 SQL 可控；Hibernate 是全自动 ORM，更强调对象关系映射和实体状态管理。

对比项 MyBatis Hibernate
SQL 控制手写 SQL，更可控多数 SQL 自动生成
学习成本相对低，接近 JDBC相对高，要理解实体状态
复杂查询更直观复杂 SQL 可能不直观
性能优化容易定位和调 SQL需要理解一级/二级缓存、懒加载
适用场景复杂业务 SQL、报表系统领域模型驱动、标准 CRUD

面试可总结：简单 CRUD Hibernate 省心，复杂 SQL 和性能优化 MyBatis 更可控。

211. #{} 和 ${} 有什么区别？

#{} 是预编译参数绑定，${} 是字符串替换。前者更安全，后者容易 SQL 注入。

对比项 #{} ${}
处理方式生成 ? 占位符直接拼接到 SQL 中
安全性高，参数按值处理低，容易被注入
适用位置where 条件、参数值动态表名、排序字段
 
<select id="selectByName">
  SELECT * FROM user WHERE name = #{name}
</select>

<select id="selectByTable">
  SELECT * FROM ${tableName}
</select>

结论很明确：能用 #{} 就不用 ${}；如果必须用 ${}，一定要做白名单校验。

212. MyBatis 如何防止 SQL 注入？

MyBatis 防止 SQL 注入的关键是：参数值使用预编译，结构片段使用白名单，而不是把用户输入直接拼进 SQL。

安全实践

  • 查询条件、参数值统一使用 #{}。
  • 动态表名、排序字段若使用 ${}，必须后端枚举或白名单映射。
  • 不要把前端传入的排序字段直接传入 Mapper。
  • 数据库使用最小权限账号，限制破坏范围。
private static final Set<String> ALLOWED_ORDER = Set.of("id", "create_time");

public String safeOrder(String orderBy) {
    return ALLOWED_ORDER.contains(orderBy) ? orderBy : "id";
}

面试可强调：MyBatis 本身不是绝对安全，错误使用 ${} 一样会注入。

213. MyBatis 一级缓存和二级缓存是什么？

MyBatis 缓存分为一级缓存和二级缓存。一级缓存是 SqlSession 级别，默认开启；二级缓存是 Mapper namespace 级别，默认关闭。

一级缓存

同一个 SqlSession 中，相同查询条件会命中本地缓存。执行增删改、调用 clearCache()、提交或关闭会话后，一级缓存会失效。

二级缓存

跨 SqlSession 共享，需要在 Mapper 上显式开启和配置。生产环境要谨慎使用，因为容易出现脏读和分布式一致性问题。

缓存级别 作用范围 默认状态
一级缓存SqlSession开启
二级缓存Mapper namespace关闭

实际项目里，热点数据更常用 Redis 等外部缓存，而不是重度依赖 MyBatis 二级缓存。

214. MyBatis 插件机制是什么？

MyBatis 插件基于拦截器 + 动态代理，可以在 SQL 执行的关键环节做增强，例如分页、审计、慢 SQL 监控。

可拦截的四大对象

  • Executor
  • StatementHandler
  • ParameterHandler
  • ResultSetHandler
@Intercepts({
    @Signature(type = Executor.class, method = "query",
        args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})
public class SlowSqlInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = invocation.proceed();
        long cost = System.currentTimeMillis() - start;
        // 记录慢 SQL
        return result;
    }
}

PageHelper 就是典型的 MyBatis 插件应用。

215. MyBatis 动态 SQL 是什么？

动态 SQL 用于根据条件动态拼接 SQL 语句，避免在 Java 代码里手工拼接字符串，既提升可读性，也降低出错概率。

常用标签

  • <if>：条件判断
  • <choose>、<when>、<otherwise>：多分支选择
  • <where>：自动处理 where 和 and
  • <set>：自动处理 set 和逗号
  • <foreach>：处理 in 列表、批量插入
<select id="searchUser">
  SELECT * FROM user
  <where>
    <if test="name != null and name != ''">
      AND name = #{name}
    </if>
    <if test="status != null">
      AND status = #{status}
    </if>
  </where>
</select>

动态 SQL 仍然要保证参数绑定安全，参数值优先使用 #{}。

216. MyBatis Mapper 接口为什么不需要实现类？

因为 MyBatis 在运行时通过JDK 动态代理为 Mapper 接口生成实现对象。调用接口方法时，代理会根据接口全限定名、方法名、参数找到对应的 Mapped Statement 并执行 SQL。

Mapper 接口方法调用
-> JDK 动态代理拦截
-> 定位 MappedStatement
-> SqlSession 执行 SQL
-> 结果映射为 Java 对象
public interface UserMapper {
    User selectById(@Param("id") Long id);
}

这也是为什么 Mapper 方法名、参数注解要和 XML 或注解 SQL 配置保持一致。Spring Boot 里通常通过 @MapperScan 把代理对象注册成 Bean。

217. MyBatis 延迟加载是什么？

延迟加载是指在查询主对象时，不立即加载关联对象，而是在真正访问关联属性时才触发额外查询。目标是减少不必要的联表或附加查询。

<setting name="lazyLoadingEnabled" value="true"/>
<setting name="aggressiveLazyLoading" value="false"/>

优点与风险

  • 优点：减少一次查询里无关关联数据的加载。
  • 风险：容易触发 N+1 查询问题。
  • 风险：在事务关闭后访问懒加载属性可能被报错。

如果明确知道业务会用到关联数据，通常直接 join 查询或分步批量查询更可控、可重复。

218. MyBatis 如何处理 N+1 查询问题？

N+1 问题指：先查 1 次主表得到 N 条记录，再对每条记录额外查 1 次关联数据，最终变成 N+1 次 SQL。常见于循环调用 Mapper 时。

常见解决方案

  • 使用 join 一次查出需要的数据。
  • 先查主表 ID 列表，再 where id in (...) 批量查关联数据。
  • 在业务层封装 map，避免循环里单条查询。
  • 谨慎使用懒加载，配合 SQL 监控定位问题。
List<Long> orderIds = orders.stream().map(Order::getId).toList();
List<OrderItem> items = orderItemMapper.selectByOrderIds(orderIds);
Map<Long, List<OrderItem>> itemMap = items.stream()
    .collect(Collectors.groupingBy(OrderItem::getOrderId));

面试回答时最好补充：不是只说“用 join”，还要强调批量查询、结果组装和 SQL 可观测性。
