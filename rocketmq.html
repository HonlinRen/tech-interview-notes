47. RocketMQ 核心组件有哪些？

RocketMQ 是典型的分布式消息中间件，整体架构围绕“路由发现、消息存储、消息投递”展开。理解各组件职责，是回答可靠性、顺序、事务、高可用等问题的基础。

组件 职责 说明
NameServer路由注册与发现无状态、轻量级，Broker 和客户端都向它注册/拉取路由信息
Broker消息存储与转发核心节点，负责接收、持久化、投递消息，支持主从部署
Producer消息生产者发送消息到指定 Topic，可选择队列、顺序键、事务消息等
Consumer消息消费者以 Push 或 Pull 方式消费，按 ConsumerGroup 分摊队列
Topic消息主题业务分类维度，例如 order_topic、payment_topic
MessageQueue主题下的队列负载均衡和顺序消费的最小单位，一个 Topic 通常包含多个 Queue
ConsumerGroup消费者组同组内分摊消费，不同组之间消费进度独立

消息流转路径

Producer
-> 从 NameServer 获取 Topic 路由
-> 选择 Broker / MessageQueue 发送消息
-> Broker 写入 CommitLog 并建立 ConsumeQueue 索引
-> Consumer 从 NameServer 获取路由
-> 拉取 / 推送消息并更新 Offset

与 Kafka 的架构差异（简要）

  • RocketMQ 用 NameServer 做路由中心，Kafka 用 ZooKeeper / KRaft 管理元数据。
  • RocketMQ 的 Broker 同时承担存储和队列索引（CommitLog + ConsumeQueue），Kafka 以 Partition 日志为核心。
  • RocketMQ 原生支持事务消息、延迟消息、死信队列，更偏业务消息场景。

48. RocketMQ 如何保证消息不丢？

消息丢失通常发生在生产、Broker 存储、消费三个阶段。RocketMQ 默认更接近“至少一次投递”，要真正做到不丢，需要三个阶段都配置正确，且消费端具备幂等能力。

三阶段保障

阶段 丢失场景 保障手段
生产端发送超时、网络抖动、Broker 未确认同步发送 + 检查 SendResult，失败重试，必要时落库补偿
Broker机器宕机、磁盘损坏、主从未复制同步刷盘 + 同步主从复制，多副本部署
消费端业务未处理完就提交 Offset业务成功后再返回 CONSUME_SUCCESS，失败返回重试

生产端

  • 关键业务优先使用同步发送，确认 Broker 返回 SEND_OK 后再认为发送成功。
  • 异步发送需实现回调，失败时记录日志并补偿重发。
  • 对极高可靠场景，可配合本地消息表：先写 DB 再发 MQ，定时任务扫描未发送记录。
SendResult result = producer.send(msg);
if (result.getSendStatus() != SendStatus.SEND_OK) {
    // 记录失败，触发重试或告警
}

Broker 端

配置项 选项 可靠性 性能
刷盘策略 flushDiskTypeSYNC_FLUSH / ASYNC_FLUSH同步刷盘更高异步刷盘更快
主从复制 brokerRoleSYNC_MASTER / ASYNC_MASTER同步复制更高异步复制更快

最高可靠组合是同步刷盘 + 同步主从复制，但吞吐和延迟会明显下降，一般只在金融、支付等强一致场景使用。

消费端

  1. 业务逻辑处理成功后再返回消费成功，不要先提交 Offset 再执行业务。
  2. 消费失败返回 RECONSUME_LATER，让 Broker 进入重试队列。
  3. 必须做幂等，因为重试和网络抖动会导致重复消费。

49. RocketMQ 顺序消息怎么实现？

RocketMQ 的顺序消息分为全局顺序和分区顺序（局部顺序）。实际业务中几乎都用局部顺序，例如同一订单、同一用户的消息按发送顺序消费。

实现原理

顺序消息的核心约束是：同一业务键的消息进入同一个 MessageQueue，且该 Queue 只被一个 Consumer 线程顺序消费。

orderId = 10001
-> Producer 按 orderId 选择固定 MessageQueue
-> 该 Queue 内消息天然有序
-> Consumer 对该 Queue 单线程顺序拉取

生产端：选择队列

producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Long orderId = (Long) arg;
        int index = (int) (orderId % mqs.size());
        return mqs.get(index);
    }
}, orderId);

消费端：顺序消费

  • 使用 MessageListenerOrderly，RocketMQ 会对每个 Queue 加锁，保证同一时刻只有一个线程消费该 Queue。
  • 消费失败会暂停当前 Queue 的消费，而不是立刻跳过，避免乱序。
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(
            List<MessageExt> msgs, ConsumeOrderlyContext context) {
        // 按 Queue 顺序处理
        return ConsumeOrderlyStatus.SUCCESS;
    }
});

全局顺序 vs 局部顺序

类型 做法 吞吐 适用场景
全局顺序Topic 只设 1 个 Queue，单消费者最低极少使用，如全局流水账
局部顺序按业务键哈希到固定 Queue较高订单状态流转、账户流水

注意点

  • Consumer 实例数不应超过 Queue 总数，否则部分实例空闲，且可能影响顺序分摊。
  • Broker 宕机、Queue 迁移时可能短暂影响顺序，需要业务侧容忍或补偿。
  • 顺序消费失败会阻塞该 Queue，需快速处理异常，避免“一条 poison message 卡死整队列”。

50. RocketMQ 事务消息原理？

事务消息用于解决本地事务执行与消息发送的一致性问题，典型场景是“创建订单后通知下游扣库存/发券”，要求两者最终一致。

完整流程

1. Producer 发送 Half Message（半消息）到 Broker
2. Broker 存储半消息，但暂不投递给消费者
3. Producer 执行本地事务（如写订单表）
4. Producer 向 Broker 发送 Commit 或 Rollback
   - Commit：半消息变为正常消息，可被消费
   - Rollback：删除半消息，消费者永远看不到
5. 若 Producer 未响应（宕机/超时），Broker 定时回查 Producer 本地事务状态
6. Producer 实现 checkLocalTransaction 返回最终状态

半消息的作用

半消息对消费者不可见，避免“本地事务还没提交，下游已经消费”的问题。只有本地事务成功后，消息才会真正投递。

代码骨架

TransactionListener listener = new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            orderService.createOrder(arg);
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // Broker 回查：查 DB 订单是否存在
        return orderService.exists(msg.getKeys())
            ? LocalTransactionState.COMMIT_MESSAGE
            : LocalTransactionState.ROLLBACK_MESSAGE;
    }
};

一致性级别

  • 保证的是最终一致性，不是强一致。
  • 回查机制解决 Producer 宕机导致状态未知的问题，但回查有延迟。
  • 消费者仍需幂等，因为 Commit 后消息可能被重复投递。

与本地消息表对比

方案 优点 缺点
RocketMQ 事务消息框架原生支持，链路清晰依赖 MQ 实现，回查逻辑需业务实现
本地消息表不依赖 MQ 特性，通用需要额外扫表任务，开发成本高

70. CommitLog 和 ConsumeQueue 是什么？

RocketMQ 的存储设计借鉴了 Kafka 的顺序写思想，但做了差异化：所有消息统一写入 CommitLog，再异步构建面向消费的 ConsumeQueue 索引。

CommitLog

  • Broker 上所有 Topic 的消息都顺序追加写入同一个 CommitLog 文件。
  • 顺序写磁盘（或 PageCache + 异步刷盘）获得高写入吞吐。
  • 每条消息在 CommitLog 中有唯一的物理偏移量（offset）。

ConsumeQueue

  • 按 Topic + QueueId 维度建立的轻量索引，不存完整消息体。
  • 每条索引约 20 字节，包含：CommitLog offset、消息 size、Tag hashcode。
  • 消费者按 ConsumeQueue 定位消息，再到 CommitLog 读取完整内容。

读取流程

Consumer 请求 Topic=order, QueueId=2, Offset=100
-> 读 ConsumeQueue[order-2][100] 得到 CommitLog 物理位置
-> 从 CommitLog 读取完整消息体
-> 返回给消费者

IndexFile（补充）

除 ConsumeQueue 外，RocketMQ 还有 IndexFile，按消息 Key 建立哈希索引，支持按 Key 快速查询消息，常用于消息追踪和问题排查。

设计优势

设计点 好处
统一 CommitLog 顺序写避免多 Topic 多文件随机写，提升 IO 效率
ConsumeQueue 轻量索引消费定位快，索引文件小，加载快
CommitLog 与索引分离写入和消费解耦，互不影响

71. ConsumerGroup 是什么？

ConsumerGroup（消费者组）是 RocketMQ 消费模型的核心概念，用于定义一组消费者的协作关系和消费进度隔离边界。

核心规则

  • 同一 ConsumerGroup 内的多个 Consumer 实例分摊 Topic 下的 MessageQueue，每条消息只被组内一个实例消费。
  • 不同 ConsumerGroup 之间消费进度完全独立，同一条消息可被多个组分别消费。
  • 一个 Consumer 实例可以订阅多个 Topic，但同一 Group 内所有实例的订阅关系应一致。

负载均衡示例

Topic order_topic 有 4 个 Queue: Q0 Q1 Q2 Q3
ConsumerGroup G1 有 2 个实例: C1, C2

分配结果（Rebalance 后）:
C1 -> Q0, Q1
C2 -> Q2, Q3

典型用法

场景 ConsumerGroup 设计
订单服务消费下单消息order-service-group
库存服务消费同一 Topicinventory-service-group（独立进度）
同一服务水平扩容同 Group 加 Consumer 实例，自动 Rebalance

Rebalance（重平衡）

当 Consumer 实例上下线、Topic Queue 数量变化时，Broker 协调 ConsumerGroup 重新分配 Queue。Rebalance 期间可能短暂停止消费，生产环境应尽量避免频繁扩缩容。

72. RocketMQ Offset 是什么？

Offset 是消费者在某一 MessageQueue 上的消费进度位点，表示“该 Queue 上下一条要消费的消息序号”。

存储维度

Offset 的粒度是 ConsumerGroup + MessageQueue，不是 Topic 级别。同一 Topic 的不同 Queue 各自维护独立 Offset。

ConsumerGroup: order-group
MessageQueue: order_topic-Q0 -> Offset = 1523
MessageQueue: order_topic-Q1 -> Offset = 891
MessageQueue: order_topic-Q2 -> Offset = 2047

Offset 存储位置

模式 存储位置 说明
集群消费（默认）Broker 端Consumer 定期向 Broker 提交 Offset，重启后从 Broker 恢复
广播消费Consumer 本地每个实例各自维护 Offset，存在本地文件

提交时机与风险

提交时机 风险
业务处理前提交Consumer 宕机后消息丢失（消费了但未处理）
业务处理后提交Consumer 宕机后可能重复消费（处理了但未提交）

RocketMQ 默认在业务返回 CONSUME_SUCCESS 后提交 Offset，属于“至少一次投递”语义，因此消费端必须幂等。

消费进度管理

  • 可通过 RocketMQ Console 或 mqadmin consumerProgress 查看各 Group 的消费延迟。
  • 重置 Offset 需谨慎，可能导致大量重复消费或跳过未消费消息。

73. 如何处理重复消费？

RocketMQ 提供的是至少一次（At Least Once）投递语义，无法保证精确一次。重复消费的来源包括：Producer 重试、Consumer 消费失败重试、Offset 提交失败、Rebalance 等。

重复消费来源

  • Producer 发送超时后重试，Broker 可能写入两条相同业务消息。
  • Consumer 业务处理成功但 Offset 提交前宕机，重启后重新消费。
  • Consumer 返回 RECONSUME_LATER 进入重试队列，再次投递。
  • Rebalance 导致消息被重新分配消费。

幂等方案

方案 实现方式 适用场景
数据库唯一索引以业务 ID 做唯一键，重复插入失败订单创建、支付流水
状态机只允许特定状态流转，重复消息被忽略订单状态变更
Redis 去重SETNX msgId 1 EX 86400高频、允许短暂窗口
去重表消费前查/插消费记录表通用、可审计

推荐做法

public void onMessage(MessageExt msg) {
    String bizId = msg.getKeys(); // 或 msgId
    if (dedupService.exists(bizId)) {
        return; // 已处理，直接返回成功
    }
    try {
        businessService.process(msg);
        dedupService.markProcessed(bizId);
    } catch (Exception e) {
        throw e; // 触发重试
    }
}

设计原则

  1. 消费逻辑设计为幂等，不要依赖“只会执行一次”。
  2. 优先使用业务唯一键（订单号、流水号），而非 MQ 的 msgId。
  3. 去重标记的写入应与业务操作在同一事务内，避免“标记成功但业务失败”。

249. RocketMQ 的 NameServer 有什么作用？

NameServer 是 RocketMQ 的轻量级路由注册中心，负责 Broker 路由信息的注册、维护和查询。它不参与消息存储和转发，职责单一。

核心功能

  • Broker 注册：Broker 启动后向所有 NameServer 注册自身信息（地址、Topic 路由、读写权限等）。
  • 路由发现：Producer 和 Consumer 启动时从 NameServer 拉取 Topic 路由表，获取 Broker 地址和 Queue 信息。
  • 心跳检测：Broker 每 30 秒向 NameServer 发心跳，NameServer 每 10 秒扫描 Broker 表，120 秒无心跳则剔除。

架构特点

特点 说明
无状态NameServer 节点之间不通信，互不依赖，可独立部署多个
轻量级仅维护路由元数据，不存储消息，资源占用小
最终一致各 NameServer 通过 Broker 心跳各自维护路由，短暂不一致可接受

与 Kafka/ZooKeeper 对比

Kafka 早期依赖 ZooKeeper 管理 Broker 和 Partition 元数据；RocketMQ 的 NameServer 更轻，无选主、无强一致要求，运维更简单。客户端会连接多个 NameServer 地址，某个节点宕机不影响整体。

Producer / Consumer 启动
-> 连接 NameServer 列表
-> 拉取 Topic 路由（Broker 地址 + Queue 列表）
-> 直接与 Broker 通信收发消息

250. RocketMQ Producer 发送消息有哪些模式？

Producer 发送消息可从两个维度理解：通信模式（同步/异步/单向）和消息类型（普通/顺序/延迟/事务）。

按通信模式

模式 方法 特点
同步发送send(msg)阻塞等待 Broker 响应，可靠性最高
异步发送send(msg, callback)非阻塞，通过回调处理结果，吞吐高
单向发送sendOneway(msg)只发不等响应，最快但可能丢失

按消息类型

类型 说明
普通消息最常用，无特殊语义
顺序消息通过 MessageQueueSelector 固定队列
延迟消息设置 delayLevel，到期后才投递
事务消息Half Message + 本地事务 + 回查
批量消息多条消息打包一次发送，提升吞吐

队列选择策略

  • 默认轮询：消息均匀分散到各 Queue。
  • 自定义 Selector：按业务键哈希，实现顺序或亲和路由。
  • 故障规避：发送失败时自动尝试其他 Broker/Queue。

251. 同步发送、异步发送、单向发送有什么区别？

三种发送方式的核心差异在于是否等待 Broker 确认以及对可靠性和吞吐的取舍。

对比项 同步发送 异步发送 单向发送
是否阻塞阻塞等待响应非阻塞，回调通知发完即返回
可靠性最高较高（需处理回调失败）最低，可能丢失
吞吐较低最高高
适用场景支付、订单等关键业务日志、通知等高频场景监控埋点、可丢数据

同步发送

SendResult result = producer.send(msg);
// 阻塞直到 Broker 返回 SEND_OK 或异常

异步发送

producer.send(msg, new SendCallback() {
    @Override
    public void onSuccess(SendResult result) { /* 记录成功 */ }

    @Override
    public void onException(Throwable e) { /* 重试或告警 */ }
});

单向发送

producer.sendOneway(msg);
// 不等待任何响应，无法感知发送是否成功

选择建议

  • 涉及资金、状态变更：同步发送。
  • 高并发、允许短暂延迟感知失败：异步发送 + 失败补偿。
  • 日志采集、指标上报等可丢场景：单向发送。

252. RocketMQ 如何做消息过滤？

RocketMQ 支持在 Broker 端和 Consumer 端两种过滤方式。Broker 端过滤可减少网络传输，Consumer 端过滤更灵活。

Tag 过滤（Broker 端，最常用）

Producer 发送时设置 Tag，Consumer 订阅时指定 Tag 表达式（支持 || 逻辑或）。

// Producer
msg.setTags("create || pay");

// Consumer 订阅
consumer.subscribe("order_topic", "create || pay");

Broker 在 ConsumeQueue 索引中存储 Tag hashcode，过滤效率高，适合 Tag 数量有限的场景。

SQL92 过滤（Broker 端）

基于消息 UserProperty 做 SQL 表达式过滤，例如 age > 18 AND city = 'beijing'。

  • 需要在 Broker 开启 enablePropertyFilter=true。
  • 灵活但性能低于 Tag 过滤，属性需设为 UserProperty。

Consumer 端过滤

Consumer 订阅全部消息（*），在 MessageListener 中自行判断，不符合则返回成功跳过。

  • 优点：逻辑灵活，可依赖外部状态。
  • 缺点：所有消息都传到 Consumer，浪费带宽和 CPU。

对比

方式 过滤位置 性能 灵活性
Tag 过滤Broker高中（Tag 数量有限）
SQL92 过滤Broker中高
Consumer 过滤Consumer低最高

253. RocketMQ Tag 和 Key 有什么区别？

Tag 和 Key 都是消息的元数据属性，但设计目的和使用方式完全不同。

对比项 Tag Key
作用消息分类/过滤消息检索/追踪
数量一条消息通常 1 个 Tag一条消息可设多个 Key（空格分隔）
Consumer 订阅支持，如 TagA || TagB不支持订阅过滤
Broker 索引ConsumeQueue 存 Tag hashcodeIndexFile 建 Key 哈希索引
典型用途区分业务类型：create/pay/cancel关联业务 ID：orderId、userId

使用示例

Message msg = new Message("order_topic", "pay", body);
msg.setKeys("order_10001 user_888");  // Key：检索追踪
// Tag 在构造函数第二个参数 "pay" 中设置

实践建议

  • Tag 用于 Consumer 订阅过滤，保持 Tag 种类可控（不宜过多）。
  • Key 用于问题排查和消息追踪，通过 Console 或 queryMsgByKey 查询。
  • 幂等去重优先用 Key（业务 ID），而非 Tag。

254. RocketMQ 延迟消息支持任意时间吗？

不支持任意时间。 RocketMQ 开源版延迟消息基于固定的延迟级别（delayLevel），不支持指定“30 分钟后”这类任意延迟。

  • Redis ZSet / 时间轮等中间件实现精确延迟。
  • 选择最接近的 delayLevel，业务容忍误差。

255. RocketMQ 死信消息如何处理？

当消息消费失败且重试次数达到上限（默认 16 次），RocketMQ 会将消息投递到死信队列（Dead Letter Queue, DLQ），避免无限重试阻塞正常消费。

死信队列命名规则

%DLQ%{ConsumerGroup}

例如 ConsumerGroup 为 order-service-group
死信 Topic 为 %DLQ%order-service-group

进入死信的条件

  1. Consumer 返回 RECONSUME_LATER 或抛异常。
  2. 消息进入重试队列（%RETRY%{ConsumerGroup}）。
  3. 重试 16 次（默认）仍失败。
  4. 消息被转移到 DLQ。

处理流程

1. 监控 DLQ 消息数量，设置告警
2. 分析失败原因（代码 Bug、依赖不可用、脏数据）
3. 修复问题后，从 DLQ 手动重投或编写补偿 Consumer
4. 对无法修复的 poison message，记录后丢弃或人工处理

工程实践

  • 单独部署 DLQ Consumer，将死信消息落库或发告警，便于人工介入。
  • 重试间隔默认递增（10s、30s、1min…2h），给依赖恢复留时间。
  • 可通过 setMaxReconsumeTimes(n) 调整最大重试次数。

256. RocketMQ 消费失败后重试流程是什么？

Consumer 消费失败时，RocketMQ 不会立刻丢弃消息，而是将其放入重试队列，按递增间隔重新投递，直到成功或进入死信队列。

重试流程

1. Consumer 消费消息，业务处理失败
2. 返回 RECONSUME_LATER（或抛异常）
3. Broker 将消息写入重试 Topic: %RETRY%{ConsumerGroup}
4. 等待重试间隔（默认 16 级递增延迟）
5. 定时任务将消息重新投递到原 Topic Queue
6. Consumer 再次消费
7. 成功 -> 正常提交 Offset；失败 -> 重复步骤 3-6
8. 超过 maxReconsumeTimes（默认 16）-> 进入 DLQ

默认重试间隔

与延迟消息共用 18 级：10s、30s、1min、2min、3min…最长 2h。第 N 次重试使用第 N 级间隔。

Consumer 端控制

// 返回成功，提交 Offset
return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;

// 返回稍后重试
return ConsumeConcurrentlyStatus.RECONSUME_LATER;

注意点

  • 重试消息可能被不同 Consumer 实例消费，必须幂等。
  • 顺序消费（Orderly）失败会阻塞当前 Queue，不会立即进入重试队列。
  • 大量重试会导致消息堆积，需监控重试队列深度。
  • 对于确定性失败（如参数校验不通过），应直接返回 SUCCESS 并记录，避免无意义重试。

257. RocketMQ 广播消费适合什么场景？

广播消费（Broadcasting）模式下，ConsumerGroup 内每个实例都会收到 Topic 下的全量消息，不做 Queue 分摊。

工作机制

  • 每条消息被 ConsumerGroup 内所有实例各消费一次。
  • Offset 存储在 Consumer 本地文件，不在 Broker 端。
  • 新加入的实例不会消费历史消息（从最新开始）。

适用场景

场景 原因
本地缓存刷新每个节点都需要收到配置变更，更新本地缓存
广播通知所有实例都需要感知的事件，如服务下线通知
边缘节点同步每个节点独立处理，无需分摊

配置方式

consumer.setMessageModel(MessageModel.BROADCASTING);

不适用场景

  • 需要负载均衡分摊消息量的业务处理（应用集群消费）。
  • 需要 Broker 端统一管理消费进度。
  • Consumer 实例动态扩缩容且需消费历史消息。

258. RocketMQ 集群消费适合什么场景？

集群消费（Clustering）是 RocketMQ 的默认模式，ConsumerGroup 内多个实例分摊 Topic 下的 MessageQueue，每条消息只被组内一个实例消费。

工作机制

  • 同 Group 内 Queue 与 Consumer 实例做 Rebalance 分配。
  • Offset 存储在 Broker，Consumer 重启后从 Broker 恢复进度。
  • 水平扩容 Consumer 实例可提升消费能力。

适用场景

场景 说明
业务消息处理订单、支付、通知等，一条消息只需处理一次
高吞吐消费多实例分摊 Queue，提升并行度
消费进度统一管理Offset 在 Broker，便于监控和重置
微服务独立消费不同 ConsumerGroup 独立消费同一 Topic

与广播消费对比

对比项 集群消费 广播消费
消息分发组内分摊，每条消息一个实例组内全量，每条消息所有实例
Offset 存储BrokerConsumer 本地
扩容效果提升吞吐不提升吞吐，仅增加副本
默认模式是否

259. RocketMQ 如何保证 Broker 高可用？

Broker 高可用通过多副本部署、主从复制、自动故障切换等手段实现，确保单点 Broker 宕机后消息服务不中断。

部署架构

Master Broker（可读写）
    |
    | 同步/异步复制
    v
Slave Broker（只读，热备）

Producer / Consumer 通过 NameServer 发现 Master 地址
Master 宕机后，Slave 可手动或自动切换为 Master

关键机制

机制 说明
主从复制Master 写入后复制到 Slave，Slave 可承担读请求
同步刷盘消息落盘后才返回 Producer 确认
同步复制Master 和 Slave 都写入成功才返回确认
DLedger 模式（4.x+）基于 Raft 的多副本自动选主，无需手动切换
多 Master 部署多个 Master-Slave 组，单组故障不影响其他组

DLedger 高可用（推荐）

RocketMQ 4.5+ 引入 DLedger CommitLog，基于 Raft 协议实现多副本自动选主：

  • 3 个或更多 Broker 组成一组，自动选举 Leader。
  • Leader 宕机后自动选出新 Leader，无需人工介入。
  • 消息写入多数派副本后才确认，保证不丢。

故障切换注意

  • 传统 Master-Slave 模式下，Master 宕机需手动将 Slave 提升为 Master，或有外部工具（如 RocketMQ Controller）自动切换。
  • 异步复制模式下，Master 宕机可能丢失未同步到 Slave 的消息。
  • Producer 应配置多个 NameServer 地址，Consumer 应能感知 Broker 变化。

260. RocketMQ 主从同步和异步复制有什么区别？

主从复制模式决定了 Master 写入消息后，是否需要等待 Slave 确认才返回 Producer。这是可靠性与性能的核心权衡点。

对比项 同步复制（SYNC_MASTER） 异步复制（ASYNC_MASTER）
写入流程Master 写 + Slave 写都成功才返回Master 写成功即返回，Slave 异步同步
Master 宕机数据丢失不丢（Slave 已有副本）可能丢失未同步的数据
写入延迟较高较低
吞吐较低较高
适用场景金融、支付等强可靠日志、通知等可容忍少量丢失

同步复制流程

Producer 发送消息
-> Master 写入 CommitLog
-> Master 等待 Slave 复制成功
-> 两者都成功，返回 SEND_OK
-> Master 宕机，Slave 可切换且数据完整

异步复制流程

Producer 发送消息
-> Master 写入 CommitLog
-> 立即返回 SEND_OK
-> 后台异步复制到 Slave
-> Master 宕机时，Slave 可能缺少最近消息

与刷盘策略的组合

组合 可靠性 性能
同步刷盘 + 同步复制最高最低
同步刷盘 + 异步复制高中
异步刷盘 + 异步复制低最高

生产环境常用“异步刷盘 + 同步复制”或“异步刷盘 + 异步复制”，在可靠性和性能间折中。

261. RocketMQ 如何处理消息堆积？

消息堆积指 Consumer 消费速度跟不上 Producer 发送速度，导致 Queue 中积压大量未消费消息。堆积过长可能引发消费延迟、Broker 磁盘压力甚至 OOM。

排查步骤

  1. 确认是全局堆积还是个别 Queue 堆积（可能是热点或顺序消费阻塞）。
  2. 检查 Consumer 是否存活、是否有异常日志、是否消费过慢。
  3. 检查 Producer 是否突发流量或重试风暴。
  4. 查看 Broker 磁盘使用和消息保留策略。

应急处理

手段 做法 注意
扩容 Consumer增加同 Group 实例数（不超过 Queue 数）触发 Rebalance，短暂停消费
提升消费并行度增加 consumeThreadMin/Max需保证幂等和 DB 连接池容量
临时 Consumer 快速消费部署只落库不处理的临时消费者后续异步补偿业务
跳过堆积消息重置 Offset 到最新会丢消息，仅极端场景
限流 Producer上游降级或限流防止堆积加剧

长期治理

  • 消费逻辑异步化：Consumer 只做校验和落库，业务处理交给线程池或下游 MQ。
  • 增加 Queue 数量并扩容 Broker，提升并行消费能力。
  • 设置消息 TTL 和过期策略，避免无限堆积。
  • 监控 Consumer Lag，堆积超过阈值自动告警和扩容。

262. RocketMQ 如何做水平扩容？

RocketMQ 的水平扩容分为Broker 扩容和Consumer 扩容两个维度，Producer 通常无状态，无需特殊扩容。

Broker 扩容

1. 部署新 Broker（Master-Slave 组）
2. 新 Broker 向 NameServer 注册
3. 在新 Broker 上创建 Topic 并增加 Queue 数量
   （或使用自动创建 Topic 机制）
4. Producer 通过 NameServer 自动发现新 Queue
5. 新消息按路由规则写入新 Broker
  • 已有消息不会自动迁移，旧 Broker 仍承担历史消息消费。
  • 扩容 Topic Queue 数需通过 updateTopic 或 Console 操作。
  • Queue 数量只增不减，规划时预留足够 Queue。

Consumer 扩容

1. 启动新 Consumer 实例，加入同一 ConsumerGroup
2. Broker 触发 Rebalance
3. 新实例分配到部分 MessageQueue
4. 并行消费，吞吐提升
  • Consumer 实例数 ≤ Queue 总数，否则多余实例空闲。
  • Rebalance 期间消费短暂暂停，避免频繁扩缩容。

扩容限制

组件 扩容方式 限制
Producer无状态，直接加实例无特殊限制
Broker加 Master-Slave 组 / 加 Queue历史消息不迁移
Consumer加同 Group 实例不超过 Queue 数
NameServer加节点无状态，线性扩展

263. RocketMQ 和 Kafka 在设计目标上有什么区别？

两者都是主流分布式消息中间件，但设计目标和适用场景有明显差异。面试中应围绕“业务消息 vs 流式平台”来对比。

对比项 RocketMQ Kafka
设计目标业务消息中间件，服务解耦、异步、事务分布式流平台，日志采集、流处理、大数据
消息语义丰富：事务消息、延迟消息、死信队列相对简单，依赖外部组件补充
消费模型Push 为主，ConsumerGroup 分摊Pull 为主，Consumer Group 分摊
存储模型CommitLog + ConsumeQueue 索引Partition 顺序日志
路由中心NameServer（轻量无状态）ZooKeeper / KRaft
顺序消息原生支持，Queue 级顺序Partition 内有序
延迟消息原生支持（固定级别）不原生支持
事务消息原生 Half Message 机制0.11+ 支持事务，但使用较少
吞吐量高（十万级 TPS）极高（百万级 TPS）
消息回溯支持，按 Offset 或时间支持，按 Offset 或时间戳
生态阿里系、国内业务场景广泛流处理生态强（Kafka Streams、Flink）

选型建议

  • 选 RocketMQ：电商订单、支付通知、事务消息、延迟消息、国内微服务解耦。
  • 选 Kafka：日志采集、指标上报、大数据管道、流式计算、超高吞吐场景。
  • 两者并非互斥，很多公司 Kafka 做数据采集，RocketMQ 做业务消息。

面试回答要点

  1. RocketMQ 偏业务消息，功能丰富，开箱即用的事务/延迟/死信能力是其优势。
  2. Kafka 偏流式数据平台，吞吐更高，与大数据生态深度集成。
  3. 具体选型看业务需求：可靠性功能 vs 吞吐和流处理能力。
