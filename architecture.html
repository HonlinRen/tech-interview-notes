84. 网关的核心价值是什么？

API 网关是微服务架构的统一入口，位于客户端与后端服务之间，集中处理横切关注点，让业务服务专注领域逻辑。

能力说明
路由转发按路径、Header、权重将请求分发到对应服务
认证授权统一校验 Token / 签名，减少各服务重复实现
限流熔断保护后端不被流量打垮
协议转换HTTP/HTTPS、gRPC/WebSocket 等协议适配
灰度发布按用户、比例切流到新版本
日志监控统一 access log、TraceId 注入

常见实现：Spring Cloud Gateway、Kong、Nginx + OpenResty、APISIX。

85. 网关如何做认证和授权？

认证（Authentication）解决「你是谁」；授权（Authorization）解决「你能做什么」。

典型流程

  1. 客户端携带 JWT / OAuth2 Access Token / API Key 请求网关。
  2. 网关校验签名、过期时间、黑名单，解析出 userId、角色、租户。
  3. 网关按路由规则做授权：路径 + HTTP 方法 + 角色/权限码。
  4. 通过后转发请求，并将用户信息写入 Header（如 X-User-Id）传给下游。
方式适用场景
JWT 无状态校验网关本地验签，高吞吐
OAuth2 Introspection需实时吊销、强管控
API Key + HMAC 签名开放 API、B2B 对接

注意：网关是第一道防线，但业务服务仍需做关键权限校验（纵深防御），不能完全信任 Header。

86. 限流、熔断、降级分别解决什么问题？

三者都是稳定性治理手段，但触发点和目标不同。

机制触发方目标典型手段
限流入口 / 调用方控制流量上限，防止过载令牌桶、漏桶、滑动窗口
熔断调用方下游故障时快速失败，避免线程/连接被拖死Sentinel、Hystrix、Resilience4j
降级被调方 / 调用方高压或故障时关闭非核心能力，保核心链路返回默认值、读缓存、关闭推荐模块
 
正常：请求 -> 限流通过 -> 调用下游 -> 返回
熔断：下游错误率超阈值 -> 熔断打开 -> 直接返回 fallback
降级：库存服务超时 -> 跳过推荐位 -> 仅展示核心商品信息

组合使用：限流防突发、熔断防故障扩散、降级保核心业务可用。

87. MQ 如何实现削峰填谷？

瞬时高峰流量远超下游处理能力时，将请求异步化：生产者快速写入 MQ，消费者按自身速率平稳处理。

秒杀场景：
用户请求 -> 网关限流 -> 写入 MQ（毫秒级返回"排队中"）
         -> 消费者按 500 QPS 扣库存、创建订单

收益与代价

收益代价 / 需处理
保护 DB / 下游服务链路变异步，用户体验需轮询或推送
平滑资源使用消息可靠性（ACK、重试、死信）
可水平扩展消费者重复消费需幂等、积压需监控告警

88. 接口幂等性为什么重要？

同一业务请求执行多次，结果与执行一次相同。网络超时重试、用户重复点击、MQ 重复投递都会导致重复请求。

不幂等的后果

  • 重复扣款、重复下单、重复发券。
  • 库存被多次扣减。

常见实现

方案原理
唯一请求号 + 去重表先 INSERT 请求号，唯一索引冲突则拒绝
数据库唯一索引订单号、业务流水号天然防重
状态机只允许特定状态流转，重复请求无效
Token 机制提交前先获取一次性 Token
分布式锁同一资源同一时刻只处理一次
 
INSERT INTO idempotent (request_id, status)
VALUES ('req-001', 'processing');
-- 重复请求触发唯一索引冲突，直接返回上次结果

89. 高并发下单接口如何设计？

核心目标：在极高并发下保证不超卖、不重复下单、核心链路可用。

分层设计

  1. 前端：按钮防重复点击、验证码/答题。
  2. 网关：全局限流 + 用户级限流。
  3. 应用：Token 防重、参数校验、异步化非核心步骤。
  4. 库存：Redis 预扣（Lua 原子）或 DB 乐观扣减 WHERE stock > 0。
  5. 订单：唯一索引 + 状态机 + MQ 异步落库/通知。
UPDATE product SET stock = stock - 1
WHERE id = ? AND stock > 0;
-- affected_rows = 0 表示库存不足

补偿机制：MQ 消费失败重试、超时关单、对账任务兜底。

90. 如何避免单点故障？

单点故障（SPOF）指某一组件宕机导致整条链路不可用。消除 SPOF 的核心是冗余 + 自动切换。

层级措施
应用多实例部署 + 负载均衡 + 健康检查
数据库主从 / 主主、MGR、跨 AZ 部署
缓存Redis Cluster / Sentinel
MQBroker 集群、同步刷盘 + 多副本
注册/配置中心集群模式（Nacos/ZK 奇数节点）
网关 / LB主备或 DNS 多 IP

还需定期演练故障切换（混沌工程），验证 failover 时间和数据一致性。

91. 重试机制有哪些风险？

  • 重试风暴：大量客户端同时重试，流量成倍放大，压垮已脆弱的下游。
  • 重复写入：请求实际已成功但响应超时，重试导致重复操作。
  • 延迟堆积：重试占用线程/连接，拖慢整体吞吐。
  • 故障扩大：短暂抖动被重试放大成长期不可用。

安全重试原则

原则说明
只重试幂等操作写操作必须有幂等键
指数退避 + 抖动1s → 2s → 4s + random
限制最大次数通常 2~3 次
配合熔断下游已熔断则不再重试
区分错误类型4xx 不重试，5xx/超时可选重试

92. K8S Service 解决什么问题？

Pod IP 随重建而变化，直接访问 Pod 不可靠。Service 为一组 Pod 提供稳定访问入口和负载均衡。

核心机制

  • 通过 selector 关联一组 Pod。
  • 分配稳定 ClusterIP（或 NodePort / LoadBalancer / Headless）。
  • kube-proxy 维护 iptables/IPVS 规则，将流量转发到健康 Pod。
  • Cluster 内通过 DNS service-name.namespace.svc.cluster.local 访问。
类型用途
ClusterIP集群内部访问（默认）
NodePort通过节点端口暴露
LoadBalancer云厂商 LB 对外暴露
Headless无 ClusterIP，直接解析 Pod IP（StatefulSet）

264. CAP 理论是什么？

分布式系统中，一致性（C）、可用性（A）、分区容错性（P）三者不可同时满足，最多满足两个。

  • C（Consistency）：所有节点同一时刻看到相同数据。
  • A（Availability）：每个请求都能得到响应（不保证最新）。
  • P（Partition tolerance）：网络分区发生时系统仍能运行。

分布式系统网络分区不可避免，因此实际是在 CP 与 AP 之间取舍：

类型代表特点
CPZooKeeper、Etcd强一致，分区时可能拒绝服务
APEureka、Cassandra高可用，允许短暂不一致

CAP 是理论模型，工程上更关注延迟、吞吐和业务可接受的不一致窗口。

265. BASE 理论是什么？

BASE 是对 CAP 中 AP 方向的工程实践总结，强调最终一致性而非强一致。

  • BA（Basically Available）：基本可用，允许损失部分可用性（如降级、限流）。
  • S（Soft state）：软状态，允许中间态数据存在。
  • E（Eventually consistent）：经过一段时间后，所有副本达到一致。

典型场景：电商下单后库存异步扣减、点赞数最终汇总、缓存与 DB 短暂不一致。

实现手段：MQ 异步、本地消息表、对账补偿、版本号/CAS。

266. Paxos 算法解决什么问题？

Paxos 解决分布式系统中多个节点如何就某个值达成一致的问题，是经典共识算法。

角色

  • Proposer：提出提案。
  • Acceptor：投票是否接受提案。
  • Learner：学习已通过的提案值。

两阶段

  1. Prepare：Proposer 发起编号 n 的 prepare，Acceptor 承诺不接受编号 ≤ n 的提案。
  2. Accept：获得多数派同意后，Proposer 发送 accept(n, value)，多数派接受则值被选定。

优点：理论完备、容错（允许少数节点故障）。缺点：理解复杂、工程实现难，Raft 是其简化替代。

267. Raft 算法解决什么问题？

Raft 通过强 Leader 简化分布式共识，与 Paxos 等价但更易理解和实现。

三种角色

Leader、Follower、Candidate（选举时 Follower 转变）。

两个核心过程

  1. Leader 选举：Follower 超时未收到心跳则成为 Candidate，拉票，获多数票成为 Leader。
  2. 日志复制：Leader 接收客户端写请求，追加日志并复制到多数 Follower，提交后应用状态机。

应用：Etcd、Consul、TiKV 等。保证已提交的日志不会丢失，且所有节点按相同顺序应用。

268. ZAB 协议是什么？

ZAB（ZooKeeper Atomic Broadcast）是 ZooKeeper 的原子广播协议，保证消息全局有序和一致性。

两种模式

模式触发作用
崩溃恢复Leader 宕机或集群启动选举 Leader、同步 epoch 和事务日志
消息广播正常运行Leader 将写请求广播给 Follower，多数 ACK 后提交

与 Raft 类似：单一 Leader 写、多数派确认、顺序一致性。ZAB 为 ZK 的协调服务（锁、选主、配置）提供基础。

269. Gossip 协议适合什么场景？

Gossip（流言协议）中节点随机选择 peer 交换状态，最终概率性传播到全网，无需中心节点。

特点

  • 去中心化、高容错、扩展性好。
  • 最终一致，不保证实时准确。
  • 收敛速度 O(log N)。

适用场景

场景示例
集群成员发现Cassandra 节点上下线感知
分布式缓存Redis Cluster 节点通信
故障检测phi accrual failure detector

不适合强一致选主、金融级实时同步等对一致性要求极高的场景。

270. RPC 是什么？

RPC（Remote Procedure Call）让调用远程服务像调用本地方法一样，屏蔽网络通信细节。

调用流程

  1. 客户端 Stub 序列化参数。
  2. 通过网络发送到服务端。
  3. 服务端 Skeleton 反序列化、执行业务、序列化结果返回。
  4. 客户端 Stub 反序列化得到返回值。

核心组件：序列化（Protobuf/Hessian）、网络传输（Netty）、服务治理（注册发现、负载均衡、熔断）。

典型框架：Dubbo、gRPC、Thrift。

选型：对外 API、浏览器通信用 HTTP；微服务内部高频调用用 RPC。

273. Dubbo 服务暴露和服务引用流程是什么？

服务暴露（Provider）

  1. Spring 容器启动，解析 @DubboService 注解。
  2. 生成 Invoker 代理，封装实现类。
  3. 通过 Protocol（如 Dubbo Protocol）打开端口监听。
  4. 将服务 URL 注册到 Registry。

服务引用（Consumer）

  1. 解析 @DubboReference，创建接口代理对象。
  2. 从 Registry 订阅 Provider 地址列表。
  3. Registry 推送变更，Consumer 更新本地 Directory。
  4. 调用时 Cluster 选 Invoker → LoadBalance 选实例 → Protocol 发网络请求。
Provider:  @DubboService -> Export -> Register URL
Consumer:  @DubboReference -> Subscribe -> Proxy.invoke()

274. ZooKeeper 适合解决什么问题？

ZK 是基于 ZAB 的分布式协调服务，提供一致性的树形命名空间（ZNode）。

典型场景

  • 服务注册与发现：临时节点，会话断开自动删除。
  • 分布式锁：临时顺序节点 + Watch。
  • Master 选举：最小序号节点为 Leader。
  • 配置管理：配置存 ZNode，变更 Watch 通知。
  • 命名服务：统一命名、分组。

特点：CP 系统，强一致、顺序一致；写性能有限，不适合大量写或海量数据存储。

275. ZooKeeper 的 ZNode 有哪些类型？

类型生命周期典型用途
Persistent客户端断开后仍存在持久配置、元数据
Ephemeral客户端会话结束即删除服务注册、心跳
Persistent Sequential持久 + 自动递增序号有序命名
Ephemeral Sequential临时 + 自动递增序号分布式锁、Leader 选举

每个 ZNode 可存少量数据（默认 ≤ 1MB），可设置 ACL 和 Watch（一次性触发，需重新注册）。

276. ZooKeeper 如何实现分布式锁？

临时顺序节点方案

  1. 所有客户端在 /lock/ 下创建临时顺序节点。
  2. 判断自己是否序号最小：是则获得锁。
  3. 否则 Watch 前一个节点，等待其删除通知。
  4. 持有锁的客户端完成业务后删除节点，或会话断开自动释放。
/lock/
  lock-0000000001  (Client A 持有)
  lock-0000000002  (Client B 等待，Watch 0001)
  lock-0000000003  (Client C 等待，Watch 0002)

优点：公平锁、无死锁（会话断开自动释放）、Watch 避免羊群效应。缺点：性能低于 Redis，ZK 集群压力需评估。

277. 分布式配置中心解决什么问题？

微服务数量增多后，配置分散在各应用本地，变更需重启、易不一致。配置中心实现集中管理 + 动态推送。

核心能力

  • 统一存储：多环境（dev/test/prod）、多命名空间。
  • 版本管理：历史回滚、灰度发布。
  • 权限审计：谁改了什么配置。
  • 实时推送：变更秒级生效，无需重启。

常见实现：Nacos、Apollo、Spring Cloud Config + Bus。

278. 配置变更如何实时推送到服务？

常见模式

模式机制代表
长轮询客户端 hold 请求，有变更立即返回Nacos 1.x
Watch + 推送客户端注册 Watch，服务端主动通知ZK、Nacos 2.x gRPC
消息广播配置变更发 MQ，各实例消费刷新Spring Cloud Bus

客户端收到通知后：更新本地缓存 → 发布 RefreshEvent → @RefreshScope Bean 重建 → 业务生效。

279. 为什么需要分布式 ID？

单体应用用数据库自增 ID 即可。分库分表、多服务独立写库后，自增 ID 会冲突、无法全局唯一、难以排序。

  • 多库多表：各分片自增 ID 重复。
  • 数据合并：无法按 ID 判断先后顺序。
  • 信息安全：连续 ID 暴露业务量，易被遍历。
  • 跨系统：订单号、流水号需全局唯一标识。

分布式 ID 生成器在应用层统一发号，保证全局唯一、高性能、趋势递增（利于索引）。

280. 分布式 ID 需要满足哪些特性？

特性说明
全局唯一任意两个 ID 不重复
趋势递增大致按时间递增，利于 B+ 树索引
高性能本地生成或低延迟，支撑高 QPS
高可用发号服务不能成为单点
信息安全不可预测、不可遍历（视业务）

不同业务权重不同：订单号重唯一和可读，日志 ID 重吞吐，对外 ID 还需不可预测。

281. UUID、数据库自增、雪花算法各有什么优缺点？

方案优点缺点
UUID本地生成、无中心、简单无序（索引碎片）、36 字符占空间、不可读
DB 自增 / 号段简单、趋势递增DB 瓶颈、分库难扩展；号段模式需批量取号
雪花算法本地生成、趋势递增、高性能依赖时钟、需分配 workerId、时钟回拨风险
 
雪花结构（64 bit）：
0 | 41 bit 时间戳 | 10 bit 机器ID | 12 bit 序列号
每毫秒每机器可生成 4096 个 ID

生产环境常用雪花或号段模式（Leaf、TinyID）；UUID 适合无索引要求的场景。

282. 雪花算法时钟回拨怎么处理？

雪花算法依赖系统时钟，若 NTP 同步导致时钟回拨，可能生成重复 ID。

处理策略

  1. 抛异常拒绝服务：简单，短暂回拨可等待追平。
  2. 借用未来时间：回拨期间用上次 timestamp + 1 继续发号（百度 UidGenerator）。
  3. 扩展 bit 存回拨序列：预留 bit 记录回拨次数。
  4. 外部时钟源：统一时间服务，减少回拨概率。

运维层面：禁止随意手动改时间、监控 clock offset、多实例 workerId 严格分配。

性能高（内存、单命令）较低（磁盘、多节点确认）
实现复杂度需 Redisson/Redlock 完善临时节点语义天然适合
锁类型非公平为主公平锁（顺序节点）
释放机制TTL 过期 + 看门狗续期会话断开自动释放

高并发、允许极小概率问题的场景用 Redis；金融、强一致选主用 ZK/Etcd。

285. 分布式事务为什么难？

  • 网络不可靠：请求可能超时、重复、乱序，无法区分「未执行」与「已执行未响应」。
  • 节点独立：各服务有独立 DB，无法共用本地 ACID 事务。
  • 部分失败：A 成功 B 失败时，缺乏全局回滚机制。
  • 性能代价：强一致方案（2PC）锁资源久、吞吐低。

工程上多数场景接受最终一致，用 Saga、TCC、本地消息表等替代强一致 2PC。

286. 2PC、TCC、Saga、本地消息表有什么区别？

方案一致性原理适用
2PC强一致协调者 Prepare → Commit/Abort跨库少、可接受阻塞
TCC最终一致Try-Confirm-Cancel 三阶段业务补偿需精细控制资源预留
Saga最终一致正向服务链 + 逆向补偿长流程、可补偿业务
本地消息表最终一致业务与消息同库事务，异步投递实现简单、常用
 
本地消息表：
BEGIN;
  UPDATE account SET balance = balance - 100;
  INSERT INTO local_msg (payload, status) VALUES (...);
COMMIT;
-- 定时任务扫描 status=pending 的消息投递 MQ

287. 最终一致性如何保证？

常用手段

  1. 异步消息：写 DB 后发 MQ，消费者更新其他系统。
  2. 本地消息表 / 事务消息：保证「写库」与「发消息」原子性。
  3. 最大努力通知：反复通知直到成功或达上限。
  4. 对账补偿：定时任务比对差异并修复。
  5. 幂等消费：重复消息不影响最终结果。

需明确「一致窗口」业务可接受时长，并配套监控告警发现长期不一致。

288. 事务消息适合解决什么问题？

解决「本地事务与消息发送」的原子性问题：不能只写库不发消息，也不能只发消息写库失败。

典型流程（RocketMQ 事务消息）

  1. 发送 Half Message（对消费者不可见）。
  2. 执行本地事务。
  3. 成功则 Commit Message，失败则 Rollback。
  4. Broker 回查本地事务状态（超时未确认时）。

适用：订单创建后发库存扣减消息、支付成功后发积分消息等跨服务最终一致场景。

289. 负载均衡有哪些常见算法？

算法原理适用
轮询（Round Robin）依次分配实例同构、无状态
加权轮询按权重比例分配机器配置不同
随机随机选实例简单场景
最少连接选连接数最少的长连接、耗时差异大
一致性哈希hash(key) 映射环有状态、缓存
源地址哈希同一客户端固定到同一节点会话保持

290. 四层负载均衡和七层负载均衡有什么区别？

对比项四层（L4）七层（L7）
依据IP + 端口HTTP URL、Header、Cookie 等
性能高，不解析应用层较低，需解析协议
能力转发、NAT路由、灰度、限流、SSL 卸载

通常 L4 做入口流量分发，L7 做细粒度路由和业务治理。

291. Nginx、Gateway、客户端负载均衡有什么区别？

方式位置特点
Nginx独立反向代理 / 边缘高性能、静态资源、SSL、通用 L7
Gateway微服务入口与注册中心集成、动态路由、鉴权限流

常见组合：Nginx → Gateway → 服务；服务间 Dubbo 客户端负载均衡。

292. CDN 是什么？适合解决什么问题？

CDN（Content Delivery Network）将静态资源缓存到离用户最近的边缘节点，就近响应，降低源站压力和延迟。

适合

  • 图片、JS/CSS、视频等静态资源加速。
  • 下载、直播、大文件分发。
  • 抗 DDoS、隐藏源站 IP。

不适合

强个性化、频繁变更的动态 API（可缓存部分 GET 接口需短 TTL + 缓存键设计）。

用户 -> 边缘节点（命中缓存直接返回）
     -> 未命中回源 -> 源站 -> 缓存并返回

293. 数据库读写分离如何设计？

架构

写请求 -> Master
读请求 -> Slave1 / Slave2（负载均衡）

关键设计点

  • 路由：中间件（ShardingSphere、MyCat）或框架注解 @ReadOnly。
  • 主从延迟：写后读走 Master 或延迟读 Slave（同步复制 / 等半同步）。
  • 故障切换：MHA、Orchestrator 自动提升从库。
  • 连接池：读写分离独立连接池，避免写连接被读占满。

294. 分库分表后如何处理跨库事务？

跨分片无法用单库 ACID，通常避免跨库事务，或采用最终一致方案。

方案说明
业务拆分按分片键设计，同一事务数据落同一库
本地消息表单库事务 + 异步通知其他分片
TCC / Saga各分片独立 Try/Confirm 或补偿
Seata AT自动代理，有性能损耗

设计阶段尽量让分片键包含在事务边界内，减少跨库需求。

295. 分库分表后如何处理 join？

  • 避免 join：冗余字段、宽表、应用层组装（推荐）。
  • 绑定表：关联表用相同分片键，可同库 join。
  • 全局表：字典等小表在每个分片复制一份。
  • ER 分片：父子表同分片键路由到同一库。
  • 中间件 join：ShardingSphere 解析 SQL 跨库查询（性能差，慎用）。

原则：分库分表后 join 成本极高，应在建模阶段消除跨库 join 需求。

296. 分库分表后如何处理唯一索引？

唯一字段若不含分片键，无法在各分片独立保证全局唯一。

方案

  1. 全局 ID：雪花等发号器，ID 本身唯一，无需跨库校验。
  2. 独立全局表：唯一字段单独一张不分片的表做校验。
  3. 冗余分片键：唯一索引包含分片键（如 user_id + 业务唯一键）。
  4. 分布式锁 + 查重：写入前全局查重（性能低）。

例：手机号全局唯一 → 以手机号为分片键，或单独 unique_phone 全局表。

297. 如何设计缓存预热？

系统启动或大促前，提前将热点数据加载到缓存，避免缓存冷启动击穿 DB。

方式

  • 启动预热：应用启动时加载 TOP N 商品/配置。
  • 定时任务：低峰期批量刷新热点 Key。
  • 发布预热：新版本上线前脚本灌缓存。
  • 渐进式：限流 + 逐步放开，配合 MQ 异步预热。

注意：预热数据量、过期时间、与 DB 一致性；监控缓存命中率验证效果。

298. 如何设计缓存降级？

缓存不可用时（Redis 故障、超时），系统仍能有限度提供服务。

策略说明
读降级走 DB 直连（加限流）或返回默认值
写降级暂写本地队列，恢复后补偿
开关降级配置中心关闭非核心缓存依赖
多级缓存本地 Caffeine 兜底，Redis 挂仍可用

必须配合熔断、限流，防止缓存全挂时 DB 被瞬间打垮（缓存雪崩）。

299. 如何设计热点 Key 保护？

  • 本地缓存：热点 Key 在 JVM 内缓存短 TTL，减少 Redis 单点 QPS。
  • Key 拆分：一个 hot key 复制为 key_1…key_n，随机读。
  • 读写分离：热点读走 Slave 或只读副本。
  • 限流：对热点接口单独限流。
  • 提前发现：Redis 4.0+ hotkeys、客户端采样上报。
// 热点 Key 本地缓存示例
if (localCache.get(key) != null) return localCache.get(key);
value = redis.get(key);
localCache.put(key, value, 1s);

300. 如何设计异步任务系统？

核心模块

  1. 任务提交：API 写入任务表 / MQ，返回 taskId。
  2. 调度执行：Worker 池拉取或消费任务。
  3. 状态管理：pending → running → success/failed。
  4. 重试与死信：指数退避、最大次数、人工介入。
  5. 进度查询：轮询 taskId 或 WebSocket 推送。

技术选型：MQ（RocketMQ/Kafka）、任务框架（XXL-Job、PowerJob）、或自研任务表 + 乐观锁抢占。

301. 如何设计 Web 实时消息推送？

方案特点适用
WebSocket全双工、低延迟IM、实时协作
SSE服务端单向推送，HTTP 友好通知、进度流
长轮询兼容性好，实现简单低频率、老浏览器

架构要点

  • 连接管理：Gateway 或独立 Push 服务维护连接映射（userId → session）。
  • 跨节点：MQ 广播，各节点推送给本机连接。
  • 心跳、断线重连、消息 ACK 与离线存储。

302. 定时任务单机和分布式实现有什么区别？

对比项单机分布式
调度@Scheduled / cron 本机触发调度中心统一触发
执行单实例多 Worker 竞争或分片
问题多实例重复执行需选主、分片、幂等
代表Spring @ScheduledXXL-Job、ElasticJob、Quartz 集群

生产环境多实例部署必须用分布式调度，避免同一任务被多个节点同时跑。

303. 分布式定时任务如何避免重复执行？

  • 调度中心选主：同一任务只分配一个执行器（XXL-Job 路由策略）。
  • 分布式锁：执行前抢锁，未获锁则跳过。
  • 数据库抢占：UPDATE task SET status=running WHERE id=? AND status=pending，affected=1 才执行。
  • 分片广播：按 mod 分片，各节点处理不同数据子集，不重复。
  • 幂等设计：即使重复执行，业务结果不变。

304. 冗余设计是什么？

通过额外副本或备用资源，在部分组件故障时仍能继续服务，是 high availability 的基础。

类型示例
硬件冗余双电源、RAID、多网卡
服务冗余多实例 + LB
数据冗余主从复制、多副本存储
地理冗余多机房、异地灾备

冗余带来成本增加，需按 RTO/RPO 目标平衡副本数量与切换机制。

305. 异地多活是什么？

在多个地理区域同时部署可对外提供服务的完整系统，用户就近接入，任一区域故障其他区域接管。

挑战

  • 跨区数据同步延迟与冲突解决。
  • 全局流量调度（DNS/GSLB）。
  • 单元化架构：用户数据按 region 划分，减少跨区写。

与「异地灾备（冷备/温备）」区别：多活是平时都在线接流量；灾备是故障时才切换。

306. 容灾和备份有什么区别？

对比项备份容灾
目标数据可恢复业务快速恢复运行
范围数据快照、binlog整套 IT 系统（应用+数据+网络）
RTO通常较长（需还原）更短（热备、多活）
手段全量/增量备份、冷存储主备切换、多活、演练

备份是容灾的数据基础，但仅有备份不等于能快速恢复业务。

307. RTO 和 RPO 是什么？

指标含义示例
RTO（Recovery Time Objective）可接受的服务中断时长故障后 30 分钟内恢复
RPO（Recovery Point Objective）可接受的数据丢失量最多丢 5 分钟数据

RTO 越小需热备/多活；RPO 越小需更频繁同步（同步复制、双写）。成本与指标成正比。

308. 服务限流有哪些算法？

算法原理特点
固定窗口每窗口计数，超限拒绝简单，边界可能突刺 2 倍
滑动窗口细粒度窗口滑动统计更平滑
令牌桶匀速放令牌，有令牌才通过允许一定突发
漏桶请求进桶，恒定速率流出输出严格平滑

网关常用令牌桶；Redis + Lua 可实现分布式滑动窗口/令牌桶。

309. 熔断器半开状态是什么？

熔断器三种状态：Closed → Open → Half-Open → Closed/Open

  1. Closed：正常调用，统计错误率/慢调用。
  2. Open：超阈值后快速失败，不调用下游。
  3. Half-Open：经过冷却期后，放行少量探测请求测试下游是否恢复。
  4. 探测成功 → Closed；仍失败 → 回到 Open。

半开避免下游刚恢复就被全量流量再次打垮，是渐进恢复的关键。

310. 降级策略如何设计？

设计步骤

  1. 梳理核心 / 非核心业务，明确保什么。
  2. 定义触发条件：错误率、RT、CPU、手动开关。
  3. 为每个非核心能力定义 fallback：默认值、缓存、静态页、关闭功能。
  4. 配合限流、熔断，避免降级逻辑本身过载。
示例降级方式
推荐模块返回空列表或热门兜底
评论只展示缓存评论
写操作排队或拒绝，保读

311. 超时设置应该遵循什么原则？

  • 端到端预算：总超时 = 各 hop 超时之和 + 余量，上游超时 > 下游超时。
  • 区分读写：读可短（100~500ms），写/复杂查询可长。
  • 避免过长：超时过长占满线程池；过短误杀正常慢请求。
  • 可配置：按接口 SLA 动态调整，支持熔断联动。
  • 传递 deadline：gRPC deadline、HTTP Header 传递剩余时间。
Gateway 3000ms > Order 2000ms > Inventory 1000ms > DB 500ms

312. 重试为什么要配合幂等？

超时重试时，无法确定第一次请求是否已成功执行：

  • 若已成功 → 重试导致重复副作用（双扣款、双下单）。
  • 若未成功 → 重试是正确行为。

幂等保证「执行 N 次 ≡ 执行 1 次」，使重试成为安全操作。写接口必须设计幂等键；读接口天然幂等。

313. 如何避免重试风暴？

  • 指数退避 + 随机抖动：分散重试时间点。
  • 限制重试次数：通常 ≤ 3 次。
  • 熔断：下游故障时停止重试。
  • 全局重试预算：单服务/集群重试 QPS 上限。
  • 只在调用链末端或关键节点重试：避免每层都重试造成乘法效应。
  • 异步重试：写入重试队列，匀速消费。

399. 雪崩、穿透、击穿在系统架构层面如何综合治理？

问题现象治理
雪崩大量 Key 同时过期 / Redis 宕机，DB 被打垮过期时间加随机、多级缓存、熔断限流、Redis 集群高可用
穿透查询不存在的数据，缓存和 DB 都拦不住布隆过滤器、空值缓存短 TTL、参数校验
击穿热点 Key 过期瞬间大量请求打到 DB互斥锁重建、逻辑过期（异步刷新）、热点永不过期

架构层统一：缓存 + 限流 + 降级 + 监控命中率与慢查询。

400. 令牌桶和漏桶限流有什么区别？

对比项令牌桶漏桶
输入请求取令牌，无令牌则拒绝或等待请求进桶，桶满则拒绝
输出有令牌即可突发通过恒定速率流出，输出平滑
突发允许（桶内预存令牌）不允许，强行平滑
典型用途API 限流、允许合理突发消息队列消费限速

Guava RateLimiter、Sentinel 默认基于令牌桶思想。

401. 单机限流和分布式限流有什么区别？

对比项单机限流分布式限流
范围本机 JVM / 进程全集群统一配额
实现Guava、Semaphore、本地计数Redis Lua、Sentinel 集群、网关集中
精度多实例时总限流 = 单机 × N（不准）精确控制全局 QPS
性能无网络开销依赖 Redis，有 RT 成本

多实例部署必须用分布式限流或网关层集中限流，否则单机限流会被突破。

402. 如何设计热点数据缓存？

  1. 识别：访问统计、Redis hotkeys、业务预知（大促商品）。
  2. 多级缓存：本地 + Redis，热点走本地。
  3. 副本扩散：hot key 拆分多副本。
  4. 预加载：活动开始前灌入缓存。
  5. 逻辑过期：数据不过期，后台异步更新，避免击穿。

403. 如何设计多级缓存？

请求 -> L1 本地缓存（Caffeine，ms 级）
     -> miss -> L2 Redis（ms~10ms）
     -> miss -> L3 DB

注意点

  • 各级 TTL：L1 短、L2 中、保证最终一致。
  • 更新策略：写 DB 后删 L2，L1 短 TTL 自然过期或广播失效。
  • 本地缓存需防 OOM：容量上限 + LRU 淘汰。

404. 缓存和数据库一致性如何保证？

Cache-Aside 模式下，强一致难且代价高，通常接受短暂不一致。

  • 读：先缓存，miss 读 DB 再写缓存。
  • 写：先更新 DB，再删缓存（推荐）或更新缓存。
  • 延迟双删、消息通知：降低并发读写不一致窗口。
  • Canal 订阅 binlog：异步删/更新缓存，解耦业务。

405. 先更新数据库还是先删除缓存？

推荐：先更新 DB，再删除缓存（Cache-Aside 写策略）。

顺序问题
先删缓存再更新 DB删后、更新前，其他读可能把旧值写回缓存
先更新 DB 再删缓存删前短暂读到旧缓存，删后即一致；极端并发仍可能不一致
先更新 DB 再更新缓存写放大、并发写顺序问题

高一致场景：删缓存 + 延迟双删，或 Canal 驱动缓存失效。

406. 延迟双删是什么？

  1. 更新 DB。
  2. 删除缓存。
  3. 延迟 N ms（如 500ms）再删一次缓存。

目的：覆盖「先删缓存后读旧 DB 写回缓存」的并发窗口。延迟时间 > 读 DB + 写缓存耗时。

缺点：延迟期间仍可能不一致；更优雅方案是 Canal/MQ 异步删缓存。

407. 如何设计秒杀系统？

核心链路

  1. 静态化商品页 + CDN。
  2. 网关 + 按钮级限流、验证码。
  3. Redis 预减库存（Lua 原子）。
  4. MQ 异步创建订单，削峰。
  5. DB 最终扣库存，超时关单释放。

防超卖

Redis decr + DB WHERE stock > 0 + 订单唯一索引 + 对账。

408. 如何设计短链接系统？

核心流程

  1. 长 URL → 发号（自增 ID Base62 或 MurmurHash）→ 短码。
  2. 存储 short_code → long_url，短码唯一索引。
  3. 访问短链：302 重定向到长 URL。

扩展

  • Redis 缓存热点短码。
  • 分库分表或按短码 hash 分片。
  • 过期时间、访问统计、防恶意链。

409. 如何设计文件上传系统？

  • 直传 OSS：客户端拿 STS 临时凭证直传对象存储，减轻应用带宽。
  • 分片上传：大文件分片 + 断点续传 + 合并。
  • 秒传：文件 hash 去重，已存在则跳过上传。
  • 异步处理：上传完成后 MQ 触发转码、缩略图、病毒扫描。
  • 限流与安全：类型/大小校验、鉴权、防 SSRF。

410. 如何设计通知推送系统？

架构

业务 -> 推送服务 -> 渠道适配（短信/邮件/App/微信）
              -> 模板引擎 + 用户偏好 + 频控
              -> 第三方 SDK / 自建 WebSocket
  • 异步 MQ 解耦，保证至少一次投递。
  • 幂等：notification_id 防重复。
  • 失败重试、死信、人工补发。
  • 用户免打扰、渠道降级（短信贵则优先 App）。

411. 一致性哈希是什么？

普通 hash(key) % N 在节点增减时几乎所有 Key 重新映射。一致性哈希将节点和 Key 映射到哈希环，只影响相邻区间。

         Node A
    Key1 ----●----
              \
               ● Node B  (新增节点只影响环上相邻 Key)
              /
    Key2 ----●----
         Node C

虚拟节点：每个物理节点多个虚拟点，解决数据倾斜。用于分库、缓存、RPC 路由。

412. 服务注册与发现的原理是什么？

  1. 注册：服务启动向 Registry 写入实例信息（IP、端口、元数据）。
  2. 心跳：定时续约，超时未心跳则剔除（临时节点）。
  3. 发现：Consumer 订阅服务名，获取实例列表。
  4. 推送/拉取：Registry 变更通知 Consumer 更新本地缓存。
  5. 调用：Client LB 选实例发起 RPC。

代表：Nacos、Eureka、Consul、ZooKeeper。K8s 内置 Service + DNS 也是一种发现机制。

413. 蓝绿发布、滚动发布、金丝雀发布有什么区别？

方式原理优点缺点
蓝绿两套完整环境，切换流量秒级回滚资源双倍
滚动逐步替换旧实例资源利用高回滚慢，版本共存
金丝雀小流量试新版本风险可控需流量控制、监控

414. 灰度发布如何做流量切分？

  • 按比例：1% → 5% → 50% → 100%。
  • 按用户：白名单 userId、内部员工先体验。
  • 按维度：Header、Cookie、地域、设备。
  • 实现层：Gateway 路由规则、Service Mesh（Istio VirtualService）、Nginx split_clients。

配套：新版本独立监控、错误率超阈值自动回滚、与配置中心联动开关。
