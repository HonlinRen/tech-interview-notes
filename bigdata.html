Elasticsearch 是什么

Elasticsearch（ES）是基于 Apache Lucene 的分布式搜索与分析引擎，擅长全文检索、日志分析、聚合统计和近实时查询，常用于 ELK 栈中的检索层。

概念 说明
Index逻辑索引，类似数据库
DocumentJSON 文档，类似一行记录
Shard分片，数据水平拆分，支持扩展
Inverted Index倒排索引，按词项快速定位文档
  • 适用：日志检索（ELK）、商品搜索、监控指标分析、RAG 混合检索中的 BM25 层。
  • 特点：近实时（NRT）、分布式、REST API、聚合能力强；不适合强事务、复杂 JOIN。

大数据常见流程

典型链路是采集 → 存储 → 计算 → 服务 → 应用，各层解耦、可独立扩展。

数据源（业务库 / 日志 / 埋点 / IoT）
  ↓ 采集
Kafka / Flume / Canal / Debezium
  ↓ 存储
HDFS / 对象存储 / Hive / Iceberg
  ↓ 计算
Spark / Flink（批处理 + 流处理）
  ↓ 服务层
Elasticsearch / ClickHouse / MySQL / Redis
  ↓ 应用
BI 报表 / 搜索 / 推荐 / 监控告警
阶段 常见组件 作用
采集Kafka、Flume、Logstash高吞吐接入、削峰、解耦
存储HDFS、Hive、Iceberg低成本存海量原始/明细数据
计算Spark（批）、Flink（流）ETL、聚合、实时指标、宽表构建

ES 在大数据链路中的位置

日志/业务数据经 Kafka 接入 → Flink/Spark 清洗聚合 → 写入 ES 供检索；原始明细仍可落 Hive/HDFS 做离线分析。ES 负责在线查询，离线仓负责全量历史。

面试回答：ES 是分布式搜索分析引擎，倒排索引 + 分片扩展，适合日志和全文检索。大数据流程：Kafka 采集 → HDFS/Hive 存储 → Spark/Flink 计算 → ES/ClickHouse 服务 → BI/搜索应用；ES 通常放在服务层做近实时检索。

Hive 数据库数据冷热分离存储怎么做？

Hive 本身不直接管理物理存储介质，冷热分离是在 HDFS / 对象存储 + Hive 分区表 之上，按数据访问频率把热数据放高性能存储、冷数据放低成本存储，兼顾查询性能与存储成本。

热数据 vs 冷数据

类型 特征 存储倾向 典型场景
热数据近 7～30 天、查询频繁、延迟敏感SSD / 标准 HDFS 三副本、高 IO 节点日报表、实时分析补数、Ad-hoc 查询
温数据1～6 个月、偶尔查询普通 HDD、可适度压缩月度汇总、审计抽查
冷数据半年以上、极少访问、合规归档纠删码（EC）、对象存储 OSS/S3、归档目录历史明细留存、监管审计

常见实现方案

1. 按时间分区 + 目录分级（最常用）

Hive 表按 dt / month 分区，不同生命周期对应不同 HDFS 路径或存储策略。

-- 热数据：近 30 天分区，标准 HDFS 路径
CREATE TABLE ods_order_hot (
  order_id STRING,
  amount   DECIMAL(10,2)
)
PARTITIONED BY (dt STRING)
LOCATION 'hdfs://ns/hot/warehouse/ods_order';

-- 冷数据：历史分区，外部表指向归档路径
CREATE EXTERNAL TABLE ods_order_cold (
  order_id STRING,
  amount   DECIMAL(10,2)
)
PARTITIONED BY (dt STRING)
LOCATION 'hdfs://ns/cold/archive/ods_order';

定时任务（Shell / Airflow / DolphinScheduler）将超过阈值的 partition 目录 DistCp 到冷存储，再 ALTER TABLE ... ADD PARTITION 挂载到冷表，或统一视图 UNION ALL 对外透明。

2. HDFS 存储策略（Storage Policy）

Hadoop 支持目录级存储策略，配合 CacheMover / Mover 自动迁移块到合适介质：

策略 说明
LAZY_PERSIST内存 + 单副本磁盘，极热临时数据
ALL_SSD / ONE_SSDSSD 加速，适合高频扫描分区
HOT默认三副本，热数据
WARM单副本 + 纠删码，温数据
COLD纠删码，冷数据，省空间
 
# 冷数据目录启用 EC + COLD 策略
hdfs storagepolicies -setStoragePolicy -path /cold/archive -policy COLD
hdfs ec -enablePolicy -policy RS-6-3-1024k -path /cold/archive

3. 对象存储承接冷层

冷数据 DistCp 到 OSS / S3，Hive 用 External Table 或 Hive on OSS 挂载；计算用 Presto/Spark 按需拉取，接受更高延迟换更低存储单价。

4. 表格式与压缩

  • 热层：ORC / Parquet + Snappy/ZSTD，列存加速扫描。
  • 冷层：更高压缩比（ZSTD level 高）、合并小文件，减少元数据与清单成本。
  • Iceberg / Hudi 可配置 snapshot 过期、orphan file 清理，辅助生命周期管理。

落地流程示例

T+1 写入热路径  /hot/warehouse/db/tbl/dt=2026-03-27
     ↓ 每日调度
30 天后 DistCp → /cold/archive/db/tbl/dt=2026-02-27
     ↓
删除热路径分区 + 冷表 ADD PARTITION
     ↓
180 天后可选：转 OSS 归档 / 降 EC 副本 / 仅保留聚合宽表

注意事项

  • 小文件：冷热迁移前合并小文件，否则 NameNode 压力和冷层查询性能都差。
  • 跨层查询：用视图或联邦查询（Presto catalog）统一热冷表，避免业务改 SQL。
  • 元数据：分区数量爆炸时考虑按月二级分区或 Iceberg 隐藏分区。
  • SLA：冷数据查询延迟高，需在架构上区分在线报表 vs 离线归档。

面试回答：Hive 冷热分离靠「时间分区 + 不同存储介质/路径」实现：热数据近 30 天放 HDFS 标准盘或 SSD，按 dt 分区高频查；冷数据 DistCp 到 EC 目录或 OSS，External Table 挂载。配合 HDFS Storage Policy（HOT/WARM/COLD）和定时调度做生命周期迁移。核心是降存储成本，同时保证热查询性能；注意小文件合并和跨层统一视图。
