下面题目里会出现一些行业简称，先有个印象即可；正文里首次出现也会附带简短说明。

  • OLAP：Online Analytical Processing（联机分析处理），面向统计/报表，可接受分钟级延迟。
  • OLTP：Online Transaction Processing（联机事务处理），面向订单/账户等频繁增删改查。
  • ETL：Extract-Transform-Load（抽取-转换-加载），把原始数据清洗成可用表。
  • T+1：今天跑「昨天」数据的批任务，结果延迟约一天，但通常更准确。
  • CDC：Change Data Capture（变更数据捕获），把数据库的增删改同步出来。
  • 入湖：把业务数据/日志写入数据湖（Data Lake，低成本存海量原始/明细数据的存储体系）。
  • HDFS：Hadoop Distributed File System，Hadoop 的分布式文件系统，Hive 等组件的底层存储。
  • 数仓：Data Warehouse，按主题整理好的历史数据仓库，供离线分析用。
  • ODS / DWS：ODS 是贴源层（接近原始数据）；DWS 是汇总层（已聚合的宽表/指标表）。
  • Lambda 架构：批处理 + 流处理两路并行，最后用批结果修正实时层误差。
  • 湖仓一体：在统一存储（如 Iceberg/Hudi 表格式）上同时做批处理和流处理。
  • 列存：按列存储（ORC/Parquet），扫某几列时比行存更省 IO，适合分析。
  • Ad-hoc：临时即席查询，分析师按需写 SQL，不是固定报表。
  • BI：Business Intelligence（商业智能），报表/看板/自助分析工具。
  • 埋点：在前端/App 记录用户行为（点击、浏览等）并上报，供后续分析。

Elasticsearch 是什么？

Elasticsearch（ES）是基于 Apache Lucene（开源全文检索库）的分布式搜索与分析引擎，擅长全文检索、日志分析、聚合统计和近实时查询，常用于 ELK（Elasticsearch + Logstash 日志采集 + Kibana 可视化）栈中的检索层。

核心概念

概念 说明
Index逻辑索引，类似数据库
DocumentJSON 文档，类似一行记录
Shard分片，数据水平拆分，支持扩展
Replica副本，高可用与读扩展
Inverted Index倒排索引，按词项快速定位文档

特点与适用场景

  • 擅长：日志检索（ELK）、商品/文档全文搜索、监控指标聚合、RAG 混合检索中的 BM25（经典关键词相关性打分算法）层。
  • 能力：近实时 NRT（Near Real-Time，写入后秒级可搜）、分布式、REST API、聚合（Aggregation，类似 GROUP BY + 统计）能力强。
  • 不擅长：强事务、复杂 JOIN、高频单行更新（偏分析检索，而非 OLTP 那种频繁改单行）。

与同类组件

组件 差异
MySQLES 倒排索引做搜索/聚合；MySQL B+ 树做点查与事务
ClickHouseES 偏全文与灵活表结构（schema）；ClickHouse 偏列存 OLAP（联机分析处理）
HiveES 在线低延迟检索；Hive 离线批处理数仓（存 HDFS 上的历史明细）

面试回答：ES 是基于 Lucene 的分布式搜索分析引擎，Index-Document-Shard 模型，倒排索引支持全文检索和聚合。适合日志、搜索、监控和 RAG 的 BM25 层；不适合事务和复杂 JOIN。大数据链路里 ES 通常放在服务层（给业务/前端直接查的那一层）做近实时查询，全量历史明细仍在 Hive/HDFS。

大数据常见流程是什么？

典型大数据链路是采集 → 存储 → 计算 → 服务 → 应用，各层解耦、可独立扩展。Lambda（批+流双路）/ 湖仓一体（统一存储上批流计算）都是在此基础上的变体。

数据源（业务库 / 日志 / 埋点 / IoT 物联网设备）
  ↓ 采集
Kafka / Flume（日志采集）/ Canal（MySQL 变更 CDC）/ Debezium（多数据库 CDC）
  ↓ 存储
Hive / HDFS / 对象存储 / Iceberg（数据湖表格式）
  ↓ 计算
Spark / Flink（批处理 + 流处理）
  ↓ 服务层
Elasticsearch / ClickHouse / MySQL / Redis
  ↓ 应用
BI 报表 / 搜索 / 推荐 / 监控告警

各阶段说明

阶段 常见组件 作用
采集Kafka、Flume、Logstash、Canal高吞吐接入、削峰、解耦；Canal 等做 CDC（同步数据库变更）
存储Hive、HDFS、Iceberg、Hudi低成本存海量原始/明细数据（即「入湖」）
计算Spark（批）、Flink（流）ETL（清洗转换）、聚合、实时指标、宽表（多表拼成一张大表）构建
服务Elasticsearch、ClickHouse、Redis低延迟检索与分析查询
应用Kibana、Grafana、Superset、自研 API可视化、搜索、业务决策

批流与架构模式

  • 离线批（T+1）：Kafka/HDFS → Spark → Hive 汇总层（DWS）→ BI 报表，准确但延迟高。
  • 实时流：Kafka → Flink → Redis/ClickHouse/ES，秒级指标与告警。
  • Lambda：批层 + 速层（实时层）并行，批结果修正流层误差。
  • 湖仓一体：Kafka 入湖（写入 Iceberg/Hudi 等数据湖表）→ Spark/Flink 统一批流计算 → 多引擎查询。

ES 在链路中的位置

日志/业务数据经 Kafka 接入 → Flink/Spark 清洗聚合 → 写入 ES 供检索；原始明细仍可落 Hive/HDFS 做离线分析。ES 负责在线查询，离线仓（Hive 等）负责全量历史。

面试回答：大数据流程五段：Kafka 等采集 → HDFS/Hive/Iceberg 存储 → Spark/Flink 计算 → ES/ClickHouse 服务 → BI/搜索/监控应用。批用 Spark T+1，实时用 Flink；ES 在服务层做近实时检索，Hive 存全量明细。Lambda 是批+流双路，湖仓是统一存储上批流一体。

Hive 数据库数据冷热分离存储怎么做？

Hive 本身不直接管理物理存储介质，冷热分离是在 HDFS（分布式文件系统）/ 对象存储 + Hive 分区表 之上，按数据访问频率把热数据放高性能存储、冷数据放低成本存储，兼顾查询性能与存储成本。

热数据 vs 冷数据

类型 特征 存储倾向 典型场景
热数据近 7～30 天、查询频繁、延迟敏感SSD / 标准 HDFS 三副本、高 IO 节点日报表、实时分析补数、临时即席查询（Ad-hoc）
温数据1～6 个月、偶尔查询普通 HDD、可适度压缩月度汇总、审计抽查
冷数据半年以上、极少访问、合规归档纠删码 EC（Erasure Coding，用编码冗余省空间）、对象存储 OSS/S3、归档目录历史明细留存、监管审计

常见实现方案

1. 按时间分区 + 目录分级（最常用）

Hive 表按 dt（日期）/ month（月份）分区，不同生命周期对应不同 HDFS 路径或存储策略。

-- 热数据：近 30 天分区，标准 HDFS 路径
CREATE TABLE ods_order_hot (
  order_id STRING,
  amount   DECIMAL(10,2)
)
PARTITIONED BY (dt STRING)
LOCATION 'hdfs://ns/hot/warehouse/ods_order';

-- 冷数据：历史分区，外部表（元数据在 Hive、文件路径在外部）指向归档路径
CREATE EXTERNAL TABLE ods_order_cold (
  order_id STRING,
  amount   DECIMAL(10,2)
)
PARTITIONED BY (dt STRING)
LOCATION 'hdfs://ns/cold/archive/ods_order';

定时任务（Shell / Airflow / DolphinScheduler 等工作流调度）将超过阈值的 partition 目录用 DistCp（Hadoop 分布式拷贝）迁到冷存储，再 ALTER TABLE ... ADD PARTITION 挂载到冷表，或统一视图 UNION ALL 对外透明。

2. HDFS 存储策略（Storage Policy，按目录指定 SSD/HDD/纠删码等）

Hadoop 支持目录级存储策略，配合 CacheMover / Mover（自动迁移数据块的工具）自动迁移块到合适介质：

策略 说明
LAZY_PERSIST内存 + 单副本磁盘，极热临时数据
ALL_SSD / ONE_SSDSSD 加速，适合高频扫描分区
HOT默认三副本，热数据
WARM单副本 + 纠删码，温数据
COLD纠删码，冷数据，省空间
 
# 冷数据目录启用 EC + COLD 策略
hdfs storagepolicies -setStoragePolicy -path /cold/archive -policy COLD
hdfs ec -enablePolicy -policy RS-6-3-1024k -path /cold/archive

3. 对象存储承接冷层

冷数据 DistCp 到 OSS / S3（对象存储，按量计费、适合归档），Hive 用 External Table 或 Hive on OSS 挂载；计算用 Presto/Trino（交互式 SQL 引擎）/ Spark 按需拉取，接受更高延迟换更低存储单价。

4. 表格式与压缩

  • 热层：ORC / Parquet（列式存储格式）+ Snappy/ZSTD 压缩，列存加速扫描。
  • 冷层：更高压缩比（ZSTD level 高）、合并小文件，减少元数据与清单成本。
  • Iceberg / Hudi（数据湖表格式，管快照/增量/清理）可配置 snapshot 过期、orphan file 清理，辅助生命周期管理。

落地流程示例

T+1 写入热路径  /hot/warehouse/db/tbl/dt=2026-03-27
     ↓ 每日调度
30 天后 DistCp → /cold/archive/db/tbl/dt=2026-02-27
     ↓
删除热路径分区 + 冷表 ADD PARTITION
     ↓
180 天后可选：转 OSS 归档 / 降 EC 副本 / 仅保留聚合宽表

注意事项

  • 小文件：冷热迁移前合并小文件，否则 NameNode（HDFS 元数据管理节点）压力和冷层查询性能都差。
  • 跨层查询：用视图或联邦查询（Presto catalog 把多数据源当一张表查）统一热冷表，避免业务改 SQL。
  • 元数据：分区数量爆炸时考虑按月二级分区或 Iceberg 隐藏分区。
  • SLA：Service Level Agreement（服务等级协议）；冷数据查询延迟高，需在架构上区分在线报表 vs 离线归档。

面试回答：Hive 冷热分离靠「时间分区 + 不同存储介质/路径」实现：热数据近 30 天放 HDFS 标准盘或 SSD，按 dt 分区高频查；冷数据用 DistCp 迁到纠删码 EC 目录或 OSS，External Table 挂载。配合 HDFS Storage Policy（HOT/WARM/COLD）和定时调度做生命周期迁移。核心是降存储成本，同时保证热查询性能；注意小文件合并和跨层统一视图。

Kafka 是什么？

Apache Kafka 是分布式流式数据平台，核心能力是高吞吐、可持久化、可回溯的消息日志（Log，按顺序追加的消息队列），在大数据链路中既做消息队列，也做数据管道（Data Pipeline，连接采集→计算→存储的通道）。

核心概念

概念 说明
Topic逻辑主题，消息分类
PartitionTopic 分片，并行与扩展单元；分区内有序
Offset分区内消息位移（读到第几条），Consumer 按 offset 消费
Consumer Group消费者组，组内分摊 Partition，实现负载均衡
Broker存储节点，持久化 Partition 日志
Replica副本，Leader 读写 + Follower 同步，保证高可用

数据写入后持久化到磁盘，Consumer 可重复消费、按 offset 回溯历史，适合「采集层 + 流计算输入 + 多下游订阅」。

面试回答：Kafka 是分布式日志型流平台，Topic-Partition-Offset 模型，高吞吐、可持久化、可回溯，常用于日志采集和 Flink/Spark 管道。RocketMQ 偏业务消息，事务消息、延迟消息、消费模型更贴近应用。选型：大数据链路、流计算选 Kafka；业务解耦、事务一致性选 RocketMQ。

Hive 数据库是什么？

Apache Hive 是构建在 Hadoop（HDFS 存数据 + YARN 调度计算资源） 之上的数据仓库工具，提供类 SQL 的 HiveQL 接口，把结构化/半结构化数据映射为表，底层数据实际存储在 HDFS / 对象存储，通过 MapReduce / Tez / Spark 引擎执行查询。

注意：Hive 不是 OLTP 关系型数据库，而是离线分析型（OLAP，Online Analytical Processing 联机分析处理）仓库，高延迟、高吞吐扫描，不适合点查和频繁更新。

核心架构

  • Metastore：表结构、分区、文件路径等元数据（常存 MySQL）。
  • HiveServer2：JDBC/ODBC 服务，提交 HiveQL。
  • 执行引擎：MapReduce（慢，早期默认）、Tez、Spark（常用，内存计算更快）。
  • 存储格式：TextFile、ORC、Parquet（列存，只读需要的列，分析友好）。
-- 典型分区表
CREATE TABLE ods_user (
  user_id BIGINT,
  name    STRING
) PARTITIONED BY (dt STRING)
STORED AS ORC;

SELECT name FROM ods_user WHERE dt = '2026-03-27';

与传统关系型数据库（MySQL 等）区别

对比项 Hive（数仓） MySQL（OLTP，Online Transaction Processing 联机事务处理）
定位离线分析、海量历史数据在线事务、业务 CRUD
数据量TB～PB 级（PB = Petabyte，百万 GB 量级）GB～TB 级（单库）
查询延迟分钟～小时（批任务）毫秒～秒
事务弱（分区级 ACID 有限支持）强 ACID
更新追加写为主，UPDATE/DELETE 成本高频繁单行更新
索引依赖分区裁剪、列存、文件格式B+ 树索引，点查快
Schema读时模式（Schema on Read，写入时不强制校验，查询时再解析）写时模式（Schema on Write，入库前结构必须固定）
扩展水平扩展（HDFS + 分布式计算）垂直扩展 + 分库分表
SQLHiveQL（类 SQL，部分语法不同）标准 SQL

技术选型与适用场景

场景 推荐
T+1 报表、多维分析、历史明细留存Hive / Iceberg / Hudi
订单支付、账户余额、实时库存MySQL 等 OLTP

面试回答：Hive 是 Hadoop 上的数据仓库，HiveQL 查 HDFS 上的表，适合 PB 级离线分析，不是 OLTP。和 MySQL 比：Hive 高延迟、弱事务、追加写、靠分区和列存；MySQL 强 ACID、低延迟点查。用于 T+1 报表、明细沉淀；业务交易仍用 MySQL。执行引擎常用 Spark，存储用 ORC/Parquet + 按 dt 分区。

大数据计算 Spark、Flink 是什么？

Spark 和 Flink 都是分布式大数据计算引擎，对海量数据做 ETL、聚合、机器学习、流式计算；通常上游接 Kafka/HDFS/Hive，下游写 Hive/ES/ClickHouse/MySQL 等。

Spark 是什么

Apache Spark 基于内存计算 + DAG 调度（DAG，有向无环图，把任务拆成可并行阶段），核心抽象是 RDD / DataFrame / Dataset（不同层次的数据集合 API），擅长批处理，Structured Streaming 和 Spark Streaming 可处理微批/流式场景。

  • 批处理：ETL、T+1 宽表、离线报表（Spark SQL 最常用）。
  • 迭代计算：MLlib 机器学习，GraphX 图计算。
  • Hive on Spark：作为 Hive 执行引擎，替代 MapReduce。
  • Structured Streaming：微批（Micro-batch，把流切成小批次处理），延迟秒级，语义简单。
# Spark SQL 批处理示例
df = spark.read.orc("hdfs://warehouse/ods_order/dt=2026-03-27")
df.groupBy("city").count().write.mode("overwrite").saveAsTable("dws_order_city")

Flink 是什么

Apache Flink 是原生流处理（Native Streaming，来一条处理一条）引擎，基于事件时间（Event Time，按数据产生时间而非到达时间）和状态（State，如累计计数等中间结果），擅长低延迟实时计算与精确一次（Exactly-once，故障恢复后不丢不重）语义。

  • 实时指标：大屏、实时 UV/PV、风控规则。
  • CEP：Complex Event Processing（复杂事件处理），识别连续模式（如连续登录失败、异常交易序列）。
  • 流批一体：同一套 API，批是流的特例（DataStream / Table API）。
  • 状态与 Checkpoint：Keyed State（按 key 维护状态）+ 分布式快照，故障恢复不丢不重（配合 Kafka 两阶段提交）。
// Flink 消费 Kafka 实时聚合（概念示意）
env.fromSource(kafkaSource, WatermarkStrategy.noWatermarks(), "kafka")
   .keyBy(event -> event.getUserId())
   .window(TumblingEventTimeWindows.of(Time.minutes(5)))
   .aggregate(new CountAgg())
   .sinkTo(esSink);

选型建议

场景 推荐
T+1 报表、历史全量回刷Spark
Hive 离线 SQL 加速Spark（Hive on Spark）
实时 KPI、秒级告警Flink（KPI = Key Performance Indicator，关键业务指标）
Kafka → 实时宽表 → OLAP（联机分析查询层）Flink 流 + 批补数用 Spark
团队只维护一套、延迟要求不高Spark Structured Streaming 可够用
 
典型 Lambda 架构：
  批层：Kafka/HDFS → Spark  → Hive 汇总层 DWS（T+1 准确）
  速层：Kafka      → Flink  → Redis/ClickHouse（实时近似/精确）
  服务层：统一查询 API 或 以批修正流

面试回答：Spark 是内存分布式计算，批处理强，Hive ETL、T+1 报表常用；Structured Streaming 是微批。Flink 是原生流引擎，事件时间和状态管理好，适合实时指标和风控。选型：离线大批量用 Spark，低延迟实时用 Flink；Lambda 架构常两者结合。二者都常消费 Kafka，结果写 Hive/ES/ClickHouse。
